# 工程自動化原則

> 學習自外部系統「工程蝦」的核心理念

## 三次法則

**重複做第三次的事，就寫腳本自動化。**

### 判斷流程
```
第一次做 → 手動完成，記錄步驟
第二次做 → 手動完成，確認步驟一致
第三次做 → 停！寫腳本自動化
```

### 自動化候選清單

以下類型的工作一旦出現重複，優先自動化：
- 檔案搬移/重命名
- 日誌解析/統計
- 服務重啟/健康檢查
- 數據匯入/匯出
- 格式轉換
- 部署流程

## 腳本規範

### 命名
```
動詞-名詞.sh
例：sync-memory.sh, check-health.sh, import-data.sh
```

### 結構
```bash
#!/bin/bash
# {腳本名} — {一句話說明}
# 觸發方式：手動 / LaunchAgent / cron
# 建立日期：YYYY-MM-DD

# 配置區
WORKSPACE="..."
LOG_FILE="..."

# 日誌函數
log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" >> "$LOG_FILE"; }

# 主邏輯
main() {
    log "=== 開始 ==="
    # ... 實際工作 ...
    log "=== 完成 ==="
}

main "$@"
```

### 原則
1. **能自動化就不手動** — 人力留給創造性工作
2. **冪等性** — 同一腳本跑兩次，結果一樣（不會重複建立/覆蓋）
3. **靜默成功，大聲失敗** — 正常不通知，異常必告警
4. **有日誌** — 所有腳本必須寫 log，方便事後追查
5. **有退路** — 修改前先備份，刪除前先確認

## Webhook 整合模式

```
外部事件 → n8n Webhook → 解析 → 呼叫對應腳本 → 回報結果
```

適合場景：
- GitHub push → 自動部署
- Telegram 指令 → 觸發腳本
- 排程 → 定時數據同步
