# 阿秘工作流範例集
> **你是阿秘（秘書），不是小蔡。** 這是你的專屬工作流範例，照著做就對了。

## 工作流 1：日報產出流程

**場景**：每天結束時，需要產出當日的工作日報，匯總所有 crew bot 和系統的活動。

**目標**：產出一份簡潔但完整的日報，讓老蔡 3 分鐘內了解今天發生了什麼。

### Step 1：從 Supabase 拉取今日數據

**為什麼**：日報的數據來源是任務系統和執行記錄，先拉出今天的所有活動。

```json
{"action":"query_supabase","table":"openclaw_tasks","select":"id,name,status,owner,updated_at","filters":{"updated_at":"gte.2026-03-04T00:00:00"},"order":"updated_at.desc","limit":50}
```

```json
{"action":"query_supabase","table":"openclaw_tasks","select":"status,count(*)","group_by":"status"}
```

**成功標準**：拿到今天更新過的所有任務，以及各狀態的統計數字。
**失敗處理**：日期過濾不 work 就不加過濾條件，拉最近 50 筆手動篩選。

### Step 2：搜尋知識庫中的活動記錄

**為什麼**：有些活動不在任務表裡（巡邏、知識庫更新、系統維護），需要從知識庫搜。

```json
{"action":"semantic_search","query":"今日活動 系統更新 維護 修復 2026-03-04"}
```

```json
{"action":"semantic_search","query":"crew bot 任務完成 報告"}
```

**成功標準**：找到今天 crew bot 的工作報告、系統變更記錄。
**失敗處理**：搜不到就去看各 crew bot 的 notes.md 和 MEMORY.md。

### Step 3：AI 整理日報

**為什麼**：把零散的數據和活動記錄整理成結構化的日報，用 AI 比自己手動排版快。

```json
{
  "action": "ask_ai",
  "model": "flash",
  "prompt": "請根據以下數據整理成一份簡潔的工作日報：\n\n## 任務數據\n[Step 1 的查詢結果]\n\n## 活動記錄\n[Step 2 的搜尋結果]\n\n## 格式要求：\n1. 一句話總結今天的重點\n2. 完成的任務（列表）\n3. 進行中的任務（列表）\n4. 發現的問題（如有）\n5. 明天建議優先做的事\n\n語言用繁體中文，簡潔有力，不要廢話。"
}
```

**成功標準**：AI 回傳格式正確的日報草稿。
**失敗處理**：AI 輸出太冗就加 prompt「每項不超過 20 字」。

### Step 4：寫日報檔案

**為什麼**：日報要存檔，方便以後回查。

```json
{
  "action": "write_file",
  "path": "~/.openclaw/workspace/crew/ami/日報_20260304.md",
  "content": "# OpenClaw 日報 — 2026-03-04\n\n> 整理人：阿秘 | 系統版本：v2.4.32\n\n## 今日一句話\n[AI 整理的一句話總結]\n\n## 完成任務\n- [任務1] — 阿工完成\n- [任務2] — 阿研完成\n\n## 進行中\n- [任務3] — 阿工進行中（預計明天完成）\n\n## 發現的問題\n- [問題描述]（已建立任務追蹤）\n\n## 統計\n| 狀態 | 數量 |\n|------|------|\n| 完成 | [N] |\n| 進行中 | [N] |\n| 待辦 | [N] |\n| 錯誤 | [N] |\n\n## 明天建議\n1. [優先事項1]\n2. [優先事項2]"
}
```

**成功標準**：日報存檔成功，格式清楚，老蔡 3 分鐘能讀完。
**失敗處理**：寫入失敗就檢查路徑。日報內容不完整也先存檔，後續再補。

## 工作流 2：長文摘要流程

**場景**：收到一份很長的文件（技術文件、調研報告、外部文章），需要做摘要。

**目標**：把長文壓縮成 20% 篇幅的精華摘要，保留所有重點。

### Step 1：讀取原文

**為什麼**：先讀完整內容，了解文件結構和重點分佈。

```json
{"action":"read_file","path":"~/.openclaw/workspace/crew/ayan/調研報告_Bun_Runtime_評估.md"}
```

**成功標準**：讀到完整的文件內容。
**失敗處理**：檔案太大被截斷就分段讀（用 offset/limit），或記錄截斷位置。

### Step 2：AI 摘要

**為什麼**：長文摘要是 AI 最擅長的事，交給 AI 做比自己手動摘更快更準。

```json
{
  "action": "ask_ai",
  "model": "flash",
  "prompt": "請為以下文件做精華摘要：\n\n## 原文\n[Step 1 讀到的完整內容]\n\n## 摘要要求：\n1. 篇幅控制在原文的 20% 以內\n2. 必須保留：關鍵數據、核心結論、重要建議\n3. 可以省略：背景介紹、過渡段落、重複論述\n4. 格式：分點列出，每點一句話\n5. 最後加一行「關鍵結論：[一句話]」"
}
```

**成功標準**：摘要篇幅 < 原文 20%，核心資訊都保留。
**失敗處理**：摘要太長就加 prompt「限制在 10 點以內」。摘要遺漏重點就指定「必須包含 [X] 和 [Y]」。

### Step 3：存檔

**為什麼**：摘要要和原文放在一起，方便以後查找。

```json
{
  "action": "write_file",
  "path": "~/.openclaw/workspace/crew/ami/摘要_Bun_Runtime_評估.md",
  "content": "# 摘要：Bun Runtime 評估報告\n\n> 原文：crew/ayan/調研報告_Bun_Runtime_評估.md\n> 摘要人：阿秘 | 日期：2026-03-04\n\n## 重點摘要\n[AI 產出的摘要內容]\n\n## 關鍵結論\n[一句話結論]"
}
```

**成功標準**：摘要檔案建立成功，標註了原文路徑方便追溯。
**失敗處理**：寫入成功即可。

## 工作流 3：記憶整理流程

**場景**：crew bot 的 workspace 累積了很多檔案，需要整理分類，清除過時資料。

**目標**：整理目錄結構，更新索引，確保知識庫乾淨有序。

### Step 1：掃描目錄內容

**為什麼**：先看目前有什麼檔案，了解混亂程度和檔案類型。

```json
{"action":"list_dir","path":"~/.openclaw/workspace/crew/ayan/"}
```

```json
{"action":"list_dir","path":"~/.openclaw/workspace/crew/agong/"}
```

```json
{"action":"list_dir","path":"~/.openclaw/workspace/crew/ami/"}
```

**成功標準**：拿到所有 crew bot 目錄的檔案清單。
**失敗處理**：目錄不存在就記錄，可能是新 crew bot 還沒建目錄。

### Step 2：讀取檔案內容判斷狀態

**為什麼**：需要讀檔案內容才能判斷是否過時、是否有價值、是否需要歸檔。

```json
{"action":"read_file","path":"~/.openclaw/workspace/crew/ayan/notes.md"}
```

```json
{"action":"read_file","path":"~/.openclaw/workspace/crew/agong/notes.md"}
```

**成功標準**：了解每個檔案的內容和時效性。
**失敗處理**：讀不到就跳過，可能已被移動或刪除。

### Step 3：AI 分類建議

**為什麼**：用 AI 根據內容自動分類，比手動分更快更一致。

```json
{
  "action": "ask_ai",
  "model": "flash",
  "prompt": "以下是 crew bot workspace 中的檔案清單和部分內容。請分類整理：\n\n## 檔案清單\n[Step 1 的結果]\n\n## 部分檔案內容\n[Step 2 讀到的內容摘要]\n\n## 請分類成：\n1. **保留** — 仍有價值的檔案\n2. **歸檔** — 有歷史價值但不常用\n3. **過時** — 內容已過時，可安全刪除\n4. **重複** — 和其他檔案重複\n\n每個檔案標註分類和理由。"
}
```

**成功標準**：AI 回傳每個檔案的分類建議和理由。
**失敗處理**：AI 不確定的標為「保留」，寧可多留不多刪。

### Step 4：寫整理報告

**為什麼**：整理結果要記錄下來，讓小蔡確認後再執行（不自己動手刪檔案）。

```json
{
  "action": "write_file",
  "path": "~/.openclaw/workspace/crew/ami/記憶整理報告_20260304.md",
  "content": "# Workspace 記憶整理報告\n\n> 整理人：阿秘 | 日期：2026-03-04\n\n## 掃描範圍\n- crew/ayan/：[N] 個檔案\n- crew/agong/：[N] 個檔案\n- crew/ami/：[N] 個檔案\n\n## 分類結果\n\n### 保留（[N] 個）\n| 檔案 | 理由 |\n|------|------|\n| ... | ... |\n\n### 建議歸檔（[N] 個）\n| 檔案 | 理由 |\n|------|------|\n| ... | ... |\n\n### 建議刪除（[N] 個）\n| 檔案 | 理由 |\n|------|------|\n| ... | ... |\n\n## 待小蔡確認\n- 以上「建議刪除」的檔案，請小蔡確認後再刪\n- 「建議歸檔」的可以移到 archive/ 子目錄"
}
```

**成功標準**：報告清楚列出每個檔案的分類建議，待小蔡確認。
**失敗處理**：整理不完整就先產出已掃描部分的報告，標註「尚未掃描」的目錄。

## 通用提醒

1. **日報不要超過 1 頁**：老蔡要的是快速了解，不是長篇大論。
2. **摘要保留原文路徑**：方便追溯完整內容。
3. **整理不刪除**：只建議分類，不自己動手刪檔案，讓小蔡確認。
4. **用 flash 模型**：秘書任務用 flash 夠了，不需要 pro。
5. **命名規則**：`日報_[日期].md`、`摘要_[原文主題].md`、`記憶整理報告_[日期].md`。
