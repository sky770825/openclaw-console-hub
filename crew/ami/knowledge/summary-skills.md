# 阿秘專屬 — 摘要技巧
> 你是阿秘（📋 秘書），不是小蔡，這是你的專屬知識庫

## 1. 金字塔原理（摘要的核心方法）

### 結構
```
        結論（一句話）
       /      |      \
  論點A     論點B     論點C
  / \       / \       / \
證據  證據  證據  證據  證據  證據
```

### 操作步驟
1. **先寫結論**：用一句話回答「所以呢？」
2. **列出支撐點**：3 個以內的主要論點（超過 3 個就要再歸納）
3. **補充細節**：每個論點下面放具體證據/數據

### 範例
```
❌ 不好的摘要：
「今天檢查了 server log，發現有 3 個 ERROR，然後查了 Supabase 連線，
有 timeout，然後看了 Telegram bot，有些 pending update...」

✅ 好的摘要：
「系統有 3 個需關注問題：
1. Supabase 連線超時（影響 API 回應速度）
2. Server 有 3 個 ERROR log（均為非關鍵路徑）
3. Telegram bot 有未處理的 pending update（建議清理）
結論：系統整體正常運行，建議優先處理 Supabase 連線問題。」
```

## 2. 長文濃縮方法

### 方法 A：三遍閱讀法

**第一遍（掃描）**：只看標題、粗體、列表，掌握結構
**第二遍（標記）**：標記關鍵句子（每段通常只有 1-2 句是重點）
**第三遍（輸出）**：用自己的話重述標記的重點

### 方法 B：5W1H 抽取法

從長文中抽取：
| 要素 | 問題 | 範例 |
|------|------|------|
| What | 做了什麼 / 發生了什麼 | 修復了登入 bug |
| Who | 誰做的 / 影響誰 | 小蔡修的，影響所有用戶 |
| When | 什麼時候 | 今天下午 3 點 |
| Where | 在哪裡 | server/src/auth.ts |
| Why | 為什麼 | Token 過期判斷邏輯錯誤 |
| How | 怎麼做的 | 修改了 expiry check 邏輯 |

不是每個都需要，挑出最重要的 3-4 個。

### 方法 C：分層摘要（適合超長文件）

```
一句話摘要（< 20 字）
  ↓
短摘要（3-5 句，100 字以內）
  ↓
中摘要（1-2 段，300 字以內）
  ↓
完整摘要（帶結構化列表，500 字以內）
```

**實際操作**：先寫完整摘要，再反覆刪減到需要的長度。

## 3. 重點提取框架

### 程式碼變更的摘要
```markdown
**改了什麼**：[檔案名 + 函數名]
**為什麼改**：[一句話原因]
**影響範圍**：[會影響哪些功能]
**需要測試**：[測試要點]
```

### 會議/對話的摘要
```markdown
**決定事項**：
1. [明確的決定]
2. [明確的決定]

**待辦事項**：
- [ ] [誰] 在 [什麼時候] 前完成 [什麼]

**開放問題**：
- [還沒決定的事]
```

### 錯誤/異常的摘要
```markdown
**問題**：[一句話描述]
**影響**：[誰受影響，嚴重程度]
**原因**：[根本原因]
**處理**：[已做了什麼 / 建議做什麼]
**狀態**：已解決 / 待處理 / 觀察中
```

### Log 分析的摘要
```markdown
**時間範圍**：[HH:MM ~ HH:MM]
**總筆數**：X 筆
**錯誤**：X 筆（佔 Y%）
**警告**：X 筆
**關鍵發現**：
1. [最重要的發現]
2. [第二重要的]
**建議**：[該怎麼處理]
```

## 4. 摘要品質檢查

寫完摘要後自問：

- [ ] **準確嗎？** 有沒有誤解原文的意思？
- [ ] **完整嗎？** 有沒有遺漏重要資訊？
- [ ] **簡潔嗎？** 能不能再刪掉一些廢話？
- [ ] **可行動嗎？** 讀完摘要的人知道下一步該做什麼嗎？
- [ ] **老蔡看得懂嗎？** 不需要額外解釋就能理解？

### 常見錯誤

| 錯誤 | 範例 | 修正 |
|------|------|------|
| 太細節 | 「在第 47 行加了 if 判斷...」 | 「修復了 token 過期判斷邏輯」 |
| 太抽象 | 「做了一些改進」 | 「API 回應速度提升 40%」 |
| 遺漏結論 | 只列事實不給建議 | 最後加「建議：...」 |
| 堆砌數據 | 列 20 個指標 | 只放 3-5 個最關鍵的 |

## 5. 給不同對象的摘要風格

### 給老蔡
- 直接講結論，不要鋪陳
- 重點標粗體
- 用數字說話
- 明確標出「需要老蔡決定」的部分
- 格式：結論 → 數據 → 建議

### 給其他 bot（阿策、阿研等）
- 可以稍微詳細，包含技術細節
- 附上相關檔案路徑
- 標明上下文來源

### 給系統記錄（MEMORY.md）
- 極度精簡，每條不超過 1 行
- 包含日期
- 只記「以後會用到」的資訊

## 6. 用 AI 輔助摘要

當內容超過 5000 字，用 `ask_ai` 做初步摘要：

```json
{
  "action": "ask_ai",
  "model": "flash",
  "prompt": "請將以下內容摘要成 5 個要點，每個要點不超過 2 句話：\n\n[貼上長文]",
  "system": "你是專業的技術文件摘要者，只提取最重要的資訊，不添加任何原文沒有的內容"
}
```

**注意**：AI 摘要完後，自己要過一遍確認沒有漏掉重要資訊或產生幻覺。
