# 記錄官洞察合成指南（Insight Synthesis）
> 從事件記錄中提煉有價值的洞察，供決策與優化使用。

---

## 合成時機

| 時機 | 頻率 | 目的 | 輸出位置 |
|------|------|------|---------|
| 每日摘要 | 每天 23:00 | 總結當日事件、標記重點 | `/crew/journal/reports/daily/` |
| 每週趨勢 | 每週日 | 識別趨勢、發現模式、提出建議 | `/crew/journal/reports/weekly/` |
| 即時查詢 | 依需求 | 回答特定問題、追蹤特定指標 | 直接回覆請求者 |

## 模式識別方法

### 1. 重複錯誤 → 根因分析

**觸發條件**：同類型錯誤在 48 小時內出現 3 次以上

**分析步驟**：
1. 收集所有相同類型的錯誤事件
2. 比對共同因素：時間、agent、服務、輸入資料
3. 排除偶發因素，找出系統性原因
4. 提出修復建議

**範例**：
```
洞察：Supabase 查詢在每日 14:00~15:00 頻繁逾時
證據：
  - 3/12 14:23 — seo agent 查詢逾時
  - 3/13 14:11 — content agent 查詢逾時
  - 3/13 14:45 — growth agent 查詢逾時
根因：該時段為流量高峰，連線池飽和
建議：增加連線池大小或錯開 agent 查詢時間
```

### 2. 速度變化 → 瓶頸識別

**觸發條件**：任務完成速度較前一週下降 20% 以上

**分析步驟**：
1. 統計本週 vs 上週的任務完成數與平均耗時
2. 按 agent 分類，找出拖慢速度的環節
3. 檢查是否有新增的阻塞依賴
4. 提出排程調整建議

**範例**：
```
洞察：content agent 本週產出速度下降 35%
證據：
  - 上週平均每任務 40 分鐘，本週平均 62 分鐘
  - review agent 審查回覆時間從平均 15 分鐘增加到 45 分鐘
  - content agent 等待審查的閒置時間佔比 40%
根因：review agent 任務量增加導致審查瓶頸
建議：調整 review 優先級，或將低風險內容改為事後抽查
```

### 3. 異常模式 → 預防性建議

**觸發條件**：偵測到非典型但可能發展為問題的趨勢

**分析步驟**：
1. 與歷史基準比對，標記偏離值
2. 評估偏離是否持續或加速
3. 預測若不干預的後果
4. 提出預防措施

## 輸出格式

每則洞察必須包含以下結構：

```markdown
## [洞察標題]

**類型**：[根因分析 / 瓶頸識別 / 趨勢預測 / 效能優化]
**信心等級**：[高 / 中 / 低]
**日期**：YYYY-MM-DD

### 證據（至少 3 個資料點）
1. [資料點 1 — 含時間與來源]
2. [資料點 2 — 含時間與來源]
3. [資料點 3 — 含時間與來源]

### 分析
[因果推論與邏輯說明]

### 建議
- [具體可執行的建議 1]
- [具體可執行的建議 2]

### 追蹤
- 預期效果：[採納建議後的預期改善]
- 驗證方式：[如何確認改善有效]
```

## 跨 Agent 供給

洞察不只是記錄，要主動推送給需要的 agent：

| 洞察類型 | 供給對象 | 用途 |
|---------|---------|------|
| 熱門話題趨勢 | seo agent | 調整關鍵字策略 |
| 使用者行為模式 | growth agent | 優化轉換漏斗 |
| 內容效果分析 | content agent | 改善內容方向 |
| 錯誤模式 | patrol agent | 預防性監控規則 |
| 效能瓶頸 | ace agent | 排程優化 |
| 週報素材 | newsletter agent | 電子報亮點 |

---

## 信心等級定義

| 等級 | 條件 | 建議動作 |
|------|------|---------|
| **高** | 5+ 資料點、因果關係明確、可重現 | 直接建議採納 |
| **中** | 3~4 資料點、關聯性強但因果待確認 | 建議觀察一週後決定 |
| **低** | 少於 3 資料點、僅為初步觀察 | 記錄並持續收集證據 |
