# 記錄官資料供給對照表（Agent Feed Mapping）
> 定義 journal 應主動提供哪些資料給其他 agent，確保資訊流通順暢。

---

## 供給總覽

```
journal ──→ content    : 熱門話題、使用者回饋主題
journal ──→ seo        : 搜尋查詢模式、熱門內容
journal ──→ growth     : 轉換事件、流失節點
journal ──→ newsletter : 週亮點、公告素材
journal ──→ patrol     : 錯誤模式、預防性監控
```

---

## 1. journal → content

| 項目 | 說明 |
|------|------|
| **供給資料** | 熱門話題趨勢、使用者回饋主題分類、內容效果排名 |
| **頻率** | 每日摘要 + 週趨勢報告 |
| **格式** | Markdown 列表，含話題名稱、熱度指數、建議角度 |
| **存放位置** | `/crew/content/inbox/journal-feed-[date].md` |

**範例輸出**：
```markdown
## Content Feed — 2026-03-14

### 本日熱門話題
1. AI Agent 自動化工作流（熱度：高）— 建議角度：實作教學
2. n8n 與 Make 比較（熱度：中）— 建議角度：深度評測
3. 小型團隊的 DevOps 實踐（熱度：中）— 建議角度：案例分享

### 使用者回饋主題
- 「教學步驟太跳躍」出現 4 次 → 建議增加中間步驟說明
- 「想看更多實際案例」出現 3 次 → 建議增加 case study 比例
```

## 3. journal → growth

| 項目 | 說明 |
|------|------|
| **供給資料** | 轉換事件統計、使用者流失節點、註冊/啟用漏斗數據 |
| **頻率** | 每日摘要 + 週趨勢 |
| **格式** | 漏斗數據表，含各階段人數、轉換率、與上期比較 |
| **存放位置** | `/crew/growth/inbox/journal-feed-[date].md` |

**範例輸出**：
```markdown
## Growth Feed — 2026-03-14

### 轉換漏斗（今日）
| 階段 | 人數 | 轉換率 | vs 昨日 |
|------|------|--------|---------|
| 訪問首頁 | 1,200 | — | +8% |
| 點擊 CTA | 180 | 15.0% | +2.1% |
| 開始註冊 | 95 | 52.8% | -3.2% |
| 完成註冊 | 72 | 75.8% | +1.5% |

### 流失節點
- 註冊頁第二步（填寫詳細資料）流失率最高（24.2%）
- 建議：簡化表單或延後蒐集非必要欄位
```

## 5. journal → patrol

| 項目 | 說明 |
|------|------|
| **供給資料** | 錯誤模式分析、異常基準線更新、預防性監控建議 |
| **頻率** | 每日 + 發現新模式時即時推送 |
| **格式** | 監控規則建議，含觸發條件、閾值、建議動作 |
| **存放位置** | `/crew/patrol/inbox/journal-feed-[date].md` |

**範例輸出**：
```markdown
## Patrol Feed — 2026-03-14

### 錯誤模式更新
- Supabase 逾時集中在 14:00~15:00（連續 3 天）
  → 建議：該時段的查詢逾時閾值從 30s 放寬到 60s，避免誤報
  → 建議：新增監控規則——該時段逾時超過 5 次才通報

### 基準線調整
- content agent 平均任務時間從 40min 調整為 50min（近兩週穩定值）
- review agent 回覆時間基準從 15min 調整為 25min

### 預防性建議
- growth agent 的 API 呼叫量持續上升，預計 3 天內可能觸及 rate limit
  → 建議提前通報 agong 評估是否需要升級方案
```

---

## 供給原則

1. **主動推送**：不等其他 agent 來問，按頻率主動供稿
2. **格式統一**：每個 feed 使用固定格式，方便消費者解析
3. **可行動**：每則供給都要包含建議動作，不只是數據堆砌
4. **時效性**：過期資料不推送，確保 feed 內容都是最新的
5. **去噪**：只推送有意義的變化，穩定不變的數據不重複推送
