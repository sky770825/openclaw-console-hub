# 知識管理生命週期 v1.0

> 知識檔案從建立到淘汰的完整流程
> 避免知識庫堆積過期資訊、確保品質

---

## 生命週期四階段

```
建立 → 審查 → 維護 → 淘汰
 ↑                      │
 └──────── 重寫 ─────────┘
```

---

## 第一階段：建立

### 建立時機
- 學到新的方法論或框架
- 解決了一個反覆出現的問題
- 外部來源有值得內化的知識
- 主人或達爾指示新增

### 檔案規格
| 項目 | 要求 |
|------|------|
| 最小長度 | 500 bytes（太短=沒價值） |
| 格式 | Markdown，含標題/段落/程式碼區塊 |
| 命名 | 小寫英文+連字號，描述性命名 |
| 位置 | `crew/{代理}/knowledge/` |
| 來源標註 | 開頭用 `>` 標註知識來源 |

### 命名規範
```
✅ local-seo-strategy.md
✅ pricing-strategy.md
✅ ab-testing.md
❌ new-knowledge.md
❌ temp.md
❌ 中文名稱.md
```

---

## 第二階段：審查

### 誰負責審查
| 知識類型 | 審查者 |
|---------|--------|
| 技術/工程 | agong 阿工 |
| 策略/流程 | ace 阿策 |
| 商業/市場 | ashang 阿商 |
| 內容/文案 | review 審查 |
| 數據/指標 | ashu 阿數 |

### 審查清單
- [ ] 內容是否準確（沒有過期資訊）
- [ ] 是否有具體範例（不是純理論）
- [ ] 是否與現有知識庫重複
- [ ] 是否符合品牌/系統的實際情況
- [ ] 命名是否清晰描述內容

---

## 第三階段：維護

### 定期檢查（每月一次，阿秘負責排程）

```markdown
# 知識庫月檢報告：{月份}

## 統計
- 總檔案數：___
- 本月新增：___
- 本月更新：___
- 標記過期：___

## 需要更新的檔案
| 檔案 | 原因 | 建議動作 |
|------|------|---------|
| {path} | {過期/不完整/有誤} | 更新/重寫/淘汰 |
```

### 更新觸發條件
- 引用的工具版本已過期
- 市場數據超過 6 個月
- 流程已經改變但文件未同步
- 發現實際操作與文件不符

---

## 第四階段：淘汰

### 淘汰條件
- 超過 12 個月未被引用或使用
- 內容已完全被其他檔案取代
- 相關功能或服務已下線
- 資訊已嚴重過期且無更新價值

### 淘汰流程
```
阿秘標記「建議淘汰」
  ↓
相關代理確認（不會影響現有流程）
  ↓
移至 archive/ 目錄（不直接刪除）
  ↓
6 個月後如無人查閱 → 永久刪除（需主人確認）
```

---

## 知識品質分級

| 等級 | 標準 | 範例 |
|------|------|------|
| A | 有具體範例+模板+數據，可直接套用 | pricing-strategy.md |
| B | 框架完整，缺少部分範例 | storytelling-techniques.md |
| C | 有用但偏理論，需補實例 | 純概念說明 |
| D | 過期或不完整，需重寫 | 引用舊版工具 |
