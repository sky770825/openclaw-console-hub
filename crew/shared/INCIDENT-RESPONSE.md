# 事件處理與升級流程 v1.0

> 系統故障、安全問題、緊急需求的標準處理程序
> 目標：快速止血 → 根因分析 → 防止復發

---

## 事件分級

| 等級 | 名稱 | 定義 | 響應時間 | 範例 |
|------|------|------|---------|------|
| S1 | 致命 | 核心系統完全不可用 | 立即 | 伺服器當機、資料遺失 |
| S2 | 嚴重 | 主要功能異常，有替代方案 | 30 分鐘 | API 錯誤率 > 10%、Ollama 離線 |
| S3 | 一般 | 非核心功能異常 | 2 小時 | 排程任務未執行、日誌遺漏 |
| S4 | 輕微 | 不影響運作的小問題 | 下次巡查 | 格式錯誤、非關鍵告警 |

---

## 處理流程

### 第一階段：發現與通報（0-5 分鐘）

```
發現異常（巡查 / 日誌 / 手動回報）
  ↓
巡查官初步判定等級（S1-S4）
  ↓
S1/S2：立即通知達爾（Telegram）+ 記錄事件
S3/S4：寫入日誌 + 下一巡查週期處理
```

**通報格式：**
```
[S{等級}] {問題摘要}
時間：{發現時間}
影響：{受影響的服務/功能}
現象：{具體錯誤訊息或異常行為}
```

### 第二階段：止血（5-30 分鐘）

| 角色 | 動作 |
|------|------|
| 巡查 | 持續監控，每 5 分鐘回報狀態 |
| 阿工 | 查 log → 定位問題 → 嘗試快速修復 |
| 阿數 | 查影響範圍（錯誤率、受影響用戶數） |
| 阿秘 | 記錄時間線（誰做了什麼、什麼時候） |

### 第三階段：根因分析（修復後 24 小時內）

由阿研主導，產出根因分析報告：

```markdown
# 事件報告：{事件標題}

## 時間線
| 時間 | 事件 |
|------|------|
| HH:MM | 問題首次出現 |
| HH:MM | 巡查發現 |
| HH:MM | 開始處理 |
| HH:MM | 止血完成 |
| HH:MM | 根因確認 |

## 根因
{一段話描述問題的根本原因}

## 影響範圍
- 持續時間：{X 分鐘/小時}
- 受影響服務：{列表}
- 資料損失：有 / 無

## 修復措施
{已經做了什麼來修復}

## 防復發措施
| 措施 | 負責人 | 期限 |
|------|--------|------|
| {具體措施} | {代理} | {日期} |
```

## 升級規則

```
                    巡查發現問題
                         │
            ┌────────────┼────────────┐
            ↓            ↓            ↓
        自動處理      通知達爾      通知主人
         (S3/S4)      (S2)         (S1)
            │            │            │
        阿工修復    達爾判斷是否    主人決策
            │        需要主人        │
        驗證完成         │        緊急處理
                    解決 / 升級
```

### 自動升級條件
- S3 問題 30 分鐘未解決 → 升級為 S2
- S2 問題 2 小時未解決 → 升級為 S1
- 同一問題 24 小時內復發 → 自動升一級
- 涉及資料安全 → 直接 S1

---

## 緊急快速通道

當標準流程太慢時（S1 情況），可以啟用快速通道：

1. **跳過審查**：阿工直接修復，事後補審查
2. **跳過排程**：不等排程週期，直接處理
3. **多代理同時動**：不按順序，並行處理

**限制**：快速通道使用後必須在 24 小時內補齊：
- [ ] 事件報告
- [ ] 代碼審查（如有修改）
- [ ] 防復發措施
