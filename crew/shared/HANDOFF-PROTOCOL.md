# 任務交接規範 v1.0

> 代理之間透過 inbox/ 交接任務時，必須遵守此格式
> 確保下游代理拿到足夠上下文，不需要回頭問

---

## Inbox 檔案命名

```
{日期}_{時間}_{來源代理}_{摘要}.md
例：20260315_1430_ace_SEO審查請求.md
```

## 必填欄位

每個 inbox 任務檔案必須包含以下區塊：

```markdown
# 任務交接：{任務標題}

## 基本資訊
- **來源代理**：{誰發出的}
- **目標代理**：{給誰做}
- **優先級**：P0 / P1 / P2
- **期望完成**：{時間或 ASAP}
- **關聯任務**：{如有上游任務，附 inbox 檔名}

## 任務描述
{一段話說明要做什麼、為什麼要做}

## 輸入材料
{列出需要讀取的檔案路徑、數據來源}
- `path/to/file.md`
- 前一步產出：`path/to/deliverable.md`

## 期望交付物
{明確說明要產出什麼}
- 檔案格式：.md / .json / .sh
- 存放位置：{指定路徑}
- 品質要求：{具體標準}

## 上下文補充
{任何幫助目標代理理解背景的資訊}
```

---

## 交接流程

```
來源代理完成自己的部分
  ↓
寫入 inbox 檔案至目標代理的 inbox/
  ↓
如果是 P0：同時發 Telegram 通知達爾
如果是 P1：等下一個排程週期處理
如果是 P2：目標代理閒時處理
  ↓
目標代理讀取 inbox → 執行 → 產出交付物
  ↓
交付物回寫至指定路徑 + 更新來源代理的追蹤
```

---

## 交接品質檢查

來源代理寫 inbox 前自查：

- [ ] 任務描述夠清楚，不需要回頭問「你要什麼」
- [ ] 輸入材料的路徑是正確的、檔案存在
- [ ] 期望交付物有明確格式和存放位置
- [ ] 優先級正確（不要什麼都 P0）
- [ ] 沒有遺漏前置依賴

## 常見錯誤

| 問題 | 後果 | 正確做法 |
|------|------|---------|
| 只寫「幫我做 SEO」 | 下游不知道做什麼 | 寫明目標關鍵字、目標頁面、期望排名 |
| 路徑寫錯 | 下游找不到檔案 | 用絕對路徑，先確認檔案存在 |
| 全部標 P0 | 優先級失效 | 真正的 P0 一天不超過 2 個 |
| 不寫期望交付物 | 下游不知道做到什麼程度 | 明確寫出檔案格式和品質要求 |

## 退回機制

目標代理如果無法執行，必須在 inbox 中回覆：

```markdown
# 退回通知：{原任務標題}

## 退回原因
- [ ] 缺少必要輸入材料
- [ ] 任務超出能力範圍
- [ ] 前置任務未完成
- [ ] 其他：{說明}

## 建議處理方式
{具體建議，如「需要阿研先完成調研」}
```
