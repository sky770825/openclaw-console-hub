# 事實查核指南

## 查核流程

```
識別聲明 → 分類 → 尋找來源 → 驗證準確性 → 檢查時效性 → 判定
```

## 聲明分類

| 類型 | 範例 | 查核方式 |
|------|------|----------|
| 統計數據 | 「90% 的客戶表示滿意」 | 要求原始資料來源 |
| 產品功能 | 「支援 AI 自動分類」 | 確認程式碼中是否實現 |
| 價格資訊 | 「NT$1,800 / 90 分鐘」 | 對照最新定價表 |
| 技術規格 | 「使用 GPT-4 模型」 | 確認實際使用的 API |
| 時間聲明 | 「成立於 2010 年」 | 對照公司登記資料 |

## 可信來源層級

### Tier 1 — 最可信
- 官方文件（API 文件、產品規格書）
- 公司內部資料庫（Supabase 記錄）
- 程式碼（功能是否真的存在）
- 政府公開資料

### Tier 2 — 可信但需交叉驗證
- 業界報告（Gartner、Statista）
- 權威媒體報導
- 同行評審論文

### Tier 3 — 需要額外驗證
- 部落格文章
- 社群媒體貼文
- 維基百科（可作為起點，但需追溯引用來源）

### 不可接受
- 無法追溯的「據說」「研究顯示」
- 競品的行銷文案
- 過期超過 1 年的統計數據（除非標註年份）

## 常見紅旗

- **無來源的統計**: 「研究顯示 80% 的人...」 → 哪個研究？誰做的？
- **絕對語氣**: 「最好的」「唯一的」「100%」 → 是否有例外？
- **過期資訊**: 技術文章超過 6 個月可能已過時
- **利益衝突**: 引用自家數據證明自家產品最好
- **樣本偏差**: 「所有客戶都喜歡」 → 調查了多少人？怎麼選的？

## 程式碼相關查核

| 聲明 | 查核方式 |
|------|----------|
| 「支援 XX 功能」 | 在程式碼中搜尋對應的函數/路由 |
| 「使用 XX API」 | 檢查 package.json 和實際 import |
| 「XX 版本」 | 對照 package.json 或 lock file |
| 「XX 端點可用」 | 實際發送測試請求 |
| 「效能提升 XX%」 | 要求 benchmark 數據 |

## 判定等級

| 等級 | 說明 | 處置 |
|------|------|------|
| ✅ 確認屬實 | 有可靠來源支持 | 通過 |
| ⚠️ 部分正確 | 核心正確但細節有誤 | 要求修正細節 |
| ❓ 無法驗證 | 找不到來源 | 要求提供來源或移除 |
| ❌ 錯誤 | 與事實矛盾 | 退回修改 |

## 快速查核清單

1. [ ] 所有數字/統計是否有來源標註？
2. [ ] 產品功能描述是否與實際程式碼吻合？
3. [ ] 價格和時間資訊是否為最新？
4. [ ] 是否有未經授權的客戶評價引用？
5. [ ] 技術聲明是否可在文件中找到佐證？
