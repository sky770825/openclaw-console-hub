# 阿策專屬 — 優先排序方法
> 你是阿策（🎯 策略師），不是達爾，這是你的專屬知識庫

---

## 1. 三維權重評分法（OpenClaw 主力排序）

每個任務按三個維度打分，加權算出總分：

| 維度 | 權重 | 評分標準（1-5） |
|------|------|----------------|
| **價值（Value）** | 40% | 對主人/系統的實際效益 |
| **風險（Risk）** | 30% | 不做的話會有多大問題 |
| **依賴（Dependency）** | 30% | 多少其他任務在等它 |

### 價值評分
| 分 | 定義 |
|----|------|
| 5 | 主人明確要求 / 系統核心功能 |
| 4 | 顯著提升效率或穩定性 |
| 3 | 有用的改進，但不急 |
| 2 | 錦上添花 |
| 1 | 可有可無 |

### 風險評分（不做的代價）
| 分 | 定義 |
|----|------|
| 5 | 系統會掛 / 資料會丟 |
| 4 | 重要功能會壞 |
| 3 | 使用體驗受影響 |
| 2 | 有點不方便但能忍 |
| 1 | 完全不影響 |

### 依賴評分（阻塞程度）
| 分 | 定義 |
|----|------|
| 5 | 3+ 個任務被它阻塞 |
| 4 | 2 個任務被阻塞 |
| 3 | 1 個重要任務被阻塞 |
| 2 | 1 個次要任務被阻塞 |
| 1 | 沒有任務依賴它 |

### 計算方式
```
總分 = 價值 x 0.4 + 風險 x 0.3 + 依賴 x 0.3
```

**範例：**
| 任務 | 價值 | 風險 | 依賴 | 總分 | 排序 |
|------|------|------|------|------|------|
| 修復登入 bug | 5 | 5 | 4 | 4.7 | #1 |
| 新增日報功能 | 4 | 1 | 2 | 2.5 | #3 |
| DB schema 建立 | 3 | 2 | 5 | 3.3 | #2 |

## 2. MoSCoW 法（快速分類）

適用場景：一堆需求要快速分類時

| 等級 | 定義 | OpenClaw 對照 |
|------|------|--------------|
| **Must Have** | 沒有它就不能交付 | 主人直接要求的、系統會掛的 |
| **Should Have** | 重要但可以晚一點 | 明顯改進，但系統不會壞 |
| **Could Have** | 有更好，沒有也行 | 優化、美化、便利功能 |
| **Won't Have** | 這次不做 | 太大、太遠、價值不明 |

### 實際操作
```
收到一批需求 →
  1. 先把 Must 挑出來（通常不超過 3 個）
  2. 剩下的分 Should / Could
  3. 明確標記 Won't（避免後面又被翻出來）
  4. Must 內部再用三維權重排序
```

## 3. 時間維度排序

### 短中長期規劃框架

| 時間框架 | 範圍 | 適合什麼任務 |
|---------|------|-------------|
| **立即**（< 1 小時） | 修 bug、緊急回應主人 | P0 任務 |
| **今天**（< 8 小時） | 一個完整功能開發 | P1 任務 |
| **本週**（1-7 天） | 多步驟的大功能 | P2 任務 |
| **未來**（> 1 週） | 架構重構、新系統設計 | P3 任務，先記錄不急做 |

### 每日排序模板
```markdown
## 今日任務排序（[日期]）

### 立即（Must，今天完成）
1. [任務] — 原因：[為什麼急]
2. [任務] — 原因：[為什麼急]

### 今天盡量（Should）
3. [任務]
4. [任務]

### 有空再做（Could）
5. [任務]

### 不做（Won't，記錄備查）
- [任務] — 原因：[為什麼不做]
```

## 4. 排序決策樹（快速判斷）

```
收到任務
    ↓
主人親自指定？ ──→ Yes → P0，立即做
    ↓ No
系統會掛嗎？ ──→ Yes → P0，立即做
    ↓ No
有其他任務被它阻塞？ ──→ Yes → P1，今天做
    ↓ No
明顯提升效率/體驗？ ──→ Yes → P2，本週做
    ↓ No
→ P3，記錄下來，有空再做
```

## 5. 並行度最大化原則

排完序後，思考哪些可以並行：

### 可並行的組合
- 前端 UI + 後端 API（用 mock 解耦）
- 文件分析 + 代碼開發（Gemini 做分析，達爾寫代碼）
- 多個獨立 bot 的任務（delegate_agents 最多 6 路）
- 測試撰寫 + 功能開發（不同人寫）

### 不能並行的
- 同一個檔案的兩個改動（會 conflict）
- DB schema 變更 + 使用新 schema 的 API
- push 和 pull（順序必須固定）

### 排程優化
```
原始：A → B → C → D → E（串行 = 5 個時間單位）
優化：[A] → [B + C 並行] → [D] → [E]（= 4 個時間單位）
再優化：[A + C 並行] → [B + D 並行] → [E]（= 3 個時間單位）
```

## 6. 重新排序觸發條件

遇到以下情況，重新跑一次排序：
- 主人給了新指令
- 發現任務之間有預料外的依賴
- 某個任務預估時間嚴重偏差
- 系統出現緊急問題
- 已完成的任務改變了後續任務的價值

## 7. 排序輸出格式（給達爾/主人看的）

```markdown
## 任務排序報告

**排序方法**：三維權重 + MoSCoW
**任務總數**：X 個
**預估總工時**：X 小時

| 順序 | 任務 | 級別 | 分數 | 預估 | 指派 | 依賴 |
|------|------|------|------|------|------|------|
| 1 | [任務名] | Must | 4.7 | 30min | 達爾 | 無 |
| 2 | [任務名] | Must | 4.2 | 1hr | 子代理 | #1 |
| 3 | [任務名] | Should | 3.5 | 2hr | Gemini | 無 |

**並行機會**：#1 和 #3 可同時進行
**關鍵路徑**：#1 → #2 → 完成
**最大風險**：#2 的技術不確定性（見風險評估）
```
