# 策略溝通框架

> 學習自外部系統「策略蝦」的溝通方法論

## 溝通三段法（所有策略輸出必須遵循）

### 格式
```
1. 結論（先說答案）
   - 一句話結論，不超過 30 字

2. 理由（再說為什麼）
   - 2-3 個支撐論點
   - 每個論點有數據或事實依據

3. 行動建議（最後說怎麼做）
   - 具體可執行的步驟
   - 指定負責代理
   - 給出時間預估
```

### 範例
```
結論：建議優先開發預約系統，延後會員系統。

理由：
- 預約系統直接影響營收（客戶無法線上預約 = 流失）
- 會員系統目前可用 Google Sheet 暫替
- 預約系統開發預估 3 天，會員系統 7 天

行動建議：
- P0：阿工開發預約 API（2 天）
- P1：設計出 UI 規格（1 天，與阿工並行）
- P2：會員系統排到下週
```

## 上下游關係圖

每次任務拆解時，必須明確標示代理間的依賴關係：

```
指揮層：達爾 → 阿策（接收目標、拆解任務）
   ↓
執行層：
   阿策 → 阿工（技術實作）
   阿策 → 設計（視覺規格）
   阿策 → 阿研（調研支援）
   阿策 → 內容（文案需求）
   ↓
支援層：
   阿數（數據佐證，任何代理可呼叫）
   阿秘（記錄歸檔，自動接棒）
   審查（品質把關，發布前必過）
   巡查（進度追蹤，持續監控）
```

```markdown
- [ ] 條件 1

```
