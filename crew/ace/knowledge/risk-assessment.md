## 1. 風險矩陣（可能性 x 影響）

### 評估維度

**可能性（Likelihood）**
| 等級 | 分數 | 定義 |
|------|------|------|
| 極低 | 1 | 幾乎不會發生，需要多重巧合 |
| 低 | 2 | 偶爾可能，但有防護措施 |
| 中 | 3 | 有一定機率，過去發生過類似情況 |
| 高 | 4 | 很可能發生，缺乏防護 |
| 極高 | 5 | 幾乎確定會發生 |

**影響程度（Impact）**
| 等級 | 分數 | 定義 |
|------|------|------|
| 微小 | 1 | 不影響核心功能，幾分鐘內修復 |
| 輕微 | 2 | 局部功能受影響，1 小時內修復 |
| 中等 | 3 | 重要功能受損，需要半天修復 |
| 嚴重 | 4 | 核心功能中斷，影響老蔡使用 |
| 災難 | 5 | 資料遺失 / 系統全掛 / 安全漏洞 |

### 風險等級 = 可能性 x 影響

```
影響 ↑
  5 │  5  10  [15] [20] [25]
  4 │  4   8  [12] [16] [20]
  3 │  3   6    9  [12] [15]
  2 │  2   4    6    8   10
  1 │  1   2    3    4    5
    └──────────────────────→ 可能性
       1   2    3    4    5
```

| 風險分數 | 等級 | 處理方式 |
|---------|------|---------|
| 1-4 | 低風險（綠） | 接受，正常執行 |
| 5-9 | 中風險（黃） | 準備備案，執行時多注意 |
| 10-15 | 高風險（橙） | 必須有備案，先報告阿策確認 |
| 16-25 | 極高風險（紅） | 停下來，報告老蔡決定 |

## 2. 風險識別清單（OpenClaw 專屬）

### 技術風險
| 風險 | 典型可能性 | 典型影響 | 預設備案 |
|------|-----------|---------|---------|
| TypeScript 編譯失敗 | 3 | 2 | 先 `npx tsc --noEmit` 檢查 |
| npm 依賴衝突 | 2 | 3 | 鎖定版本，`npm ci` |
| API 回傳格式變更 | 2 | 3 | 加 schema 驗證 |
| Supabase 連線逾時 | 2 | 4 | 加 retry + fallback |
| 向量搜尋結果不準 | 3 | 2 | 調 threshold + 加同義詞 |

### 部署風險
| 風險 | 典型可能性 | 典型影響 | 預設備案 |
|------|-----------|---------|---------|
| push 覆蓋老蔡代碼 | 3 | 5 | 先 `git pull` + `git diff` |
| build 失敗 server 掛 | 3 | 4 | launchd KeepAlive 自動重啟 |
| 版本號忘記更新 | 4 | 1 | Push SOP 裡有強制步驟 |
| 老蔡目錄忘記 pull | 3 | 3 | Push SOP 有提醒 |

### 運維風險
| 風險 | 典型可能性 | 典型影響 | 預設備案 |
|------|-----------|---------|---------|
| EADDRINUSE port 衝突 | 3 | 3 | `lsof -i :3011` 找 PID kill |
| Telegram bot 斷線 | 2 | 3 | 清 sessions + 重啟 gateway |
| Token 超限（Claude） | 3 | 2 | 切 Gemini 子代理 |
| 磁碟空間不足 | 1 | 4 | 清 log + node_modules |

### 安全風險
| 風險 | 典型可能性 | 典型影響 | 預設備案 |
|------|-----------|---------|---------|
| API Key 外洩 | 1 | 5 | `.env` 不進 git，`.gitignore` 確認 |
| Prompt injection | 2 | 3 | security-sentinel-skill |

|---|---------|-------|------|------|------|------|

## 4. 備案規劃框架

### 備案三要素
1. **觸發條件**：什麼情況下啟動備案？（不要等到全壞才反應）
2. **具體行動**：備案的步驟是什麼？（要可執行，不是「想辦法」）
3. **回復時間**：預估多久能恢復？

### 備案模板
```markdown
### 備案：[風險 ID] — [風險名稱]

**觸發條件**：[具體描述什麼時候啟動]
**行動步驟**：
1. [第一步]
2. [第二步]
3. [第三步]
**預估回復時間**：[X 分鐘/小時]
**負責人**：[小蔡/老蔡/子代理]
```

### 常用備案模式

**回滾（Rollback）**
```bash
# git 回到上一版
git revert HEAD
git push origin main
# 老蔡目錄同步
cd /Users/caijunchang/openclaw任務面版設計 && git pull origin main && cd server && npm run build
launchctl stop com.openclaw.taskboard && sleep 2 && launchctl start com.openclaw.taskboard
```

**降級（Graceful Degradation）**
- 功能掛了 → 回傳預設值，不要讓整個 API 500
- 外部服務掛了 → 切 fallback provider
- AI 模型超限 → 降級到更便宜的模型

**隔離（Isolation）**
- 新功能出問題 → feature flag 關掉
- 某個 bot 異常 → 只停那個 bot
- 某個 API 過載 → 限流（rate limit）

## 5. 風險決策流程圖

```
發現風險
    ↓
評估分數 = 可能性 x 影響
    ↓
┌─ 分數 ≤ 9（低/中）→ 記錄 + 正常執行
│
├─ 分數 10-15（高）→ 準備備案 → 告知阿策 → 執行（帶備案）
│
└─ 分數 ≥ 16（極高）→ 停下 → 報告老蔡 → 等指示
```
