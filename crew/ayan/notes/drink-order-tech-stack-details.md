# 訂購飲料網站詳細技術選型分析

**日期：** 2026-03-07
**研究員：** 阿研

## 總體架構

延續先前提案，採用 **Headless (無頭式) 架構**，前後端分離，透過 API 溝通。此架構具備高度彈性，有利於未來擴展至 App 或其他終端。

## 技術選型詳解

### 1. 前端框架：Next.js

*   **選用理由：**
    *   **SEO 優化：** 內建伺服器端渲染 (SSR) 與靜態網站生成 (SSG)，對商品列表等需要被搜尋引擎索引的頁面至關重要。
    *   **開發體驗：** 基於 React，生態系成熟，開發工具鏈完整，上手快速。
    *   **效能：** 圖片優化、程式碼分割等內建功能可顯著提升網站載入速度與使用者體驗。
*   **替代方案：** Vue (Nuxt.js)。雖然也很優秀，但 React 在台灣的社群與人才庫更為龐大。

### 2. 後端框架：NestJS

*   **選用理由：**
    *   **架構清晰：** 採用 TypeScript 並借鑒 Angular 的模組化、依賴注入等設計模式，使大型專案的架構清晰、易於維護與擴展。
    *   **整合性高：** 輕鬆整合 TypeORM/Prisma (資料庫)、GraphQL、WebSocket 等常用工具。
    *   **類型安全：** 完全支援 TypeScript，減少執行時錯誤。
*   **替代方案：** Express.js。更輕量自由，但大型專案中容易缺乏統一架構，增加維護成本。

### 3. 資料庫：PostgreSQL

*   **選用理由：**
    *   **功能強大：** 開源、穩定，支援複雜查詢與 JSONB 格式，適合處理商品規格、訂單等半結構化資料。
    *   **擴展性佳：** 具備良好的水平與垂直擴展能力，足以應對未來業務增長。
    *   **團隊協作：** 阿數已在使用 Supabase (基於 PostgreSQL)，採用相同技術有利於數據整合與分析。
*   **替代方案：** MySQL。同樣穩定，但在處理複雜資料類型與地理資訊方面，PostgreSQL 略勝一籌。

### 4. 部署方案

*   **前端 (Vercel)：**
    *   **理由：** 專為 Next.js 設計，提供無縫的 CI/CD 整合、全球 CDN 加速與自動擴展，極大簡化前端部署流程。
*   **後端 (GCP Cloud Run / Render)：**
    *   **理由：** 採用容器化部署 (Docker)，可按需擴展，符合成本效益。Render 提供更簡便的開發者體驗，GCP 則提供更完整的雲端生態系。

## 結論

此技術棧 (Next.js + NestJS + PostgreSQL) 是一套經過市場驗證的現代化組合，兼顧了開發效率、系統效能與長期維護性，非常適合本次「訂購飲料網站」專案的需求。
