# A/B 測試框架

## 測試設計流程

每個 A/B 測試必須遵循以下流程：

1. **假設 (Hypothesis)**：明確陳述「如果我們做 X，那麼指標 Y 會因為 Z 原因而改善」
2. **指標 (Metric)**：定義主要指標（primary）和護欄指標（guardrail）
3. **變體 (Variant)**：描述控制組與實驗組的具體差異
4. **樣本量 (Sample Size)**：根據預期效果大小計算所需最小樣本
5. **持續時間 (Duration)**：最少運行 7 天，覆蓋完整週期

## 統計嚴謹度要求

| 要求 | 標準 |
|------|------|
| 信賴水準 | 95%（p < 0.05） |
| 每個變體最少事件數 | 100 events |
| 最短運行天數 | 7 天（含週末） |
| 不可提前結束 | 即使提前達到顯著性也不可提前停止 |

## 併行測試限制

- **最多同時 3 個測試**，避免交互效應（interaction effects）
- 若測試涉及同一頁面或同一用戶群，必須排除重疊
- 若必須超過 3 個，需記錄潛在的交互風險

## 測試文件記錄

每個測試必須記錄：

| 欄位 | 說明 |
|------|------|
| Test ID | 唯一識別碼（格式：`AB-YYYYMM-NNN`） |
| 假設 | 完整假設陳述 |
| 開始日期 | |
| 結束日期 | |
| 結果 | 統計顯著性 + 效果大小（effect size） |
| 學習 | 無論成敗都要記錄學到了什麼 |

## 常見測試方向

| 測試類別 | 範例 |
|----------|------|
| CTA 文案 | 「免費試用」vs「立即開始」vs「馬上體驗」 |
| 定價展示 | 月費 vs 年費優先、價格錨定效果 |
| 新手引導流程 | 步驟數量、是否強制完成、引導內容順序 |
| Email 主旨行 | 緊迫感 vs 好奇心 vs 利益導向 |
| 功能曝光 | 功能入口位置、提示時機、引導方式 |

## 決策準則

- **Ship**：統計顯著 + 效果大小有意義 + 護欄指標未惡化
- **Iterate**：方向對但效果不夠大，調整後再測
- **Kill**：無效果或負面影響，停止並記錄學習
