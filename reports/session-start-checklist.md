# 新對話啟動檢查清單 (Session Start Checklist)

## 快速檢查清單 (Quick Start)

```
□ 1. 檢查 taskboard/running/ 是否有任務正在執行
□ 2. 檢查是否有子代理 session 正在運作（running/active）
□ 3. 如果 running 為空且無子代理運作：
   → 啟動子代理執行自檢流程
   → 檢查專案進度
   → 識別缺口
   → 生成新任務
□ 4. 如果發現任務積壓：
   → 啟動子代理並行處理
□ 5. 加載上次對話的上下文（如有）
□ 6. 確認優先任務列表
□ 7. 提示用戶最新狀態
```

---

## 詳細檢查流程

### Phase 1: 系統狀態檢查 (5 分鐘)

#### 1.1 檢查運行中的任務

```bash
# 掃描 taskboard/running 目錄
ls -la taskboard/running/

# 分析運行狀態
# - 有多少任務在執行？
# - 每個任務的進度是多少？
# - 是否有卡住超過 1 小時的任務？

# 輸出：
# running_tasks: {count}
# stuck_tasks: {list}
# estimated_time_to_clear: {時間}
```

#### 1.2 檢查子代理狀態

```bash
# 查詢當前運行的子代理 session
# - 有多少個活躍 session？
# - 每個 session 的任務是什麼？
# - 預計何時完成？

# 若有活躍代理，等待其報告
# 若無活躍代理，準備啟動新的

# 狀態指標：
# - IDLE: 無代理運行
# - RUNNING: {n} 個代理在執行
# - WAITING: {n} 個代理等待決策
```

#### 1.3 檢查掛起任務

```bash
# 掃描 taskboard/pending 目錄
ls -la taskboard/pending/

# 分析待處理任務
# - 有多少任務等待分配？
# - 優先級分佈如何？
# - 最早應完成的是哪個？

# 判斷是否需要立即處理
# - 是否有任務期限已過？
# - 是否有阻塞其他任務？
# - 是否有新進入的高優先級任務？
```

---

### Phase 2: 快速診斷 (3 分鐘)

#### 2.1 健康度評估

```
健康度 = (已完成任務 / 總任務) × 進度評分
進度評分 = 1 - (阻塞任務 / 活動任務)

計算範例：
- 總任務: 10
- 已完成: 6
- 活動中: 3
- 阻塞: 1

健康度 = (6 / 10) × (1 - 1/3) = 0.6 × 0.67 = 40.2% ⚠️ WARNING
```

#### 2.2 關鍵瓶頸識別

```
優先檢查：
1. 是否有任務被多個阻塞因素卡住？
2. 是否有關鍵路徑上的任務被延遲？
3. 是否有 24 小時未有進展的任務？
4. 是否有資訊缺口導致任務無法開始？

輸出：
- blockers: [{task_id, reason, impact}]
- critical_path: [task_id_list]
- stuck_duration: [{task_id, hours}]
```

#### 2.3 資源狀態

```
檢查項目：
- 所有需要的人員是否可用？
- 是否有技能缺陷阻塞了任務？
- 是否有外部依賴未滿足？
- 是否有計算/存儲資源瓶頸？

輸出：
- resource_gaps: [{type, severity, impact}]
- skill_gaps: [{skill, missing_count}]
- external_deps: [{dep, status, owner}]
```

---

### Phase 3: 決策與行動 (2 分鐘)

#### 情景 A: 一切正常 (HEALTHY)

```
✅ 條件
- running/ 中的任務進展順利
- 無關鍵阻塞
- 所有人員分配明確
- 沒有超期任務

✅ 行動
1. 加載上次對話的上下文
2. 列出當日優先任務 (TOP 3)
3. 建議用戶下一步方向
4. 等待新指令
```

#### 情景 B: 輕微警告 (WARNING)

```
⚠️  條件
- 有 1-2 個任務卡住 < 2 小時
- 有未分配任務但可快速分配
- 有信息缺口但影響不大

⚠️  行動
1. 列出待解決的問題
2. 自動生成 1-2 個「解除阻塞」任務
3. 建議優先順序
4. 詢問用戶是否啟動子代理處理
```

情景內容：
```markdown
### 發現的問題
- [ ] TASK-001: 資訊缺失，需蒐集需求文檔
- [ ] TASK-005: 分配給 Person-A，但其目前忙於 TASK-003
- [ ] TASK-010: 等待 API Key，可向 DevOps 催促

### 建議行動
1. 立即生成 TASK-001-RESEARCH (HIGH 優先級)
2. 與 Person-A 重新協商 TASK-005 進度
3. 向 DevOps 催促 API Key，預計 2 小時內完成

### 需要用戶決策
- 是否啟動子代理並行處理以上 3 個任務？
- 如是，估計 2-3 小時內全部解決
```

#### 情景 C: 嚴重警告 (CRITICAL)

```
🔴 條件
- 有 3+ 個任務被阻塞
- 有任務超期未完成
- 關鍵路徑上有重大障礙
- 資訊嚴重缺失或技能缺陷難以補救

🔴 行動
1. 立即生成詳細問題報告
2. 自動啟動子代理進行全面自檢
3. 生成 3-5 個「解除阻塞」高優先級任務
4. 標記為需要 escalation 或決策
5. 向用戶報警並請求指導
```

報警內容：
```markdown
🚨 CRITICAL: 系統進度卡頓

### 現狀分析
- 進行中的任務: 5 個
- 被阻塞的任務: 3 個 (TASK-001, TASK-005, TASK-010)
- 未分配的任務: 2 個
- 超期任務: 1 個 (TASK-003, 4 小時)

### 根本原因
1. 資訊缺失 (2 個任務)
2. 技能缺陷 (1 個任務)
3. 外部依賴 (1 個任務)
4. 人力不足 (1 個任務)

### 立即建議
1. 啟動並行自檢 → 15 分鐘內得出完整分析
2. 生成 5 個解除阻塞的子任務 → 4 小時內清除
3. 重新分配人力 → 避免瓶頸

### 需要用戶決策
- [ ] 是否授權自動重新分配人力?
- [ ] 是否允許啟動外包或臨時資源?
- [ ] 是否需要向管理層報告?
```

---

### Phase 4: 對話上下文恢復 (2-3 分鐘)

#### 4.1 加載上次對話記憶

```bash
# 檢查是否存在上次對話的上下文
if [ -f "taskboard/session-context-latest.json" ]; then
    # 加載並顯示：
    - 上次對話的日期/時間
    - 執行的任務列表
    - 懸停的決策
    - 待解決的問題
    - 下一步建議的行動
fi
```

#### 4.2 檢查是否有待決策項

```
待決策項檢查清單：
□ 上次提出了什麼問題需要決策？
□ 用戶是否已經給出指導？
□ 是否有 WAITING_FOR_DECISION 狀態的任務？
□ 是否有 escalation 等待批准？

若有待決策項：
→ 優先呈現
→ 說明背景與選項
→ 等待用戶明確指示
```

#### 4.3 列出當日優先清單

```
優先任務清單 (TODAY'S FOCUS):

🔴 立即行動 (HIGH)
1. TASK-XXX: [標題]
   狀態: [進行中/阻塞/待分配]
   負責: [Person]
   完成度: [%]
   下一步: [具體行動]

2. TASK-YYY: [標題]
   ...

🟡 今日完成 (MEDIUM)
1. TASK-ZZZ: ...

🟢 靈活安排 (LOW)
1. TASK-WWW: ...

📊 整體進度
- 完成率: X%
- 進行中: N 個任務
- 阻塞中: M 個任務
- 下次檢查時間: [建議時間]
```

## 何時跳過檢查

```
⏭️  可以跳過完整檢查的情況：

1. 用戶明確說「繼續上次工作」且上次記憶清晰
   → 只做 Phase 4 (上下文恢復)

2. 上次檢查距今不超過 30 分鐘且無新任務進入
   → 只做快速狀態確認

3. 系統上次檢查結果為 HEALTHY 且無預期變化
   → 快速掃描，確認無新問題

⚠️  但以下情況必須做完整檢查：
- 新開對話 (首次 session)
- 上次檢查 > 4 小時前
- 用戶報告問題或異常
- 有新任務進入系統
- 上次健康度為 WARNING 或 CRITICAL
```
