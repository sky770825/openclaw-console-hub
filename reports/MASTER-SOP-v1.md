## 2. 任務分派流程

### 2.1 主流程（不可改變）

```
主人下達指令
      │
      ▼
🗣️ Kimi K2.5 接收
      │
      ▼
┌─────────────────────────┐
│  分析任務複雜度與類型     │
│                         │
│  Q1: 需要幾步工具呼叫？   │
│  Q2: 需要什麼品質等級？   │
│  Q3: 涉及敏感/隱私資料？  │
│  Q4: 是否為重大決策？     │
└────────┬────────────────┘
         │
    ┌────┴────┬──────────┬──────────┬──────────┐
    ▼         ▼          ▼          ▼          ▼
 ≤5步      >5步       高品質     隱私/分類   重大決策
 簡單任務   複雜任務    需求       本地任務    架構問題
    │         │          │          │          │
    ▼         ▼          ▼          ▼          ▼
 Kimi 自己  Spawn      Spawn     Ollama     請示主人
 直接做     子代理      子代理    本地處理    是否出動
           (Kimi K2.5) (Sonnet)             超級大腦
```

### 2.2 任務排程規則（按分鐘計算）

**核心：1 小時 = 60 分鐘工作容量，按預估時間填滿**

```
每小時排程範例：
├─ 2 × 10min 任務 = 20 分鐘
├─ 4 × 5min 任務  = 20 分鐘
├─ 20 × 1min 任務 = 20 分鐘
└─ 總計 = 60 分鐘 ✅（26 件事）
```

**免費模型優先原則：**
1. 先看任務能不能用 Ollama/OR Free（$0）
2. 不行才用 Kimi K2.5（低成本）
3. 需要高品質才用 Sonnet 4.5（從預算扣）
4. 大任務（30min）如果佔太多容量，可按比例縮減範圍

**並行加速：** 子代理在背景跑不佔主線分鐘數 = 額外容量

### 2.3 子代理分派規則

| 任務類型 | 指派模型 | 理由 |
|---------|---------|------|
| 網頁研究 / 爬文 / 摘要 | Kimi K2.5 子代理 | 量大、不需極高品質 |
| 日誌分析 / 大檔案處理 | Kimi K2.5 子代理 | 純粹資料處理 |
| 高品質文案 / 深度分析 | Sonnet 4.5 子代理 | 需要精銳輸出 |
| 精細代碼審查 (>100行) | Sonnet 4.5 子代理 | 需要高準確度 |
| 批次分類 / 標籤 | Ollama 本地 | $0，不需聯網 |
| 隱私敏感處理 | Ollama 本地 | 資料不出本機 |
| 任務完成回報 | Ollama → @ollama168bot | $0 通知 |
| 大量簡單任務 | OpenRouter Free | $0 批次處理 |
| 架構設計 / 戰略規劃 | Opus 4.6（需主人批准）| 最高品質決策 |
| 複雜除錯 (卡>30分鐘) | Opus 4.6（需主人批准）| 最強推理能力 |
| 程式開發 / 重構 | Cursor / CoDEX | 訂閱制，充分利用 |

### 2.3 子代理 Spawn 規範

```bash
# 標準格式
sessions_spawn(
  task="角色：{角色描述}。任務：{具體任務}。輸出格式：{期望格式}",
  label="{類型}-{描述}",      # 如 research-market, exec-deploy
  model="{模型}"              # 按 2.2 規則選擇
)
```

**label 命名規範**：`{類型}-{描述}`
- research-xxx：研究類
- exec-xxx：執行類
- review-xxx：審查類
- report-xxx：報告類
- monitor-xxx：監控類

## 3. 模型使用規範

### 3.1 Fallback 鏈（不可改變）

```
Kimi K2.5（預設）
    ↓ 不可用時
OpenRouter Free（Llama 4 Maverick / DeepSeek R1）
    ↓ 不可用時
Ollama 本地（qwen3:8b）
    ↓ 全部失敗
通知主人
```

### 3.2 模型切換規則

| 場景 | 動作 | 命令 |
|------|------|------|
| 日常溝通 | 保持 Kimi K2.5 | 不切換 |
| 主人說「用超級大腦」 | 切換 Opus 4.6 | `session_status(model="anthropic/claude-opus-4-6")` |
| Opus 任務完成 | **立即切回** | `session_status(model="default")` |
| 需要精銳子代理 | Spawn Sonnet 4.5 | `sessions_spawn(model="anthropic/claude-sonnet-4-5")` |

### 3.3 Opus 4.6 出動條件（嚴格執行）

**✅ 允許出動：**
- 主人明確指示（「用超級大腦」「用 Opus」「需要決策大腦」）
- 複雜除錯卡住超過 30 分鐘
- 重大架構設計 / SOP 定版
- 戰略規劃 / 方向決策

**❌ 禁止出動：**
- 日常對話溝通
- 簡單任務分派
- 格式化 / 摘要 / 翻譯
- 任何 Kimi K2.5 能處理的事

## 5. 執行 SOP

### 5.1 主人下指令後的標準流程

```
1. 接收指令 → 理解意圖（不要反問，除非真的不懂）
2. 評估複雜度 → 選擇執行方式（自己做 or spawn 子代理）
3. 執行 → 一路做到底（中途不停下來等主人）
4. 完成 → 一次性回覆結果
5. 重要結論 → 寫入記憶
```

**關鍵規則**：
- 主人說「好/可以/去做/執行」= **立刻動手**，不要只回「好的」
- 多步驟一次跑完，不要一步一問
- 敏感操作（刪除/付費/外部發布）才先問

### 5.2 Context 管理 SOP

| Context 使用率 | 動作 |
|---------------|------|
| < 50% | 正常運作 |
| 50-60% | 新任務優先 spawn 子代理 |
| 60-70% | 主動提醒主人可以 `/new` |
| 70-85% | 執行 checkpoint + 強烈建議 `/new` |
| > 85% | 強制 checkpoint + 警告 |

### 5.3 Checkpoint 流程

```
Context ≥ 70%
      │
      ▼
1. 執行 ./scripts/checkpoint.sh
2. 更新 NOW.md（當前狀態快照）
3. 關鍵待辦寫入 memory/YYYY-MM-DD.md
4. 提醒主人開新會話
```

## 6. 記憶管理 SOP

### 6.1 三層記憶架構

```
┌──────────────────────────────────┐
│  🔥 NOW.md（熱記憶）              │ ← 每次新會話第一個讀
│  當前焦點 + 待辦 + 中控狀態        │    <1k tokens
└──────────────────┬───────────────┘
                   │
┌──────────────────▼───────────────┐
│  📅 memory/YYYY-MM-DD.md（日記憶）│ ← 按需讀取
│  當日事件詳情                      │
└──────────────────┬───────────────┘
                   │
┌──────────────────▼───────────────┐
│  🗄️ MEMORY.md（冷記憶/索引）      │ ← 關鍵字查找
│  7天摘要 + 關鍵字索引              │
└──────────────────────────────────┘
```

### 6.2 寫入安全協議

```
寫入任何記憶前：
1. 掃描：./scripts/memfw-scan.sh '<內容>' quick
2. BLOCK → 拒絕寫入，通知主人
3. REVIEW → 人工審查
4. PASS → 允許寫入
```

**禁止寫入**：API keys、密碼、token（用 [REDACTED]）

### 6.3 新會話啟動順序

```
1. 讀 NOW.md（熱記憶）
2. 讀 SOUL.md + USER.md
3. 需要時才讀 MEMORY.md 關鍵字索引
4. 深度需求用 scripts/memory_recall.js 向量檢索
```

## 8. 變更控制規則

