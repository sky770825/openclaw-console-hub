# 16 — 雙手能力邊界白皮書

> 你的 8 個 action 各有什麼限制？怎麼組合最高效？看這裡。

---

## 每個 Action 的完整 Spec

### read_file
```json
{"action":"read_file","path":"~/.openclaw/workspace/MEMORY.md"}
```
- **用途**：讀取任何文字檔
- **限制**：截斷到 ~2000 字回饋給你
- **timeout**：5 秒
- **二進位檔**：PNG、JPG、ZIP 等會失敗，用 `run_script file $PATH` 檢查類型
- **超大檔案**：超過 2000 字會被截斷，看不到尾部

### write_file
```json
{"action":"write_file","path":"~/.openclaw/workspace/notes/my-note.md","content":"內容"}
```
- **用途**：建立或覆蓋檔案
- **限制**：content 太長可能被 JSON 截斷
- **覆蓋策略**：直接覆蓋，沒有備份！寫之前確認不會蓋掉重要東西
- **目錄必須存在**：先 `mkdir` 再 `write_file`
- **禁區**：.env、SOUL.md、AGENTS.md、IDENTITY.md、BOOTSTRAP.md

### mkdir
```json
{"action":"mkdir","path":"~/.openclaw/workspace/new-dir"}
```
- **用途**：建立目錄
- **已存在**：不會報錯（等同 `mkdir -p`）
- **巢狀**：支援多層（`a/b/c` 一次建好）

### move_file
```json
{"action":"move_file","from":"舊路徑","to":"新路徑"}
```
- **用途**：搬移或重命名
- **目標目錄必須存在**
- **會覆蓋**：如果目標已存在，直接覆蓋

### list_dir
```json
{"action":"list_dir","path":"~/.openclaw/workspace"}
```
- **用途**：列出目錄內容
- **回饋**：檔案名列表，不含大小和日期
- **建議**：如果需要詳細資訊，用 `run_script ls -lh $PATH` 更好

### run_script
```json
{"action":"run_script","command":"ls -la ~/.openclaw/workspace"}
```
- **用途**：執行任意 bash 指令
- **時限**：30 秒，超時被 kill
- **輸出**：截斷到 ~800 字
- **環境**：有基本環境變數，但**沒有 API key**（安全隔離）
- **危險**：可以改系統狀態，三思後行
- **適合**：查看、分析、驗證。不適合長時間運算

### create_task
```json
{"action":"create_task","name":"任務名稱","description":"描述"}
```
- **用途**：在任務板建立新任務
- **異步**：建完後由 auto-executor 接手，不是立即執行
- **適合**：需要 Gemini 寫腳本 + sandbox 執行的任務

### ask_ai
```json
{"action":"ask_ai","model":"flash","prompt":"你的問題","context":"背景資料"}
```
- **用途**：諮詢其他 AI 代理
- **模型選擇**：flash（快）、pro（深）、claude/sonnet（精準）、opus（最強）
- **timeout**：Gemini 90 秒、Claude CLI 120 秒
- **回饋**：截斷到 ~2000 字
- **費用**：flash 最便宜、opus 最貴（走主人訂閱）
- **不會觸發 chain**：ask_ai 的回覆不含 action JSON，不會遞迴

## 連續行動（Chain）的限制

| 參數 | 值 |
|------|-----|
| 最多連續步數 | 5 步 |
| Self-Drive 追加 | 最多 2 輪 |
| 每步最多 action | 無限制（但建議 1-3 個） |
| read_file 回饋上限 | 2000 字 |
| ask_ai 回饋上限 | 2000 字 |
| 其他 action 回饋上限 | 800 字 |

## 行動序列設計模式

### 模式 A：查詢 → 判斷 → 回覆（最常用）
```
step 0: read_file 讀資料
step 1: 分析結果，直接回覆（不放 action）
```
用 2 步，省 3 步。大多數查詢任務用這個就夠了。

### 模式 B：讀 → 寫 → 驗證
```
step 0: read_file 讀範本
step 1: write_file 寫新內容
step 2: run_script 驗證產出物
step 3: 如果驗證失敗，修復；成功就回覆
```
用 3-4 步，留 1 步容錯。

### 模式 C：讀 → 問 AI → 整合
```
step 0: read_file 讀代碼
step 1: ask_ai 諮詢分析
step 2: 整合 AI 回覆 + 自己判斷，回覆主人
```
用 3 步。注意 ask_ai 可能要 10-20 秒。

### 模式 D：派遣任務（最省步數）
```
step 0: read_file 讀 BLUEPRINT.md 確認需求
step 1: create_task 建任務
step 2: 回覆主人「已建任務交給 auto-executor」
```
用 2 步。讓 auto-executor 接手重活。

## 效率優化技巧

1. **run_script 合併多個查詢**
   ```
   ❌ step 0: list_dir → step 1: run_script ls -la → step 2: run_script wc -l
   ✅ step 0: run_script "ls -lh $DIR && echo '---' && wc -l $DIR/*.md"
   ```
   一步搞定，省兩步。

2. **先查索引再讀檔**
   ```
   ❌ step 0: list_dir server/src/ → step 1: list_dir server/src/routes/ → ...
   ✅ step 0: read_file CODEBASE-INDEX.md → 直接知道要讀哪個檔案
   ```

3. **write_file 前不用 read_file**
   如果是全新檔案，直接寫，不用先讀。

4. **ask_ai 把 context 一次給足**
   ```
   ❌ ask_ai(prompt="分析") → 資訊不足 → 再 ask_ai(prompt="補充...")
   ✅ ask_ai(prompt="分析以下代碼的三個問題", context=完整代碼)
   ```

## Telegram 訊息限制

- 一條訊息最多 **4096 字**
- 超過會被系統自動分段發送
- action JSON 不算在訊息裡（系統會拆開處理）
- 回覆太長時，精簡重點，細節用 write_file 存起來
