## A. 主力模型（優先使用）

### 1️⃣ Kimi K2.5（日常主模型）
| 項目 | 內容 |
|------|------|
| **用途** | 日常協作、任務調度、一般分析 |
| **成本** | 低（~$0.0005/1k tokens input） |
| **優勢** | 速度快、上下文理解佳、成本低 |
| **限制** | 複雜推理能力一般 |
| **何時用** | 默認模型，所有日常任務優先 |

### 2️⃣ Grok 4.1（深度推理專用）
| 項目 | 內容 |
|------|------|
| **用途** | 深度推理、架構設計、複雜決策 |
| **成本** | 高（$0.03/1k input, $0.15/1k output） |
| **優勢** | 推理能力強、決策質量高 |
| **限制** | 需付費額度 |
| **何時用** | 僅 P0/P1 且有明確複雜度需求時啟用 |

### 3️⃣ Gemini 2.5 Flash（快速摘要專用）
| 項目 | 內容 |
|------|------|
| **用途** | 快速摘要、監控報告、低成本短任務 |
| **成本** | 免費額度豐富 |
| **優勢** | 超快速、免費、適合短文本 |
| **限制** | 深度分析能力有限 |
| **何時用** | 監控、日報、快速查詢 |

### 4️⃣ Ollama（本地模型）
| 項目 | 內容 |
|------|------|
| **用途** | 本地可離線任務、低敏感資訊、固定格式輸出 |
| **成本** | $0（完全免費） |
| **優勢** | 隱私安全、離線可用、零成本 |
| **限制** | 能力較弱，推理不足 |
| **何時用** | 監控報告、格式轉換、無網路場景 |

## B. 其他可用模型（條件啟用）

### 5️⃣ Claude Sonnet/Opus（高風險高價值）
| 項目 | 內容 |
|------|------|
| **用途** | 僅用於高風險高價值任務（需明確指定） |
| **成本** | 中高（Sonnet 便宜，Opus 更貴） |
| **優勢** | 推理能力強、文本理解深 |
| **重要** | ❌ 不作預設模型，避免成本失控 |

| 項目 | 內容 |
|------|------|
| **用途** | 前端/UI 微調、程式修改與驗證 |
| **成本** | 訂閱制（按月） |
| **特點** | 專門用於程式改碼 |
| **重要** | ❌ 由 Cursor Agent 接手，不作聊天模型 |
| **流程** | 小蔡分派 → Cursor Agent 執行 |

### 7️⃣ Codex Native（系統故障專用）
| 項目 | 內容 |
|------|------|

| **特點** | 專業故障排查能力 |
| **重要** | ❌ 優先走訂閱，明確指定時才走額外付費 API |

## C. 路由優先順序（決策樹）

### 日常任務
```
優先級: Kimi K2.5 → Gemini Flash → Ollama
例子: 
  - 日報生成 → Ollama / Gemini Flash
  - 任務調度 → Kimi K2.5
  - 快速查詢 → Gemini Flash
```

### 深度任務（複雜分析、架構設計）
```
優先級: Grok 4.1 → Kimi K2.5 → Sonnet/Opus（必要時）
例子:
  - 商業模式分析 → Kimi K2.5（成本考量）
  - 複雜決策 → Grok 4.1（老蔡明確指定）
  - 高風險決策 → Opus（必要時）
```

### 監控任務（定時巡檢）
```
優先級: Ollama → Gemini Flash → Kimi K2.5
例子:
  - Cron 監控報告 → Ollama
  - 儀表板更新 → Gemini Flash
  - 複雜監控邏輯 → Kimi K2.5
```

### 工程修復（程式改碼、故障排查）
```
優先級: Codex Native → Cursor Agent（依任務類型）
例子:
  - 系統故障排查 → Codex Native
  - UI/前端改碼 → Cursor Agent
  - 後端修復 → Codex Native
```

## D. 風險 & 成本保護機制

### 超時處理
```
連續 2 次 timeout:
  1. 先加 timeout（例：120s → 180s）
  2. 再嘗試 1 次
  3. 仍失敗 → 切換模型

連續 3 次 failed:
  觸發 [ESCALATE_Codex] 立即升級
```

### 成本控制
```
高成本模型（Grok/Opus）啟用條件:
  ✅ P0 或 P1 任務
  ✅ 有明確複雜度需求
  ✅ 預期 ROI 收益 > 成本
  ❌ 日常任務禁用
  ❌ 無明確收益時禁用

成本超高警告:
  - Grok 連續用超過 3 次 → 自動改 Kimi
  - Opus 任何時候 → 必須老蔡明確批准
```

### 回報規範（必帶欄位）
```
【小蔡執行-TASK_NAME】
task_id: task_xxx
run_id: run_xxx
model_used: kimi/kimi-k2.5 | xai/grok-4-1-fast | gemini-2.5-flash | ollama/xxx
duration: XXs
status: success | failed | timeout
retry: 0 | 1 | 2
summary: 任務摘要
✅ 完成
```

## E. 實踐範例

### 日常日報（低成本）
```
推薦: Ollama 或 Gemini Flash
理由: 快速、免費、格式固定
回報: task_id/run_id/model_used/status
```

### 商業決策分析（中等複雜）
```
推薦: Kimi K2.5（成本與質量平衡）
備案: Grok 4.1（老蔡明確指定 + P0/P1）
理由: Kimi 足以勝任，成本低
```

### 複雜系統故障排查（高複雜）
```
推薦: Codex Native
備案: Cursor Agent（若涉及程式改碼）
理由: 專業故障排查能力
```

### 高風險高價值決策（極特殊）
```
推薦: Opus（老蔡明確指定）
成本: 必須預先批准
理由: 最高推理能力
```
