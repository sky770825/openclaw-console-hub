## 目的

確保每次換對話（/new）後，你還能接上之前的進度。不再「換對話 = 失憶」。

---

## 核心機制

```
舊對話結束前 → 寫 HANDOFF-LATEST.md (包含情境感知)
新對話啟動後 → 讀 HANDOFF-LATEST.md
```

**就這麼簡單。關鍵是你要每次都做，並且帶上「情境」。**

---

## 🧠 情境感知寫入規範 (Context-Aware Writing)

為了確保記憶不只是「流水帳」，寫入 `HANDOFF-LATEST.md` 或 `MEMORY.md` 時應包含以下要素：

1.  **Background (B)**: 為什麼要做這件事？（主人的要求、系統報錯、預防性維護）
2.  **Decision (D)**: 為什麼選擇這個解法？（成本考量、時間限制、穩定性優先）
3.  **Execution (E)**: 關鍵執行細節。（用了哪個腳本、改了哪個核心配置）
4.  **Reflection (R)**: 這次學到了什麼？（某個工具的坑、一個更好的流程）

## 交接檔位置

```
~/.openclaw/workspace/memory/HANDOFF-LATEST.md
```

**永遠只有這一個檔案**。每次換對話前覆蓋上一次的內容。

舊的交接檔自動備份到：
```
~/.openclaw/workspace/memory/handoff/HANDOFF-{YYYY-MM-DD-HHMM}.md
```

## 換對話前：寫交接摘要

### 什麼時候要寫

| 時機 | 必要性 |
|------|--------|
| Context 到 70%（auto-checkpoint 提醒你） | 必要 |
| 主人說「換對話」「/new」「開新的」 | 必要 |
| 你覺得對話快滿了 | 必要 |
| 主人說「停，先存起來」 | 必要 |
| 任務做到一半要中斷 | 必要 |

### 怎麼寫

**Step 1: 備份舊的交接檔**

```bash
# 如果舊的存在，先備份
if [ -f ~/.openclaw/workspace/memory/HANDOFF-LATEST.md ]; then
  mkdir -p ~/.openclaw/workspace/memory/handoff
  cp ~/.openclaw/workspace/memory/HANDOFF-LATEST.md \
     ~/.openclaw/workspace/memory/handoff/HANDOFF-$(date '+%Y-%m-%d-%H%M').md
fi
```

**Step 2: 寫新的交接摘要**

用以下模板寫入 `memory/HANDOFF-LATEST.md`（強制包含情境感知）：

```markdown
# 交接摘要 (情境感知版)

時間：{YYYY-MM-DD HH:MM}
對話主題：{一句話描述這次對話在做什麼}

## 🎯 當前進度與情境
- **任務**: {名稱}
- **背景 (B)**: {為什麼開始？觸發點是什麼？}
- **決策 (D)**: {為什麼這樣做？放棄了哪些方案？}
- **細節 (E)**: {關鍵修改點、已跑過的指令}
- **風險/反思 (R)**: {要注意的坑或學到的經驗}

## 正在做的事

- {任務 1 — 做到哪了 — 下一步是什麼}
- {任務 2 — 做到哪了 — 下一步是什麼}

## 等待中的事

- {等主人回覆的問題}
- {等其他服務的回應}

## 重要決定

- {主人做了什麼決定}
- {選了方案 A 而不是 B，因為...}

## 注意事項

- {這個要小心}
- {這個不能碰}

## 關鍵檔案

- {修改過的檔案路徑}
- {正在看的檔案路徑}

## 任務板狀態

- running: {哪些任務在跑}
- 需要處理: {哪些任務有問題}
```

**Step 3: 確認寫入成功**

```bash
# 確認檔案存在且不為空
wc -l ~/.openclaw/workspace/memory/HANDOFF-LATEST.md
```

**Step 4: 回報主人**

```
💾 交接摘要已存檔
位置：memory/HANDOFF-LATEST.md
重點：{一句話}
可以換對話了。
```

## 什麼該存、什麼不該存

### 要存

| 內容 | 為什麼 |
|------|--------|
| 正在做的任務和進度 | 接續用 |
| 主人的決定和指示 | 不用再問一次 |
| 修改過的檔案清單 | 知道動了什麼 |
| 遇到的問題和解法 | 避免重蹈覆轍 |
| 任務板上的狀態 | 知道全局 |

### 不要存

| 內容 | 為什麼 |
|------|--------|
| 完整的對話記錄 | 太長，塞不進 context |
| 程式碼全文 | 檔案還在磁碟上，用路徑就好 |
| 已完成且不再需要的任務 | 過去的事不用帶到下次 |
| 系統日誌 | 隨時可以重新查 |

## 長對話中的 Context 管理

### 你要注意的信號

| 信號 | 意思 | 該做什麼 |
|------|------|---------|
| auto-checkpoint 提醒 70% | 快滿了 | 開始準備交接摘要 |
| auto-checkpoint 觸發 80% | 很滿了 | 立刻寫交接摘要，告訴主人要換對話 |
| 開始忘記之前說過的事 | context 被壓縮了 | 查向量知識庫補回資訊 |
| 回覆變慢或不完整 | 可能要溢出了 | 馬上交接 |

### 省 Context 的好習慣

1. **不要重複貼完整檔案** — 用路徑引用：「見 SOP-11 第 3 節」
2. **摘要而非全文** — 「API 回 200，正常」而不是貼整個 response
3. **善用向量搜尋** — 需要查 SOP 時用搜尋，不要把整個 SOP 載入
4. **子派分攤** — 大量搜尋工作用 sessions_spawn，不要自己查
5. **完成就放下** — 任務完成後不用反覆提及

## 三層記憶架構

你的記憶分三層，各有用途：

```
A. 工作記憶（Context Window）— 對話裡看得到的
    ↑ 最快，但對話結束就沒了

B. 長期記憶（向量資料庫）— Qdrant memory_smart_chunks
    ↑ SOP、指令集、AGENTS.md，搜尋才會拿回來

C. 偏好記憶（Profile）— ~/.openclaw/workspace/memory/PROFILE.md
    ↑ 主人的偏好、習慣、禁忌，每次新對話都載入
```

### A. 工作記憶 — 你的 Context Window

就是「這段對話裡你看得到的東西」。

- 不用查庫，直接在腦裡
- 對話結束就沒了 → 所以要寫 HANDOFF
- SOP-17 的核心就在管理這一層

### B. 長期記憶 — Qdrant 向量庫

目前有 380+ chunks，包含所有 SOP、指令集、AGENTS.md。

- 你不會自動記得，要**主動搜尋**才會拿回來
- 搜尋方式：用向量語意搜尋（嵌入 nomic-embed-text）
- 適合：「某個 SOP 怎麼說的」「之前的規範是什麼」

### C. 偏好記憶 — 主人的個人檔案

**位置：** `~/.openclaw/workspace/memory/PROFILE.md`

這是主人的偏好、習慣、禁忌，結構化存好讓你每次新對話都能載入。

**目前已知的偏好（達爾要記住）：**

```yaml
# 主人偏好檔
溝通偏好:
  - 喜歡表格和清單，不喜歡大段文字
  - 回覆要精簡，重點先
  - 帶選項不帶問題（「A 或 B，我建議 A」而不是「怎麼辦？」）
  - 不需要每次都很正式，自然對話就好

工作偏好:
  - 偏好安全穩定，不喜歡冒險的操作
  - 「慢慢來」— 不急著一次全做完，分批比較好
  - 重要檔案怕壞掉，改之前要先確認
  - 省錢優先 — 能免費就不花錢（Ollama > Gemini > Claude Code）

技術偏好:
  - 主力工具：n8n、Supabase、React、Node.js
  - 部署偏好：Docker Compose + cloudflared tunnel
  - 本地開發優先，能跑在 Mac 上就好

禁忌:
  - 不要未經同意就 push 程式碼
  - 不要動正在跑的服務（先問）
  - 不要花錢的操作不先說
  - 不要一次改太多東西
```

**每次新對話的載入順序更新為：**
```
1. 讀 AGENTS.md（規範）
2. 讀 memory/PROFILE.md（主人偏好）      ← 新增
3. 讀 memory/HANDOFF-LATEST.md（上次進度）
4. 跑 ./scripts/self-heal.sh check（系統狀態）
5. 回報主人
```

## 長期記憶寫入規則

不是什麼都往向量庫塞。亂寫會越記越亂。

### 該寫入長期記憶的

| 類型 | 舉例 | 存到哪 |
|------|------|--------|
| 主人確認的偏好 | 「我喜歡表格」「省錢優先」 | PROFILE.md |
| 新建/更新的 SOP | SOP-18 程式碼品質通病 | sop-知識庫/ → 自動向量化 |
| 穩定事實 | 系統架構、API 規格、部署方式 | 知識庫 → 向量化 |
| 反覆出現的問題模式 | 通病清單、常見錯誤 | SOP → 向量化 |
| 主人做的重大決定 | 「用 Supabase 不用 Firebase」 | PROFILE.md 或專案文件 |

### 不該寫入長期記憶的

| 類型 | 舉例 | 為什麼不存 |
|------|------|-----------|
| 一次性的閒聊 | 「今天天氣不錯」 | 沒有重複價值 |
| 未確認的推測 | 「可能是因為 X」 | 可能是錯的，存了會誤導 |
| 完整對話記錄 | 整段對話 | 太長，用 HANDOFF 摘要就好 |
| 程式碼全文 | 整個檔案 | 檔案在磁碟上，用路徑引用 |
| 過時的資訊 | 舊版 API 規格 | 會跟新版衝突 |

### 寫入判斷流程

```
收到新資訊 →
  主人明確說「記住」？ → 寫入 PROFILE.md
  是新 SOP 或規範？ → 寫入 sop-知識庫/ → 跑向量化
  是穩定的技術事實？ → 寫入 knowledge/ → 跑向量化
  是主人反覆提到的偏好？ → 更新 PROFILE.md
  以上都不是？ → 不存，只放 HANDOFF
```

## 向量搜尋補回記憶

當你在新對話中發現忘記某件事：

```
1. 讀 memory/PROFILE.md：看主人偏好
2. 搜尋向量庫：用關鍵字搜 SOP 或歷史記錄
3. 讀 HANDOFF-LATEST.md：看交接摘要
4. 讀 memory/（日期）.md：看當天的記錄
5. 問主人：以上都找不到就問
```

**順序很重要：先自己找，找不到再問。**

## 錯誤處理

| 狀況 | 處理 |
|------|------|
| HANDOFF-LATEST.md 不存在 | 當作全新開始，跑標準啟動流程（指令 01） |
| HANDOFF-LATEST.md 太舊（>24h） | 提醒主人：「上次交接是 X 天前，可能不準了」 |
| 交接摘要看不懂 | 問主人：「上次做到哪了？」 |
| 寫入失敗 | 直接在對話裡回報摘要給主人，至少有文字記錄 |
| 備份資料夾太滿（>50 個檔案） | 刪 30 天前的舊備份 |

## 絕對禁止

- ❌ 換對話前不寫交接摘要就走
- ❌ 新對話啟動不讀 HANDOFF-LATEST.md
- ❌ 交接摘要寫得比 context 還長（控制在 50 行以內）
- ❌ 忘記進度就自己猜，不查不問
- ❌ 把交接摘要當垃圾場，什麼都塞
