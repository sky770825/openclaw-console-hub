# ArchGuard - Pull Request 審查檢查清單 v1.0

> 本檢查清單用於標準化 PR 審查流程，確保所有合併到主分支的程式碼都符合品質標準。

---

## 審查前準備

- [ ] **理解變更範圍**: 閱讀 PR 描述，理解本次變更的目的和範圍。
- [ ] **關聯 Issue**: 確認 PR 是否關聯了相關的 Issue 或任務。

---

## 程式碼品質檢查

### 功能性 (Functionality)
- [ ] **功能正確性**: 程式碼是否正確實現了預期的功能？
- [ ] **邊界情況處理**: 是否考慮了異常輸入、空值、極端情況等邊界條件？
- [ ] **錯誤處理**: 錯誤處理機制是否完善？錯誤訊息是否對用戶友好？

### 可讀性與維護性 (Readability & Maintainability)
- [ ] **命名清晰**: 變數、函數、類別的命名是否清晰、具描述性？
- [ ] **程式碼結構**: 程式碼是否組織良好？函數是否單一職責？
- [ ] **註解完整**: 複雜邏輯是否有適當的註解說明？
- [ ] **遵循風格指南**: 是否遵循了 [CODE_STYLE_GUIDE.md](./CODE_STYLE_GUIDE.md) 中的規範？

### 安全性 (Security)
- [ ] **敏感資訊**: 程式碼中是否意外包含了 API Keys、密碼或其他敏感資訊？
- [ ] **注入風險**: 是否有可能的指令注入或程式碼注入風險？
- [ ] **權限控管**: 新的功能是否正確地實現了權限檢查？

### 效能 (Performance)
- [ ] **資源使用**: 是否有明顯的資源浪費（例如重複的資料庫查詢、不必要的迴圈）？
- [ ] **複雜度**: 演算法的時間/空間複雜度是否合理？

---

## 測試檢查

- [ ] **單元測試**: 新的功能是否有對應的單元測試？
- [ ] **測試覆蓋率**: 測試是否覆蓋了主要的功能路徑和邊界情況？
- [ ] **測試通過**: 所有的自動化測試是否都通過了？

---

## 文件檢查

- [ ] **README 更新**: 如果功能有變更，README 文件是否已同步更新？
- [ ] **API 文件**: 如果有 API 變更，API 文件是否已更新？
- [ ] **CHANGELOG**: 重大的變更是否已記錄在 CHANGELOG 中？

---

## 最終確認

- [ ] **本地測試通過**: 審查者是否在本地環境驗證過功能正常？
- [ ] **無衝突**: PR 是否與目標分支沒有程式碼衝突？
- [ ] **可回滾**: 如果出問題，這次變更是否可以安全地回滾？

---

## 審查結論

- [ ] **Approve (通過)**: 程式碼品質符合標準，可以合併。
- [ ] **Request Changes (請求變更)**: 有需要修改的地方，請作者修正後再次提交。
- [ ] **Comment (僅評論)**: 有一些建議，但不妨礙合併（由作者決定是否採納）。

## 給審查者的建議

1.  **保持友善**: 審查的目的是提升品質，不是批評個人。請使用建設性的語氣。
2.  **解釋「為什麼」**: 當提出修改建議時，請解釋為什麼這樣改會更好。
3.  **區分「必須修改」和「建議」**: 明確標註哪些是阻擋合併的問題，哪些只是優化建議。
4.  **及時回應**: 盡量在 24 小時內給出初步回饋。
