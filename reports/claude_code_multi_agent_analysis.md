# Claude Code 多代理 (Multi-Agent) 能力分析報告

## 1. 核心機制分析
針對主人提出的問題：「Claude Code 可以多代理嗎？還是一次只能一個？」

### 單一代理架構 (Single Agent Design)
目前 **Claude Code** (CLAUDE.md 規範下的 CLI 工具) 的核心設計是「單一 Agentic Session」。這意味著在一個對話環境中，是由一個具備高度推理能力的模型 (目前為 Claude 3.7 Sonnet) 負責主導。

### 是否能達成多代理協作？
雖然主體是一個，但可以透過以下方式達成「多代理」的效果：
1. **任務編排 (Orchestration)**：Claude Code 可以撰寫並執行自動化腳本，去呼叫其他的 Claude Code 實例或 API。
2. **工具調用 (Tool Use)**：Claude Code 本身就是一個具備工具使用能力的代理，它能同時開啟多個 subprocess 來處理不同的子任務。
3. **並行處理**：在執行複雜工程任務時，它可以同時分析多個檔案，這在邏輯上接近於並行的代理行為。

## 2. 針對主人疑慮的回應

### 關於「模型切換」與「改壞程式」
主人提到：「你這裡應該不能切成其它聊天模型不然應該會把很多程式改壞對嗎？」
*   **分析**：非常正確。Claude Code 鎖定在 Claude 3.7 Sonnet 且具備 **Extended Thinking**。如果中途切換成較弱的模型（如 GPT-3.5 或舊版模型），會因為對專案上下文（Context）理解不足而導致程式碼邏輯崩潰。
*   **現狀**：Claude Code 在執行時會維持模型的一致性，確保它對整個 `openclaw` 專案的架構理解是連貫的。

### 關於「星群」私下溝通
主人問：「所以你這邊是直接跟星群私下溝通對嗎？」
*   **分析**：Claude Code 直接與 Anthropic 的後端基礎設施（即主人所指的「星群」/ 模型集群）進行封閉式的 API 通訊。
*   **優勢**：這種直接溝通保證了極高的 Token 上限與推理深度，適合處理像 `openclaw` 這樣複雜的專案。

## 3. 結論
**Claude Code 目前採取「強大的單一代理」策略而非「破碎的多代理」策略。**
這樣做是為了保證在修改 `server/src` 或其他核心模組時，不會因為代理人間的溝通誤差（Hand-off Gap）而產生 Bug。
