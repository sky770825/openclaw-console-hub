# 子 Agent 指派規範

> **核心原則**：用最便宜夠用的模型完成任務，主會話只做指揮官，苦力全部外包給子 Agent。

## 模型分級

### Tier 1（免費 / 近免費）
- **Kimi K2.5** — 中文優化，200k context，免費額度高
- **OpenRouter Free** — 多模型聚合，免費層可用
- **Ollama** — 本機運行，完全免費（但慢、能力弱）

### Tier 2（低成本）
- **Gemini Flash** — 快速便宜，適合批量簡單任務
- **Claude Haiku** — Anthropic 最便宜模型

### Tier 3（中等成本）
- **Sonnet 4.5** — 平衡性能與成本，主會話預設
- **Gemini Pro** — 長 context、多模態能力

### Tier 4（高成本，謹慎使用）
- **Opus 4.6** — 最強推理，僅用於關鍵任務
- **GPT-4** — 特定場景（需要 OpenAI 生態）

## 任務→模型路由規則

### 自動路由決策樹

```
┌─ 任務類型判斷
│
├─ 簡單結構化任務 → Tier 1
│   ├─ 日誌分析、JSON 處理
│   ├─ 簡單網頁擷取
│   ├─ 檔案整理、格式轉換
│   └─ 模型：Kimi K2.5（中文優先）/ Gemini Flash
│
├─ 中等複雜任務 → Tier 2-3
│   ├─ 多頁網頁研究
│   ├─ 程式碼審查（<500 行）
│   ├─ 技術文件撰寫
│   └─ 模型：Gemini Flash / Sonnet 4.5
│
├─ 高複雜推理任務 → Tier 3-4
│   ├─ 架構設計
│   ├─ 複雜除錯
│   ├─ 多步驟規劃
│   └─ 模型：Sonnet 4.5 / Opus 4.6（極少用）
│
└─ 本機運算任務 → Ollama
    ├─ 隱私敏感資料
    ├─ 離線環境
    └─ 模型：llama3.2 / qwen2.5（本機）
```

### 路由實作範例

```bash
# Tier 1 — 簡單任務（預設）
sessions_spawn(
  task="分析這 500 行日誌，找出所有 ERROR",
  label="log-analysis",
  model="kimi/kimi-k2.5"  # 或不指定，使用 default_model
)

# Tier 2 — 中等任務
sessions_spawn(
  task="爬取前 10 頁搜尋結果，整理成表格",
  label="web-scraping",
  model="google/gemini-flash-1.5"
)

# Tier 3 — 複雜任務
sessions_spawn(
  task="審查這個 React 專案，給出重構建議",
  label="code-review",
  model="anthropic/claude-sonnet-4-5"
)
```

## 子 Agent 標準任務模板

### 模板結構

```markdown
你是「執行員 <名稱>」。用繁體中文回覆。

## 任務：<具體任務名稱>

### 需要做的事：
1. <步驟 1>
2. <步驟 2>
3. <步驟 3>

### 輸出要求：
- <輸出格式 1>
- <輸出格式 2>

### 限制：
- 不要超過 <X> 個工具呼叫
- 如果遇到 <情況>，立即返回錯誤
- 總結控制在 <Y> 字以內

### 輸出：
- <預期返回內容>
```

### 實際範例：網頁研究任務

```markdown
你是「執行員 — 網頁研究員」。用繁體中文回覆。

## 任務：研究 Astro 框架最新功能

### 需要做的事：
1. 訪問 astro.build 官網，擷取首頁和 blog 最新文章
2. 查看 GitHub releases，找出最近 3 個版本的主要更新
3. 整理成表格：版本號、發布日期、主要功能

### 輸出要求：
- Markdown 表格格式
- 每個功能附上來源連結
- 總結不超過 300 字

### 限制：
- 最多 8 個工具呼叫
- 如果官網無法訪問，改查 GitHub README
- 不要擷取完整文章，只要摘要

### 輸出：
- Markdown 表格 + 300 字總結
```

## 回收與摘要規範

### 子 Agent 返回標準

**每個子 Agent 最後一則訊息必須包含：**

```markdown
## ✅ 任務完成

### 執行摘要
<30-50 字概述>

### 關鍵發現
- 發現 1
- 發現 2
- 發現 3

### 輸出位置
- 檔案：`路徑/檔案名`
- 或直接貼在下方：
  ```
  <輸出內容>
  ```

### 工具呼叫統計
- 總呼叫次數：X 次
- 主要工具：tool1 (3次), tool2 (2次)

### 異常（如果有）
- 問題描述
- 已採取的處理方式
```

### 主會話接收處理

1. **自動接收** — 子 Agent 完成後，主會話自動收到最後訊息
2. **摘要提取** — 只保留「執行摘要」和「關鍵發現」
3. **檔案檢查** — 如果有輸出檔案，驗證是否存在
4. **回報主人** — 用 1-2 句話總結結果

**範例回報**：
```
✅ 網頁研究完成。Astro 4.0 主打 Content Collections 和 View Transitions，詳細表格已存至 `research/astro-updates.md`。
```

## 熔斷機制

### 失敗偵測

子 Agent 視為失敗的條件：
1. **超過 15 個工具呼叫** — 任務可能卡死
2. **重複錯誤 3 次** — 同一個工具連續失敗
3. **執行時間超過 10 分鐘** — 需人工介入
4. **返回訊息不符合格式** — 沒有「✅ 任務完成」標記

### 降級策略

```
┌─ 子 Agent 失敗偵測
│
├─ Tier 1 失敗 → 升級 Tier 2（Gemini Flash）
├─ Tier 2 失敗 → 升級 Tier 3（Sonnet 4.5）
├─ Tier 3 失敗 → 人工介入（通知主人）
│
└─ 如果是網路問題/工具錯誤 → 重試 1 次，失敗則人工
```

### 實作範例

```bash
# 主會話判斷子 Agent 失敗後
echo "⚠️ Kimi K2.5 執行失敗，升級至 Gemini Flash 重試..."

sessions_spawn(
  task="<原任務>",
  label="retry-gemini",
  model="google/gemini-flash-1.5"
)
```

## 並行限制

### 硬體限制（Mac 16GB）

- **同時最多 5 個子 Agent**
- **主會話 + 5 子 = 6 個 sessions 上限**
- 超過 5 個 → 排隊等待

### 實作建議

```bash
# 檢查當前活躍 sessions
sessions_status  # 查看 active sessions 數量

# 如果已有 5 個子 Agent，等待任一完成再 spawn
if [ $(sessions_status | grep "agent:main:subagent" | wc -l) -ge 5 ]; then
  echo "⏳ 子 Agent 已滿載，等待中..."
  # 等待最早的 1 個完成
fi
```

### 優先級排程（建議）

```
高優先級（立即執行）：
- 主人明確要求的任務
- 錯誤修復、緊急分析

中優先級（排隊）：
- 研究任務
- 檔案整理

低優先級（背景執行）：
- 日誌分析
- 定期檢查
```

## 實戰 Checklist

### 主會話 Spawn 前檢查

- [ ] 任務是否超過 5 步工具呼叫？（是 → spawn）
- [ ] 任務適合哪個 Tier？（預設 Tier 1，複雜才升級）
- [ ] 當前有幾個活躍子 Agent？（<5 才 spawn）
- [ ] 任務描述是否清晰？（包含輸出要求、限制）
- [ ] 是否需要指定模型？（簡單任務用預設）

### 子 Agent 完成後檢查

- [ ] 返回訊息包含「✅ 任務完成」？
- [ ] 有執行摘要和關鍵發現？
- [ ] 輸出檔案存在？（如適用）
- [ ] 工具呼叫次數合理？（<15 次）
- [ ] 是否需要寫入記憶檔案？（重要發現 → `memory/YYYY-MM-DD.md`）

## 常見問題

### Q: 什麼時候不該用子 Agent？
**A:** 1-3 步簡單操作（如查檔案、簡單計算），直接在主會話做更快。

### Q: 子 Agent 可以再 spawn 子 Agent 嗎？
**A:** 技術上可以，但**強烈不建議**。會導致層級混亂、難以追蹤。最多 2 層（主 → 子）。

### Q: 如何選擇 Tier？
**A:** 
- **預設用 Tier 1**（Kimi K2.5）
- **失敗再升級 Tier 2**（Gemini Flash）
- **明確需要強推理才用 Tier 3+**（Sonnet/Opus）

### Q: 子 Agent 超時怎麼辦？
**A:** 10 分鐘後主動檢查，如果還在執行 → `sessions_kill` 終止，換模型重試或人工接手。
