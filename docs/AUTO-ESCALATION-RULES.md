# 自動升級規則（Auto-Escalation Rules）

> 當系統出現異常時，自動升級給 Codex 處理的判定條件與流程

---

## 一、觸發條件（任一成立即升級）

| # | 類型 | 判定條件 |
|---|------|---------|
| 1 | **服務不可用** | API 健康檢查連續 2 次失敗（間隔 10~15 秒） |
| 2 | **任務超時/卡住** | 單任務執行 > 90 秒；或 running > 120 秒無進度更新 |
| 3 | **失敗過多** | 同類任務連續失敗 ≥3 次；或 10 分鐘內 failed > 30% |
| 4 | **任務池異常** | ready=0 持續 ≥3 輪；且 created=0 持續 ≥3 輪 |
| 5 | **重複執行** | 同 taskId 短時間重複執行（duplicate > 0） |
| 6 | **系統錯誤** | DB constraint / migration / schema / TS build error |
| 7 | **成本異常** | 單小時成本 > 預算 120% |

---

## 二、升級後立即動作

1. **暫停高風險排程**（保留 health check 與核心監控）
2. **保留現場**（不清空 log、不覆蓋錯誤）
3. **送出完整升級訊息給 Codex**

---

## 三、升級訊息格式（必填 10 項）

```
[ESCALATE_Codex]
1. 問題摘要
2. 預期結果
3. 實際結果
4. 執行指令（完整）
5. 工作目錄（cwd）
6. 錯誤原文（完整）
7. 最近 80 行 log
8. 影響範圍（taskId/scheduleId/API）
9. 已嘗試處理
10. 可否回滾（方式）
```

---

## 四、解除升級條件（恢復正常）

- [ ] API 健康檢查連續 3 次成功
- [ ] running 卡住數 = 0
- [ ] failed rate 回到 < 10%
- [ ] ready 任務池恢復到 ≥ 3
- [ ] 再恢復分批上線（P0→P1→P2）
