## 📋 審核流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   我發想     │ ──→ │  待審核區    │ ──→ │  主人審核    │
│  (寫 Markdown)│     │  (pending/) │     │ (讀檔+決定)  │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                                │
                   ┌────────────────────────────┼────────────────────────────┐
                   │                            │                            │
                   ▼                            ▼                            ▼
            ┌─────────────┐             ┌─────────────┐               ┌─────────────┐
            │   ✅ 通過    │             │  ❌ 未通過   │               │  💬 需修改   │
            │ (approved/) │             │ (rejected/) │               │ (pending/)  │
            └──────┬──────┘             └─────────────┘               └──────┬──────┘
                   │                                                         │
                   ▼                                                         ▼
            變成正式任務執行                                              我修改後重送
```

## ✅ 審核方式選項

### 方式一：Telegram 按鈕（推薦）

我發送發想摘要時，附上 Inline Buttons：

```
[💡 新發想] Token 消耗優化策略

簡介：發現 5 個高 Token 情境，提出 5 個解決方案...

[查看詳情] [✅ 通過] [❌ 未通過] [💬 需修改]
```

- 點「查看詳情」→ 我讀取完整文件內容回覆
- 點「通過/未通過/需修改」→ 我更新狀態並移動文件

### 方式二：任務板審核

任務板新增「審核中」狀態，主人在 UI 上點擊審核按鈕。

### 方式三：簡單指令

主人直接回覆：
- `通過 #001` → 我執行通過流程
- `未通過 #001 原因：xxx` → 我記錄原因
- `修改 #001` → 我標記需修改

---

## 🔄 通過後的流程

1. 文件從 `pending/` 移到 `approved/`
2. 更新文件狀態為「已通過」+ 審核日期
3. 任務板任務狀態改為「ready」或「running」
4. 開始執行發想內容
