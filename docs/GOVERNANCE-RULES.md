## A. SLO 告警線（超線自動降速/升級）

### 監控指標與門檻

| 指標 | 告警線 | 嚴重線 |
|------|--------|--------|
| **Success Rate** | < 90% | < 80% |
| **Failed Rate** | > 10% | > 20% |
| **Duplicate Tasks** | > 0 | > 3 |
| **Avg Duration** | > 60s | > 120s |

### 自動動作流程

```
檢測到異常（任一指標達告警線）
    ↓
[第 1 輪] 自動降速
    ├─ maxTasksPerMinute 降一級（例：2→1）
    ├─ 記錄降速事件至 /memory/slo-alerts/
    └─ 通知 Telegram（輕度告警）
    ↓
[連續 2 輪異常] 自動升級
    ├─ 觸發 [ESCALATE_Codex]
    ├─ 暫停高風險任務（保留 health check、核心監控）
    ├─ 鎖定 maxTasksPerMinute = 0（暫停新任務）
    └─ 通知 Telegram（緊急告警）
```

### 降速級別對照表

| 當前級別 | 告警後級別 | 嚴重後級別 |
|---------|-----------|-----------|
| 5 | 3 | 0（暫停）|
| 3 | 2 | 0（暫停）|
| 2 | 1 | 0（暫停）|
| 1 | 0（暫停）| 0（暫停）|

## B. 風險執行分級（v2.0）

### 執行權限表

| 風險等級 | 執行方式 | 前置條件 | 失敗處理 |
|---------|---------|---------|---------|
| **低風險** | ✅ 自動執行 | 無 | 記錄日誌 |
| **中風險** | ✅ 自動執行 | 快照備份 | 自動回滾 |
| **高風險** | ✅ 自動執行 | 快照備份 + 冷卻檢查 | 自動回滾 + 冷卻期 |

### 共同前置條件（中/高風險）

1. **執行前必做快照**
   - 調用 `./scripts/checkpoint.sh create "任務名稱" "執行前"`
   - 檢查點 ID 寫入任務元數據

2. **失敗時自動回滾**
   - 失敗後立即調用 `./scripts/checkpoint.sh rollback`
   - 回滾後執行健康驗證（見下方）

### 新增保險機制

#### B1. 高風險冷卻期

```
任一高風險任務失敗
    ↓
暫停同類高風險任務 15 分鐘
    ↓
冷卻期結束後，先執行 1 個低風險測試任務
    ↓
測試通過 → 恢復高風險執行
測試失敗 → [ESCALATE_Codex]
```

**冷卻期設定**：**15 分鐘**（平衡安全與效率）

#### B2. 回滾後健康驗證

回滾完成後必須執行以下檢查，**全部通過**才解除告警：

| 檢查項目 | 驗證方式 | 通過標準 |
|---------|---------|---------|
| **API 可用性** | `curl /api/health` | HTTP 200，response time < 2s |
| **資料一致性** | 比對 checksum / 記錄數 | 與快照一致 |
| **關鍵流程** | 執行 smoke test | 核心功能正常 |
| **任務池狀態** | `GET /api/tasks` | ready ≥ 0，running 無卡住 |

**驗證失敗處理**：
- 任一項未通過 → 立即 `[ESCALATE_Codex]`
- 標記系統狀態為 `degraded`，暫停所有非核心任務

## C. 每週回顧機制（固定排程）

### 排程設定

```cron
# 每週一早上 9:00 執行
0 9 * * 1
```

**排程名稱**：`weekly-governance-review`

### 報告內容（Telegram 摘要版）

```
📊 小蔡週治理報告（YYYY-MM-DD）

🎯 SLO 趨勢
• Success Rate: XX%（上週 XX%）
• Failed Rate: XX%（上週 XX%）
• 異常事件：X 次

💰 成本優化
• 調整模型路由：XXX → XXX（省 $X）
• 關閉低價值排程：X 個

📈 產出優化
• 任務來源權重調整：internal-ops X% / business-model X% / external-radar X%
• 本週產出任務：X 個

⚠️ 待處理異常
• [異常 1] ...
• [異常 2] ...
```

### 自動優化動作

報告產出後，系統自動執行（低風險）：

1. **關閉低價值排程**
   - 過去 7 天執行次數 = 0 且非核心監控 → 自動禁用

2. **調整模型路由**
   - 成功率 < 95% 的模型 → 降級為 fallback
   - 成本效益比最優的模型 → 升級為預設

3. **更新任務來源權重**
   - 產出率低的來源 → 權重 -5%
   - 產出率高的來源 → 權重 +5%
