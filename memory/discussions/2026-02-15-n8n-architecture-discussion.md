# n8n 架构设计讨论记录
**日期**: 2026-02-15  
**参与者**: 老蔡, 小蔡  
**主题**: n8n 与 OpenClaw 整合架构、Token 节省策略

## 核心问题讨论

### 1. 信息流程架构
**老蔡提问**: 訊息是先給小蔡還是 n8n？

**小蔡澄清**:
- 目前: 你 → 小蔡 (直接)，n8n 未參與
- n8n 是「被動待命」，只有主動呼叫才會動作

### 2. 成本优化思路
**老蔡提出**:
```
Input 層（便宜）→ 決策層（中價）→ 執行層
你 → 小蔡 → 決定派誰 → n8n/Codex
          ↓
    小蔡是 Output，對外分配
    決定後直接指派，結果直傳給你
```

**小蔡確認**: 這個思路正確！
- 小蔡理解+決策 ($0.05)
- 指派後不參與後續 ($0 節省)
- 執行者直接回報 (省掉轉述層)

### 3. n8n 回复成本归属
**老蔡确认**: 走 n8n 路线，是小蔡回复还是 n8n 回复？

**结论**:
- n8n 只是「收件箱」，不能直接回 Telegram（除非额外配置）
- 目前架构: 结果存文件 → 小蔡读取 → 回复你
- 要让 n8n 直发，需要给 n8n Telegram Bot Token

### 4. n8n OpenAI 路径检查
**发现**:
- OPENAI_API_KEY= (空的，未配置)
- TELEGRAM_BOT_TOKEN= (空的，未配置)
- 目前 n8n 是「干净」实例，无高成本 API 连接

### 5. 记忆快搜可行性
**可行方案**:
- n8n → Qdrant (已有，port 6333)
- n8n → PostgreSQL pgvector (已有)
- n8n → OpenClaw API (http://localhost:3011)
