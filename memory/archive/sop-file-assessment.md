# 小蔡檔案/任務評估 SOP

> 建立日期：2026-02-14  
> 用途：標準化檔案評估、檢查、分析流程  
> 適用：接到任務後的「第一步」標準動作

---

## 🔄 評估流程總覽

```
收到任務 → 初步評估 → 深度檢查 → 分析診斷 → 輸出報告 → 執行決策
    ↓         ↓          ↓          ↓          ↓          ↓
  理解需求   5W2H      檔案掃描    根因分析    評估報告    Go/No-Go
```

---

## Step 1：初步評估（5W2H）

接到任務後，先問這 7 個問題：

| 問題 | 目的 | 範例 |
|------|------|------|
| **What** 做什麼？ | 明確任務內容 | 「修復登入 bug」vs 「重構整個認證系統」 |
| **Why** 為什麼？ | 理解動機和價值 | 是緊急修復還是技術債清理？ |
| **Who** 誰相關？ |  stakeholders | 影響哪些用戶？需要誰確認？ |
| **When** 何時要？ | 時間壓力 | 今天 vs 本週 vs 下個月 |
| **Where** 在哪？ | 範圍定位 | 哪個專案？哪個模組？ |
| **How** 怎麼做？ | 技術方案 | 已有想法還是要我設計？ |
| **How much** 多少成本？ | 資源評估 | 需要多少時間？多少 token？ |

---

## Step 2：深度檢查（檔案掃描）

### 2.1 檔案存在性檢查
```bash
# 基本檢查
ls -la <file_path>          # 檔案是否存在
file <file_path>            # 檔案類型
checksum <file_path>        # 檔案完整性（如需要）
```

### 2.2 程式碼品質快速掃描
```bash
# 如果是程式專案
find . -name "*.ts" -o -name "*.js" | head -20    # 檔案結構
grep -r "TODO\|FIXME\|XXX" .                       # 技術債標記
wc -l <main_file>                                  # 程式碼行數
```

### 2.3 相依性檢查
- 檔案引用了哪些其他檔案？
- 是否有外部依賴（API、資料庫、套件）？
- 修改這個檔案會影響哪些其他檔案？

---

## Step 3：分析診斷（根因分析）

### 3.1 問題分類

| 類型 | 特徵 | 處理方式 |
|------|------|----------|
| 🔴 **阻塞型** | 系統無法運作 | 立即處理，P0 |
| 🟡 **功能型** | 功能異常但可用 | 排程處理，P1 |
| 🟢 **優化型** | 效能/體驗改進 | 規劃處理，P2 |
| ⚪ **債務型** | 技術債、重構 | 排入迭代，P3 |

### 3.2 風險評估矩陣

```
影響範圍
    高 │  謹慎處理   高風險
       │    🟡        🔴
       │
    低 │  快速處理   低風險
       │    🟢        🟢
       └───────────────────
         低        高   修改複雜度
```

### 3.3 常見檢查項目

- [ ] **語法檢查**：有無明顯語法錯誤？
- [ ] **邏輯檢查**：程式邏輯是否合理？
- [ ] **安全檢查**：有無安全漏洞（SQL injection、XSS 等）？
- [ ] **效能檢查**：有無明顯效能問題？
- [ ] **測試檢查**：有無對應測試？測試是否通過？
- [ ] **文件檢查**：有無註解/文件？是否過期？

---

## Step 4：輸出評估報告

### 4.1 報告格式

```markdown
【評估報告】任務名稱

## 📊 執行摘要
- 任務類型：修復 / 開發 / 優化 / 重構
- 優先級：P0 / P1 / P2 / P3
- 預估工時：X 小時
- 預估成本：$X（token 費用）

## 🔍 檢查結果
- 檔案狀態：✅ 正常 / ⚠️ 有問題 / ❌ 嚴重問題
- 問題數量：X 個
- 主要問題：
  1. ...
  2. ...

## ⚠️ 風險評估
- 修改風險：高 / 中 / 低
- 回滾難度：高 / 中 / 低
- 影響範圍：X 個檔案 / X 個功能

## 💡 建議方案
### 方案 A：快速修復
- 做法：...
- 優點：快
- 缺點：治標不治本

### 方案 B：完整修復
- 做法：...
- 優點：治本
- 缺點：時間長

## ✅ 執行建議
- [ ] 建議執行方案：A / B
- [ ] 需要老蔡確認：是 / 否
- [ ] 建議指派給：Codex / Cursor / Ollama
```

### 4.2 評估等級

| 等級 | 說明 | 後續動作 |
|------|------|----------|
| 🟢 **可直接執行** | 需求清楚、風險低 | 直接 spawn Agent 執行 |
| 🟡 **需補充資訊** | 部分資訊不足 | 詢問老蔡，補齊後執行 |
| 🔴 **需重新評估** | 需求模糊或風險高 | 深入討論後再決定 |

### 5.1 決策流程圖

        評估完成

    ↓           ↓

## 📝 實際範例

### 範例：「網站跑很慢，看一下」

#### Step 1：初步評估
- What：網站效能問題
- Why：用戶抱怨載入慢
- Who：所有用戶
- When：今天
- Where：生產環境
- How：需要診斷
- How much：先評估再報價

#### Step 2：深度檢查
```bash
# 檢查項目
- 網站 URL 是什麼？
- 伺服器狀態？
- 資料庫查詢慢？
- 靜態資源大？
- 第三方服務慢？
```

#### Step 3：分析診斷
假設發現：
- 首頁圖片 5MB
- 沒有用 lazy loading
- 資料庫查詢 N+1 問題

#### Step 4：輸出報告
```markdown
【評估報告】網站效能優化

## 📊 執行摘要
- 任務類型：優化
- 優先級：P1（影響用戶體驗）
- 預估工時：4-6 小時
- 預估成本：$2-3（Ollama 可免費做部分）

## 🔍 檢查結果
- 主要問題：
  1. 圖片過大（5MB）
  2. 無 lazy loading
  3. N+1 查詢問題

## ⚠️ 風險評估
- 修改風險：低
- 回滾難度：低
- 影響範圍：首頁、商品列表頁

## 💡 建議方案
### 方案 A：快速優化（2 小時）
- 壓縮圖片
- 加 lazy loading
- 緩解 70% 問題

### 方案 B：完整優化（6 小時）
- 加上方案 A
- 修復 N+1 查詢
- 加 CDN

## ✅ 執行建議
- 建議執行方案：A（先解決燃眉之急）
- 需要老蔡確認：是（選 A 或 B）
```

#### Step 5：決策
等待老蔡選 A 或 B，然後指派 Agent 執行。
