## biz_realestate春酒專案 - 部署失敗教訓（重要）

### 問題概述
協助修復「biz_realestate鑽石春酒」報名系統時發生嚴重流程錯誤：
1. **Schema 錯誤**：程式碼查詢 `huadrink` schema，但資料在 `public` schema
2. **付款狀態遺失**：CSV 備份顯示 52 筆已付款，但資料庫全部顯示未付款
3. **未測試就部署**：直接部署導致後台無法登入

### 根本原因
- **缺乏部署前檢查清單**
- **未執行本地測試**（`npm run dev`）
- **無回滾計畫**（Rollback Plan）
- **Schema 變更未驗證**

### 解決方案
- 已建立完整 SOP：`PROJECT-DEPLOYMENT-SOP.md`
- 關鍵原則：**「先測試 → 再部署 → 驗證後回報」**

### 學習重點
| 錯誤 | 改進 |
|------|------|
| 直接修改程式碼 | 先確認 schema 和資料庫狀態 |
| 沒測試就部署 | 本地完整測試後才部署 |
| 部署後不回報驗證 | 線上逐一驗證後才回報 |
| 沒有回滾計畫 | 部署前準備回滾方案 |

#### Workflow 工具選擇（對 OpenClaw 的啟示）
| 工具 | 最佳場景 | 市場地位 |
|------|----------|----------|
| **Temporal** | AI Agent 編排 | 狀態持久化、業界最佳 UI |
| **Dagster** | LLM/ML 管道 | 資產追蹤、數據血緣 |
| **Airflow** | 批處理 ETL | 2025 仍是主導力量 |

**啟示**: Temporal 模式最適合 Agent 編排，可考慮增加 checkpoint 機制。

#### LLM 成本優化（可立即套用）
| 策略 | 效果 | 實施難度 |
|------|------|----------|
| **Anthropic prefix caching** | 90% 成本降低 | 低 |
| **OpenAI automatic caching** | 50% 成本降低 | 低 |
| **Prompt 優化** | 70% token 節省 | 低 |
| **模型路由** | 額外 20-30% 節省 | 中 |

**輸出 token 成本是輸入的 2-5 倍** — 限制輸出長度對成本影響巨大。
