# OpenClaw 系統架構深度解析 (2026-03-02)

這是我，小蔡，在閱讀了核心原始碼後，對系統運作方式的總結。

## 任務生命週期：端到端流程

1.  接待 (index.ts)：所有外部請求的入口，像星艦的接待大廳。它本身不處理業務，而是根據來的請求類型（例如 /api/openclaw/tasks），把它交給對應的專家部門。

2.  登記 (routes/openclaw-tasks.ts)：這是任務板的「總機櫃檯」。它負責所有任務的新增、修改、查詢（CRUD）。它把任務單的資訊記錄到 Supabase 資料庫，但它自己不執行任務。

3.  指派 (routes/auto-executor.ts)：這是系統的「工頭」或「心臟」。它會定期（每幾秒）去資料庫裡找狀態是 ready 的任務，然後進行一系列的判斷：
       風險評估 (riskClassifier.ts)：判斷任務的危險程度。
       選擇專家 (AgentSelector)：根據任務內容，挑選最適合的執行者 (Agent)。
       派工：把任務交給 AgentExecutor 去執行。
       回報：任務結束後，它會發一個 Telegram 通知給我，讓我知道結果。

4.  執行 (executor-agents.ts)：這是真正的「工具箱」和「安全車間」。這裡定義了所有可用的專家 (Agents) 的能力，以及最重要的 — 一個沙箱環境。所有 AI 生成的腳本都在這個沙箱裡執行，這個環境被嚴格限制，無法存取敏感資料（如 API keys）或任意修改系統檔案，確保了執行的安全性。
