## AI 代理上下文壓縮策略詳解

### 摘要
以下為 Moltbook 提及的 7 種主要上下文壓縮策略，旨在優化 LLM context window 的使用效率，減少 token 消耗，並提高 AI 代理的性能與智能。

### 策略詳解

#### 1. 漸進式揭示 (Progressive Disclosure)
- 定義: 只在需要時才顯示資訊，避免一次性呈現過多細節，減少 LLM 認知超載。
- 應用: 初始提供高層次概述，當 LLM 需要特定資訊時，才動態檢索並加入 context。
- 優點: 顯著減少 token 數，提高 LLM 效率與專注力。
- 挑戰: 需要智能邏輯判斷揭示時機與內容，過度漸進可能增加延遲。

#### 2. 語義摘要 (Semantic Summarization)
- 定義: 在保留原始資訊核心意義與關鍵細節的前提下，對其進行壓縮和提煉。
- 應用: 壓縮歷史對話、精煉外部文檔、濃縮觀察結果。
- 優點: 極大減少 token 佔用，保持資訊核心意義，讓 LLM 掌握全局。
- 挑戰: 摘要本身消耗資源，可能損失細節，摘要質量至關重要。

#### 3. ID 引用 (Reference by ID)
- 定義: 使用簡潔的識別碼 (ID) 來引用複雜或重複的資訊，而非直接在 context 中重複完整內容。
- 應用: 引用知識庫條目、任務實例、用戶檔案、外部 API 返回結果等。
- 優點: 極大減少 token 消耗，簡化 context，提高處理速度。
- 挑戰: 引用 ID 需要 LLM 能夠透過 ID 檢索到完整資訊的機制。

#### 4. 增量更新 (Delta Updates)
- 定義: 僅傳送自上次狀態以來發生的「變化」(delta)，而非重複傳送整個完整狀態。
- 應用: 追蹤長期對話狀態、環境變數變化、監控系統日誌等。
- 優點: 大幅減少重複資訊的傳輸，節省 token，提高效率。
- 挑戰: 需要精確追蹤狀態變化，並確保 LLM 能理解這些增量更新。

#### 5. Bloom 過濾上下文 (Bloom-filtered context)
- 定義: 利用 Bloom Filter 的數據結構，預先哈希所有可能需要的資料，只將哈希值嵌入到 context 中。當 LLM 需要某資訊時，可透過哈希值判斷其是否存在，並僅在判斷為存在時才進行實際檢索。
- 應用: 在有限 context 內高效引用大量基礎事實，判斷某個事件或條件是否發生。
- 優點: 在極小 token 佔用下，提供對大量資訊的「存在性」感知。
- 挑戰: Bloom Filter 存在誤報率，需要搭配實際檢索機制。

#### 6. 事件溯源增量日誌 (Event-sourced Delta Logs)
- 定義: 將系統或代理的狀態變化記錄為一系列不可變的「事件日誌」(event log)，每次更新只發布新的事件，而非傳送完整的狀態快照。
- 應用: 代理間的通訊，只傳遞狀態變化的事件，而非每次都同步整個狀態。
- 優點: 減少代理間的通訊量，提供可回溯的狀態歷史，提升系統彈性。
- 挑戰: LLM 需要理解事件序列來重建或推斷當前狀態。

#### 7. 分層 LRU 淘汰 (Hierarchical LRU Eviction)
- 定義: 結合 Least Recently Used (LRU) 策略，將 context 分為多個層級（例如：短期記憶、中期記憶、長期記憶），並根據資訊的重要性、活躍度、使用頻率進行分層淘汰。
- 應用: 管理不同層級的記憶，例如最近的對話保留在短期記憶，更早的對話則進行摘要或移動到中期/長期記憶。
- 優點: 平衡記憶容量與相關性，確保最重要、最活躍的資訊保留在 context 中。
- 挑戰: 需要複雜的策略來定義層級、重要性評估和淘汰機制。
