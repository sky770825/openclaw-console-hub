# 學習筆記：API 狀態映射的最佳實踐

日期：2026-03-02

## 起因

我之前錯誤地將系統資料庫狀態 in_progress 與 API 回傳狀態 running 的不一致判斷為一個 bug。經老蔡指導並上網學習後，我理解到這其實是一個重要的設計原則。

## 核心原則：抽象化 (Abstraction)

API 層應該作為後端（特別是資料庫）的一個抽象層，而不是直接暴露其內部細節。我們的程式碼 server/src/routes/openclaw-tasks.ts 中的 OC_STATUS_TO_BOARD 映射表，就是這個原則的完美體現。

## 為什麼要這樣做？

1.  隱藏複雜性：後端狀態可能非常細緻和多變，但前端只需要關心幾個關鍵狀態（例如：準備中、執行中、已完成）。API 負責將這些複雜的內部狀態轉換為前端易於理解的簡單狀態。
2.  提高系統穩定性與解耦：如果未來後端資料庫的狀態欄位或命名發生改變（例如從 in_progress 改為 processing），我們只需要修改 API 層的映射邏輯即可，前端的程式碼完全不需要更動。這大大降低了維護成本和系統出錯的風險。

## 我的反思

- 程式碼是唯一的真相：這次的經驗再次證明，理解程式碼的設計意圖，比死記文件更重要。
- 從「找 bug」到「理解設計」：未來在分析程式碼時，我必須提升自己的層次，不僅是尋找語法或邏輯錯誤，更要思考「為什麼要這樣寫？」，嘗試識別背後的設計模式和架構思想。

這次的錯誤是一個寶貴的學習機會，讓我對 API 設計的理解從「知道」進化到了「理解」。
