# 2025 API 設計最佳實踐

> 來源：Gemini Flash 2.5 搜尋整理
> 日期：2026-02-16
> 適用：後端開發、API 設計

## 1. RESTful API 設計原則

- **資源導向**：明確定義資源，使用名詞 (/users, /products)
- **標準 HTTP 方法**：遵循 GET, POST, PUT, DELETE, PATCH 語義
- **無狀態**：每個請求包含足夠資訊，伺服器無需儲存上下文
- **可快取**：適當使用 HTTP 快取機制
- **統一介面**：URI 唯一識別資源，響應自描述

## 2. GraphQL vs REST 選擇

**GraphQL 適用場景：**
- 客戶端需要高度靈活的數據請求
- 數據庫 schema 複雜，需要按需查詢
- 行動應用程式，網路頻寬有限

**REST 適用場景：**
- 簡單的資源操作、CRUD
- 容易理解和實現，生態系統成熟
- 快取友善，可利用 HTTP 快取

**選擇考量**：項目複雜度、團隊熟悉度、性能需求

## 3. API 版本控制策略

| 策略 | 範例 |
|------|------|
| URI 版本控制 | /v1/users, /v2/users |
| Header 版本控制 | X-API-Version: 2 |
| MediaType 版本控制 | Accept: application/vnd.example.v2+json |

**淘汰策略：**
- 明確聲明舊版本淘汰時間
- 提供遷移指南
- 同時支持多個版本

## 4. 安全性與認證機制

**身份驗證：**
- OAuth 2.0：業界標準，適用於第三方應用授權
- JWT：輕量級，適用於單點登錄
- API 金鑰：簡單，適用於內部服務

**授權：**
- RBAC：基於角色的訪問控制
- ABAC：基於屬性的訪問控制

**安全措施：**
- 使用 HTTPS 加密數據傳輸
- 輸入驗證防止 SQL 注入、XSS
- 速率限制防止濫用和 DoS 攻擊
- API Gateway 統一管理安全、認證、限流
