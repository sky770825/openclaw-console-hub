# RAG 戰術內部工作流程 (Ollama + 串流 + 異常處理)

## 前言
這是一份為 OpenClaw 平台量身打造的 RAG (Retrieval Augmented Generation) 戰術內部工作流程指南，旨在高效結合 semantic_search 和 run_script 能力與本地 Ollama 模型。本指南詳細闡述了從接收用戶查詢到處理模型回應，乃至於全面的異常處理策略。

## 核心工作流程步驟

### 1. 接收使用者查詢 (Query)
• 系統接收來自用戶的原始問題或指令。
• 這是整個 RAG 流程的起點。

### 2. 查詢嵌入 (Query Embedding)
• 使用一個嵌入模型（例如本地的 Ollama 嵌入模型或專門的嵌入服務）將使用者查詢轉換為向量表示。
• 確保嵌入模型與知識庫的向量資料庫使用的模型一致，以保證向量空間的兼容性。

### 3. 向量相似性搜尋 (Vector Similarity Search)
• 使用查詢向量在向量資料庫中執行相似性搜尋。
• 目標是找出與用戶查詢語義最相關的 K 個文件區塊 (chunks)。

### 4. 檢索相關上下文 (Context Retrieval)
• 從向量資料庫檢索到的文件區塊中提取原始文本內容。
• 這些文本將作為 LLM 的上下文資訊。

### 5. 建構最終提示詞 (Prompt Engineering)
• 將使用者原始查詢與檢索到的上下文資訊整合成一個結構化的最終提示詞 (final_prompt)。
• 提示詞設計應清晰指示 LLM 根據提供的上下文來回答問題，並處理未找到相關資訊的情況。

### 6. 調用 Ollama LLM 並處理串流輸出 (Streaming)

直接等待 LLM 生成完整答案可能會花費很長時間，尤其是在答案很長的情況下，這會導致使用者體驗不佳（感覺像應用程式卡住了）。*串流輸出 (Streaming)* 讓 LLM 每生成一小部分內容（例如一個 token 或一個詞），就立刻將其傳回給前端，實現類似 ChatGPT 的即時打字效果。

這是透過在 API 調用中設定 stream=True 參數來實現的。

#### *內部工作流程詳解：*

1.  *發起串流請求：*
    當你使用 Ollama 的 Python 客戶端 ollama.chat 或 ollama.generate 時，將 stream 參數設為 True。

    ``python
    import ollama

    # final_prompt 來自上一步
    # model_name 是你希望使用的模型，例如 'llama3:8b' 或 'mistral'
    
    # 我們將使用 ollama.chat 因為它更適合對話格式
    response_stream = ollama.chat(
        model=model_name,
        messages=[
            {'role': 'user', 'content': final_prompt}
        ],
        stream=True  # 關鍵參數！
    )
    `

2.  *迭代處理回應區塊 (Chunks)：*
    設定 stream=True 後，API 的返回值不再是單一的 JSON 物件，而是一個*迭代器 (iterator)* 或*生成器 (generator)*。你需要遍歷這個迭代器來接收一系列的資料區塊 (chunks)。

    每個 chunk 都是一個小的 JSON 物件，包含了當前生成的部分內容。

    `python
    full_response = ""
    print("AI:", end="", flush=True) # 準備在同一行上持續輸出

    for chunk in response_stream:
        # 提取每個 chunk 中的內容
        content = chunk['message']['content']
        
        # 1. 即時顯示給使用者
        print(content, end="", flush=True)
        
        # 2. 累加內容以獲得完整回應
        full_response += content

    print() # 輸出完成後換行
    `

3.  *理解 Chunk 的結構：*
    一個典型的 chunk 看起來像這樣：
    
    {
      "model": "llama3:8b",
      "created_at": "2024-05-21T10:30:00.123Z",
      "message": {
        "role": "assistant",
        "content": "根據" // 可能是「根據您提供的資料...」的第一部分
      },
      "done": false // 這表示還有更多內容即將到來
    }
    `
    當 "done": true 時，表示這是回應的最後一個區塊。你可以利用這個標誌來做一些收尾工作，例如關閉串流連接或觸發最終的 UI 更新。

### 7. 異常處理 (Error Handling)

### *目標*

此階段的核心目標是確保應用程式在遇到非預期情況時，不會直接崩潰，而是能夠優雅地處理錯誤，並向使用者或開發者提供有意義的回饋。一個健壯的 RAG 系統必須能應對從資料庫連接到 LLM API 調用的各種潛在失敗點。

### *常見的異常情境與處理策略*

在 RAG 流程中，主要有以下幾個環節可能會出錯：

1.  *向量資料庫操作失敗 (Vector Database Errors)*
    *   *情境*:
        *   無法連接到向量資料庫（例如，網路問題、資料庫服務未啟動）。
        *   執行檢索（Search/Query）時發生內部錯誤。
        *   檢索到的文件為空或格式不符預期。
    *   *處理策略*:
        *   *連接失敗*: 使用 try-except 區塊包裹資料庫連接和查詢的程式碼。若捕捉到連接異常，應記錄詳細錯誤日誌，並向使用者顯示一條通用的錯誤訊息，例如「暫時無法訪問知識庫，請稍後再試」。可以實作重試機制（Retry Logic），在短時間內嘗試重新連接數次。
        *   *檢索無結果*: 如果查詢沒有返回任何相關文件，系統不應將空的上下文傳遞給 LLM。此時應有備用策略：
            *   *策略一（告知使用者）*: 直接回覆使用者「在我們的資料庫中找不到與您問題相關的資訊」。
            *   *策略二（無 RAG 生成）*: 不使用檢索到的上下文，僅將原始問題直接發送給 LLM 進行回答，並可能標註此答案未經內部資料驗證。

2.  *大型語言模型 (LLM) API 調用失敗 (LLM API Errors)*
    *   *情境*:
        *   無法連接到 LLM 服務（例如，Ollama 服務未在本機運行，或網路不通）。
        *   API 返回錯誤碼（如 4xx 用戶端錯誤、5xx 伺服器端錯誤）。
        *   請求超時（Timeout）。
        *   模型生成的回應為空或格式錯誤。
    *   *處理策略*:
        *   *連接與 API 錯誤*: 同樣使用 try-except 區塊來捕獲 API 請求過程中的異常（如 requests.exceptions.ConnectionError）。記錄錯誤後，向使用者顯示「無法連接至語言模型，請稍後再試」。
        *   *超時*: 在 API 調用中設定合理的超時時間。如果發生超時，可以選擇重試或直接向使用者回報錯誤。
        *   *空回應*: 如果 LLM 返回空內容，應將其視為一次失敗的生成，並告知使用者「模型未能生成有效回覆，請嘗試更換提問方式」。

3.  *資料處理與串流錯誤 (Data Processing & Streaming Errors)*
    *   *情境*:
        *   在處理從 LLM 返回的串流（Stream）資料時，資料塊（Chunk）的格式可能不正確，導致解析 JSON 或其他格式時出錯。
        *   串流在傳輸過程中意外中斷。
    *   *處理策略*:
        *   *解析錯誤*: 在處理每個資料塊時，都應加上錯誤處理。如果某個資料塊解析失敗，可以選擇忽略它（如果對整體影響不大）或終止整個串流並回報錯誤。
        *   *串流中斷*: 監控串流的結束標記。如果串流在未收到結束標記前就中斷了，應清理資源並通知前端，讓使用者知道回應可能不完整。

### *實作層面的建議*

*   *全面的 try-except 結構*: 將整個 RAG 的核心邏輯（從接收請求、檢索、生成提示、調用 LLM 到返回結果）都包裹在一個大的 try-except` 區塊中，以捕捉任何未被預料到的錯誤。
*   *使用者友善的錯誤訊息*: 避免將原始的技術性錯誤堆疊（Stack Trace）直接暴露給終端使用者。應將其轉換為簡潔、易懂的訊息。
*   *日誌記錄 (Logging)*: 詳細記錄所有錯誤，包括時間戳、錯誤類型、相關輸入、堆疊追蹤等，以便於後續的除錯和系統監控。
*   *監控與告警*: 結合系統監控工具，對 RAG 流程中的關鍵指標（如錯誤率、響應時間）設定告警，以便及時發現和處理問題。
*   *重試機制 (Retry Mechanisms)*: 對於暫時性的錯誤（如網路波動、服務器瞬時過載），實作帶有指數退避 (Exponential Backoff) 的重試機制，可以顯著提高系統的穩定性。
