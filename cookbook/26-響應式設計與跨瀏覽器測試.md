## 1. 響應式設計原則

### 1.1 Mobile First vs Desktop First

| 比較 | Mobile First | Desktop First |
|------|-------------|---------------|
| 起點 | 從最小螢幕開始設計 | 從桌面版開始設計 |
| Media Query | `min-width`（向上擴展） | `max-width`（向下壓縮） |
| 效能 | 手機只載入必要 CSS | 手機載入全部再覆蓋 |
| 適合場景 | 新專案、行動優先產品 | 改版既有桌面網站 |
| 業界趨勢 | **主流推薦** | 漸漸淡出 |

**結論：新專案一律 Mobile First。**

原因很實際：先在小螢幕上把核心內容做好，再往大螢幕擴展，比從大螢幕往下塞容易十倍。客戶驗收時手機體驗也是第一個被挑的。

```css
/* Mobile First：預設 = 手機，往上加 */
.card {
  padding: 1rem;
  font-size: 0.875rem;
}

@media (min-width: 768px) {
  .card {
    padding: 2rem;
    font-size: 1rem;
  }
}

@media (min-width: 1024px) {
  .card {
    padding: 3rem;
    font-size: 1.125rem;
  }
}
```

```css
/* Desktop First：預設 = 桌面，往下減（不推薦新專案） */
.card {
  padding: 3rem;
  font-size: 1.125rem;
}

@media (max-width: 1023px) {
  .card {
    padding: 2rem;
    font-size: 1rem;
  }
}

@media (max-width: 767px) {
  .card {
    padding: 1rem;
    font-size: 0.875rem;
  }
}
```

### 1.3 CSS Media Query 實戰範例

#### 範例一：三欄 → 兩欄 → 一欄佈局

```css
.grid-layout {
  display: grid;
  gap: 1rem;
  /* 手機：一欄 */
  grid-template-columns: 1fr;
}

@media (min-width: 768px) {
  .grid-layout {
    /* 平板：兩欄 */
    grid-template-columns: repeat(2, 1fr);
  }
}

@media (min-width: 1024px) {
  .grid-layout {
    /* 桌面：三欄 */
    grid-template-columns: repeat(3, 1fr);
  }
}
```

#### 範例二：側邊欄佈局

```css
.page-layout {
  display: grid;
  grid-template-columns: 1fr;
}

.sidebar {
  order: 2; /* 手機上側邊欄放下面 */
}

.main-content {
  order: 1;
}

@media (min-width: 1024px) {
  .page-layout {
    grid-template-columns: 280px 1fr;
  }

  .sidebar {
    order: unset; /* 桌面恢復正常順序 */
  }

  .main-content {
    order: unset;
  }
}
```

#### 範例三：根據方向（Orientation）

```css
/* 手機橫向時，減少上下 padding */
@media (max-height: 500px) and (orientation: landscape) {
  .hero {
    padding: 1rem 2rem;
    min-height: auto;
  }
}
```

#### 範例四：深色模式偏好

```css
/* 系統深色模式 */
@media (prefers-color-scheme: dark) {
  :root {
    --bg-color: #1a1a2e;
    --text-color: #e0e0e0;
    --card-bg: #16213e;
  }
}

/* 系統淺色模式（預設）*/
@media (prefers-color-scheme: light) {
  :root {
    --bg-color: #ffffff;
    --text-color: #333333;
    --card-bg: #f5f5f5;
  }
}
```

#### 範例五：減少動態效果偏好

```css
/* 使用者系統設定「減少動態效果」*/
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

#### 範例六：高解析度螢幕（Retina）

```css
/* 2x 像素密度以上的螢幕 */
@media (-webkit-min-device-pixel-ratio: 2),
       (min-resolution: 192dpi) {
  .logo {
    background-image: url('logo@2x.png');
    background-size: 200px 60px;
  }
}
```

### 1.4 Container Queries（新標準）

Container Queries 是 2023 年後主流瀏覽器都支援的新標準。和 Media Query 不同的是：**Media Query 看的是整個視窗的寬度，Container Query 看的是父容器的寬度**。

這解決了元件化開發的大痛點：同一個 Card 元件放在側邊欄和主內容區，應該要有不同的呈現方式，但 Media Query 做不到（因為視窗寬度一樣）。

```css
/* 第一步：把父容器標記為 container */
.card-wrapper {
  container-type: inline-size;
  container-name: card-container; /* 可選，方便指名 */
}

/* 第二步：用 @container 寫子元素的響應規則 */
.card {
  display: flex;
  flex-direction: column;
  padding: 1rem;
}

.card__image {
  width: 100%;
  aspect-ratio: 16 / 9;
  object-fit: cover;
}

.card__title {
  font-size: 1rem;
}

/* 當「父容器」寬度 >= 400px */
@container card-container (min-width: 400px) {
  .card {
    flex-direction: row;
    gap: 1.5rem;
  }

  .card__image {
    width: 200px;
    aspect-ratio: 1;
  }

  .card__title {
    font-size: 1.25rem;
  }
}

/* 當「父容器」寬度 >= 700px */
@container card-container (min-width: 700px) {
  .card {
    padding: 2rem;
  }

  .card__title {
    font-size: 1.5rem;
  }
}
```

**實際應用場景：**

```html
<!-- 同一個 Card 元件，放在不同寬度的容器中 -->
<div class="main-content">
  <div class="card-wrapper">
    <!-- 這裡的 card-wrapper 寬度大，card 會用橫向佈局 -->
    <div class="card">...</div>
  </div>
</div>

<aside class="sidebar">
  <div class="card-wrapper">
    <!-- 這裡的 card-wrapper 寬度小，card 會用直向佈局 -->
    <div class="card">...</div>
  </div>
</aside>
```

**瀏覽器支援（2026 年）：** Chrome 105+、Safari 16+、Firefox 110+、Edge 105+。基本上可以放心用了。

## 2. 常見 RWD 問題與解法

### 2.1 圖片跑版

**問題：** 圖片在不同螢幕寬度下溢出容器、比例變形、留白太多。

**解法一：object-fit（最常用）**

```css
/* 圖片填滿容器，裁切多餘部分 */
.card-image {
  width: 100%;
  height: 200px;
  object-fit: cover;       /* 填滿並裁切 */
  object-position: center; /* 裁切時以中心為準 */
}

/* object-fit 的所有值 */
.img-cover   { object-fit: cover; }    /* 填滿裁切（最常用）*/
.img-contain { object-fit: contain; }  /* 完整顯示，可能留白 */
.img-fill    { object-fit: fill; }     /* 強制填滿，會變形 */
.img-none    { object-fit: none; }     /* 原始尺寸 */
```

**解法二：aspect-ratio（固定比例）**

```css
/* 不管容器多寬，圖片永遠維持 16:9 */
.card-image {
  width: 100%;
  aspect-ratio: 16 / 9;
  object-fit: cover;
}

/* 正方形頭像 */
.avatar {
  width: 80px;
  aspect-ratio: 1;          /* 等同 1 / 1 */
  border-radius: 50%;
  object-fit: cover;
}

/* 直式海報 */
.poster {
  width: 100%;
  aspect-ratio: 2 / 3;
  object-fit: cover;
}
```

**解法三：`<picture>` + srcset（不同裝置載入不同圖片）**

```html
<picture>
  <!-- 桌面：大圖 -->
  <source media="(min-width: 1024px)" srcset="hero-large.webp" type="image/webp">
  <source media="(min-width: 1024px)" srcset="hero-large.jpg">

  <!-- 平板：中圖 -->
  <source media="(min-width: 768px)" srcset="hero-medium.webp" type="image/webp">
  <source media="(min-width: 768px)" srcset="hero-medium.jpg">

  <!-- 手機：小圖（預設）-->
  <img src="hero-small.jpg"
       alt="首頁主視覺"
       loading="lazy"
       decoding="async"
       width="800"
       height="450">
</picture>
```

**解法四：背景圖片響應式**

```css
.hero {
  background-image: url('hero-mobile.jpg');
  background-size: cover;
  background-position: center;
  min-height: 50vh;
}

@media (min-width: 768px) {
  .hero {
    background-image: url('hero-tablet.jpg');
    min-height: 60vh;
  }
}

@media (min-width: 1024px) {
  .hero {
    background-image: url('hero-desktop.jpg');
    min-height: 80vh;
  }
}
```

### 2.2 表格在手機上怎麼處理

表格是 RWD 的頭號難題。橫向資料一多，手機上根本看不完。

**方法一：水平捲動（最簡單）**

```css
.table-wrapper {
  overflow-x: auto;
  -webkit-overflow-scrolling: touch; /* iOS 平滑捲動 */
}

/* 加個漸層提示使用者可以左右滑 */
.table-wrapper {
  position: relative;
}

.table-wrapper::after {
  content: '';
  position: absolute;
  right: 0;
  top: 0;
  bottom: 0;
  width: 30px;
  background: linear-gradient(to left, rgba(255,255,255,0.9), transparent);
  pointer-events: none;
}
```

**方法二：卡片式轉換（推薦小資料量）**

```css
/* 桌面：正常表格 */
.responsive-table thead {
  display: table-header-group;
}

/* 手機：每行變成一張卡片 */
@media (max-width: 767px) {
  .responsive-table,
  .responsive-table thead,
  .responsive-table tbody,
  .responsive-table tr,
  .responsive-table td {
    display: block;
  }

  .responsive-table thead {
    display: none; /* 隱藏表頭 */
  }

  .responsive-table tr {
    margin-bottom: 1rem;
    padding: 1rem;
    border: 1px solid #e2e8f0;
    border-radius: 8px;
    background: #fff;
  }

  .responsive-table td {
    padding: 0.5rem 0;
    display: flex;
    justify-content: space-between;
    border-bottom: 1px solid #f0f0f0;
  }

  .responsive-table td:last-child {
    border-bottom: none;
  }

  /* 用 data-label 顯示欄位名稱 */
  .responsive-table td::before {
    content: attr(data-label);
    font-weight: 600;
    color: #666;
    flex-shrink: 0;
    margin-right: 1rem;
  }
}
```


### 2.4 字體大小 clamp() 用法

`clamp()` 讓字體大小根據視窗寬度自動流動，不需要一堆 media query。

**語法：`clamp(最小值, 偏好值, 最大值)`**

```css
/* 標題：最小 1.5rem，最大 3.5rem，中間根據視窗寬度流動 */
h1 {
  font-size: clamp(1.5rem, 4vw + 0.5rem, 3.5rem);
}

/* 副標題 */
h2 {
  font-size: clamp(1.25rem, 3vw + 0.25rem, 2.5rem);
}

/* 內文 */
body {
  font-size: clamp(0.875rem, 1.5vw + 0.25rem, 1.125rem);
}

/* 實用公式：
   偏好值 = (最大值 - 最小值) / (最大視窗 - 最小視窗) * 100vw + 補正
   但通常不用算，直接用以下經驗值：
*/
```

**常用 clamp 速查表：**

```css
/* 元素        最小      偏好             最大     */
/* 大標題 */   clamp(2rem,    5vw + 0.5rem,   4rem);
/* 中標題 */   clamp(1.5rem,  3vw + 0.5rem,   2.5rem);
/* 小標題 */   clamp(1.125rem, 2vw + 0.5rem,  1.5rem);
/* 內文   */   clamp(0.875rem, 1.2vw + 0.5rem, 1.125rem);
/* 小字   */   clamp(0.75rem,  1vw + 0.25rem,  0.875rem);
```

**不只字體，間距也可以用 clamp：**

```css
.section {
  /* 手機 2rem，桌面 6rem，中間流動 */
  padding: clamp(2rem, 5vw, 6rem) clamp(1rem, 3vw, 4rem);
}

.container {
  max-width: clamp(320px, 90vw, 1200px);
  margin: 0 auto;
}
```

### 2.5 vh 在手機上的問題（dvh / svh / lvh）

**問題：** `100vh` 在手機瀏覽器上不等於「看得到的畫面高度」。因為手機瀏覽器的網址列和底部工具列會動態出現/消失，`100vh` 包含了被工具列遮住的部分。

結果就是：`height: 100vh` 的全螢幕 hero 區塊，在手機上會多出一截需要滑動。

**解法：使用新的視口單位**

```css
/* 舊寫法（有問題）*/
.hero-old {
  height: 100vh; /* 手機上會超出可見區域 */
}

/* 新寫法 */
.hero {
  height: 100dvh; /* Dynamic Viewport Height：跟著工具列動態變化 */
}

/* 三種新單位的差異 */
.full-screen-a {
  height: 100svh; /* Small Viewport：工具列展開時的高度（最保守） */
}

.full-screen-b {
  height: 100lvh; /* Large Viewport：工具列收起時的高度（最大） */
}

.full-screen-c {
  height: 100dvh; /* Dynamic Viewport：隨工具列動態變化（推薦） */
}
```

**選擇建議：**

| 場景 | 用什麼 | 原因 |
|------|--------|------|
| 全螢幕 hero | `100dvh` | 動態適應，視覺最佳 |
| 固定滿版背景 | `100svh` | 避免抖動 |
| 全螢幕 overlay | `100lvh` | 確保遮住底部 |
| 一般佈局 | `min-height: 100dvh` | 內容多時可以超出 |

**向下相容寫法：**

```css
.hero {
  /* Fallback：舊瀏覽器用 100vh */
  height: 100vh;
  /* 支援 dvh 的瀏覽器會覆蓋 */
  height: 100dvh;
```

**另一個常見的 iOS Safari 修法（如果不能用 dvh）：**

```css
.hero {
  height: 100vh;
  /* iOS Safari 特殊處理 */
  height: -webkit-fill-available;
}

/* 或用 JavaScript */
```

```js
// 用 JS 動態設定（舊方案，dvh 出來後不太需要了）

  const vh = window.innerHeight * 0.01;
  document.documentElement.style.setProperty('--vh', `${vh}px`);
}

setVH();
window.addEventListener('resize', setVH);
```


## 3. Flexbox vs Grid 快速決策指南

### 一句話決策

> **一維排列用 Flexbox，二維排列用 Grid。不確定的話先用 Flexbox。**

### 詳細決策表

| 場景 | 用什麼 | 原因 |
|------|--------|------|
| 導航列（水平排列） | Flexbox | 一維、項目數量不固定 |
| 頁腳連結群組 | Flexbox | 簡單水平排列 |
| 按鈕群組 | Flexbox | 一維、間距控制簡單 |
| 卡片網格（等寬等高） | **Grid** | 二維、需要對齊行列 |
| 整頁佈局（header/sidebar/main/footer） | **Grid** | 二維、區域命名清楚 |
| 表單佈局（label + input 對齊） | **Grid** | 兩欄對齊 |
| 瀑布流 | Grid + `masonry`（實驗中）或 JS | 目前純 CSS 仍有限制 |
| 置中一個元素 | 都可以 | Grid 更簡潔 |
| 項目順序需要動態調整 | Flexbox | `order` 更直覺 |
| 項目大小需要按內容自適應 | Flexbox | `flex-grow/shrink` |
| 精確控制行列位置 | **Grid** | `grid-row/column` |

### 常用 Pattern

**Flexbox — 導航列：**

```css
.navbar {
  display: flex;
  align-items: center;
  justify-content: space-between;
  gap: 1rem;
}

.navbar__links {
  display: flex;
  gap: 2rem;
}
```

**Flexbox — 垂直水平置中：**

```css
.center-me {
  display: flex;
  align-items: center;
  justify-content: center;
  min-height: 100dvh;
}
```

**Grid — 垂直水平置中（更簡潔）：**

```css
.center-me {
  display: grid;
  place-items: center;
  min-height: 100dvh;
}
```

**Grid — 整頁佈局：**

```css
.page {
  display: grid;
  grid-template-areas:
    "header  header"
    "sidebar main"
    "footer  footer";
  grid-template-columns: 280px 1fr;
  grid-template-rows: auto 1fr auto;
  min-height: 100dvh;
}

.header  { grid-area: header; }
.sidebar { grid-area: sidebar; }
.main    { grid-area: main; }
.footer  { grid-area: footer; }

@media (max-width: 767px) {
  .page {
    grid-template-areas:
      "header"
      "main"
      "sidebar"
      "footer";
    grid-template-columns: 1fr;
  }
}
```

**Grid — 自動填充卡片（auto-fill / auto-fit）：**

```css
/* auto-fill：盡量塞滿，多的留空 */
.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(280px, 1fr));
  gap: 1.5rem;
}

/* auto-fit：盡量塞滿，多的空間分給現有項目 */
.card-grid-stretch {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
  gap: 1.5rem;
}

/*
  差異：3 個卡片在 1200px 寬的容器中
  auto-fill：3 個 280px 卡片 + 1 個空欄位
  auto-fit：3 個卡片平均分配 1200px（每個 ~380px）
  通常用 auto-fit 比較符合預期。
*/
```

### 可以混用

Flexbox 和 Grid 不是互斥的，很多時候會混著用：

```css
/* 外層用 Grid 控制整體佈局 */
.page {
  display: grid;
  grid-template-columns: 1fr 3fr;
}

/* 內層用 Flexbox 排列項目 */
.sidebar__nav {
  display: flex;
  flex-direction: column;
  gap: 0.5rem;
}

.main__toolbar {
  display: flex;
  align-items: center;
  justify-content: space-between;
}
```

#### Contrast Ratio（對比度）

WCAG 2.1 AA 標準：
- **一般文字**：對比度 >= 4.5:1
- **大文字**（18px bold 或 24px 以上）：對比度 >= 3:1
- **UI 元件**（按鈕邊框、表單輸入框）：對比度 >= 3:1

**檢查工具：**

```
- Chrome DevTools：Elements > 點色彩值 > 顯示對比度
- WebAIM Contrast Checker：https://webaim.org/resources/contrastchecker/
- Figma 外掛：Stark、A11y - Color Contrast Checker
```

**常見問題色彩（對比度不及格）：**

```css
/* 不及格：淺灰字 on 白底（對比度 ~2.3:1） */

/* 及格：深灰字 on 白底（對比度 ~7.4:1） */
.good { color: #555; background: #fff; }

/* 不及格：白字 on 淺藍底（對比度 ~2.8:1） */
.bad { color: #fff; background: #5dade2; }

/* 及格：白字 on 深藍底（對比度 ~7.5:1） */
.good { color: #fff; background: #1a5276; }
```

## 7. 常見客戶問題 FAQ

### Q1：「為什麼 IE 不能用？」

**簡短回答：** 微軟在 2022 年 6 月已正式停止支援 Internet Explorer，全球市占率 < 0.3%。所有主流網站（Google、Facebook、YouTube）都不再支援 IE。

**給客戶的話：**

> IE 已經在 2022 年被微軟正式淘汰了。目前全球只有不到 0.3% 的使用者還在用 IE，而且微軟自己的 Microsoft 365、Outlook 網頁版都不支援 IE 了。
>
> 支援 IE 需要額外寫大量的相容性代碼，開發成本會增加 30-50%，而且很多現代網頁功能（例如動畫、佈局、互動效果）在 IE 上完全無法實現。
>
> 建議引導使用者升級到 Edge（微軟的新瀏覽器，Windows 10/11 都內建）。如果貴司內部系統必須用 IE，那是 IT 部門需要處理的獨立問題。

**如果客戶堅持要 IE：** 加收 50% 以上的費用，而且事先說明很多動畫和佈局效果會降級。

---

### Q2：「為什麼 Safari 和 Chrome 長得不一樣？」

**簡短回答：** Safari 和 Chrome 使用不同的瀏覽器引擎（WebKit vs Blink），對 CSS 的解讀細節有差異。就像 Word 和 Pages 開同一份文件，排版會有微小差異。

**給客戶的話：**

> Safari 和 Chrome 是不同公司做的瀏覽器，底層引擎不同，就像 Microsoft Word 和 Apple Pages 打開同一份文件會有微小的排版差異一樣。
>
> 我們的 QA 流程會在兩個瀏覽器上都測試過，確保主要功能和視覺呈現一致。但像是字體粗細、按鈕圓角、捲動效果等細節，可能會有 1-2 像素的差異，這是業界正常現象。
>
> 如果您發現某個功能在 Safari 上有明顯問題（不是 1-2 像素的差異，而是跑版或不能用），請截圖告訴我們，我們會優先修正。

**技術上的預防措施：**

```css
/* 全域重置，減少瀏覽器差異 */
*,
*::before,
*::after {
  box-sizing: border-box;
  margin: 0;
  padding: 0;
}

/* 或者用 CSS Reset / Normalize.css */
```

### Q3：「手機上看起來字太小 / 太大？」

**給客戶的話：**

> 網站已經針對不同螢幕尺寸做了響應式設計。如果您覺得字體大小不合適，請告訴我您使用的裝置型號和看到的畫面截圖，我們可以微調。
>
> 另外，請確認您手機的「顯示設定」>「字體大小」是否調整過。有些使用者會把系統字體放大，這會影響網頁的顯示。

**技術提醒：**

```html
<!-- 確保有設定 viewport -->
<meta name="viewport" content="width=device-width, initial-scale=1.0">

<!-- 不要加 maximum-scale=1.0 或 user-scalable=no，這違反無障礙規範 -->
<!-- 錯誤：-->
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
```

### Q4：「為什麼動畫在我手機上卡卡的？」

**給客戶的話：**

> 動畫效能取決於裝置的硬體能力。較新的手機（iPhone 13 以後、近兩年的 Android 旗艦機）通常不會有問題。較舊的手機或中低階 Android 可能會有些許延遲。
>
> 我們的動畫都使用了 GPU 加速技術來確保流暢度。如果您在特定裝置上遇到卡頓，請告訴我們裝置型號，我們可以針對低效能裝置做降級處理。

**技術優化：**

```css
/* 用 transform 和 opacity 做動畫（GPU 加速）*/
.animate-good {
  /* 好：觸發 GPU 合成層 */
  transform: translateX(100px);
  opacity: 0.5;
  transition: transform 0.3s ease, opacity 0.3s ease;
}

.animate-bad {
  /* 不好：會觸發 layout 重新計算 */
  left: 100px;
  width: 200px;
  margin-top: 20px;
}

/* 明確告訴瀏覽器「這個元素會變化」 */
.will-animate {
  will-change: transform, opacity;
}
```

### Q5：「為什麼 LINE 裡面開網站跟用瀏覽器開不一樣？」

**給客戶的話：**

> LINE、Facebook、Instagram 等 App 內建的瀏覽器，和一般的 Chrome/Safari 是不同的。它們通常使用的是精簡版的瀏覽器引擎，可能不支援某些較新的網頁功能。
>
> 如果您的主要流量來源是 LINE 分享，我們會額外在 LINE 內建瀏覽器中測試。最穩妥的做法是在頁面上放一個「用瀏覽器開啟」的按鈕。

**技術處理：**

```js
// 偵測 LINE 內建瀏覽器
const isLINE = /Line/i.test(navigator.userAgent);

if (isLINE) {
  // 顯示「用瀏覽器開啟」按鈕
  document.getElementById('open-in-browser')?.style.display = 'block';
}
```

```html
<!-- LINE 分享的 meta tags -->
<meta property="og:title" content="頁面標題">
<meta property="og:description" content="頁面描述">
<meta property="og:image" content="https://example.com/og-image.jpg">
<meta property="og:image:width" content="1200">
<meta property="og:image:height" content="630">
```

### Q6：「為什麼印出來的網頁和螢幕上不一樣？」

**給客戶的話：**

> 網頁預設是為螢幕設計的，印出來會有差異。如果您需要列印功能，我們可以加入列印樣式，讓列印結果更接近螢幕上的呈現。

**列印樣式：**

```css
@media print {
  /* 隱藏不需要列印的元素 */
  .navbar,
  .footer,
  .sidebar,
  .no-print,
  button {
    display: none !important;
  }

  /* 頁面用白底黑字 */
  body {
    background: #fff !important;
    color: #000 !important;
    font-size: 12pt;
    line-height: 1.5;
  }

  /* 連結顯示 URL */
  a[href]::after {
    content: " (" attr(href) ")";
    font-size: 0.8em;
    color: #666;
  }

  /* 避免表格和圖片斷頁 */
  table, img, figure {
    break-inside: avoid;
  }

  /* 標題不要出現在頁面最底部 */
  h1, h2, h3 {
    break-after: avoid;
  }
}
```

### Q7：「網站能不能支援到 Windows 7？」

**給客戶的話：**

> Windows 7 已在 2020 年停止支援。在 Windows 7 上能執行的最新 Chrome 是 109 版（2023 年初停止更新）。我們的網站使用了 2023 年以後的新標準，在舊版 Chrome 上可能有部分功能無法正常使用。
>
> 如果貴司有大量 Windows 7 使用者，建議先確認他們的 Chrome 版本，我們可以評估需要做多少降級處理。但長期來看，升級作業系統才是根本解法。

### Q8：「我用手機看 OK，為什麼客戶的手機有問題？」

**排查步驟（跟客戶要以下資訊）：**

```markdown
1. 裝置型號（iPhone 幾？Android 什麼牌子型號？）
2. 作業系統版本（iOS 17？Android 14？）
3. 使用什麼瀏覽器開啟的？（Safari / Chrome / LINE / Facebook）
4. 截圖或螢幕錄影
5. 系統字體有沒有放大？
```

**常見原因：**
- **LINE / Facebook 內建瀏覽器**：引擎不同，見 Q5
- **舊版 iOS / Android**：不支援新 CSS
- **系統字體放大**：使用者把系統字體調到 120-150%，破壞佈局
- **省電模式**：停用動畫和背景圖片載入
- **網路問題**：CSS 沒載入完，看到的是沒有樣式的頁面
