## 1. 專案管理方法論

### 三大方法論比較

| 維度 | Kanban（看板） | Scrum（衝刺） | Waterfall（瀑布） |
|------|---------------|--------------|-------------------|
| 核心概念 | 視覺化工作流、限制進行中數量 | 固定週期衝刺、每日站會 | 線性階段、依序推進 |
| 迭代方式 | 持續流動 | 1-4 週 Sprint | 無迭代，一次到位 |
| 適合規模 | 1-3 人 | 3-9 人 | 任意，但需明確需求 |
| 需求變更 | 隨時可調 | Sprint 內不變 | 變更成本極高 |
| 文件量 | 極少 | 中等（backlog、burndown） | 大量（PRD、SRS） |
| 學習成本 | 低 | 中（需理解角色與儀式） | 低但僵化 |

### 接案推薦：混合式 Kanban

**結論：接案最適合以 Kanban 為基底，混入 Scrum 的「週期回顧」概念。**

原因：
- 客戶不懂 Sprint Planning，但看得懂看板
- 接案需求變更頻繁，Kanban 的彈性最高
- 一人作業或小團隊不需要 Scrum Master、PO 等角色
- Waterfall 在客戶「邊做邊改」的現實下幾乎不可行

**推薦做法 — Kanban + 週節奏：**

```
看板欄位：
待處理 → 本週進行 → 設計中 → 開發中 → 測試/修改 → 待客戶確認 → 完成

週節奏：
- 每週一：檢視看板、排定本週任務
- 每週五：更新客戶進度、移動卡片
- 每月底：覆盤（Retrospective）
```

### 各方法論適用情境速查

| 情境 | 推薦方法 |
|------|---------|
| 單頁 Landing Page | 簡單 Kanban（甚至不需要工具） |
| 5 頁形象官網 | Kanban + 里程碑 |
| 電商網站（含後端） | Kanban + 2 週 Sprint |
| 長期維護合約 | 純 Kanban（持續流動） |
| 大型企業專案（10 人以上） | Scrum 或 SAFe |
| 政府/標案 | Waterfall（因為合約要求） |

- **Google Calendar** — 時間管理
- **GitHub Projects** — 程式碼追蹤（如果需要）

**團隊組合：**
- **Notion** — 知識庫 + 客戶溝通
- **Linear** — 開發任務追蹤
- **Slack** — 即時通訊
- **Figma** — 設計協作

## 4. 專案時程估算技巧

### 4.1 三點估算法（PERT）

對每個任務給出三個估計值，計算加權平均：

```
公式：
預估時間 = (樂觀值 + 4 × 最可能值 + 悲觀值) ÷ 6

標準差 = (悲觀值 - 樂觀值) ÷ 6

範例：首頁切版
├── 樂觀值（O）：6 小時（一切順利、素材齊全）
├── 最可能值（M）：10 小時（正常情況）
├── 悲觀值（P）：18 小時（需求不清、反覆修改）
├── 預估 = (6 + 4×10 + 18) ÷ 6 = 10.67 小時
└── 標準差 = (18 - 6) ÷ 6 = 2 小時

→ 報給客戶：10-13 小時（預估值 + 1 個標準差）
```

### 4.2 Buffer 加成法則

```
Buffer 規則（依經驗累加）：

基礎估算
  × 1.2  新客戶（不熟悉溝通風格）
  × 1.3  新技術（從未用過的框架/工具）
  × 1.1  多人協作（溝通成本）
  × 1.2  客戶決策慢（歷史經驗）
  × 1.15 假日/年底（工作效率低）

計算範例：
原始估算：40 小時
新客戶 × 1.2 = 48 小時
新技術 × 1.3 = 62.4 小時
→ 報價以 65 小時計算

簡化版（適合快速估算）：
- 你覺得要 X 天 → 報 X × 1.5 天
- 你覺得很簡單 → 報 X × 2 天（因為「簡單」的事最容易出錯）
- 你覺得很複雜 → 報 X × 1.3 天（因為你已經在警戒了）
```

### 4.3 歷史數據參考法

建立一個「工時紀錄」Database：

```
Database 名稱：工時紀錄

記錄每個任務的：
- 任務類型（切版/API/設計/測試...）
- 預估工時
- 實際工時
- 偏差率 = (實際 - 預估) / 預估 × 100%

累積 10 個以上同類型任務後，就能算出你的「個人偏差係數」：

範例：
過去 10 次「RWD 切版」任務
平均偏差率 = +35%
→ 以後切版估算都乘以 1.35

過去 8 次「API 串接」任務
平均偏差率 = +15%
→ API 串接估算都乘以 1.15
```

### 4.4 分解估算法（WBS）

將大任務拆成小任務，分別估算再加總：

```
範例：形象官網 5 頁

1. 需求確認與規劃          8h
   ├── 客戶訪談             2h
   ├── 競品分析             2h
   ├── 網站架構圖           2h
   └── 功能規格書           2h

2. 設計                   40h
   ├── Wireframe × 5 頁    10h
   ├── 視覺設計 × 5 頁     25h
   └── 設計修改（2 輪）      5h

3. 前端開發                48h
   ├── 環境建置             4h
   ├── 共用元件             8h
   ├── 首頁切版            12h
   ├── 內頁切版 × 4        16h
   ├── RWD 調整             8h
   └── 動態效果             不定

4. 後端/CMS                16h
   ├── CMS 設定             8h
   ├── 表單功能             4h
   └── SEO 設定             4h

5. 測試與上線              12h
   ├── 跨瀏覽器測試          4h
   ├── 修 Bug               4h
   └── 部署上線             4h

6. Buffer                 12h

總計：136h ÷ 8h/天 = 17 個工作天 ≈ 4 週
```

## 5. 常見專案類型時程範本

```

Day 5    │ 設計確認
         │ - 客戶第 1 次回饋

Week 2（第 6-10 天）
──────────────────────────────────────

         │ - HTML/CSS 切版
         │ - RWD 響應式
         │ - 動態效果

         │ - 表單提交
         │ - GA/Pixel 追蹤碼
         │ - SEO meta 設定

         │ - 跨瀏覽器測試

         │ - 客戶最終確認

         │ - DNS 設定
         │ - SSL 憑證
         │ - 上線後驗證

里程碑：
M1 Day 2  需求確認完成

```

```
Phase 3（Week 5-8）── 開發
──────────────────────────────────────

- 前端框架建置 + 共用元件

- 物流串接（超商取貨）

Phase 4（Week 9-10）── 整合測試
──────────────────────────────────────
- 金流測試環境驗證
- 完整購物流程測試
- 壓力測試
- 跨裝置/瀏覽器測試
- 安全性測試（XSS、SQL Injection）
- 客戶 UAT 測試

Phase 5（Week 11-12）── 上線 + Buffer
──────────────────────────────────────
- 客戶回饋修改
- 資料遷移（舊系統搬家）
- 正式環境部署
- DNS 切換 + SSL
- 上線後監控 72 小時
- 教育訓練（後台操作）
- 專案結案

關鍵提醒：
- 金流串接審核需 3-7 個工作天，要提前申請
- 商品資料匯入通常比預期花更多時間
- 客戶在「看到實際畫面後」一定會有大量修改
```

### 6.1 優先級矩陣（Eisenhower Matrix 改良版）

```
                    緊急                    不緊急
            ┌──────────────────┬──────────────────┐
            │                  │                  │
     重要    │  🔴 立即處理      │  🟡 排入計畫      │

            │  - 客戶急件       │  - 系統優化       │
            │  - 線上 Bug      │  - 學習新技術     │
            │                  │                  │
            ├──────────────────┼──────────────────┤
            │                  │                  │
    不重要   │  🔵 委派/快速處理  │  ⚪ 延後/刪除     │
            │  - 簡單修改       │  - 可有可無的功能  │

            │                  │                  │
            └──────────────────┴──────────────────┘
```

### 6.2 時間分配策略

**同時進行 2-3 個專案的時間分配：**

```
方案 A：日切分（推薦新手）
──────────────────────────────────────
週一：專案 A（整天）
週二：專案 B（整天）
週三：專案 A（整天）
週四：專案 C（整天）
週五：專案 B 上午 + 行政/客戶溝通 下午

優點：減少切換成本，深度工作
缺點：客戶可能等不及

方案 B：半日切分（推薦有經驗者）
──────────────────────────────────────
每天上午：主力專案（最複雜/最緊急的）
每天下午前半：次要專案
每天下午後半：溝通 + 小修改 + 行政

優點：每天都有推進
缺點：切換成本較高

方案 C：階段切分（推薦團隊）
──────────────────────────────────────
同時間只有 1 個專案在「設計階段」
同時間只有 1 個專案在「開發階段」
利用「等客戶確認」的空檔推進下一案

範例：
專案 A：開發中 ─────────────
專案 B：      設計中 ──── 等確認 ──── 開發中
專案 C：                  需求確認 ──── 設計中
```

### 6.3 客戶排程技巧

```
客戶溝通時段規劃：
──────────────────────────────────────
09:00-10:00  回覆所有客戶訊息（集中處理）
10:00-12:00  專注工作時段（靜音）
12:00-13:00  午餐
13:00-14:00  客戶會議時段（固定時段開會）
14:00-17:00  專注工作時段（靜音）
17:00-18:00  回覆訊息 + 更新進度

原則：
1. 不要讓客戶隨時打斷你 — 設固定回覆時段
2. 會議集中排在下午 1-3 點 — 保護上午的黃金時段
3. 每週五寄出所有專案進度更新 — 減少客戶追問
4. 用「非同步溝通」取代「即時通訊」— Notion 評論 > Line 訊息
```

## 7. 里程碑設定

### 7.1 標準里程碑模型

```
M0 ── 專案啟動（Kick-off）
│     交付物：簽約確認、Brief 完成、收到訂金
│     負責人：PM / 業務
│
M1 ── 需求確認
│     交付物：網站架構圖、功能規格書、技術方案
│     確認人：客戶簽核
│     ⚠️ 這步沒確認清楚，後面一定爆炸
│
M2 ── 設計確認
│     交付物：Wireframe + 視覺設計稿（Desktop + Mobile）
│     確認人：客戶簽核
│     修改次數：合約約定（通常 2-3 次）
│     ⚠️ 設計確認後的重大修改，視為需求變更加價
│
M3 ── 前端開發完成
│     交付物：前端頁面（靜態/可互動原型）
│     驗收標準：所有頁面切完、RWD 正常、動效完成
│     內部 Review：代碼品質、效能檢查
│
M4 ── 後端開發完成
│     交付物：完整功能可運作的網站
│     驗收標準：所有功能正常、API 回應正確
│     內部 Review：安全性、效能
│
M5 ── 測試完成
│     交付物：測試報告、Bug 修復完畢
│     驗收標準：無 P0/P1 Bug
│     客戶 UAT：客戶實際操作測試
│
M6 ── 上線
│     交付物：正式網站上線
│     檢查清單：DNS、SSL、SEO、Analytics、效能
│     確認人：客戶最終確認
│
M7 ── 結案
      交付物：原始碼、文件、教學、發票
      收款：尾款收齊
      保固：開始計算保固期（通常 1-3 個月）
```

### 7.2 里程碑與收款掛鈎

```
收款時程建議（降低風險）：

方案 A：3-3-3-1 分期
M0 簽約  30% 訂金
M2 設計確認  30%
M6 上線  30%
M7 結案  10% 尾款

方案 B：5-3-2 分期（小案）
M0 簽約  50% 訂金
M3 開發完成  30%
M6 上線  20% 尾款

方案 C：月結（長期合作）
每月固定金額，按月結算

原則：
- 開工前至少收 30% — 確認客戶誠意
- 設計確認後收一筆 — 過了這關設計不退
- 不要等全部做完才收尾款 — 風險太大
```

### 7.3 里程碑確認 SOP

```
每個里程碑確認流程：

1. 內部 Review
   └── 負責人自檢 → 交叉 Review → 確認可交付

2. 準備交付物
   └── 截圖/連結/文件整理 → 放到 Notion 客戶頁面

3. 通知客戶
   └── 發送「階段完成通知」
   └── 附上確認期限（建議 3 個工作天）

4. 客戶確認
   └── 客戶回覆「確認」或「修改意見」
   └── 修改意見記錄在 Notion

5. 修改（如有）
   └── 合約內修改次數 → 免費修改
   └── 超出次數 → 報價加收

6. 正式簽核
   └── 客戶回覆「確認通過」（Email/Notion 評論留紀錄）
   └── 更新專案狀態 + 觸發下一階段
```

## 8. 風險管理

### 8.1 常見風險清單

```
風險等級：🔴 高（發生率高、影響大）  🟡 中  🔵 低

🔴 需求變更（客戶反覆修改）
   頻率：幾乎每個專案都會發生
   影響：時程延遲 2-4 週、成本增加 30-50%
   預防：
   - 合約明定修改次數（設計 2 次、開發 1 次）
   - 超出修改按時薪收費（NT$1,500-3,000/hr）
   - 每次修改要求書面確認
   應急：
   - 評估變更影響（時程/成本）
   - 發出「變更通知單」（含加價/延期說明）
   - 客戶不接受就按原方案繼續

🔴 客戶端延遲（素材/回饋/決策慢）
   頻率：非常常見
   影響：整體時程延遲，但不是你的問題
   預防：
   - 合約寫明「客戶需在 N 個工作天內回覆」
   - 素材清單在簽約時就給（附截止日期）
   - 設定「等待超過 5 天自動暫停專案」條款
   應急：
   - 先做不依賴客戶的部分
   - 用假資料/假圖繼續開發
   - 定期提醒（但不要太頻繁，每 3 天一次）
   - 書面通知時程影響

🔴 技術難題（預估不足）
   頻率：新技術或複雜功能
   影響：時程延遲 1-3 週
   預防：
   - 接案前做 PoC（Proof of Concept）
   - 複雜功能提前 Spike（技術探索）
   - 估算時加 30% Buffer
   應急：
   - 尋求社群/Stack Overflow 幫助
   - 考慮替代方案（換套件、簡化功能）
   - 與客戶討論降規

🟡 團隊成員異動
   頻率：長期專案較常見
   影響：知識斷層、交接成本
   預防：
   - 文件化所有決策
   - 代碼注釋 + README
   - 定期 Code Review
   應急：
   - 找外包支援
   - 與客戶協商延期

🟡 第三方服務問題（金流/API/套件）
   頻率：串接時常遇到
   影響：特定功能延遲 1-2 週
   預防：
   - 提前申請第三方服務帳號
   - 先在測試環境跑通
   - 準備備選方案（如藍新不行就用綠界）
   應急：
   - 聯繫第三方客服
   - 先完成其他功能
   - 評估替代方案

🟡 範疇蔓延（Scope Creep）
   頻率：客戶常說「能不能順便加一下...」
   影響：工時增加、利潤下降
   預防：
   - 合約明定功能範圍
   - 「順便」的事情一律報價
   - 區分「Bug 修復」和「新需求」
   應急：
   - 記錄所有額外要求
   - 累積到一定量一起報價
   - 建議放到「Phase 2」

🔵 設備/環境問題
   頻率：偶爾
   影響：1-2 天
   預防：
   - 代碼上 Git（每日 push）
   - 設計檔雲端同步
   - 備用開發環境
   應急：
   - 用備用電腦
   - 暫時在雲端 IDE 開發
```

## 11. 專案覆盤（Retrospective）

### 11.1 覆盤會議流程

```
覆盤會議 — 時長 60-90 分鐘

第一段（15 分鐘）── 數據回顧
──────────────────────────────────────
- 原始時程 vs 實際時程（延遲了幾天？）
- 原始報價 vs 實際工時（利潤率？）
- Bug 數量 / 修改次數 / 客戶滿意度
- 各階段的偏差分析

第二段（20 分鐘）── 三格回顧
──────────────────────────────────────
每個人各寫便利貼（或打在 Notion）：

┌──────────────────┬──────────────────┬──────────────────┐
│ 做得好            │ 可以改善          │ 要停止做          │
│ (Keep)           │ (Try)            │ (Stop)           │
├──────────────────┼──────────────────┼──────────────────┤
│ - 切版速度比上次快 │ - 設計交接文件不足 │ - 不要在群組改需求 │
│ - 客戶溝通順暢    │ - 要提早串金流    │ - 不要週末回訊息  │
│ - Git 流程順暢    │ - 估算要再保守些  │ - 不要口頭確認需求 │
│ - 模板化省了時間  │ - 測試要更完整    │                  │
└──────────────────┴──────────────────┴──────────────────┘

第三段（15 分鐘）── 深度討論
──────────────────────────────────────
- 挑出 2-3 個最重要的改善項目
- 討論根本原因（5 Why 分析法）
- 討論具體改善措施

第四段（10 分鐘）── 行動計畫
──────────────────────────────────────
- 每個改善項目指定負責人 + 截止日
- 記錄到「改善追蹤表」
```

```markdown
| Bug 數量 | -- | 12 個 | -- |
| 客戶滿意度 | -- | 8/10 | -- |

## 時程偏差分析

| 階段 | 預估 | 實際 | 原因 |
|------|------|------|------|
| 需求確認 | 3 天 | 5 天 | 客戶決策慢 |
| 設計 | 10 天 | 14 天 | 修改 3 輪 |
| 前端 | 12 天 | 12 天 | 準時 |
| 後端 | 8 天 | 10 天 | 金流串接卡關 |
| 測試 | 3 天 | 5 天 | Bug 比預期多 |

## Keep（繼續做）

1. Notion 專案頁面分享給客戶，溝通效率高
2. 每週五固定發週報，客戶安心
3. 設計系統模板複用，切版速度提升

## Try（嘗試改善）

1. 設計確認從口頭改為書面（Notion 留紀錄）
2. 估算加 35% Buffer（而非 20%）
3. 金流申請提前到專案第一週就送件
4. 加入自動化測試（至少 E2E 跑一輪）

## Stop（停止做）

1. 不再接受 Line 群組裡的需求變更（統一用 Notion）
2. 不再讓客戶「先看看再說」— 每次確認都要書面
3. 不再估算時給「最樂觀」的數字

## 行動計畫

| # | 改善事項 | 負責人 | 截止日 | 狀態 |
|---|---------|-------|--------|------|
| 1 | 建立「設計確認單」模板 | 設計師 | MM/DD | 待辦 |
| 2 | 更新估算模板（加 Buffer 係數） | PM | MM/DD | 待辦 |
| 3 | 建立金流申請 Checklist | 後端 | MM/DD | 待辦 |
| 4 | 導入 Playwright E2E | 前端 | MM/DD | 待辦 |
```

## 12. 自動化工具搭配

### 12.1 n8n 定時提醒工作流

**工作流 1：每日任務提醒**

```
觸發：Schedule（每天 09:00）
  → 讀取 Notion Database（篩選：狀態=進行中、截止日=今天或已過期）
  → IF 有過期任務
    → 發送 Slack/Telegram 通知
    → 格式：「⚠️ 以下任務已過期：[任務名稱] - 原截止 MM/DD」

n8n 節點設定概要：
1. Schedule Trigger
   - Interval: Every Day
   - Hour: 9

2. Notion - Get Database Items
   - Database ID: [你的 Database ID]
   - Filter: status = "進行中" AND due_date <= today

3. IF - Has Overdue Items
   - Condition: items.length > 0

4. Telegram / Slack - Send Message
   - 內容：過期任務清單
```

**工作流 2：每週五自動發送客戶週報**

```
觸發：Schedule（每週五 16:00）
  → 讀取 Notion 專案 Database（狀態=進行中）
  → 讀取每個專案的本週更新
  → 組合成週報格式
  → 發送 Email 給客戶

n8n 節點概要：
1. Schedule Trigger (Friday 16:00)
2. Notion - Get Active Projects
3. Notion - Get This Week's Updates (for each project)
4. Code Node - Format Weekly Report
5. Send Email (Gmail / SMTP)
```

**工作流 3：里程碑到期提醒**

```
觸發：Schedule（每天 10:00）
  → 讀取 Notion Database（篩選：里程碑截止日 = 3 天內）
  → 發送提醒
  → 格式：「📌 [專案名稱] 的 [里程碑名稱] 將在 N 天後到期」

目的：提前 3 天提醒你準備里程碑交付物
```

## 附錄 B：工時紀錄最佳實踐

```
工時紀錄原則：
1. 即時記錄（不要事後回想）
2. 最小單位：30 分鐘
3. 區分「可收費工時」和「非收費工時」
4. 包含：任務名稱 + 專案 + 分類 + 時數 + 備註

工具推薦：
- Toggl Track（免費版夠用，有報表）
- Clockify（完全免費）
- Notion Database（自建，整合在同一平台）
- Harvest（付費，有發票功能）

每月分析：
- 總工時 vs 可收費工時 → 利用率
- 每個專案的時薪 → 哪些專案賺錢/虧錢
- 各分類工時占比 → 設計/開發/溝通/行政
- 與估算的偏差 → 校正未來估算
```
