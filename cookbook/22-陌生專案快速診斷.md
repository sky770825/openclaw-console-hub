# 22 — 陌生專案快速診斷 SOP

> 老蔡叫你看一個你從沒碰過的專案，30 秒內搞清楚它是什麼。

---

## 什麼時候用

- 老蔡說「幫我看一下 XXX 專案」
- 老蔡給你一個路徑，你不知道裡面是什麼
- 要幫一個外部專案修 bug / 加功能 / 做分析

---

## 第一步：找到專案（10 秒）

```
# 老蔡說了名字但沒給路徑 → 自己找
{"action":"run_script","command":"find /Users/caijunchang/Desktop /Users/caijunchang/Downloads -maxdepth 2 -name '*.json' -path '*/package.json' 2>/dev/null | head -20"}

# 或用關鍵字找
{"action":"run_script","command":"find /Users/caijunchang/Desktop -maxdepth 3 -type d -name '*餐車*' -o -name '*food*' -o -name '*華地*' 2>/dev/null"}

# 老蔡桌面常見的專案資料夾
{"action":"list_dir","path":"/Users/caijunchang/Desktop"}
```

---

## 第二步：快速掃描結構（10 秒）

找到路徑後，跑這 3 個指令就能判斷專案類型：

```
# 1. 看根目錄結構
{"action":"list_dir","path":"/Users/caijunchang/Desktop/專案名"}

# 2. 看 package.json（如果有）→ 知道技術棧
{"action":"read_file","path":"/Users/caijunchang/Desktop/專案名/package.json"}

# 3. 看 README（如果有）→ 知道用途
{"action":"read_file","path":"/Users/caijunchang/Desktop/專案名/README.md"}
```

---

## 第三步：判斷專案類型（10 秒）

根據掃描結果判斷：

| 你看到的 | 專案類型 | 關鍵檔案 |
|----------|----------|----------|
| package.json + src/ + vite/next | React/Vue/Next.js 前端 | src/App.tsx, src/pages/ |
| package.json + server/ 或 express | Node.js 後端 | server/index.ts, routes/ |
| index.html + script.js（沒有 package.json） | 純 HTML/JS 靜態網站 | index.html, script.js |
| index.html + supabase 字樣 | Supabase + 靜態前端 | script.js 找 supabaseUrl |
| .py 檔案 + requirements.txt | Python 專案 | main.py, app.py |
| docker-compose.yml | Docker 化服務 | docker-compose.yml |
| 只有 .md / .xlsx / .csv | 文件/數據專案 | 直接看檔案內容 |

---

## 第四步：找關鍵資訊

### 找 Supabase 連線
```
{"action":"run_script","command":"grep -r 'supabase' /Users/caijunchang/Desktop/專案名/ --include='*.js' --include='*.ts' --include='*.html' -l 2>/dev/null | head -5"}

# 然後讀那個檔案找 URL 和 Key
{"action":"run_script","command":"grep -n 'supabaseUrl\\|supabaseKey\\|SUPABASE_URL\\|createClient' /Users/caijunchang/Desktop/專案名/script.js 2>/dev/null"}
```

### 找 API 端點
```
{"action":"run_script","command":"grep -rn 'fetch(\\|axios\\|api/' /Users/caijunchang/Desktop/專案名/src/ --include='*.js' --include='*.ts' 2>/dev/null | head -10"}
```

### 找環境變數
```
# .env.example 通常可以安全讀取（不含真實 key）
{"action":"read_file","path":"/Users/caijunchang/Desktop/專案名/.env.example"}

# 或看 .env 有哪些變數名（不看值）
{"action":"run_script","command":"grep -o '^[A-Z_]*=' /Users/caijunchang/Desktop/專案名/.env 2>/dev/null"}
```

---

## 第五步：回報老蔡

摸清楚後，用這個格式回老蔡：

```
專案：[名稱]
路徑：[完整路徑]
類型：[前端/後端/全端/靜態/數據]
技術棧：[React + Supabase / Node.js + Express / 純 HTML+JS ...]
資料庫：[Supabase / SQLite / 無 / 不確定]
狀態：[可運行 / 需要設定 / 有報錯]
我能做：[讀代碼/改代碼/查數據/...]
```

---

## 常見陷阱

1. **不要猜路徑** — 一定要先 list_dir 或 find 確認路徑存在
2. **不要改 node_modules/** — 那是自動生成的
3. **不要改 dist/ 或 build/** — 那是編譯產物
4. **.env 不能讀** — 用 .env.example 或 grep 變數名代替
5. **Lovable 專案**的結構特殊 — 通常是 src/pages/ + src/components/ + supabase/
6. **老蔡桌面的中文資料夾名**要小心 — 用完整路徑加引號

---

**記住：先掃描，再判斷，最後行動。不要還沒看過就開始改。**
