## 1. OWASP Top 10（2025 版）

> OWASP（Open Worldwide Application Security Project）每隔數年更新一次 Web 應用最常見的十大安全風險。
> 2025 版基於 2021 版架構進行調整，以下為最新分類。

### A01 — Broken Access Control（存取控制失效）

**說明**：使用者能存取不屬於自己權限的資源。例如修改 URL 中的 ID 就能看到別人的訂單、普通會員能存取管理後台。

**常見場景**：
- 水平越權：使用者 A 看到使用者 B 的資料（改 `/api/orders/123` 中的 ID）
- 垂直越權：一般會員存取 `/admin` 路徑
- IDOR（Insecure Direct Object Reference）：直接用資料庫 ID 當 API 參數，未驗證擁有權

**防護方法**：
```typescript
// Express.js 中間件 — 驗證資源擁有權
async function checkOwnership(req: Request, res: Response, next: NextFunction) {
  const resourceId = req.params.id;
  const userId = req.user.id; // 從 JWT/session 取得

  const resource = await db.query(
    'SELECT owner_id FROM orders WHERE id = $1',
    [resourceId]
  );

  if (!resource.rows[0] || resource.rows[0].owner_id !== userId) {
    return res.status(403).json({ error: '無權限存取此資源' });
  }
  next();
}

// 路由綁定
app.get('/api/orders/:id', authenticate, checkOwnership, getOrder);
```

```typescript
// 角色權限中間件
function requireRole(...roles: string[]) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!roles.includes(req.user.role)) {
      return res.status(403).json({ error: '權限不足' });
    }
    next();
  };
}

app.get('/api/admin/users', authenticate, requireRole('admin'), listUsers);
app.delete('/api/admin/users/:id', authenticate, requireRole('admin'), deleteUser);
```

**檢查清單**：
- [ ] 所有 API 端點都有認證中間件
- [ ] 資源存取有擁有權驗證（不能只靠 ID）
- [ ] 管理功能有角色檢查
- [ ] 前端隱藏不等於後端安全（前端只做 UX，後端做真正的權限控制）
- [ ] 預設拒絕（deny by default），明確授權才放行

### A02 — Cryptographic Failures（加密失效）

**說明**：敏感資料（密碼、信用卡號、個資）沒有正確加密，或使用了過時的加密演算法。

**防護方法**：
- 密碼必須用 bcrypt / argon2 雜湊（不是 MD5/SHA1）
- 傳輸中資料強制 HTTPS（TLS 1.2+）
- 資料庫中的個資欄位加密儲存
- 不要在 log 中記錄敏感資料
- API Key / 密碼不要寫死在程式碼中

```typescript
// 正確：使用 bcrypt 儲存密碼
import bcrypt from 'bcrypt';

const SALT_ROUNDS = 12;

async function hashPassword(password: string): Promise<string> {
  return bcrypt.hash(password, SALT_ROUNDS);
}

async function verifyPassword(password: string, hash: string): Promise<boolean> {
  return bcrypt.compare(password, hash);
}
```

```
# 錯誤做法（絕對不要）
❌ MD5(password)
❌ SHA1(password)
❌ SHA256(password)  ← 雖然更好但沒有加鹽、沒有慢雜湊，仍不安全
❌ 明文儲存密碼
❌ Base64 編碼（那不是加密）
```

### A04 — Insecure Design（不安全的設計）

**說明**：應用程式在設計階段就存在缺陷，不是實作的 bug 而是架構問題。

**常見問題**：
- 密碼找回問題用「你的寵物叫什麼名字」（容易被社工）
- 沒有交易限額（一次轉帳可以轉一億）
- 沒有業務邏輯驗證（折扣碼可以無限次使用）

**防護方法**：
- 威脅建模（Threat Modeling）：開發前先想「攻擊者會怎麼攻擊」
- 業務邏輯限制：每日交易上限、折扣碼使用次數、帳號建立速率限制
- 最小權限原則：每個元件只給它需要的最少權限
- 分層防護（Defense in Depth）：不依賴單一防線

### A05 — Security Misconfiguration（安全設定錯誤）

**說明**：伺服器、框架、雲端服務的預設設定不安全，或安全設定被遺漏。

**常見問題**：
- 生產環境開著 debug 模式（錯誤訊息洩漏程式碼路徑）
- 預設帳密沒改（admin/admin）
- 不需要的 HTTP methods 沒關（PUT、DELETE、TRACE）
- 目錄列表（Directory Listing）開著
- 雲端 S3 bucket 設成 public

**防護方法**：
```typescript
// Express.js 生產環境設定
if (process.env.NODE_ENV === 'production') {
  // 不要洩漏框架資訊
  app.disable('x-powered-by');

  // 錯誤處理不要回傳 stack trace
  app.use((err: Error, req: Request, res: Response, next: NextFunction) => {
    console.error(err.stack); // 記到 log，不回給用戶
    res.status(500).json({ error: '伺服器錯誤，請稍後再試' });
  });
}
```

```nginx
# Nginx — 關閉目錄列表、隱藏版本號
server {
    autoindex off;
    server_tokens off;

    # 關閉不需要的 HTTP methods
    if ($request_method !~ ^(GET|HEAD|POST)$ ) {
        return 405;
    }
}
```

### A08 — Software and Data Integrity Failures（軟體與資料完整性失效）

**說明**：CI/CD pipeline 被注入惡意代碼、未驗證更新來源、反序列化漏洞。

**防護方法**：
- 使用 lockfile（package-lock.json）固定依賴版本
- 驗證 CI/CD 步驟中的外部指令碼來源
- 啟用 npm 的 `npm audit signatures` 驗證套件簽章
- 避免不安全的反序列化（不要 `eval(JSON.parse(untrustedData))`）

```json
// package.json — 固定版本，避免自動升級到有問題的版本
{
  "dependencies": {
    "express": "4.21.2",
    "lodash": "4.17.21"
  }
}
// 用 "4.21.2" 而非 "^4.21.2" 或 "~4.21.2"
// 或至少確保 package-lock.json 有 commit 到 repo
```

### A09 — Security Logging and Monitoring Failures（安全日誌與監控不足）

**說明**：沒有記錄安全事件、沒有告警機制，攻擊發生了也不知道。

**防護方法**：
```typescript
// 記錄所有認證事件
function logSecurityEvent(event: {
  type: 'login_success' | 'login_failure' | 'password_change' | 'permission_denied';
  userId?: string;
  ip: string;
  userAgent: string;
  details?: string;
}) {
  const logEntry = {
    timestamp: new Date().toISOString(),
    ...event,
  };

  // 寫到安全專用 log（與應用 log 分開）
  securityLogger.info(JSON.stringify(logEntry));

  // 短時間內多次登入失敗 → 告警
  if (event.type === 'login_failure') {
    trackFailedLogin(event.ip);
  }
}

function trackFailedLogin(ip: string) {
  const key = `failed_login:${ip}`;
  const count = cache.incr(key);
  cache.expire(key, 600); // 10 分鐘窗口

  if (count >= 5) {
    alertAdmin(`IP ${ip} 在 10 分鐘內登入失敗 ${count} 次，可能正在暴力破解`);
  }
}
```

**必須記錄的事件**：
- 登入成功/失敗
- 密碼變更
- 權限變更
- 存取被拒
- 輸入驗證失敗
- 管理操作（新增/刪除使用者、修改設定）

### A10 — Server-Side Request Forgery (SSRF)（伺服器端請求偽造）

**說明**：攻擊者讓伺服器發出未預期的內部請求，存取內部網路資源。

**常見場景**：
- 圖片 URL 預覽功能，攻擊者傳入 `http://169.254.169.254/latest/meta-data/`（AWS metadata）
- Webhook URL 設定，攻擊者傳入內部 IP

**防護方法**：
```typescript
import { URL } from 'url';
import dns from 'dns/promises';

async function isUrlSafe(urlString: string): Promise<boolean> {
  try {
    const url = new URL(urlString);

    // 只允許 http/https
    if (!['http:', 'https:'].includes(url.protocol)) return false;

    // 阻擋內部 IP 和 metadata 端點
    const blockedHosts = [
      'localhost', '127.0.0.1', '0.0.0.0',
      '169.254.169.254',  // AWS/GCP metadata
      'metadata.google.internal',
    ];
    if (blockedHosts.includes(url.hostname)) return false;

    // 解析 DNS，確保不是指向內部 IP
    const addresses = await dns.resolve4(url.hostname);
    for (const addr of addresses) {
      if (addr.startsWith('10.') ||
          addr.startsWith('172.16.') ||
          addr.startsWith('192.168.') ||
          addr.startsWith('127.') ||
          addr.startsWith('169.254.')) {
        return false;
      }
    }

    return true;
  } catch {
    return false;
  }
}

// 使用
app.post('/api/preview', async (req, res) => {
  const { url } = req.body;
  if (!await isUrlSafe(url)) {
    return res.status(400).json({ error: '不允許的 URL' });
  }
  // 安全地 fetch...
});
```

## 2. XSS 防護

> XSS（Cross-Site Scripting）是最常見的前端安全漏洞。攻擊者注入惡意 JavaScript 到網頁，竊取用戶的 cookie、session、個資。

### XSS 三種類型

```
Stored XSS（儲存型）
  攻擊者把惡意腳本存到資料庫（留言板、個人簡介）
  其他用戶瀏覽時就會執行
  危害最大

Reflected XSS（反射型）
  惡意腳本放在 URL 參數中
  用戶點了惡意連結就會觸發
  常見於搜尋結果頁面

DOM-based XSS（DOM 型）
  純前端 JavaScript 操作 DOM 時引入惡意內容
  不經過伺服器
  常見於 SPA 應用
```

### 2.1 React 的自動防護

React 預設會對 JSX 中的變數做 escape，防止大部分 XSS。

```tsx
// 安全：React 會自動 escape
function Comment({ text }: { text: string }) {
  return <p>{text}</p>;
  // 如果 text 是 "<script>alert('xss')</script>"
  // React 會輸出 &lt;script&gt;alert('xss')&lt;/script&gt;
  // 不會執行腳本
}

// ⚠️ 危險：dangerouslySetInnerHTML 繞過自動防護
function UnsafeComment({ html }: { html: string }) {
  return <div dangerouslySetInnerHTML={{ __html: html }} />;
  // 這會直接插入 HTML，必須先 sanitize！
}
```

**React 中仍然可能 XSS 的地方**：
```tsx
// 1. dangerouslySetInnerHTML — 必須 sanitize
<div dangerouslySetInnerHTML={{ __html: userContent }} />

// 2. href 中的 javascript: 協定
<a href={userInput}>連結</a>
// 如果 userInput 是 "javascript:alert('xss')" → 危險

// 3. eval / Function 構造器
eval(userInput);          // 絕對不要
new Function(userInput);  // 絕對不要

// 4. 動態 style（CSS 注入）
<div style={{ background: userInput }} />
// 如果 userInput 含有 expression() 或 url() → 可能危險
```

### 2.2 DOMPurify（HTML 淨化）

當你**必須**渲染使用者提供的 HTML（例如富文本編輯器內容），用 DOMPurify。

```bash
npm install dompurify
npm install --save-dev @types/dompurify  # TypeScript
```

```tsx
import DOMPurify from 'dompurify';

// 基本用法
function SafeHtmlRenderer({ html }: { html: string }) {
  const cleanHtml = DOMPurify.sanitize(html);
  return <div dangerouslySetInnerHTML={{ __html: cleanHtml }} />;
}

// 嚴格模式 — 只允許特定標籤和屬性
function StrictHtmlRenderer({ html }: { html: string }) {
  const cleanHtml = DOMPurify.sanitize(html, {
    ALLOWED_TAGS: ['b', 'i', 'u', 'em', 'strong', 'a', 'p', 'br', 'ul', 'ol', 'li', 'h1', 'h2', 'h3'],
    ALLOWED_ATTR: ['href', 'title', 'target'],
    ALLOW_DATA_ATTR: false,
  });
  return <div dangerouslySetInnerHTML={{ __html: cleanHtml }} />;
}

// 移除所有 HTML，只留純文字
const plainText = DOMPurify.sanitize(userInput, { ALLOWED_TAGS: [] });
```

### 2.3 後端 Escape / Sanitize

```typescript
// 後端 — 在儲存前 sanitize（Node.js）
import createDOMPurify from 'dompurify';
import { JSDOM } from 'jsdom';

const window = new JSDOM('').window;
const DOMPurify = createDOMPurify(window as any);

function sanitizeInput(input: string): string {
  return DOMPurify.sanitize(input, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'br'],
    ALLOWED_ATTR: ['href'],
  });
}

// 或使用 xss 套件
import xss from 'xss';

const cleanHtml = xss(userInput, {
  whiteList: {
    a: ['href', 'title', 'target'],
    b: [],
    i: [],
    p: [],
    br: [],
  },
});
```

```typescript
// 手動 escape（當你只需要純文字時）
function escapeHtml(str: string): string {
  const map: Record<string, string> = {
    '&': '&amp;',
    '<': '&lt;',
    '>': '&gt;',
    '"': '&quot;',
    "'": '&#039;',
  };
  return str.replace(/[&<>"']/g, (m) => map[m]);
}
```

### 2.4 Content Security Policy (CSP)

CSP 是最強的 XSS 防線，即使有 XSS 漏洞，CSP 也能阻止腳本執行。

```typescript
// Express.js — 用 Helmet 設定 CSP
import helmet from 'helmet';

app.use(
  helmet.contentSecurityPolicy({
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],          // 只允許自己的腳本
      styleSrc: ["'self'", "'unsafe-inline'"],  // 允許 inline style（CSS-in-JS 需要）
      imgSrc: ["'self'", 'data:', 'https:'],    // 圖片來源
      connectSrc: ["'self'", 'https://api.example.com'],  // AJAX 請求
      fontSrc: ["'self'", 'https://fonts.gstatic.com'],
      objectSrc: ["'none'"],          // 禁止 Flash 等外掛
      upgradeInsecureRequests: [],    // HTTP 自動升級 HTTPS
    },
  })
);
```

```html
<!-- 或直接在 HTML meta 標籤設定 -->
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:;">
```

**CSP 常見問題**：
| 問題 | 解決方案 |
|------|---------|
| Google Analytics 被擋 | `scriptSrc` 加 `https://www.googletagmanager.com` |
| Google Fonts 被擋 | `styleSrc` 加 `https://fonts.googleapis.com`，`fontSrc` 加 `https://fonts.gstatic.com` |
| inline style 被擋 | 加 `'unsafe-inline'`（或用 nonce） |
| inline script 被擋 | 不要用 inline script，改用外部檔案。如果必須用，使用 nonce |
| 圖片 CDN 被擋 | `imgSrc` 加上 CDN 網域 |

**CSP Nonce（進階）**：
```typescript
import crypto from 'crypto';

app.use((req, res, next) => {
  // 每次請求產生新的 nonce
  res.locals.nonce = crypto.randomBytes(16).toString('base64');
  next();
});

app.use(
  helmet.contentSecurityPolicy({
    directives: {
      scriptSrc: ["'self'", (req, res) => `'nonce-${(res as any).locals.nonce}'`],
    },
  })
);
```

```html
<!-- 只有帶正確 nonce 的 script 才能執行 -->
<script nonce="abc123随机值">
  // 這個可以執行
</script>
<script>
  // 這個被 CSP 擋掉
</script>
```

## 3. SQL 注入防護

> SQL 注入讓攻擊者修改你的 SQL 查詢，讀取、修改、刪除資料庫中的任何資料。

### 攻擊範例

```
// 有漏洞的代碼
const query = `SELECT * FROM users WHERE email = '${email}' AND password = '${password}'`;

// 攻擊者輸入：
// email: admin@example.com
// password: ' OR '1'='1

// 實際執行的 SQL：
SELECT * FROM users WHERE email = 'admin@example.com' AND password = '' OR '1'='1'
// → 永遠為 true，攻擊者登入成功
```

### 3.1 Parameterized Queries（參數化查詢）

```typescript
// PostgreSQL (pg 套件) — 正確做法
import { Pool } from 'pg';
const pool = new Pool();

// ✅ 參數化查詢 — $1, $2 是參數佔位符
async function getUser(email: string, password: string) {
  const result = await pool.query(
    'SELECT * FROM users WHERE email = $1 AND password_hash = $2',
    [email, passwordHash]
  );
  return result.rows[0];
}

// ✅ INSERT
```

## 4. CSRF 防護

> CSRF（Cross-Site Request Forgery）讓攻擊者利用已登入使用者的身份，發出未授權的請求。

### 攻擊原理

```
1. 使用者登入了 bank.com（瀏覽器存有 session cookie）
2. 使用者瀏覽惡意網站 evil.com
3. evil.com 中有一個隱藏表單，自動提交到 bank.com：
   <form action="https://bank.com/transfer" method="POST">
     <input type="hidden" name="to" value="attacker" />
     <input type="hidden" name="amount" value="1000000" />
   </form>
   <script>document.forms[0].submit();</script>
4. 瀏覽器會自動帶上 bank.com 的 cookie → 轉帳成功
```

### 4.1 CSRF Token

```typescript
// Express.js — csurf 套件（經典做法）
// 注意：csurf 已被棄用，推薦用 csrf-csrf 或 csrf-sync
import { doubleCsrf } from 'csrf-csrf';

const { doubleCsrfProtection, generateToken } = doubleCsrf({
  getSecret: () => process.env.CSRF_SECRET!,
  cookieName: '__Host-psifi.x-csrf-token',
  cookieOptions: {
    httpOnly: true,
    sameSite: 'strict',
    secure: true,
    path: '/',
  },
  size: 64,
  getTokenFromRequest: (req) => req.headers['x-csrf-token'] as string,
});

// 取得 token 的端點
app.get('/api/csrf-token', (req, res) => {
  const token = generateToken(req, res);
  res.json({ token });
});

// 保護 POST/PUT/DELETE 端點
app.use('/api', doubleCsrfProtection);
```

```tsx
// React 前端 — 每次請求帶上 CSRF token
// 1. 先取得 token
const response = await fetch('/api/csrf-token');
const { token } = await response.json();

// 2. 之後的請求帶上 token
await fetch('/api/transfer', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-CSRF-Token': token,
  },
  body: JSON.stringify({ to: 'recipient', amount: 100 }),
});
```

### 4.2 SameSite Cookie（現代瀏覽器的最佳防線）

```typescript
// Express.js — 設定 cookie 的 SameSite 屬性
app.use(session({
  secret: process.env.SESSION_SECRET!,
  cookie: {
    httpOnly: true,       // JavaScript 無法讀取 cookie
    secure: true,         // 只在 HTTPS 傳送
    sameSite: 'strict',   // 跨站請求不帶 cookie（最嚴格）
    maxAge: 3600000,      // 1 小時過期
  },
}));
```

**SameSite 三個值的差別**：
| 值 | 跨站 GET | 跨站 POST | 適用場景 |
|---|----------|----------|---------|
| `Strict` | 不帶 cookie | 不帶 cookie | 銀行、後台管理（最安全） |
| `Lax` | 帶 cookie | 不帶 cookie | 一般網站（Chrome 預設值） |
| `None` | 帶 cookie | 帶 cookie | 第三方服務（需搭配 `Secure`） |

### 4.3 Double Submit Cookie

```typescript
// 不需要 server-side state 的 CSRF 防護
import crypto from 'crypto';

// 設定 CSRF cookie
app.use((req, res, next) => {
  if (!req.cookies['csrf-token']) {
    const token = crypto.randomBytes(32).toString('hex');
    res.cookie('csrf-token', token, {
      httpOnly: false,  // 前端需要讀取
      secure: true,
      sameSite: 'strict',
    });
  }
  next();
});

// 驗證：cookie 中的 token 和 header 中的 token 必須一致
function csrfCheck(req: Request, res: Response, next: NextFunction) {
  if (['POST', 'PUT', 'DELETE', 'PATCH'].includes(req.method)) {
    const cookieToken = req.cookies['csrf-token'];
    const headerToken = req.headers['x-csrf-token'];
    if (!cookieToken || cookieToken !== headerToken) {
      return res.status(403).json({ error: 'CSRF token 驗證失敗' });
    }
  }
  next();
}
```

### 4.4 SPA / API 場景的 CSRF

```
如果你的前後端是完全分離的（React + API），並且：
- 認證用 JWT 放在 Authorization header（不是 cookie）
- 所有 API 都用 JSON body

那你天生就不容易被 CSRF 攻擊，因為：
1. 攻擊者無法用 <form> 送 Authorization header
2. 攻擊者無法用 <form> 送 JSON content-type（只能送 form-urlencoded）

但如果 JWT 存在 cookie 裡，就仍然需要 CSRF 防護。
```

```typescript
function isPasswordStrong(password: string): boolean {
  const rules = [
    { test: (p: string) => p.length >= 8, msg: '至少 8 個字元' },
    { test: (p: string) => p.length <= 128, msg: '最多 128 個字元' },
    { test: (p: string) => /[a-z]/.test(p), msg: '至少一個小寫字母' },
    { test: (p: string) => /[A-Z]/.test(p), msg: '至少一個大寫字母' },
    { test: (p: string) => /[0-9]/.test(p), msg: '至少一個數字' },
    { test: (p: string) => /[^a-zA-Z0-9]/.test(p), msg: '至少一個特殊字元' },
  ];

  const failures = rules.filter(r => !r.test(password));
  return failures.length === 0;
}

// 檢查常見弱密碼（可用 have-i-been-pwned API）
async function isPasswordBreached(password: string): Promise<boolean> {
  const crypto = require('crypto');
  const hash = crypto.createHash('sha1').update(password).digest('hex').toUpperCase();
  const prefix = hash.substring(0, 5);
  const suffix = hash.substring(5);

  const response = await fetch(`https://api.pwnedpasswords.com/range/${prefix}`);
  const text = await response.text();
  return text.includes(suffix);
}
```

### 5.5 JWT 安全

```typescript
import jwt from 'jsonwebtoken';

const JWT_SECRET = process.env.JWT_SECRET!; // 至少 256 bits
const JWT_EXPIRY = '1h';                    // 短期有效
const REFRESH_TOKEN_EXPIRY = '7d';          // Refresh token 長一點

// 產生 Access Token + Refresh Token
function generateTokens(user: { id: string; email: string; role: string }) {
  const accessToken = jwt.sign(
    { sub: user.id, email: user.email, role: user.role },
    JWT_SECRET,
    { expiresIn: JWT_EXPIRY, algorithm: 'HS256' }
  );

  const refreshToken = jwt.sign(
    { sub: user.id, type: 'refresh' },
    JWT_SECRET,
    { expiresIn: REFRESH_TOKEN_EXPIRY }
  );

  return { accessToken, refreshToken };
}

// 驗證 JWT 中間件
function authenticate(req: Request, res: Response, next: NextFunction) {
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: '未提供認證 token' });
  }

  try {
    const token = authHeader.slice(7);
    const payload = jwt.verify(token, JWT_SECRET) as jwt.JwtPayload;
    req.user = { id: payload.sub!, email: payload.email, role: payload.role };
    next();
  } catch (err) {
    if (err instanceof jwt.TokenExpiredError) {
      return res.status(401).json({ error: 'Token 已過期', code: 'TOKEN_EXPIRED' });
    }
    return res.status(401).json({ error: 'Token 無效' });
  }
}
```

**JWT 安全要點**：
- [ ] Secret 至少 256 bits，從環境變數讀取
- [ ] Access token 有效期短（15 分鐘 ~ 1 小時）
- [ ] Refresh token 存在 httpOnly cookie 中
- [ ] Refresh token 可以被撤銷（存在資料庫中）
- [ ] 不要在 JWT payload 放敏感資料（JWT 只是 base64，不是加密）
- [ ] 使用 RS256（非對稱）比 HS256（對稱）更安全（適合微服務）

## 7. CORS 設定

> CORS（Cross-Origin Resource Sharing）控制哪些網域可以從瀏覽器呼叫你的 API。

### 7.1 什麼時候需要 CORS

```
需要 CORS 的情況：
  前端 https://app.example.com 呼叫 API https://api.example.com
  → 不同子網域 = 跨源 = 需要 CORS

不需要 CORS 的情況：
  前端和 API 在同一個網域（https://example.com/api）
  → 同源 = 不需要 CORS

  伺服器對伺服器的請求（Node.js 呼叫另一個 API）
  → CORS 是瀏覽器的限制，伺服器不受影響
```

### 7.2 Express.js CORS 設定

| 錯誤訊息 | 原因 | 解法 |
|---------|------|------|
| `No 'Access-Control-Allow-Origin' header` | 後端沒設 CORS | 加 cors 中間件 |
| `The value of the 'Access-Control-Allow-Origin' header must not be the wildcard '*'` | `credentials: true` 時用了 `*` | 改成明確的網域列表 |
| `Request header field X-Custom-Header is not allowed` | 自訂 header 沒在 allowedHeaders | 把 header 名稱加到 allowedHeaders |
| `Method PUT is not allowed` | PUT 不在 methods 列表 | 加 'PUT' 到 methods |
| Preflight (OPTIONS) 被擋 | 伺服器沒處理 OPTIONS 請求 | cors 中間件會自動處理，確認中間件順序正確 |

```typescript
import multer from 'multer';
import path from 'path';
import crypto from 'crypto';
import { fileTypeFromBuffer } from 'file-type';

// 允許的檔案類型
const ALLOWED_MIME_TYPES = [
  'image/jpeg',
  'image/png',
  'image/gif',
  'image/webp',
  'application/pdf',
] as const;

const ALLOWED_EXTENSIONS = ['.jpg', '.jpeg', '.png', '.gif', '.webp', '.pdf'];

const MAX_FILE_SIZE = 5 * 1024 * 1024; // 5MB

// 1. Multer 設定 — 檔名隨機化、大小限制
const storage = multer.diskStorage({
  destination: (req, file, cb) => {
    cb(null, '/var/uploads/tmp'); // 暫存目錄，不要放在 web root
  },
  filename: (req, file, cb) => {
    // 隨機檔名，防止路徑穿越攻擊
    const randomName = crypto.randomBytes(16).toString('hex');
    const ext = path.extname(file.originalname).toLowerCase();
    cb(null, `${randomName}${ext}`);
  },
});

const upload = multer({
  storage,
  limits: {
    fileSize: MAX_FILE_SIZE,
    files: 5,  // 最多 5 個檔案
  },
  fileFilter: (req, file, cb) => {
    // 檢查副檔名
    const ext = path.extname(file.originalname).toLowerCase();
    if (!ALLOWED_EXTENSIONS.includes(ext)) {
      return cb(new Error(`不允許的檔案類型: ${ext}`));
    }

    // 檢查 MIME type
    if (!ALLOWED_MIME_TYPES.includes(file.mimetype as any)) {
      return cb(new Error(`不允許的 MIME type: ${file.mimetype}`));
    }

    cb(null, true);
  },
});
```

### 8.2 Magic Bytes 驗證（不要只信副檔名）

```typescript
import { fileTypeFromFile } from 'file-type';
import fs from 'fs/promises';

// 上傳後的二次驗證
async function validateUploadedFile(filePath: string): Promise<boolean> {
  // 用 magic bytes 判斷真實檔案類型
  const type = await fileTypeFromFile(filePath);

  if (!type) {
    // 無法辨識的檔案類型 → 刪除
    await fs.unlink(filePath);
    return false;
  }

  if (!ALLOWED_MIME_TYPES.includes(type.mime as any)) {
    // 真實類型不在允許清單 → 刪除
    // 例如：.jpg 副檔名但實際是 .exe
    await fs.unlink(filePath);
    return false;
  }

  return true;
}

// 路由
app.post('/api/upload', authenticate, upload.single('file'), async (req, res) => {
  if (!req.file) {
    return res.status(400).json({ error: '未提供檔案' });
  }

  // Magic bytes 驗證
  const isValid = await validateUploadedFile(req.file.path);
  if (!isValid) {
    return res.status(400).json({ error: '檔案類型不合法' });
  }

  // 移到正式目錄
  const finalPath = path.join('/var/uploads/files', req.file.filename);
  await fs.rename(req.file.path, finalPath);

  res.json({
    url: `/files/${req.file.filename}`,
    originalName: req.file.originalname,
    size: req.file.size,
  });
});
```

### 9.2 API Key 管理

```typescript
// API Key 驗證中間件
function validateApiKey(req: Request, res: Response, next: NextFunction) {
  const apiKey = req.headers['x-api-key'] as string
    || req.headers.authorization?.replace('Bearer ', '');

  if (!apiKey) {
    return res.status(401).json({ error: '未提供 API Key' });
  }

  // 用 timing-safe 比較，防止 timing attack
  const crypto = require('crypto');
  const validKeys = process.env.API_KEYS!.split(',');

  const isValid = validKeys.some(key => {
    try {
      return crypto.timingSafeEqual(
        Buffer.from(apiKey),
        Buffer.from(key.trim())
      );
    } catch {
      return false;
    }
  });

  if (!isValid) {
    return res.status(401).json({ error: 'API Key 無效' });
  }

  next();
}
```

- 不同 key 有不同權限（讀取 / 寫入 / 管理）

### 11.4 漏洞管理流程

```
發現漏洞
    │
    ├── Critical / High → 24 小時內修復
    │    ├── 有修復版本 → npm audit fix 或手動升級
    │    └── 沒有修復版本 → 找替代套件或加 workaround
```

### 11.5 lockfile 的重要性

```
package-lock.json（npm）/ yarn.lock（yarn）/ pnpm-lock.yaml（pnpm）

為什麼重要：
1. 鎖定確切版本 — 每台機器安裝一樣的版本
2. 防止供應鏈攻擊 — 有人偷偷發了惡意版本，lockfile 不會自動升級
3. 可重現的建置 — CI/CD 和本機環境一致

必做：
  [x] lockfile 一定要 commit 到 git
  [x] CI/CD 用 npm ci（不是 npm install）
  [x] 不要手動修改 lockfile
  [x] 不要把 lockfile 加到 .gitignore
```

```
│                                         │

│                                         │
```

### 應急指令

```bash
# 1. Nginx 快速切換維護模式
cat > /etc/nginx/conf.d/maintenance.conf << 'EOF'

    listen 80;
    listen 443 ssl;
    server_name example.com;

    # SSL 憑證（如果有的話）

    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    index index.html;

    location / {
        return 503;
    }

    error_page 503 @maintenance;
    location @maintenance {
        rewrite ^(.*)$ /index.html break;
    }
}

grep -rl "passthru" /var/www/html --include="*.php"
grep -rl "shell_exec" /var/www/html --include="*.php"

# 4. 比對核心檔案完整性（WordPress）
wp core verify-checksums
```

## 13. SSL/TLS 最佳實踐

### 13.1 Let's Encrypt 免費憑證

```bash
# 安裝 Certbot
sudo apt install certbot python3-certbot-nginx

# 取得憑證（自動設定 Nginx）
sudo certbot --nginx -d example.com -d www.example.com

# 取得憑證（手動設定）
sudo certbot certonly --webroot -w /var/www/html -d example.com

# 測試自動更新
sudo certbot renew --dry-run

# 自動更新（Certbot 安裝後通常自動設定 cron/systemd timer）
# 手動加 cron：
# 0 3 * * * certbot renew --quiet --post-hook "nginx -s reload"
```

### 13.2 Nginx SSL 最佳設定

```nginx
server {
    listen 443 ssl http2;
    server_name example.com;

    # 憑證
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # TLS 版本 — 只啟用 TLS 1.2 和 1.3
    ssl_protocols TLSv1.2 TLSv1.3;

    # 加密套件 — 強加密優先
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;  # TLS 1.3 時讓 client 選

    # OCSP Stapling — 加速 SSL 握手
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 8.8.8.8 8.8.4.4 valid=300s;
    resolver_timeout 5s;

    # Session 快取 — 減少重複握手
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;

    # HSTS
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;

    # DH 參數（如果用 DHE 加密套件）
    # 產生：openssl dhparam -out /etc/nginx/dhparam.pem 2048
    ssl_dhparam /etc/nginx/dhparam.pem;
}

# HTTP 自動跳轉 HTTPS
server {
    listen 80;
    server_name example.com www.example.com;
    return 301 https://$host$request_uri;
}
```

### 13.3 SSL 測試

```bash
# 用 SSL Labs 測試（目標 A+ 評分）
# https://www.ssllabs.com/ssltest/

# 用 testssl.sh 本機測試
git clone --depth 1 https://github.com/drwetter/testssl.sh.git
cd testssl.sh
./testssl.sh example.com

# 用 openssl 快速檢查
openssl s_client -connect example.com:443 -tls1_2
openssl s_client -connect example.com:443 -tls1_3

# 檢查憑證到期日
echo | openssl s_client -connect example.com:443 2>/dev/null | openssl x509 -noout -dates
```

### 13.4 SSL/TLS 清單

```
必做：
  [x] 使用 TLS 1.2 以上（停用 TLS 1.0/1.1 和 SSL）
  [x] 使用強加密套件（AES-GCM、ChaCha20-Poly1305）
  [x] 啟用 HSTS（含 preload）
  [x] HTTP 自動跳轉 HTTPS（301）
  [x] 憑證自動更新（Let's Encrypt + cron）
  [x] 啟用 OCSP Stapling
  [x] SSL Labs 測試 A 或 A+

不要做：
  [ ] 不要用自簽憑證（除了開發環境）
  [ ] 不要用 SSL 2.0/3.0 或 TLS 1.0/1.1
  [ ] 不要用弱加密（RC4、DES、3DES）
  [ ] 不要在 mixed content 頁面（HTTPS 頁面載入 HTTP 資源）
  [ ] 不要忽略憑證到期提醒
```

## 14. 安全檢查工具

### 14.1 OWASP ZAP（Zed Attack Proxy）

> 免費開源的 Web 應用安全掃描工具，適合滲透測試。

```bash
# Docker 快速啟動
docker run -u zap -p 8080:8080 -p 8090:8090 \
  ghcr.io/zaproxy/zaproxy:stable \
  zap-webswing.sh

# 自動掃描（Baseline Scan — 被動掃描，不會攻擊網站）
docker run --rm ghcr.io/zaproxy/zaproxy:stable \
  zap-baseline.py -t https://example.com -r report.html

# 完整掃描（Full Scan — 含主動攻擊，只在測試環境使用）
docker run --rm ghcr.io/zaproxy/zaproxy:stable \
  zap-full-scan.py -t https://staging.example.com -r report.html

# API 掃描（OpenAPI/Swagger）
docker run --rm ghcr.io/zaproxy/zaproxy:stable \
  zap-api-scan.py -t https://api.example.com/openapi.json -f openapi -r report.html
```

### 14.2 Nikto

> Web 伺服器安全掃描工具，檢查已知漏洞、不安全的設定。

```bash
# 安裝
sudo apt install nikto
# 或 Mac
brew install nikto

# 基本掃描
nikto -h https://example.com

# 指定 port
nikto -h example.com -p 443 -ssl

# 只掃特定項目
nikto -h example.com -Tuning 1234

# 輸出 HTML 報告
```

**Nikto 常見發現與修復**：
| Nikto 發現 | 修復方法 |
|-----------|---------|
| Server 洩漏版本號 | Nginx: `server_tokens off;` |
| X-Powered-By header | Express: `app.disable('x-powered-by')` |
| 缺少 X-Frame-Options | 加 `X-Frame-Options: DENY` header |
| 缺少 X-Content-Type-Options | 加 `X-Content-Type-Options: nosniff` header |
| 目錄列表 | Nginx: `autoindex off;` |
| 存在 /phpinfo.php | 刪除這個檔案 |
