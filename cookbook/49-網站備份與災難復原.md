## 一、備份策略（3-2-1 原則）

### 核心原則

```
3 份備份 — 至少保存 3 份資料副本（1 份正式 + 2 份備份）
2 種媒介 — 至少使用 2 種不同的儲存媒介（雲端 + 本地硬碟）
1 份異地 — 至少 1 份存放在不同地理位置（不同資料中心/不同國家）
```

### 實際應用範例

| 備份層級 | 儲存位置 | 頻率 | 保留期限 |
|----------|----------|------|----------|
| 第 1 份（正式環境） | 主機/VPS | 即時 | - |
| 第 2 份（雲端備份） | AWS S3 / Google Cloud Storage | 每日 | 30 天 |
| 第 3 份（異地備份） | Backblaze B2 / 外接硬碟 | 每週 | 90 天 |

### 各種專案建議備份頻率

| 專案類型 | 完整備份 | 增量備份 | 資料庫備份 |
|----------|----------|----------|------------|
| 形象官網（少更新） | 每週 | 每日 | 每日 |
| 電商網站 | 每日 | 每小時 | 每小時 |
| 部落格/內容網站 | 每週 | 每日 | 每日 |
| SaaS 應用程式 | 每日 | 每小時 | 即時（WAL） |
| 客戶資料密集型 | 每日 | 每 6 小時 | 每小時 |

## 二、備份類型

### 完整備份（Full Backup）

**什麼是**：複製所有資料，不管有沒有變更過。

**優點**：恢復最快、最簡單
**缺點**：佔空間、執行時間長

**適用時機**：
- 專案首次上線後
- 重大更新/版本升級前
- 每週定期備份

```bash
# 完整備份範例 — 整站打包
tar -czf backup-full-$(date +%Y%m%d).tar.gz \
  --exclude='node_modules' \
  --exclude='.next' \
  --exclude='.cache' \
  /var/www/mysite/
```

### 增量備份（Incremental Backup）

**什麼是**：只備份上次備份後「有變更」的檔案。

**優點**：最省空間、最快
**缺點**：恢復時要從完整備份 + 所有增量逐步還原

**適用時機**：
- 每日備份
- 頻繁更新的網站

```bash
# rsync 增量備份（只傳變更的檔案）
rsync -avz --delete \
  --exclude='node_modules' \
  --exclude='.next' \
  /var/www/mysite/ \
  /backup/incremental/mysite/

# 搭配 --link-dest 做硬連結增量備份（省空間又能獨立恢復）
LATEST=$(ls -td /backup/daily/mysite-* 2>/dev/null | head -1)
DEST="/backup/daily/mysite-$(date +%Y%m%d)"

rsync -avz --delete \
  --exclude='node_modules' \
  ${LATEST:+--link-dest="$LATEST"} \
  /var/www/mysite/ \
  "$DEST/"
```

### 差異備份（Differential Backup）

**什麼是**：備份自上次「完整備份」以來所有變更的檔案。

**優點**：恢復只需完整備份 + 最新差異備份（比增量快）
**缺點**：比增量備份佔空間（每次都對比完整備份）

**適用時機**：
- 折衷方案 — 恢復速度和空間的平衡
- 中型專案

```bash
# 用 find 找出自上次完整備份後變更的檔案
LAST_FULL="/backup/full/mysite-20260301.tar.gz"
LAST_FULL_DATE="2026-03-01"

find /var/www/mysite/ \
  -newer "$LAST_FULL" \
  -not -path '*/node_modules/*' \
  -not -path '*/.next/*' \
  -print0 | tar -czf "backup-diff-$(date +%Y%m%d).tar.gz" --null -T -
```

### 比較總結

| | 完整備份 | 增量備份 | 差異備份 |
|---|---------|---------|---------|
| 備份速度 | 慢 | 最快 | 中等 |
| 備份大小 | 最大 | 最小 | 中等 |
| 恢復速度 | 最快 | 最慢 | 中等 |
| 恢復複雜度 | 最簡單 | 需全部增量 | 需完整+最新差異 |
| 建議頻率 | 每週 | 每日 | 每日 |

## 三、WordPress 備份

### 3.1 UpdraftPlus 設定（推薦方案）

**安裝**：WordPress 後台 → 外掛 → 安裝新外掛 → 搜尋「UpdraftPlus」

**設定自動排程 + 雲端儲存**：

```
後台路徑：設定 → UpdraftPlus 備份

【排程設定】
  檔案備份排程：每日，保留 14 份
  資料庫備份排程：每日，保留 30 份

【遠端儲存】（擇一設定）
  ✅ Google Drive — 最簡單，免費 15GB
  ✅ Amazon S3 — 專業方案，便宜又穩定
  ✅ Dropbox — 客戶最容易理解

【Google Drive 設定步驟】
  1. 選擇「Google Drive」
  2. 點擊「使用 Google 帳號登入」
  3. 授權 UpdraftPlus 存取
  4. 資料夾名稱：UpdraftPlus-客戶網站名稱
  5. 儲存設定

【建議排除的目錄】
  ✅ 排除 /wp-content/cache/
  ✅ 排除 /wp-content/updraft/（備份本身）
  ✅ 排除 /wp-content/upgrade/（更新暫存）
```

**用 WP-CLI 觸發 UpdraftPlus 備份**：

```bash
# 手動觸發備份（需安裝 WP-CLI）
wp updraftplus backup --include=db,plugins,themes,uploads,others

# 查看備份歷史
wp updraftplus list_backups

# 恢復指定備份
wp updraftplus restore <backup_id> --components=db,plugins,themes,uploads
```

### 3.2 All-in-One WP Migration

**適用場景**：整站搬家、本機測試環境、客戶交接

```
安裝：外掛 → 搜尋「All-in-One WP Migration」

【匯出】
  All-in-One WP Migration → 匯出 → 檔案
  → 自動產生 .wpress 檔案（包含資料庫 + 檔案 + 外掛 + 主題）

【匯入】
  All-in-One WP Migration → 匯入 → 拖曳 .wpress 檔案
  → 自動覆蓋整站（含搜尋取代網址）

【注意事項】
  - 免費版限制 512MB（付費版無限制）
  - 大型網站：先排除 uploads 資料夾，單獨用 rsync 搬
  - 匯入後務必重設固定網址：設定 → 固定網址 → 儲存
```

**提高上傳限制**（免費版常見問題）：

```php
// wp-config.php 加入
@ini_set('upload_max_filesize', '512M');
@ini_set('post_max_size', '512M');
@ini_set('memory_limit', '512M');
@ini_set('max_execution_time', '300');
@ini_set('max_input_time', '300');
```

### 3.3 WP-CLI 備份腳本

```bash
#!/bin/bash
# wp-backup.sh — WordPress 自動備份腳本
# 用法：bash wp-backup.sh /var/www/wordpress
# cron：0 3 * * * bash /opt/scripts/wp-backup.sh /var/www/wordpress

set -euo pipefail

# ===== 設定 =====
WP_PATH="${1:?請指定 WordPress 路徑}"
BACKUP_DIR="/backup/wordpress/$(basename "$WP_PATH")"
DATE=$(date +%Y%m%d-%H%M%S)
KEEP_DAYS=30
S3_BUCKET="s3://my-backups/wordpress"

mkdir -p "$BACKUP_DIR"

echo "[$(date)] 開始備份 $WP_PATH ..."

# 1. 資料庫備份
echo "  → 匯出資料庫..."
wp db export "$BACKUP_DIR/db-$DATE.sql" --path="$WP_PATH" --quiet
gzip "$BACKUP_DIR/db-$DATE.sql"
echo "  ✓ 資料庫備份完成：db-$DATE.sql.gz"

# 2. wp-content 備份（主題+外掛+上傳）
echo "  → 打包 wp-content..."
tar -czf "$BACKUP_DIR/wp-content-$DATE.tar.gz" \
  --exclude='cache' \
  --exclude='updraft' \
  --exclude='upgrade' \
  -C "$WP_PATH" wp-content/
echo "  ✓ wp-content 備份完成"

# 3. wp-config.php 備份
cp "$WP_PATH/wp-config.php" "$BACKUP_DIR/wp-config-$DATE.php"

# 4. 上傳到 S3（異地備份）
if command -v aws &>/dev/null; then
  echo "  → 上傳到 S3..."
  aws s3 cp "$BACKUP_DIR/db-$DATE.sql.gz" "$S3_BUCKET/db-$DATE.sql.gz"
  aws s3 cp "$BACKUP_DIR/wp-content-$DATE.tar.gz" "$S3_BUCKET/wp-content-$DATE.tar.gz"
  echo "  ✓ S3 上傳完成"
fi

# 5. 清理舊備份
echo "  → 清理 $KEEP_DAYS 天前的備份..."
find "$BACKUP_DIR" -type f -mtime +$KEEP_DAYS -delete
echo "  ✓ 舊備份已清理"

# 6. 驗證備份
DB_SIZE=$(stat -f%z "$BACKUP_DIR/db-$DATE.sql.gz" 2>/dev/null || stat -c%s "$BACKUP_DIR/db-$DATE.sql.gz")
WC_SIZE=$(stat -f%z "$BACKUP_DIR/wp-content-$DATE.tar.gz" 2>/dev/null || stat -c%s "$BACKUP_DIR/wp-content-$DATE.tar.gz")

if [ "$DB_SIZE" -lt 1000 ]; then
  echo "  ⚠ 警告：資料庫備份過小（${DB_SIZE} bytes），可能有問題！"
  exit 1
fi

echo "[$(date)] 備份完成！"
echo "  資料庫：$(numfmt --to=iec "$DB_SIZE" 2>/dev/null || echo "${DB_SIZE} bytes")"
echo "  wp-content：$(numfmt --to=iec "$WC_SIZE" 2>/dev/null || echo "${WC_SIZE} bytes")"
```

**Cron 排程設定**：

```bash
# 編輯 crontab
crontab -e

# 每天凌晨 3 點執行備份
0 3 * * * /opt/scripts/wp-backup.sh /var/www/client-site-a >> /var/log/wp-backup-a.log 2>&1
0 3 * * * /opt/scripts/wp-backup.sh /var/www/client-site-b >> /var/log/wp-backup-b.log 2>&1

# 每週日凌晨 2 點做完整備份（含 WordPress core）
0 2 * * 0 tar -czf /backup/full/site-a-full-$(date +\%Y\%m\%d).tar.gz --exclude='node_modules' /var/www/client-site-a/
```

### 3.4 手動備份（FTP + phpMyAdmin）

**步驟一：用 FTP 下載檔案**

```
1. 開啟 FileZilla（或任何 FTP 用戶端）
2. 連線到主機：
   - 主機：ftp.example.com
   - 使用者：你的 FTP 帳號
   - 密碼：你的 FTP 密碼
   - 埠：21（FTP）或 22（SFTP，建議用這個）

3. 進入網站根目錄（通常是 /public_html/ 或 /var/www/html/）

4. 下載以下重要資料夾/檔案：
   ├── wp-content/          ← 最重要！主題+外掛+上傳檔
   │   ├── themes/
   │   ├── plugins/
   │   └── uploads/
   ├── wp-config.php        ← 資料庫設定
   └── .htaccess            ← 網址重寫規則

5. 存到本地：~/Backups/客戶名稱/YYYY-MM-DD/
```

**步驟二：用 phpMyAdmin 匯出資料庫**

```
1. 登入 cPanel → phpMyAdmin（或直接開 phpMyAdmin 網址）

2. 左側選擇你的 WordPress 資料庫

3. 點擊上方「匯出」頁籤

4. 匯出方式：選「自訂」
   - 格式：SQL
   - 表格：全選
   - 輸出：✅ 壓縮 → gzip
   - SQL 選項：
     ✅ 加入 CREATE TABLE 語法
     ✅ 加入 DROP TABLE
     ✅ 完整的 INSERT 語法

5. 按「執行」→ 下載 .sql.gz 檔案

6. 存到本地：~/Backups/客戶名稱/YYYY-MM-DD/database.sql.gz
```

**手動恢復流程**：

```
1. 用 FTP 上傳 wp-content/ 回主機
2. 上傳 wp-config.php（注意修改資料庫設定）
3. phpMyAdmin → 匯入 → 選擇 .sql.gz 檔案
4. 開啟網站確認，重設固定網址
```

## 四、Node.js / Next.js 專案備份

### 4.1 備份四要素

```
Node.js 專案完整備份 = 4 樣東西

1. 代碼（Git）        → 推到 GitHub/GitLab，這是基本功
2. 資料庫              → pg_dump / mongodump 定期匯出
3. 環境變數（.env）    → 加密儲存，絕對不進 git
4. 上傳檔案（uploads） → rsync 同步到備份伺服器或 S3
```

### 4.2 完整自動備份腳本

```bash
#!/bin/bash
# node-backup.sh — Node.js/Next.js 專案自動備份
# 用法：bash node-backup.sh
# cron：0 2 * * * bash /opt/scripts/node-backup.sh >> /var/log/node-backup.log 2>&1

set -euo pipefail

# ===== 設定區 =====
PROJECT_NAME="my-nextjs-app"
PROJECT_DIR="/var/www/$PROJECT_NAME"
BACKUP_DIR="/backup/$PROJECT_NAME"
DATE=$(date +%Y%m%d-%H%M%S)
BACKUP_PATH="$BACKUP_DIR/$DATE"
KEEP_DAYS=30

# 資料庫
DB_HOST="localhost"
DB_PORT="5432"
DB_NAME="myapp_production"
DB_USER="myapp"

# S3（異地備份）
S3_BUCKET="s3://my-backups/$PROJECT_NAME"
S3_REGION="ap-northeast-1"

# 通知（備份完成/失敗時通知）
WEBHOOK_URL="https://hooks.slack.com/services/xxx/yyy/zzz"  # 或用 n8n webhook

# ===== 函式 =====
notify() {
  local status="$1" message="$2"
  curl -s -X POST "$WEBHOOK_URL" \
    -H "Content-Type: application/json" \
    -d "{\"text\":\"[$PROJECT_NAME] 備份${status}: ${message}\"}" \
    >/dev/null 2>&1 || true
}

cleanup_on_error() {
  echo "[ERROR] 備份失敗！清理暫存..."
  rm -rf "$BACKUP_PATH"
  notify "失敗" "$(date) — 請立即檢查！"
  exit 1
}
trap cleanup_on_error ERR

# ===== 開始備份 =====
mkdir -p "$BACKUP_PATH"
echo "========================================"
echo "[$(date)] 開始備份 $PROJECT_NAME"
echo "========================================"

# 1. 資料庫備份（PostgreSQL）
echo "[1/6] 備份資料庫..."
PGPASSWORD="${DB_PASSWORD:-}" pg_dump \
  -h "$DB_HOST" \
  -p "$DB_PORT" \
  -U "$DB_USER" \
  -d "$DB_NAME" \
  --format=custom \
  --compress=6 \
  --file="$BACKUP_PATH/database.dump"
echo "  ✓ 資料庫大小：$(du -sh "$BACKUP_PATH/database.dump" | cut -f1)"

# 2. 環境變數備份（加密）
echo "[2/6] 備份環境變數..."
if [ -f "$PROJECT_DIR/.env" ]; then
  # 用 openssl 加密 .env 檔案
  openssl enc -aes-256-cbc -salt -pbkdf2 \
    -in "$PROJECT_DIR/.env" \
    -out "$BACKUP_PATH/env.enc" \
    -pass pass:"${BACKUP_ENCRYPTION_KEY:-changeme}"
  echo "  ✓ .env 已加密備份"
else
  echo "  ! 找不到 .env，跳過"
fi

# 3. 上傳檔案備份
echo "[3/6] 備份上傳檔案..."
UPLOADS_DIR="$PROJECT_DIR/public/uploads"
if [ -d "$UPLOADS_DIR" ]; then
  tar -czf "$BACKUP_PATH/uploads.tar.gz" \
    -C "$PROJECT_DIR/public" uploads/
  echo "  ✓ uploads 大小：$(du -sh "$BACKUP_PATH/uploads.tar.gz" | cut -f1)"
else
  echo "  ! 無 uploads 目錄，跳過"
fi

# 4. package.json + lock 檔（記錄依賴版本）
echo "[4/6] 備份依賴設定..."
cp "$PROJECT_DIR/package.json" "$BACKUP_PATH/"
[ -f "$PROJECT_DIR/package-lock.json" ] && cp "$PROJECT_DIR/package-lock.json" "$BACKUP_PATH/"
[ -f "$PROJECT_DIR/yarn.lock" ] && cp "$PROJECT_DIR/yarn.lock" "$BACKUP_PATH/"
[ -f "$PROJECT_DIR/pnpm-lock.yaml" ] && cp "$PROJECT_DIR/pnpm-lock.yaml" "$BACKUP_PATH/"
echo "  ✓ 依賴設定已備份"

# 5. 壓縮整包
echo "[5/6] 壓縮備份..."
ARCHIVE="$BACKUP_DIR/$PROJECT_NAME-$DATE.tar.gz"
tar -czf "$ARCHIVE" -C "$BACKUP_DIR" "$DATE/"
rm -rf "$BACKUP_PATH"
echo "  ✓ 壓縮檔：$(du -sh "$ARCHIVE" | cut -f1)"

# 6. 上傳到 S3
echo "[6/6] 上傳到 S3..."
if command -v aws &>/dev/null; then
  aws s3 cp "$ARCHIVE" "$S3_BUCKET/$PROJECT_NAME-$DATE.tar.gz" \
    --region "$S3_REGION" \
    --storage-class STANDARD_IA
  echo "  ✓ S3 上傳完成"
else
  echo "  ! aws CLI 未安裝，跳過 S3 上傳"
fi

# 清理舊備份（本地）
echo "清理 ${KEEP_DAYS} 天前的本地備份..."
find "$BACKUP_DIR" -name "*.tar.gz" -type f -mtime +$KEEP_DAYS -delete

# 完成通知
FINAL_SIZE=$(du -sh "$ARCHIVE" | cut -f1)
echo "========================================"
echo "[$(date)] 備份完成！大小：$FINAL_SIZE"
echo "========================================"
notify "成功" "大小 $FINAL_SIZE — $(date +%Y-%m-%d)"
```

### 4.3 恢復腳本

```bash
#!/bin/bash
# node-restore.sh — 從備份恢復 Node.js 專案
# 用法：bash node-restore.sh /backup/my-nextjs-app/my-nextjs-app-20260305.tar.gz

set -euo pipefail

ARCHIVE="${1:?請指定備份檔案路徑}"
PROJECT_DIR="/var/www/my-nextjs-app"
DB_NAME="myapp_production"
DB_USER="myapp"

echo "[$(date)] 開始恢復..."

# 1. 解壓備份
TEMP_DIR=$(mktemp -d)
tar -xzf "$ARCHIVE" -C "$TEMP_DIR"
BACKUP_PATH=$(find "$TEMP_DIR" -maxdepth 1 -mindepth 1 -type d | head -1)

# 2. 恢復資料庫
echo "恢復資料庫..."
pg_restore \
  -U "$DB_USER" \
  -d "$DB_NAME" \
  --clean --if-exists \
  "$BACKUP_PATH/database.dump"
echo "  ✓ 資料庫恢復完成"

# 3. 恢復環境變數
if [ -f "$BACKUP_PATH/env.enc" ]; then
  echo "解密 .env..."
  openssl enc -aes-256-cbc -d -pbkdf2 \
    -in "$BACKUP_PATH/env.enc" \
    -out "$PROJECT_DIR/.env" \
    -pass pass:"${BACKUP_ENCRYPTION_KEY:-changeme}"
  echo "  ✓ .env 恢復完成"
fi

# 4. 恢復上傳檔案
if [ -f "$BACKUP_PATH/uploads.tar.gz" ]; then
  echo "恢復上傳檔案..."
  tar -xzf "$BACKUP_PATH/uploads.tar.gz" -C "$PROJECT_DIR/public/"
  echo "  ✓ uploads 恢復完成"
fi

# 5. 安裝依賴 + 重建
echo "安裝依賴..."
cd "$PROJECT_DIR"
npm ci
npm run build

# 6. 重啟服務
echo "重啟服務..."
pm2 restart "$PROJECT_DIR/ecosystem.config.js" || systemctl restart myapp

# 7. 驗證
sleep 3
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/api/health)
if [ "$HTTP_CODE" = "200" ]; then
  echo "✓ 恢復完成，服務正常運行！"
else
  echo "✗ 警告：服務回應 HTTP $HTTP_CODE，請手動檢查"
  exit 1
fi

# 清理
rm -rf "$TEMP_DIR"
```

## 五、資料庫備份

### 5.1 PostgreSQL

#### pg_dump（邏輯備份 — 最常用）

```bash
# 基本匯出（SQL 格式）
pg_dump -U postgres -d mydb > backup.sql

# 自訂格式（壓縮 + 可選擇性恢復）— 推薦
pg_dump -U postgres -d mydb \
  --format=custom \
  --compress=6 \
  --file=mydb-$(date +%Y%m%d).dump

# 只匯出特定 schema
pg_dump -U postgres -d mydb \
  --schema=public \
  --format=custom \
  --file=mydb-public.dump

# 只匯出結構（不含資料）
pg_dump -U postgres -d mydb --schema-only > schema.sql

# 只匯出資料（不含結構）
pg_dump -U postgres -d mydb --data-only > data.sql

# 匯出特定表格
pg_dump -U postgres -d mydb \
  -t users -t orders -t products \
  --format=custom \
  --file=selected-tables.dump
```

#### pg_restore（恢復）

```bash
# 從自訂格式恢復
pg_restore -U postgres -d mydb --clean --if-exists mydb-20260305.dump

# 恢復到新資料庫
createdb -U postgres mydb_restored
pg_restore -U postgres -d mydb_restored mydb-20260305.dump

# 只恢復特定表格
pg_restore -U postgres -d mydb -t users mydb-20260305.dump

# 從 SQL 格式恢復
psql -U postgres -d mydb < backup.sql
```

#### pg_basebackup（物理備份 — 大型資料庫）

```bash
# 完整的物理備份（含 WAL）
pg_basebackup \
  -h localhost \
  -U replication_user \
  -D /backup/pg-base-$(date +%Y%m%d) \
  -Ft -z -P \
  --wal-method=stream

# 參數說明：
#   -Ft  → tar 格式
#   -z   → gzip 壓縮
#   -P   → 顯示進度
#   --wal-method=stream → 串流 WAL（確保一致性）
```

#### WAL Archiving（持續歸檔 — SaaS 等級）

```ini
# postgresql.conf
wal_level = replica
archive_mode = on
archive_command = 'test ! -f /backup/wal/%f && cp %p /backup/wal/%f'
archive_timeout = 300  # 最多 5 分鐘歸檔一次
```

```bash
# 結合 pg_basebackup + WAL 做 PITR（Point-In-Time Recovery）
# 恢復到某個時間點
pg_restore --target-time="2026-03-05 14:30:00" ...
```

#### PostgreSQL 自動備份 cron

```bash
# /opt/scripts/pg-backup.sh
#!/bin/bash
set -euo pipefail

DB_NAME="${1:-mydb}"
BACKUP_DIR="/backup/postgres"
DATE=$(date +%Y%m%d-%H%M%S)
KEEP_DAYS=30

mkdir -p "$BACKUP_DIR"

# 備份
PGPASSWORD="${PGPASSWORD:-}" pg_dump \
  -h localhost -U postgres \
  -d "$DB_NAME" \
  --format=custom \
  --compress=9 \
  --file="$BACKUP_DIR/${DB_NAME}-${DATE}.dump"

# 驗證備份完整性
pg_restore --list "$BACKUP_DIR/${DB_NAME}-${DATE}.dump" > /dev/null 2>&1
if [ $? -eq 0 ]; then
  echo "[$(date)] ✓ ${DB_NAME} 備份成功：$(du -sh "$BACKUP_DIR/${DB_NAME}-${DATE}.dump" | cut -f1)"
else
  echo "[$(date)] ✗ ${DB_NAME} 備份驗證失敗！"
  exit 1
fi

# 清理
find "$BACKUP_DIR" -name "${DB_NAME}-*.dump" -mtime +$KEEP_DAYS -delete
```

### 每季一次恢復測試 SOP

```
╔═══════════════════════════════════════════════════╗
║   恢復演練 SOP（建議每季執行一次）               ║
║   預計時間：2-4 小時                              ║
╚═══════════════════════════════════════════════════╝
```

#### 演練流程

```bash
#!/bin/bash
# restore-drill.sh — 恢復演練腳本
# 用法：bash restore-drill.sh <project-name>

set -euo pipefail

PROJECT="${1:?請指定專案名稱}"
DATE=$(date +%Y%m%d)
LOG="/tmp/restore-drill-$PROJECT-$DATE.log"
PASS=0
FAIL=0

log() { echo "[$(date +%H:%M:%S)] $*" | tee -a "$LOG"; }
pass() { log "✓ PASS: $1"; ((PASS++)); }
fail() { log "✗ FAIL: $1"; ((FAIL++)); }

log "========================================"
log "恢復演練開始：$PROJECT"
log "========================================"

# ===== 測試 1：確認備份存在 =====
log "--- 測試 1：備份檔案檢查 ---"

# 檢查本地備份
LATEST_LOCAL=$(ls -t /backup/$PROJECT/*.dump 2>/dev/null | head -1)
if [ -n "$LATEST_LOCAL" ]; then
  BACKUP_AGE=$(( ($(date +%s) - $(stat -c %Y "$LATEST_LOCAL" 2>/dev/null || stat -f %m "$LATEST_LOCAL")) / 86400 ))
  if [ "$BACKUP_AGE" -le 1 ]; then
    pass "本地備份存在且小於 1 天（$BACKUP_AGE 天前）"
  else
    fail "本地備份太舊（$BACKUP_AGE 天前）"
  fi
else
  fail "找不到本地備份"
fi

# 檢查 S3 備份
if command -v aws &>/dev/null; then
  S3_COUNT=$(aws s3 ls "s3://my-backups/$PROJECT/" --recursive | wc -l)
  if [ "$S3_COUNT" -gt 0 ]; then
    pass "S3 備份存在（$S3_COUNT 個檔案）"
  else
    fail "S3 無備份檔案"
  fi
fi

# ===== 測試 2：備份完整性 =====
log "--- 測試 2：備份完整性驗證 ---"

# PostgreSQL dump 驗證
if [ -n "$LATEST_LOCAL" ]; then
  if pg_restore --list "$LATEST_LOCAL" > /dev/null 2>&1; then
    pass "PostgreSQL dump 結構正常"
  else
    fail "PostgreSQL dump 損壞"
  fi
fi

# ===== 測試 3：實際恢復到測試環境 =====
log "--- 測試 3：恢復到測試資料庫 ---"

TEST_DB="${PROJECT}_drill_test"

# 建立測試資料庫
dropdb -U postgres --if-exists "$TEST_DB" 2>/dev/null
createdb -U postgres "$TEST_DB"

# 恢復
if pg_restore -U postgres -d "$TEST_DB" "$LATEST_LOCAL" 2>/dev/null; then
  pass "資料庫恢復成功"

  # 驗證資料完整性
  TABLE_COUNT=$(psql -U postgres -d "$TEST_DB" -t -c "SELECT count(*) FROM information_schema.tables WHERE table_schema='public';")
  log "  表格數量：$TABLE_COUNT"

  if [ "$TABLE_COUNT" -gt 0 ]; then
    pass "恢復後有 $TABLE_COUNT 個表格"
  else
    fail "恢復後沒有表格"
  fi
else
  fail "資料庫恢復失敗"
fi

# 清理測試資料庫
dropdb -U postgres --if-exists "$TEST_DB"

# ===== 測試 4：檔案備份恢復 =====
log "--- 測試 4：檔案恢復測試 ---"

LATEST_ARCHIVE=$(ls -t /backup/$PROJECT/*.tar.gz 2>/dev/null | head -1)
if [ -n "$LATEST_ARCHIVE" ]; then
  TEMP_DIR=$(mktemp -d)
  if tar -xzf "$LATEST_ARCHIVE" -C "$TEMP_DIR" 2>/dev/null; then
    FILE_COUNT=$(find "$TEMP_DIR" -type f | wc -l)
    pass "檔案備份解壓成功（$FILE_COUNT 個檔案）"
  else
    fail "檔案備份解壓失敗"
  fi
  rm -rf "$TEMP_DIR"
fi

# ===== 測試 5：.env 解密 =====
log "--- 測試 5：環境變數備份 ---"
if [ -f "/backup/$PROJECT/env.enc" ] || find /backup/$PROJECT/ -name "env.enc" -print -quit 2>/dev/null | grep -q .; then
  pass "加密的 .env 備份存在"
else
  fail "找不到 .env 備份（如遺失需重建）"
fi

# ===== 總結 =====
log "========================================"
log "演練完成！通過：$PASS / 失敗：$FAIL"
log "報告：$LOG"
log "========================================"

if [ "$FAIL" -gt 0 ]; then
  log "⚠ 有失敗項目，請立即修復備份流程！"
  exit 1
else
  log "所有測試通過，備份系統正常。"
fi
```

```typescript
// server/src/routes/health.ts — 完整的健康檢查 API
import { Router, Request, Response } from 'express';
import { Pool } from 'pg';

const router = Router();

// 基本健康檢查（給 load balancer / UptimeRobot 用）
router.get('/api/health', (req: Request, res: Response) => {
  res.json({
    status: 'ok',
    timestamp: new Date().toISOString(),
    uptime: process.uptime(),
    version: process.env.npm_package_version || 'unknown',
  });
});

// 詳細健康檢查（給內部監控用，需認證）
router.get('/api/health/detailed', async (req: Request, res: Response) => {
  const checks: Record<string, any> = {};
  let overall = 'healthy';

  // 1. 資料庫連線
  try {
    const pool = new Pool({ connectionString: process.env.DATABASE_URL });
    const result = await pool.query('SELECT NOW() as time, pg_database_size(current_database()) as db_size');
    checks.database = {
      status: 'ok',
      responseTime: Date.now(),
      size: result.rows[0].db_size,
    };
    await pool.end();
  } catch (err) {
    checks.database = { status: 'error', error: (err as Error).message };
    overall = 'degraded';
  }

  // 2. 記憶體使用
  const mem = process.memoryUsage();
  checks.memory = {
    heapUsed: Math.round(mem.heapUsed / 1024 / 1024) + 'MB',
    heapTotal: Math.round(mem.heapTotal / 1024 / 1024) + 'MB',
    rss: Math.round(mem.rss / 1024 / 1024) + 'MB',
    status: mem.heapUsed / mem.heapTotal > 0.9 ? 'warning' : 'ok',
  };

  // 3. 磁碟空間（簡易檢查）
  try {
    const { execSync } = require('child_process');
    const df = execSync("df -h / | tail -1 | awk '{print $5}'").toString().trim();
    const usagePercent = parseInt(df);
    checks.disk = {
      usage: df,
      status: usagePercent > 90 ? 'critical' : usagePercent > 80 ? 'warning' : 'ok',
    };
    if (usagePercent > 90) overall = 'critical';
  } catch {
    checks.disk = { status: 'unknown' };
  }

  res.json({
    status: overall,
    timestamp: new Date().toISOString(),
    uptime: Math.round(process.uptime()) + 's',
    checks,
  });
});

export default router;
```

### 10.4 整合監控腳本

```bash
#!/bin/bash
# monitor-all-sites.sh — 監控所有客戶網站
# cron: */5 * * * * /opt/scripts/monitor-all-sites.sh

# 網站列表
SITES=(
  "客戶A|https://clienta.com"
  "客戶B|https://clientb.com"
  "客戶C|https://clientc.com/api/health"
)

ALERT_WEBHOOK="https://n8n.example.com/webhook/site-alert"
ALERT_EMAIL="你的email"

for SITE_INFO in "${SITES[@]}"; do
  NAME=$(echo "$SITE_INFO" | cut -d'|' -f1)
  URL=$(echo "$SITE_INFO" | cut -d'|' -f2)

  # 檢查
  HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" --max-time 15 "$URL" || echo "000")
  RESPONSE_TIME=$(curl -s -o /dev/null -w "%{time_total}" --max-time 15 "$URL" 2>/dev/null || echo "timeout")

  if [ "$HTTP_CODE" != "200" ]; then
    echo "[$(date)] 異常：$NAME ($URL) → HTTP $HTTP_CODE"

    # 發送告警
    curl -s -X POST "$ALERT_WEBHOOK" \
      -H "Content-Type: application/json" \
      -d "{
        \"site\": \"$NAME\",
        \"url\": \"$URL\",
        \"status\": $HTTP_CODE,
        \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"
      }" >/dev/null 2>&1
  else
    # 檢查回應時間是否太慢（超過 3 秒）
    if (( $(echo "$RESPONSE_TIME > 3.0" | bc -l 2>/dev/null || echo 0) )); then
      echo "[$(date)] 慢回應：$NAME → ${RESPONSE_TIME}s"
    fi
  fi
done
```

## 附錄：常用指令速查

```bash
# ===== PostgreSQL =====
pg_dump -Fc -f backup.dump mydb              # 備份
pg_restore -d mydb --clean backup.dump        # 恢復
pg_restore --list backup.dump                 # 查看備份內容

# ===== MySQL =====
mysqldump --single-transaction mydb > bk.sql  # 備份
mysql mydb < bk.sql                           # 恢復

# ===== MongoDB =====
mongodump --gzip --archive=bk.gz              # 備份
mongorestore --gzip --archive=bk.gz --drop    # 恢復

# ===== 檔案 =====
tar -czf backup.tar.gz --exclude=node_modules /var/www/site/  # 打包
tar -xzf backup.tar.gz -C /restore/           # 解壓

# ===== rsync =====
rsync -avz --delete src/ dest/                # 同步（刪除目的地多餘檔案）
rsync -avz --exclude=node_modules src/ dest/  # 同步（排除 node_modules）

# ===== AWS S3 =====
aws s3 cp file.gz s3://bucket/path/           # 上傳
aws s3 cp s3://bucket/path/file.gz ./         # 下載
aws s3 sync /local/ s3://bucket/prefix/       # 同步

# ===== Backblaze B2 =====
b2 upload-file bucket file.gz remote/file.gz  # 上傳
b2 download-file-by-name bucket path ./file   # 下載
b2 sync /local/ b2://bucket/prefix/           # 同步

# ===== cron 時間格式 =====
# 分  時  日  月  星期
# 0   3   *   *   *      每天凌晨 3 點
# 0   */6 *   *   *      每 6 小時
# 0   2   *   *   0      每週日凌晨 2 點
# 0   2   1   *   *      每月 1 號凌晨 2 點
```
