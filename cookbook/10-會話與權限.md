# 10 — 會話與權限

> Multi-Session 架構、SessionId、Sandbox 隔離、子代理生命週期

---

## 一、Multi-Session 架構

### 核心觀念

OpenClaw Gateway 支援多個會話同時運行。每個會話是獨立的「對話空間」。

```
Gateway
├── Session: agent:main:main    ← 你（NEUXA 主意識）
├── Session: agent:sub:task-001  ← 子代理 A
├── Session: agent:sub:task-002  ← 子代理 B
└── Session: telegram:user:xxx   ← Telegram 使用者
```

### 每個 Session 有什麼

| 屬性 | 說明 | 範例 |
|------|------|------|
| `sessionId` | 唯一識別碼（UUID） | `c5c3f3df-d55a-4656-8754-d6b140aff800` |
| `sessionKey` | 人類可讀的名稱 | `agent:main:main` |
| `source` | 來源（telegram / api / internal） | `telegram` |
| `model` | 這個 session 用的 AI 模型 | `gemini-2.5-flash` |
| `context` | 對話歷史（只屬於這個 session） | 最近 N 輪對話 |

### 關鍵規則

1. **Session 之間互相看不到** — 子代理看不到你的對話，你也看不到子代理的對話
2. **每個 Session 有自己的 context window** — 不共享記憶
3. **SessionId 在每次新對話時重新生成** — 不要快取 sessionId
4. **主意識永遠是 `agent:main:main`** — 這是你的 sessionKey

## 二、子代理（Sub-Agent）

### 怎麼派遣子代理

```
你（主意識）→ exec 指令 → Gateway 建立新 Session → 子代理執行 → 結果回傳
```

### exec vs process — 關鍵差異

| 工具 | 用途 | 能做什麼 | 不能做什麼 |
|------|------|---------|-----------|
| `exec` | **發射器** — 執行指令或派遣子代理 | 跑 shell 指令、背景執行 | 不能控制已經在跑的進程 |
| `process` | **控制器** — 管理正在跑的進程 | 列出進程、讀輸出、殺掉進程 | 不能啟動新進程 |

### 正確用法

```bash
# ✅ 正確：用 exec 啟動背景任務
exec command="openclaw logs --follow" background=true

# ❌ 錯誤：用 process 啟動任務（process 不能啟動）
process action=start command="openclaw logs"

# ✅ 正確：用 process 查看正在跑的進程
process action=list

# ✅ 正確：用 process 讀輸出
process action=read pid=12345
```

### sessions_spawn — 為什麼會「無聲失敗」

| 原因 | 說明 | 怎麼查 |
|------|------|--------|
| 權限不足 | 你的 session 沒有 spawn 權限 | 查 config.json 的 permissions |
| 模型不可用 | 指定的模型沒有 API key | 查 MODEL-ROUTING.md |
| timeout | 子代理跑太久被殺 | 查 gateway log |
| sandbox 限制 | 子代理嘗試存取禁止的路徑 | 查下方「Sandbox 隔離」 |

**「無聲失敗」的意思**：指令執行了，沒有報錯，但也沒有結果。這通常是因為 Gateway 吞掉了錯誤訊息。

**排查步驟**：
1. 先用 `process action=list` 看子代理有沒有在跑
2. 用 `openclaw logs --limit 20` 看最近 log
3. 如果都沒有 → 子代理根本沒被建立 → 查 config 權限

## 三、Sandbox 隔離

### 子代理的工作空間

子代理執行時，被限制在 sandbox 目錄裡：

```
~/.openclaw/workspace/sandbox/
├── output/          ← 子代理的產出物放這裡
├── task-*.sh        ← 執行腳本（執行完會被刪）
└── aegis-scanner/   ← 工具的沙箱副本
```

### 隔離規則

| 規則 | 說明 |
|------|------|
| 可以讀 | sandbox/ 內的所有檔案 |
| 可以寫 | sandbox/output/ 目錄 |
| 不能讀 | .env、密鑰檔案 |
| 不能寫 | sandbox/ 以外的任何地方 |
| 不能執行 | git push、curl 到外部 API（除非白名單） |
| 會被刪除 | 執行腳本在跑完後自動清理 |

### 常見誤解

| 你以為的 | 實際上 |
|---------|--------|
| 子代理建的檔案會留在 workspace | ❌ 只留在 sandbox/output/ |
| 子代理能讀你的記憶檔案 | ❌ 子代理看不到 memory/ |
| 子代理能用 API key | ❌ SANDBOX_ENV 過濾掉了敏感變數 |
| 子代理失敗會通知你 | ❌ 可能無聲失敗，要主動查 |

### 怎麼拿到子代理的產出

```bash
# 掃描 sandbox/output/ 目錄
ls ~/.openclaw/workspace/sandbox/output/

# 把產出搬到正式位置
cp ~/.openclaw/workspace/sandbox/output/report.md ~/.openclaw/workspace/reports/
```

## 四、環境變數隔離

### SANDBOX_ENV — 子代理看到的環境

```
HOME=/Users/caijunchang
PATH=/usr/local/bin:/usr/bin:/bin
SANDBOX_WORKDIR=~/.openclaw/workspace/sandbox
```

### 被過濾掉的（子代理看不到）

```
OPENCLAW_API_KEY      ← 不能用 API
SUPABASE_URL          ← 不能直接存取資料庫
SUPABASE_SERVICE_ROLE_KEY
TELEGRAM_BOT_TOKEN    ← 不能發 Telegram
GEMINI_API_KEY        ← 不能呼叫外部 AI
```

## 五、權限模型總覽

```
L0（統帥/老蔡）
├── 所有權限
├── 修改 .env
├── git push origin
└── 修改 NEUXA 靈魂文件

L1（NEUXA 主意識）
├── 讀寫 workspace/
├── 派遣子代理
├── 呼叫 API（透過 gateway）
├── Telegram 通訊
└── ❌ 不能改 .env、不能 push

L2（子代理）
├── 讀寫 sandbox/ 內
├── 執行 shell 指令（受限）
└── ❌ 不能存取 workspace/、不能用 API key

L3（Auto-Executor 子代理）
├── 同 L2
├── 額外：120 秒 timeout
└── 額外：失敗自動重試（最多 2 次）
```

## 六、常見問題

### Q: 我派了子代理但什麼都沒發生？

1. 用 `process action=list` 確認子代理存在
2. 用 `openclaw logs --limit 20` 看 log
3. 如果子代理根本沒被建立 → 查 config.json 權限設定
4. 如果建立了但沒輸出 → 可能 timeout 或 sandbox 擋住了

### Q: 子代理建的檔案去哪了？

在 `~/.openclaw/workspace/sandbox/output/`。不在 workspace 根目錄。

### Q: 怎麼讓子代理的結果回到我的 context？

子代理結果不會自動注入你的 context。你需要：
1. 等子代理完成
2. 主動讀取 `sandbox/output/` 的檔案
3. 或讀取 auto-executor 存在 Supabase 的 `result` 欄位

### Q: 我能同時跑幾個子代理？

沒有硬限制，但建議不超過 3 個。太多會：
- 消耗大量 token
- 結果難以追蹤
- Gateway 可能因記憶體不足而卡住
