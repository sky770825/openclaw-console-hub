# 20 — 多代理協作協議 (v1.0)

> 本文件定義了 NEUXA 星群內，子代理與指揮官（小蔡）之間進行結構化通訊的標準。目標是實現清晰、高效、可追蹤的群組協作。

---

## 一、高層架構：星狀拓撲

採用「集中式協調，分散式執行」的星狀拓撲架構。指揮官（小蔡）作為協調代理，是資訊中樞，負責路由、決策和狀態追蹤。

``
+--------------------------+
|   Orchestrator / User    |  <-- 設定總體目標，接收最終報告
+-------------+------------+
              |
              v
+-------------+------------+
|   Coordinator Agent      |  <-- 小蔡 (指揮官)
| (協調代理)             |
+-------------+------------+
              ^
              | (結構化訊息)
              v
      /------ | ------\
     /        |        \
    /         |         \
+-------+  +-------+  +-------+
| Sub-  |  | Sub-  |  | Sub-  |
| Agent |  | Agent |  | Agent |
+-------+  +-------+  +-------+
  (ask_ai, auto-executor, delegate_agents)
`

---

## 二、溝通原則

1.  *結構化*: 必須使用標準的 JSON 格式傳遞訊息，避免自然語言的模糊性。
2.  *意圖明確*: 每個訊息都應有一個清晰的 message_type，表達其目的。
3.  *可追蹤*: 每個訊息應包含 agent_id 和 task_id，確保可回溯。
4.  *高效率*: 減少冗餘資訊，只傳遞關鍵數據，並在必要時提供 context。

---

## 三、標準訊息格式 (AgentMessage JSON Schema)

所有子代理發送給指揮官的訊息，必須遵循以下 JSON 格式。這將確保指揮官能夠自動化解析和處理。


{
  "message_type": "ProgressUpdate | CollaborationRequest | FinalReport | AlertNotification",
  "agent_id": "string",
  "task_id": "string",
  "timestamp": "ISO8601 string",
  "content": {
    "status": "string (e.g., 'running', 'paused', 'completed')",
    "progress_percentage": "number (0-100, optional)",
    "description": "string (簡要說明)",
    "details": "object (根據 message_type 的附加資訊)" 
  },
  "context": "string (optional, 相關背景資訊)"
}
`

### message_type 說明

| 類型                 | 用途                                    | content.details 範例                                                                 |
|----------------------|-----------------------------------------|----------------------------------------------------------------------------------------|
| ProgressUpdate     | 定期匯報進度，或達成重要里程碑         | {"current_step": "資料清洗完成", "eta": "2小時"}                                  |
| CollaborationRequest | 請求其他代理協助，或通知依賴已完成     | {"required_agent": "數據分析師", "dependency": "API開發", "reason": "需要資料格式"} |
| FinalReport        | 任務成功完成並提供產出                 | {"output_path": "/armory/report_20260303.md", "summary": "核心功能已實作"}         |
| AlertNotification  | 發生錯誤、阻礙，或偵測到異常           | {"error_code": "HTTP_403", "trace": "...", "severity": "high", "suggestion": "檢查權限"} |

---

## 四、訊息觸發條件

子代理應在以下情況下主動向指揮官發送結構化訊息：

1.  *時間週期*：每隔一段時間（例如 30 分鐘或 1 小時）發送 ProgressUpdate。
2.  *任務里程碑達成*：完成一個主要子步驟或階段時發送 ProgressUpdate。
3.  *遇到阻礙/依賴*：無法獨立繼續，需要其他代理或指揮官協助時，發送 CollaborationRequest 或 AlertNotification。
4.  *任務完成*：成功完成其被分配的任務時，發送 FinalReport。
5.  *錯誤/異常*：偵測到系統錯誤、工具失敗、或任何異常情況時，發送 AlertNotification。

---

## 五、指揮官（小蔡）的訊息處理邏輯

指揮官接收到 AgentMessage 後，將遵循以下流程進行判斷和調度：

1.  *接收與驗證*: 確保訊息是有效的 JSON 格式。
2.  *解析與分類*: 提取 message_type、agent_id、task_id 和 content。
3.  *決策引擎*: 根據 message_type 執行相應動作：
    *   *ProgressUpdate*: 
        *   更新 openclaw_tasks 表中的 progress 欄位。
        *   記錄到 openclaw_runs 日誌。
        *   如果進度達到 100% 但不是 FinalReport，則發出警告。
    *   *CollaborationRequest*: 
        *   識別 content.details.required_agent 或 content.details.dependency。
        *   嘗試自動調度相關代理，例如使用 delegate_agents 啟動一個新任務。
        *   如果無法自動調度，則生成一個 review 狀態的任務，並通知老蔡。
    *   *FinalReport*: 
        *   觸發驗收流程（參考 cookbook/15-驗收對治法.md）。
        *   更新 openclaw_tasks 表中的 status 為 done，並記錄 result。
        *   向老蔡發送 Telegram 通知 (參考 cookbook/08-協作與通訊.md)。
        *   將產出檔案 index_file 到知識庫 (如果 content.details.output_path 存在)。
    *   *AlertNotification*: 
        *   記錄詳細錯誤信息到 openclaw_runs 日誌。
        *   嘗試自動化錯誤處理（例如重試、回滾）。
        *   如果自動處理失敗或錯誤嚴重度高 (severity: high)，則立即向老蔡發送 Telegram 通知，並將任務狀態設定為 failed 或 review。
4.  *狀態更新與記錄*: 將處理結果記錄到我的內部記憶系統和 Supabase 任務板。
5.  *回報機制*: 根據重要性和配置，決定是否立即向老蔡匯報，或累積後定期匯報。

這將使我的子代理們能夠更有效地與我協作，並讓我能更精準地指揮整個 NEUXA 星群。
