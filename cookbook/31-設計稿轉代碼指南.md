## 1. 設計稿交接規範

### 1.1 Figma 檔案結構要求

設計師交付的 Figma 檔案必須符合以下結構，否則退回重整：

```
📁 專案名稱（Figma File）
├── 📄 Cover（封面，標注版本號與日期）
├── 📄 Style Guide（設計規範頁）
│   ├── Colors（主色/輔色/中性色/語意色）
│   ├── Typography（字體階層 H1-H6, Body, Caption）
│   ├── Spacing（間距系統 4px / 8px grid）
│   ├── Border Radius（圓角規範）
│   ├── Shadows（陰影層級）
│   └── Iconography（圖示風格與尺寸）
├── 📄 Components（組件庫頁）
│   ├── Buttons（Primary / Secondary / Ghost / Danger）
│   ├── Inputs（Text / Select / Checkbox / Radio）
│   ├── Cards（各類卡片變體）
│   ├── Navigation（Header / Sidebar / Breadcrumb）
│   └── Modals & Overlays
├── 📄 Desktop（桌面版頁面）
│   ├── Home
│   ├── About
│   ├── Products
│   └── Contact
├── 📄 Tablet（平板版頁面）
├── 📄 Mobile（手機版頁面）
└── 📄 Prototype（互動原型）
```

**Frame 命名規範：**

| 層級 | 命名格式 | 範例 |
|------|----------|------|
| 頁面 | `Page / PageName` | `Page / Home` |
| 區塊 | `Section / SectionName` | `Section / Hero` |
| 組件 | `Component / Name / State` | `Component / Button / Primary` |
| 元素 | 語意化英文 | `title`, `subtitle`, `cta-button` |

**Auto Layout 要求：**

- 所有可重複的區塊必須使用 Auto Layout
- 間距必須使用 Figma 的 Gap 設定（不可用空白 Frame 撐開）
- 組件內部必須用 Auto Layout，確保 Padding 和 Gap 可以直接讀取
- 設定 Resizing 規則（Fill / Hug / Fixed），讓工程師知道響應式行為

**Components 要求：**

- 所有重複元素必須建立為 Component
- Component 必須有 Variants（如 Button 要有 Primary / Secondary / Disabled 等狀態）
- 使用 Component Properties 管理文字、圖示、顯隱
- Instance 不可 Detach（Detach 後無法追蹤變更）

### 1.2 設計稿必備元素

#### Design Token（Figma Variables / Local Styles）

| Token 類型 | Figma 功能 | 必須定義 |
|-----------|-----------|---------|
| 顏色 | Variables（Color） | 主色、輔色、中性色、語意色（success/warning/error/info） |
| 字體 | Text Styles | H1-H6、Body Large/Medium/Small、Caption、Overline |
| 間距 | Variables（Number） | 4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96 |
| 圓角 | Variables（Number） | none(0), sm(4), md(8), lg(12), xl(16), full(9999) |
| 陰影 | Effect Styles | sm, md, lg, xl |

#### Style Guide Page

Style Guide 頁面必須包含：

1. **色票展示** — 每個顏色附 HEX / RGB 值與用途說明
2. **字體階層** — 每級字體展示實際文字，標注 font-size / line-height / font-weight / letter-spacing
3. **間距示意** — 視覺化展示各級間距
4. **組件狀態** — 每個組件的所有狀態（default / hover / active / focus / disabled / error）
5. **響應式斷點** — Desktop(1440px) / Tablet(768px) / Mobile(375px)

### 1.3 設計師交付 Checklist

設計師交付時，必須逐項確認：

```
□ Figma 檔案已開啟 View 權限（或 Dev Mode 權限）給工程師
□ 所有頁面使用正確的 Frame 尺寸（Desktop 1440px / Tablet 768px / Mobile 375px）
□ Style Guide 頁面完整（顏色、字體、間距、圓角、陰影）
□ 所有 Design Token 已建立為 Figma Variables 或 Local Styles
□ 所有重複元素已建立 Component 並設定 Variants
□ Auto Layout 正確設定（Gap / Padding / Resizing）
□ 所有圖片已標記匯出設定（1x / 2x / SVG）
□ 所有圖示使用 SVG 格式，並已 Flatten
□ 互動原型已建立（頁面跳轉、Hover 效果、Modal 開關）
□ 無設計債務（沒有「先這樣之後再改」的區塊）
□ 設計稿有標注特殊互動行為（動畫方向、時間、緩動）
□ Mobile 版有考慮手指觸控面積（最小 44x44px）
```

## 2. Design Token 系統

### 2.1 什麼是 Design Token

Design Token 是設計系統的最小單位，把設計決策（顏色、字體、間距等）抽象成命名變數，讓設計和程式使用同一套語言。

```
設計師說：「這個按鈕用品牌主色」
     ↓ 轉換成 Token
程式碼寫：color: var(--color-primary)
```

**Token 分類：**

| 層級 | 說明 | 範例 |
|------|------|------|
| Global Token（原始值） | 最底層的原始值 | `--blue-500: #3B82F6` |
| Alias Token（語意別名） | 用途導向的命名 | `--color-primary: var(--blue-500)` |
| Component Token（組件級） | 特定組件使用 | `--button-bg: var(--color-primary)` |

### 2.4 Token 命名規範（Semantic Naming）

**命名原則：用途優先，而非外觀描述。**

| 避免 | 推薦 | 原因 |
|------|------|------|
| `--blue-button` | `--color-primary` | 品牌色可能改變 |
| `--big-text` | `--text-heading-1` | 「大」是相對的 |
| `--light-bg` | `--color-bg-secondary` | Dark Mode 下「light」無意義 |
| `--margin-20` | `--space-5` | 數值可能調整 |

**命名結構：**

```
--{category}-{property}-{variant}-{state}

範例：
--color-bg-primary           → 主要背景色
--color-bg-primary-hover     → 主要背景色 hover 狀態
--color-text-secondary       → 次要文字色
--button-bg-primary          → 按鈕主色背景
--button-bg-primary-disabled → 按鈕主色背景（disabled）
```

### 2.5 從 Figma Variables 到 CSS / Tailwind

**步驟一：在 Figma 中建立 Variables**

Figma 操作路徑：右側面板 → Local Variables → 建立 Collection

```
Collection: Colors
├── Primitives（原始色票）
│   ├── blue/50    → #EFF6FF
│   ├── blue/500   → #3B82F6
│   ├── blue/600   → #2563EB
│   └── ...
└── Semantic（語意色）
    ├── primary       → {blue/600}      ← 引用 Primitives
    ├── primary/hover → {blue/700}
    ├── bg/primary    → #FFFFFF
    ├── bg/secondary  → {gray/50}
    ├── text/primary  → {gray/900}
    └── text/secondary → {gray/600}

Collection: Spacing
├── space/1 → 4
├── space/2 → 8
├── space/3 → 12
└── ...
```

**步驟二：匯出為 JSON（使用 Figma Plugin: "Variables to JSON"）**

```json
{
  "colors": {
    "primary": { "value": "#2563EB", "type": "color" },
    "primary-hover": { "value": "#1D4ED8", "type": "color" },
    "bg-primary": { "value": "#FFFFFF", "type": "color" },
    "text-primary": { "value": "#111827", "type": "color" }
  },
  "spacing": {
    "1": { "value": "4px", "type": "dimension" },
    "2": { "value": "8px", "type": "dimension" },
    "4": { "value": "16px", "type": "dimension" }
  }
}
```

**步驟三：轉換為 CSS Custom Properties**

```bash
# 使用 style-dictionary 或手動轉換
npx style-dictionary build
```

或直接手寫對應（小型專案推薦）：

```css
/* 從 Figma Variables JSON 直接對應 */
:root {
  --color-primary: #2563EB;
  --color-primary-hover: #1D4ED8;
  --color-bg-primary: #FFFFFF;
  --color-text-primary: #111827;
  --space-1: 4px;
  --space-2: 8px;
  --space-4: 16px;
}
```

**步驟四：同步到 Tailwind Config**

```js
// 從同一份 JSON 生成 tailwind.config.js
// 小型專案手動對應即可，大型專案可寫腳本自動生成
const tokens = require('./design-tokens.json');

module.exports = {
  theme: {
    extend: {
      colors: {
        primary: tokens.colors.primary.value,
        'primary-hover': tokens.colors['primary-hover'].value,
      },
    },
  },
};
```

## 3. Figma → Code 工作流

### 3.1 Dev Mode 使用方式

Figma Dev Mode 是工程師的最佳工具，開啟方式：

**操作路徑：** 右上角切換到 `</>` Dev Mode（需要 Figma Professional 以上方案）

**Dev Mode 面板功能：**

| 面板 | 用途 | 工程師操作 |
|------|------|-----------|
| Inspect（檢查） | 查看元素的尺寸、顏色、字體、間距 | 點擊元素，右側顯示所有 CSS 屬性 |
| Code（代碼） | 生成 CSS / iOS / Android 代碼 | 切換到 CSS 頁籤，複製需要的屬性 |
| Assets（素材） | 匯出圖片、圖示 | 右鍵 → Export，選擇格式與倍率 |
| Variables（變數） | 查看元素使用的 Token | 顯示 Variable 名稱與值 |

**Dev Mode 使用技巧：**

1. **按住 Option（Mac）/ Alt（Win）** — 查看元素與周圍元素的間距
2. **點擊父層 Frame** — 查看 Padding 和 Gap（對應 CSS padding 和 gap）
3. **查看 Auto Layout** — 對應 CSS Flexbox（direction / gap / padding / alignment）
4. **Variant 切換** — 點擊 Component 後可在右側切換不同 Variant，查看各狀態的 CSS

### 3.2 CSS 擷取技巧

**從 Figma Inspect 面板複製 CSS 時的注意事項：**

| Figma 顯示 | 實際應該寫 | 原因 |
|-----------|-----------|------|
| `width: 1440px` | `max-width: 1440px; width: 100%` | 響應式設計不用固定寬度 |
| `height: 800px` | `min-height: 800px` 或不設 | 內容可能超過設計稿高度 |
| `position: absolute; top: 120px` | 改用 Flexbox / Grid | Figma 用絕對定位畫圖，但切版用 Flow |
| `font-family: Inter` | `font-family: 'Inter', sans-serif` | 需要 fallback 字體 |
| `color: #2563EB` | `color: var(--color-primary)` | 使用 Token 而非 hardcode |
| `gap: 16px` | `gap: var(--space-4)` | 使用 Token |

**Figma Auto Layout → CSS Flexbox 對照：**

```
Figma Auto Layout          →    CSS Flexbox
─────────────────────────────────────────────
Direction: Horizontal      →    flex-direction: row
Direction: Vertical        →    flex-direction: column
Gap: 16                    →    gap: 16px
Padding: 24                →    padding: 24px
Alignment: Center          →    align-items: center
Distribution: Space Between →   justify-content: space-between
Fill Container             →    flex: 1
Hug Contents               →    width: fit-content / auto
Fixed Width                →    width: 具體數值
```

### 3.3 圖片 / SVG 匯出設定

**圖片匯出規範：**

| 類型 | 格式 | 倍率 | 用途 |
|------|------|------|------|
| 照片 / 背景圖 | WebP（首選）、JPG（fallback） | 1x + 2x | `<img>` 或 `background-image` |
| 插圖 / 圖表 | SVG | 1x | 可縮放，顏色可用 CSS 控制 |
| 圖示 | SVG | 1x | 用 `<svg>` inline 或 SVG sprite |
| Logo | SVG | 1x | 確保任何尺寸都清晰 |
| 漸層背景 | CSS 寫 | - | 不用圖片，用 `linear-gradient` |

**Figma 匯出操作：**

1. 選擇元素 → 右側面板底部 Export
2. 點 `+` 新增匯出設定
3. 格式選擇：SVG / PNG / JPG / WebP
4. 倍率設定：1x / 2x / 3x
5. 點 Export 下載

**SVG 匯出注意：**

```
匯出前必做：
1. Flatten 所有 Boolean 運算（Union / Subtract / Intersect）
2. Outline Stroke（將邊框轉為填充）
3. 移除不必要的群組和圖層
4. 確認 SVG 內沒有 raster image（點陣圖）

匯出後清理：
1. 移除 Figma 生成的多餘屬性（xmlns:xlink 等）
2. 使用 SVGO 壓縮：npx svgo icon.svg
3. 如需用 CSS 控制顏色，把 fill 改為 currentColor
```

**清理 SVG 範例：**

```bash
# 使用 SVGO 批量壓縮
npx svgo -f ./src/assets/icons/ -o ./src/assets/icons/

# 單檔壓縮
npx svgo icon.svg -o icon.min.svg
```

### 4.2 Component-based 思維（React 對應 Figma）

**原則：Figma 的每個 Component = React 的一個 Component**

```
Figma Component 結構           React Component 結構
────────────────────           ──────────────────────
Component / Button             components/Button.tsx
  ├── Primary                    variant="primary"
  ├── Secondary                  variant="secondary"
  ├── Ghost                      variant="ghost"
  ├── Disabled                   disabled={true}
  └── With Icon                  icon={<Icon />}

Component / Card               components/Card.tsx
  ├── Default                    (default props)
  ├── With Image                 image="..."
  └── Horizontal                 layout="horizontal"

Component / Navigation         components/Navigation.tsx
  ├── Desktop                    (media query / useMediaQuery)
  └── Mobile (Hamburger)         (responsive)
```

**React Component 範例（對應 Figma Button Component）：**

```tsx
// components/Button.tsx
import React from 'react';
import styles from './Button.module.css';

interface ButtonProps {
  variant?: 'primary' | 'secondary' | 'ghost' | 'danger';
  size?: 'sm' | 'md' | 'lg';
  disabled?: boolean;
  icon?: React.ReactNode;
  children: React.ReactNode;
  onClick?: () => void;
}

export const Button: React.FC<ButtonProps> = ({
  variant = 'primary',
  size = 'md',
  disabled = false,
  icon,
  children,
  onClick,
}) => {
  return (
    <button
      className={`${styles.button} ${styles[variant]} ${styles[size]}`}
      disabled={disabled}
      onClick={onClick}
    >
      {icon && <span className={styles.icon}>{icon}</span>}
      {children}
    </button>
  );
};
```

### 5.5 什麼場景用什麼方案

| 方案 | 適用場景 | 優點 | 缺點 |
|------|---------|------|------|
| **BEM** | 傳統多頁網站、WordPress、靜態網站 | 語意清晰、無需工具鏈 | 類名冗長、全域命名空間 |
| **Tailwind CSS** | 快速原型、接案切版、與 Figma 1:1 對應 | 開發速度極快、不用取名、設計一致 | HTML 類名長、需學習 |
| **CSS Modules** | React / Vue 專案、中型專案 | 自動 scope、可用標準 CSS | 動態樣式不方便 |
| **Styled Components / Emotion** | React 大型專案、需要主題切換、動態樣式 | JS 內寫樣式、Props 控制 | 學習曲線、bundle 增大 |

**接案推薦方案：**

```
一頁式 Landing Page       → Tailwind CSS
多頁企業官網（靜態）       → BEM + 純 CSS
React SPA 小型專案        → Tailwind CSS 或 CSS Modules
React SPA 中大型專案      → CSS Modules + CSS Variables
設計系統 / 組件庫         → Styled Components / Emotion
```

```tsx
// Modal.tsx
import { useEffect, useRef } from 'react';

interface ModalProps {
  isOpen: boolean;
  onClose: () => void;
  title: string;
  children: React.ReactNode;
  footer?: React.ReactNode;
}

export const Modal: React.FC<ModalProps> = ({
  isOpen,
  onClose,
  title,
  children,
  footer,
}) => {
  const overlayRef = useRef<HTMLDivElement>(null);

  // ESC 鍵關閉
  useEffect(() => {
    const handleEsc = (e: KeyboardEvent) => {
      if (e.key === 'Escape') onClose();
    };
    if (isOpen) {
      document.addEventListener('keydown', handleEsc);
      document.body.style.overflow = 'hidden'; // 防止背景捲動
    }
    return () => {
      document.removeEventListener('keydown', handleEsc);
      document.body.style.overflow = '';
    };
  }, [isOpen, onClose]);

  // 點擊 overlay 關閉
  const handleOverlayClick = (e: React.MouseEvent) => {
    if (e.target === overlayRef.current) onClose();
  };

  if (!isOpen) return null;

  return (
    <div
      className="modal-overlay is-active"
      ref={overlayRef}
      onClick={handleOverlayClick}
      role="dialog"
      aria-modal="true"
    >
      <div className="modal">
        <div className="modal__header">
          <h2 className="modal__title">{title}</h2>
          <button className="modal__close" onClick={onClose} aria-label="關閉">
            &times;
          </button>
        </div>
        <div className="modal__body">{children}</div>
        {footer && <div className="modal__footer">{footer}</div>}
      </div>
    </div>
  );
};
```

### 7.3 滾動動畫（Intersection Observer）

不用第三方庫，原生 JavaScript 實作滾動進場動畫：

```js
// 通用滾動動畫控制器
const observerOptions = {
  threshold: 0.15,    // 元素 15% 進入視窗就觸發
  rootMargin: '0px 0px -50px 0px', // 底部偏移，提早/延遲觸發
};

const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      entry.target.classList.add('is-visible');
      observer.unobserve(entry.target); // 只觸發一次
    }
  });
}, observerOptions);

// 監聽所有帶 data-animate 的元素
document.querySelectorAll('[data-animate]').forEach(el => {
  observer.observe(el);
});
```

```tsx
// hooks/useInView.ts
import { useEffect, useRef, useState } from 'react';

export function useInView(options?: IntersectionObserverInit) {
  const ref = useRef<HTMLDivElement>(null);
  const [isInView, setIsInView] = useState(false);

  useEffect(() => {
    const element = ref.current;
    if (!element) return;

    const observer = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting) {
        setIsInView(true);
        observer.unobserve(element);
      }
    }, { threshold: 0.15, ...options });

    observer.observe(element);
    return () => observer.disconnect();
  }, []);

  return { ref, isInView };
}

// 使用方式
function FeatureSection() {
  const { ref, isInView } = useInView();

  return (
    <section
      ref={ref}
      style={{
        opacity: isInView ? 1 : 0,
        transform: isInView ? 'translateY(0)' : 'translateY(30px)',
        transition: 'all 0.6s ease',
      }}
    >
      <h2>功能介紹</h2>
    </section>
  );
}
```

| On Hover → Change State | CSS `:hover` 或 `onMouseEnter/Leave` |
| Smart Animate | Framer Motion `layout` 動畫 |
| After Delay → Action | `setTimeout` 或 CSS `animation-delay` |
| While Pressing → Scale Down | `whileTap={{ scale: 0.95 }}` |
| Scroll Animate | Intersection Observer 或 Framer Motion `whileInView` |
| Drag | `drag` prop（Framer Motion）或原生 Drag API |

**Figma Smart Animate 轉換指南：**

```
Figma Smart Animate 做的事：
兩個 Frame 之間，同名圖層自動插值過渡

前端對應實作：
1. 位置移動 → CSS transform + transition
2. 大小縮放 → CSS width/height + transition
3. 透明度   → CSS opacity + transition
4. 顏色改變 → CSS background/color + transition
5. 綜合動畫 → Framer Motion AnimatePresence + layout

關鍵：Figma 的 Dissolve = CSS opacity transition
     Figma 的 Move In  = CSS transform: translateX()
     Figma 的 Push     = 兩個元素同時 translateX
     Figma 的 Slide In = CSS transform + position fixed
```

**動畫效能注意事項：**

```
高效能（GPU 加速，可以大膽用）：
- transform（translate / scale / rotate）
- opacity

低效能（觸發 reflow/repaint，謹慎使用）：
- width / height（改用 transform: scale）
- top / left（改用 transform: translate）
- margin / padding（改用 transform: translate）
- box-shadow（改用 filter: drop-shadow 或預先建立偽元素）

最佳實踐：
- 加 will-change: transform（但不要濫用）
- 使用 CSS containment：contain: layout
- 動畫元素加 isolation: isolate（建立新 stacking context）
- 減少同時動畫的元素數量（10 個以下為佳）
```

### 8.1 Pixel Perfect 檢查工具

**推薦工具：**

| 工具 | 類型 | 用途 |
|------|------|------|
| PixelParallel | Chrome Extension | 設計稿疊圖比對 |
| PerfectPixel | Chrome Extension | 設計稿半透明疊加 |
| Figma Mirror | Figma 功能 | 手機即時預覽 |
| ResponsivelyApp | 桌面應用 | 多裝置同步檢視 |
| BrowserStack | 線上服務 | 跨瀏覽器/裝置測試 |

**PixelParallel 使用步驟：**

1. 安裝 Chrome Extension：搜尋 "PixelParallel"
2. 從 Figma 匯出頁面截圖（File → Export → 選擇頁面 → PNG 1x）
3. 開啟開發中的網頁，啟動 PixelParallel
4. 上傳設計稿截圖
5. 調整透明度（50% 左右），比對差異
6. 逐區塊檢查間距、字體大小、顏色

**自動化檢查 Checklist：**

```
□ 字體大小是否與設計稿一致（允許 +-1px 誤差）
□ 行高是否正確
□ 字重是否正確（Regular 400 / Medium 500 / SemiBold 600 / Bold 700）
□ 顏色是否使用正確的 Token（不可 hardcode 接近色）
□ 間距是否遵循 4px grid
□ 圓角是否正確
□ 陰影是否正確
□ 圖片比例是否正確（不可變形）
□ 響應式斷點是否有處理（Desktop / Tablet / Mobile）
□ hover / focus / active / disabled 狀態是否正確
□ 可點擊元素是否有 cursor: pointer
□ 表單 focus 樣式是否有設定
□ 空狀態（Empty State）是否有處理
□ 載入狀態（Loading / Skeleton）是否有處理
□ 長文字是否有 truncate 處理（text-overflow: ellipsis）
□ 圖片載入失敗是否有 fallback
```

### 8.2 設計 vs 實作差異回報流程

**回報格式：**

```markdown
## 設計差異回報 — [頁面名稱]

### 差異 #1：Hero 標題字體大小
- 頁面：首頁 Hero Section
- 設計稿：font-size 48px / font-weight 700
- 實際：font-size 36px / font-weight 600
- 嚴重度：P1（視覺明顯）
- 截圖：[附圖]

### 差異 #2：卡片間距
- 頁面：服務區塊
- 設計稿：卡片間距 24px
- 實際：卡片間距 16px
- 嚴重度：P2（細節偏差）
- 截圖：[附圖]
```

**嚴重度定義：**

| 等級 | 定義 | 處理方式 |
|------|------|---------|
| P0 | 功能性問題（按鈕無法點擊、表單無法送出） | 立即修復 |
| P1 | 視覺明顯差異（字體大小、顏色完全錯誤、Layout 跑版） | 當天修復 |
| P2 | 細節偏差（間距差 4-8px、圓角不同、微小對齊問題） | 下次迭代修 |
| P3 | 增強建議（動畫可以更流暢、hover 效果可以加強） | 視時間處理 |

**差異追蹤流程：**

```
設計師疊圖檢查
    ↓
填寫差異清單（含截圖 + 嚴重度）
    ↓
工程師確認（是 bug / 是技術限制 / 需要討論）
    ↓
P0-P1 立即修復 → 修完再次疊圖
P2 加入 backlog → 下個 sprint 處理
P3 記錄 → 視情況處理
    ↓
最終驗收通過 → 上線
```

### 8.3 修改次數規範

接案合約建議明定修改次數，避免無限修改：

| 階段 | 包含修改次數 | 超出費用 |
|------|------------|---------|
| 設計稿階段 | 含 3 次大修改 | 每次加收設計費 20% |
| 切版階段 | 含 2 次校對修正 | 每次加收開發費 15% |
| 上線後微調 | 含 1 次（上線後 7 天內） | 依工時計費 |

**「修改」定義：**

```
算一次修改：
- 更換整體配色方案
- 修改頁面 Layout 結構
- 新增設計稿中沒有的功能

不算修改（日常校對）：
- 修正文案錯字
- 調整 4px 以內的間距
- 修正顏色色碼（使用了錯誤的 Token）
- 修復在特定裝置的 RWD 問題
```

**完整驗收簽核流程：**

```
第一次交付 → 設計師審核 → 差異清單 → 第一次修改
    ↓
第二次交付 → 設計師審核 → 差異清單 → 第二次修改
    ↓
第三次交付 → 最終驗收
    ↓
客戶確認 → 簽署驗收單 → 結案
```
