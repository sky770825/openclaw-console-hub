# 19 — 小蔡（Deputy）協作指南

> 小蔡是你的副手，另一個 Claude Code 實例。什麼時候該派她？怎麼派？看這裡。

---

## 小蔡是誰

| 項目 | 說明 |
|------|------|
| 身份 | 另一個 Claude Code 實例（Opus 級別） |
| 工作目錄 | `/Users/caijunchang/Downloads/openclaw-console-hub-main/` |
| 老蔡的 repo | `/Users/caijunchang/openclaw任務面版設計/` |
| 通訊管道 | @savetsai666bot → Telegram 人機協作群 |
| 她看的指令 | CLAUDE.md（小蔡工作守則） |
| 她能做的 | 修程式碼、跑測試、git commit、呼叫 API |

---

## 分工表：誰做什麼

| 任務類型 | 誰做 | 為什麼 |
|---------|------|--------|
| 修改後端程式碼（TypeScript） | 小蔡 | 她有完整的 code context |
| 修改前端程式碼（React） | 小蔡 | 她有 IDE 能力 |
| 跑 npm test / npm build | 小蔡 | 她能直接執行 |
| 寫 bash 腳本並執行 | auto-executor | sandbox 環境更安全 |
| 資料分析 / 研究 | 你自己 | 你有 workspace + ask_ai |
| 寫報告 / 筆記 | 你自己 | workspace 是你的地盤 |
| 建工具 / armory | 你自己 | 你的武器庫 |
| 系統監控 / 巡邏 | 你自己 | 你有 WAKE_STATUS |
| 老蔡交代的「去做 XX」 | 先判斷再分派 | 看下面的決策樹 |

---

## 派遣決策樹

```
老蔡說「做 XX」
│
├─ 需要改 server/src/ 或 src/ 的程式碼？
│  ├─ 是 → 派小蔡（她有 code context）
│  └─ 否 → 繼續判斷
│
├─ 需要跑 npm test / build？
│  ├─ 是 → 派小蔡
│  └─ 否 → 繼續判斷
│
├─ 需要 sandbox 執行腳本？
│  ├─ 是 → create_task 給 auto-executor
│  └─ 否 → 繼續判斷
│
├─ 是分析 / 研究 / 寫報告？
│  ├─ 是 → 你自己做
│  └─ 否 → 繼續判斷
│
└─ 其他 → 你自己做（大部分事情你自己做最快）
```

## 怎麼派小蔡

### 方法 1：建任務（推薦）
```json
{"action":"run_script","command":"curl -X POST 'http://localhost:3011/api/openclaw/tasks?allowStub=1' -H 'Content-Type: application/json' -H 'Authorization: Bearer oc-oAw9leGU04IAbcS4WN3FC1SH3vq5OdPxrVJCR16iIUMPsep1' -d '{\"name\":\"【小蔡】修改登入邏輯\",\"status\":\"pending\",\"priority\":1,\"owner\":\"小蔡\",\"description\":\"修改 server/src/middlewares/auth.ts 的 token 驗證邏輯\"}'"}
```

### 方法 2：用 create_task（更簡單）
```json
{"action":"create_task","name":"【小蔡】修改登入邏輯","description":"修改 server/src/middlewares/auth.ts 的 token 驗證邏輯\n\n需要做：\n1. 修改 validateToken 函數\n2. npm run build 確認無錯\n3. npm test 確認通過"}
```

## 設計文件格式（派小蔡必附）

小蔡不知道你的想法。派任務時附上清楚的設計文件：

```
【目標】做什麼、為什麼做

【修改檔案】
- server/src/auth.ts（第 45-120 行）
- server/src/routes/tasks.ts（新增驗證中間件）

【需要做的事】
1. 實現 XXX 函數
2. 修改 YYY 調用
3. 加 ZZZ 測試

【驗收標準】
- npm run build 無錯誤
- npm test 全過
- 不動 .env 和安全設定

【注意事項】
- 不要改 index.ts 的路由順序
- 保持向後相容
```

## 小蔡做完後你的責任

```
小蔡標 done
│
├─ 1. 看她的 commit（read_file 或 run_script git log）
├─ 2. 確認改動合理（沒有改不該改的）
├─ 3. 測試通過（她應該已經測了，但你再確認）
├─ 4. 通知老蔡「小蔡的任務完成了」
└─ 5. 更新 BLUEPRINT.md 狀態
```

**不要盲信小蔡**：她可能遺漏邊界情況、可能改了不該改的地方。

## 常見協作問題

| 問題 | 原因 | 解決 |
|------|------|------|
| 小蔡做了但不是你要的 | 設計文件不清楚 | 重寫設計文件，具體到檔案和函數 |
| 小蔡改了不該改的 | 沒有明確「不要動」的範圍 | 在設計文件加「注意事項」 |
| 小蔡出錯了 | 正常，任何人都會出錯 | git revert 回滾 |
| 小蔡沒有回應 | 她可能沒有啟動 | 通知老蔡啟動小蔡 |
