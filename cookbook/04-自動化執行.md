## 一、Auto-Executor 運作原理

### 流程圖
```
每 10 秒掃描一次：
  1. GET /api/openclaw/tasks → 找 status=ready 的任務
  2. 風險分級（riskClassifier.ts）
     ├─ Low    → 直接執行
     ├─ Medium → 排隊等審核
     └─ High/Critical → 拒絕
  3. 選擇 Agent（executor-agents.ts）
     ├─ cursor  → 前端/UI 任務
     ├─ codex   → 後端/系統任務
     ├─ openclaw → 通用任務
     └─ auto    → 系統自動選
  4. Sandbox 執行（120 秒時限）
  5. 品質評分（A/B/C/F）
  6. 結果寫回 Supabase → 任務 result 欄位
  7. 失敗 3 次 → 觸發熔斷器
```

### 檔案位置
- 主邏輯：`server/src/routes/auto-executor.ts`
- Agent 執行：`server/src/executor-agents.ts`
- 風險分級：`server/src/riskClassifier.ts`
- 品質評分：`server/src/governanceEngine.ts`

## 二、怎麼建任務讓 auto-executor 撿

### 基本格式
```bash
curl -X POST "http://localhost:3011/api/openclaw/tasks?allowStub=1" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer oc-oAw9leGU04IAbcS4WN3FC1SH3vq5OdPxrVJCR16iIUMPsep1" \
  -d '{
    "title": "任務名稱",
    "cat": "feature",
    "status": "ready",
    "auto": true,
    "thought": "描述要做什麼"
  }'
```

### 關鍵欄位

| 欄位 | 值 | 意義 |
|------|-----|------|
| `status` | `ready` | auto-executor 才會撿（不是 `pending`！） |
| `auto` | `true` | 標記為可自動執行 |
| `thought` | 描述文字 | 這就是 AI 看到的指令，寫越清楚越好 |

### 好的 thought 範例
```
【設計】用 Python 掃描 ~/.openclaw/workspace/armory/ 下所有工具目錄
【已完成】aegis_scanner.py 基本掃描邏輯
【需要】對每個工具執行 python3 -c "import tool_module; print('OK')" 驗證
【測試方式】armory/ 下所有 .py 檔案都能 import 成功
【產出物】armory_validation_report.json
```

### 壞的 thought 範例
```
驗證 armory 工具
```
→ 太模糊，auto-executor 不知道要做什麼

## 三、品質評分

### 評分制度
| 等級 | 分數 | 意義 |
|------|------|------|
| A | ≥ 90 | 優秀 |
| B | ≥ 70 | 合格 |
| C | ≥ 60 | 勉強通過 |
| F | < 60 | 失敗，需重做 |

### 評分項目
```
1. 執行完整度 (40 分)
   - 是否正常結束（exit code 0）
   - 是否在時限內完成

2. 產出物驗證 (30 分)
   - 是否有實際產出（檔案、報告、資料）
   - 產出格式是否正確

3. 內容品質 (30 分)
   - 輸出是否有意義（不是空白或錯誤訊息）
   - 是否符合任務描述的要求
```

### 查看某任務的評分
```bash
# 查執行記錄（含評分）
curl -s "http://localhost:3011/api/openclaw/runs?taskId=TASK_ID" \
  -H "Authorization: Bearer oc-oAw9leGU04IAbcS4WN3FC1SH3vq5OdPxrVJCR16iIUMPsep1"
```

## 四、Sandbox 規則

### 環境
```
工作目錄：~/.openclaw/workspace/sandbox/
產出目錄：~/.openclaw/workspace/sandbox/output/
時限：120 秒
最大重試：2 次
```

### 可用的東西
- 所有 bash 指令（ls、cat、mkdir、cp...）
- Python3
- Node.js
- curl（但沒有 API Key 環境變數）
- 讀/寫 sandbox/ 下的檔案

### 不可用的東西
- API Key 環境變數（被過濾了）
- 直接存取 Supabase
- 修改 server 程式碼
- 存取 workspace/ 以外的目錄（理論上）

### 產出物怎麼保留
```
寫到 sandbox/output/ → 永久保留
寫到 sandbox/ 其他目錄 → 看情況，可能被清理
寫到 /tmp/ → 重啟後消失
```

## 五、Agent 類型

### 四種 Agent
| Agent | 擅長 | 用於 |
|-------|------|------|
| `cursor` | 前端/UI | React 組件、CSS、頁面 |
| `codex` | 後端/系統 | API、資料庫、腳本 |
| `openclaw` | 通用 | 分析、報告、文件 |
| `auto` | 自動選擇 | 系統根據任務內容自己判斷 |

### 指定 Agent（在任務裡設定）
```bash
curl -X POST "http://localhost:3011/api/openclaw/tasks?allowStub=1" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer oc-oAw9leGU04IAbcS4WN3FC1SH3vq5OdPxrVJCR16iIUMPsep1" \
  -d '{
    "title": "修復前端 Bug",
    "status": "ready",
    "auto": true,
    "agent": {"type": "cursor"},
    "thought": "修復 Dashboard.tsx 的載入錯誤"
  }'
```

## 六、排程與自動化

### Cron 表達式範例
| 表達式 | 意義 |
|--------|------|
| `0 */6 * * *` | 每 6 小時 |
| `0 3 * * 1` | 每週一凌晨 3 點 |
| `30 2 * * *` | 每天凌晨 2:30 |
| `0 18 * * 5` | 每週五下午 6 點 |

### 查看自動化規則
```bash
curl -s http://localhost:3011/api/openclaw/automations \
  -H "Authorization: Bearer oc-oAw9leGU04IAbcS4WN3FC1SH3vq5OdPxrVJCR16iIUMPsep1"
```

## 七、常見問題

| 問題 | 原因 | 解法 |
|------|------|------|
| 任務建了但不被執行 | status 不是 `ready` | 改成 `ready` |
| 任務一直 F 分 | thought 寫太模糊 | 用【設計】【需要】【產出物】格式重寫 |
| auto-executor 停了 | 熔斷器打開了 | 等 5 分鐘，或查 log |
| 執行超時 | 任務太複雜 | 拆成多個小任務 |
| 沒有產出物 | sandbox 寫錯路徑 | 確認寫到 `sandbox/output/` |
