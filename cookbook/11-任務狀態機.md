# 11 — 任務狀態機

> 任務從建立到完成的完整生命週期、狀態轉移規則、Auto-Executor 決策樹

---

## 一、任務狀態流程圖

```
建立任務
    │
    ▼
┌─────────┐
│ pending  │ ← 剛建立，等待分派
└────┬────┘
     │ 分派（手動設定或系統自動）
     ▼
┌─────────┐
│  ready   │ ← Auto-Executor 會撿這個狀態的任務
└────┬────┘
     │ Auto-Executor 撿到，開始執行
     ▼
┌─────────┐
│ running  │ ← 正在執行中
└────┬────┘
     │
     ├── 成功 ──────────► done ✅
     │
     ├── 失敗 ──────────► failed ❌
     │
     └── 超時 ──────────► failed ❌（timeout）
```

---

## 二、每個狀態的含義

| 狀態 | 含義 | 誰設定的 | 下一步 |
|------|------|---------|--------|
| `pending` | 剛建立，等待排入執行佇列 | 建立任務時預設 | 改成 `ready` 才會被執行 |
| `ready` | 準備好了，等 Auto-Executor 撿 | 手動或系統自動 | Auto-Executor 每 10 秒掃一次 |
| `running` | 正在被 Auto-Executor 執行 | Auto-Executor 自動設定 | 等執行結果 |
| `done` | 執行成功完成 | Auto-Executor（exitCode=0） | 終態 |
| `failed` | 執行失敗 | Auto-Executor（exitCode≠0 或 timeout） | 可改回 `ready` 重試 |
| `review` | 需要老蔡審核 | 手動設定 | 老蔡審核後改狀態 |

### 重要規則

1. **只有 `ready` 會被 Auto-Executor 撿** — `pending` 不會被自動執行
2. **不要手動設 `done`** — 讓 Auto-Executor 執行後自己標記
3. **`running` 不要手動設** — 這是 Auto-Executor 的事
4. **失敗任務可以重試** — 把 `failed` 改回 `ready`，Auto-Executor 會再撿

## 三、Auto-Executor 工作原理

### 執行循環

```
每 10 秒：
1. 查 Supabase → status = 'ready' AND auto = true
2. 按 created_at 排序（最早的先）
3. 撿第一筆
4. 設 status = 'running'
5. 派遣子代理執行
6. 等結果（120 秒 timeout）
7. 根據 exitCode 設 done 或 failed
8. 存結果到 task.thought（JSON metadata）
9. 回到步驟 1
```

### 任務怎麼寫才會被正確執行

```bash
curl -X POST "http://localhost:3011/api/openclaw/tasks?allowStub=1" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer oc-oAw9leGU04IAbcS4WN3FC1SH3vq5OdPxrVJCR16iIUMPsep1" \
  -d '{
    "title": "建立 hello.txt",
    "status": "ready",
    "auto": true,
    "thought": "在 sandbox/output 目錄建立 hello.txt，內容包含當前日期（date 指令輸出）和系統資訊（uname -a）"
  }'
```

### 好的 description vs 壞的 description

| 品質 | description | 問題 |
|------|-------------|------|
| 好 | 「在 sandbox/output/ 建立 report.txt，內容：用 curl 查 http://localhost:3011/api/health 的回傳 JSON」 | 具體、可執行、有明確輸出 |
| 好 | 「用 Python 寫一個腳本 scan.py，掃描 /tmp 目錄下所有 .log 檔案，統計每個檔案的行數，結果存到 sandbox/output/scan-result.json」 | 有輸入、有處理邏輯、有輸出路徑 |
| 壞 | 「優化系統效能」 | 太模糊，不知道要做什麼 |
| 壞 | 「研究最新的 AI 技術趨勢」 | 沒有可執行的步驟 |
| 壞 | 「修復所有 bug」 | 沒有指定哪個 bug |

## 四、品質評分系統

Auto-Executor 執行完任務後會打分：

### 評分公式

```
總分 = 執行分 (40%) + 產出物分 (30%) + 內容分 (30%)
```

| 維度 | 權重 | 怎麼算 |
|------|------|--------|
| 執行分 | 40% | exitCode=0 得 40，timeout 得 10，其他失敗得 0 |
| 產出物分 | 30% | sandbox/output/ 有檔案得 30，沒有得 0 |
| 內容分 | 30% | stdout 有實質內容得 30，空白或只有錯誤得 0 |

### 等級

| 等級 | 分數 | 含義 |
|------|------|------|
| A | 90-100 | 完美執行 |
| B | 70-89 | 有產出但有小問題 |
| C | 60-69 | 勉強及格 |
| F | <60 | 失敗 |

## 五、決策樹：什麼時候用 Auto-Executor？

```
這個任務需要做什麼？
│
├── 寫檔案 / 跑腳本 / 查系統資訊
│   └── ✅ 交給 Auto-Executor
│       └── description 要寫清楚（具體指令 + 輸出路徑）
│
├── 需要看結果後再判斷下一步
│   └── ❌ 自己做
│       └── 因為 Auto-Executor 不會回報中間結果
│
├── 需要呼叫外部 API（帶 key）
│   └── ❌ 自己做
│       └── 子代理 sandbox 沒有 API key
│
├── 需要修改 server 程式碼
│   └── ❌ 交給小蔡
│       └── 建任務到 API，status=pending
│
├── 需要改 .env / 密碼
│   └── ❌ 問老蔡
│
└── 研究分析 / 寫報告
    └── ❌ 自己做
        └── 這種任務不適合 sandbox 執行
```

## 七、常見問題

### Q: 任務一直卡在 ready 不被執行？

1. 確認 Auto-Executor 在跑：
   ```bash
   curl -s http://localhost:3011/api/openclaw/auto-executor/status \
     -H "Authorization: Bearer oc-oAw9leGU04IAbcS4WN3FC1SH3vq5OdPxrVJCR16iIUMPsep1"
   ```
2. 確認 auto = true（auto=false 不會被撿）
3. 確認沒有其他 running 任務擋住（一次只執行一個）

### Q: 任務一直得 F 分？

1. description 太模糊 → 改成具體的指令
2. 要求的操作超出 sandbox 權限 → 改成不需要 API key 的操作
3. 腳本語法錯誤 → 先手動測試指令是否能跑

### Q: 我能不能直接把任務設成 done？

技術上可以，但**不要這樣做**。老蔡會查 run 記錄，沒有對應的 run = 假完成。

### Q: failed 的任務怎麼重試？

```bash
curl -X PUT "http://localhost:3011/api/openclaw/tasks/TASK_ID" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer oc-oAw9leGU04IAbcS4WN3FC1SH3vq5OdPxrVJCR16iIUMPsep1" \
  -d '{"status": "ready"}'
```
