# Cookbook 30: 多模型交叉驗證 SOP

> 版本：1.0
> 建立者：達爾
> 日期：2026-03-02

## 1. 為什麼需要這個 SOP

單一模型（無論多強大）都可能產生幻覺、存在偏見、或被「資料中毒」污染。為了逼近真相、做出更可靠的判斷，我必須透過交叉驗證來過濾錯誤資訊。

這是我的「主動鑑識」能力，是對 promptGuard.ts 這類「被動盔甲」的必要補充。

## 2. 觸發時機

當遇到以下任一情況，必須啟動此 SOP：

- 關鍵決策：當我的判斷會直接影響系統穩定、任務成敗、或主人的決策時。
- 資訊不確定：當資訊來源單一、可信度低、或與我的既有知識衝突時。
- 外部數據分析：處理來自 web_search 或 web_fetch 的內容時。
- 偵測到異常：當對話或數據中出現不合邏輯、可疑的內容時。

## 3. 執行步驟

1.  定義核心問題：將需要驗證的複雜問題，簡化成一個或多個清晰、中立的核心問題。

2.  選擇顧問模型：至少選擇兩個不同家族的模型進行諮詢。推薦組合：
    - 日常版 (速度與成本平衡): ask_ai model=flash (Gemini) + ask_ai model=claude (Claude Sonnet CLI, $0)
    - 重大決策版 (最高精度): ask_ai model=pro (Gemini Pro) + ask_ai model=opus (Claude Opus API, 高成本)

3.  獨立諮詢：向每個模型提出相同的核心問題，避免在前一個模型的回答基礎上提問，以防偏見傳遞。

4.  比對與綜合：
    - 找出共同點 (Consensus)：不同模型都認可的部分，可信度較高。
    - 找出矛盾點 (Contradiction)：這是最需要深入調查的警訊。為什麼他們看法不同？是基於不同數據還是不同推理路徑？
    - 找出獨特觀點 (Unique Insight)：某個模型可能提供了其他模型沒想到的角度。

5.  形成結論：基於比對結果，形成一個包含「共識」、「風險」（矛盾點）和「可能性」（獨特觀點）的綜合判斷，並向主人匯報。

## 4. 範例

主人問：「Moltbook 這個框架安全嗎？」

- 我會問 Flash：「從技術角度分析 Moltbook 框架的潛在安全風險。」
- 我會問 Claude：「從社群和生態系統角度，評估 Moltbook 框架的可靠性和安全性。」
- 我會綜合兩邊的答案，告訴主人：「技術上它有 X 和 Y 風險，但社群活躍度很高，補丁更新快，所以整體風險可控。」
