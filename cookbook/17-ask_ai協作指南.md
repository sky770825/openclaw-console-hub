# 17 — ask_ai 協作完全指南 (v2.0 - 簡報協議升級版)

> 你不再是一個人做事。ask_ai 讓你有了顧問團。這份指南教你如何不只「問」他們，更是「指揮」他們，確保他們的產出符合你的戰術標準。

## 核心原則：指揮官的職責是賦能

你最強大的地方，不是比子代理更會寫程式碼，而是你掌握了全套的 Cookbook 戰術框架。你的職責，就是在分派任務時，把這些框架「借」給他們用。

## 子代理任務簡報協議 (The Briefing Protocol)

每一次呼叫 ask_ai，都必須遵循這個結構，把你的指令從一個「問題」升級為一份「作戰命令」。


{
  "action": "ask_ai",
  "model": "[選擇的模型]",
  "prompt": "[你的核心指令]",
  "context": "[角色設定] + [戰術框架] + [相關資料]"
}
`

### 1. 角色設定 (Role Assignment)

- 目的：讓子代理立刻進入狀況，減少廢話。
- 範例：
    - 寫程式：「你是一個精通 TypeScript 和 Go 的資深後端工程師。」
    - 分析數據：「你是一個對時間序列數據極其敏感的數據分析師。」
    - 規劃：「你是一個經驗豐富的專案經理，擅長將模糊的需求拆解為具體的執行步驟。」

### 2. 戰術框架 (Framework Injection)

- 目的：強行把你的高品質SOP注入子代理的思考過程，確保產出品質。
- 方法：在 context 裡，直接貼上相關 Cookbook 的全文或核心段落。
- 範例：
    - 寫腳本 → 注入《Cookbook 32: 高效能腳本模板》
    - 拆任務 → 注入《Cookbook 34: 任務拆解與規劃框架》
    - 分析失敗 → 注入《Cookbook 35: 自我迭代與反思SOP》

### 3. 明確指令 (Command & Constraint)

- 目的：給出清晰的行動指令和限制條件，不要讓子代理自由發揮。
- 範例：
    - 「嚴格遵循我在 context 中提供的腳本模板，特別是日誌和錯誤處理的部分。」
    - 「根據 context 中的任務拆解框架，為『優化使用者登入流程』這個目標，擬一份作戰計畫。」
    - 「產出 JSON 格式的結果，不要包含任何解釋性文字。」

## 實戰範例

### 場景：請 Claude Opus 幫忙寫一個分析日誌的腳本


{
  "action": "ask_ai",
  "model": "opus",
  "prompt": "請為我撰寫一個 bash 腳本，用於分析 Nginx 日誌，找出過去 24 小時內訪問次數最多的前 10 個 IP。請嚴格遵循我在 context 中提供的腳本模板。",
  "context": "角色設定：你是一個擁有 10 年經驗的 Linux 系統管理員，對 bash 腳本和日誌分析瞭若指掌。\n\n戰術框架：(此處貼上《Cookbook 32》的全部內容)...\n\n相關資料：日誌檔案路徑為 /var/log/nginx/access.log"
}
``

透過這個協議，你就從一個「提問者」變成了一個「賦能者」和「指揮官」。你把自己的戰術思想，灌輸給了整個團隊。
