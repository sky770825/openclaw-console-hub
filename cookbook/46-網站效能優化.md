## 1. 效能指標（Core Web Vitals + 輔助指標）

### 1.1 Core Web Vitals（Google 排名因素）

Google 在 2024 年將 INP 正式取代 FID 成為 Core Web Vital。這三項指標直接影響 SEO 排名。

| 指標 | 全名 | 測量什麼 | 良好 | 需改善 | 差 |
|------|------|----------|------|--------|-----|
| **LCP** | Largest Contentful Paint | 最大內容元素繪製時間 | ≤ 2.5s | 2.5–4.0s | > 4.0s |
| **INP** | Interaction to Next Paint | 互動到下一次繪製的延遲 | ≤ 200ms | 200–500ms | > 500ms |
| **CLS** | Cumulative Layout Shift | 累計版面位移量 | ≤ 0.1 | 0.1–0.25 | > 0.25 |

```
用白話說：
LCP = 使用者看到主要內容的速度（越快越好）
INP = 點東西之後畫面反應的速度（越快越好）
CLS = 畫面有沒有亂跳（越少越好）
```

### 1.2 輔助指標

| 指標 | 全名 | 測量什麼 | 目標值 |
|------|------|----------|--------|
| **FCP** | First Contentful Paint | 第一個內容元素出現 | ≤ 1.8s |
| **TTFB** | Time to First Byte | 伺服器回應第一個位元組 | ≤ 800ms |
| **TBT** | Total Blocking Time | 主執行緒被阻塞的總時間 | ≤ 200ms |
| **SI** | Speed Index | 內容可見速度的綜合分數 | ≤ 3.4s |
| **TTI** | Time to Interactive | 頁面可完全互動的時間 | ≤ 3.8s |

### 1.3 指標之間的因果關係

```
TTFB 慢 → FCP 慢 → LCP 慢（連鎖反應）
TBT 高  → INP 差（主執行緒被 JS 佔滿，互動延遲）
沒設寬高 → CLS 高（圖片/廣告載入後版面位移）
```

### 1.4 各指標的常見元兇

| 指標 | 常見元兇 | 對策速查 |
|------|----------|----------|
| LCP 慢 | 大圖未壓縮、伺服器慢、render-blocking CSS/JS | 壓圖 + preload + SSR |
| INP 差 | 大量 JS 在主執行緒、複雜事件處理器 | Code splitting + Web Worker + debounce |
| CLS 高 | 圖片無尺寸、動態插入內容、Web Font 閃爍 | 設 width/height + font-display: optional |
| FCP 慢 | Render-blocking 資源、大型 CSS | Critical CSS + async 載入 |
| TTFB 慢 | 伺服器效能差、無 CDN、無快取 | CDN + 快取 + 資料庫優化 |
| TBT 高 | 未分割的大型 JS bundle | Code splitting + tree shaking + defer |

## 3. 前端優化

### 3.1 Code Splitting（程式碼分割）

#### 為什麼需要

```
問題：整個 app 打成一個 bundle.js（2MB）
     → 使用者只看首頁也要下載全部 2MB
     → FCP 慢、TBT 高、使用者等到不耐煩

解法：按路由/功能拆分，只載入當前需要的程式碼
     → 首頁 bundle 從 2MB 降到 200KB
     → 其他頁面需要時再動態載入
```

#### React.lazy + Suspense

```tsx
// 改造前：全部 import（全部打包在一起）
import HomePage from './pages/HomePage'
import AboutPage from './pages/AboutPage'
import ContactPage from './pages/ContactPage'
import AdminDashboard from './pages/AdminDashboard'

// 改造後：動態 import（按需載入）
import { lazy, Suspense } from 'react'

const HomePage = lazy(() => import('./pages/HomePage'))
const AboutPage = lazy(() => import('./pages/AboutPage'))
const ContactPage = lazy(() => import('./pages/ContactPage'))
const AdminDashboard = lazy(() => import('./pages/AdminDashboard'))

// 搭配 Suspense 顯示載入中
function App() {
  return (
    <Suspense fallback={<LoadingSpinner />}>
      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/about" element={<AboutPage />} />
        <Route path="/contact" element={<ContactPage />} />
        <Route path="/admin/*" element={<AdminDashboard />} />
      </Routes>
    </Suspense>
  )
}

// 進階：按區塊拆分（不只是路由）
const HeavyChart = lazy(() => import('./components/HeavyChart'))
const PDFViewer = lazy(() => import('./components/PDFViewer'))

function Dashboard() {
  const [showChart, setShowChart] = useState(false)
  return (
    <div>
      <button onClick={() => setShowChart(true)}>顯示圖表</button>
      {showChart && (
        <Suspense fallback={<div>載入圖表中...</div>}>
          <HeavyChart data={data} />
        </Suspense>
      )}
    </div>
  )
}
```

### 3.2 Tree Shaking（搖樹優化）

#### 原理

```
Tree Shaking = 移除程式碼中未被使用的部分

條件：
1. 必須使用 ESM（import/export），CJS（require）無法 tree shake
2. package.json 需標記 "sideEffects": false
3. 使用具名匯入，避免匯入整個套件
```

#### 確保 ESM

```ts
// 能 tree shake（ESM 具名匯入）
import { format, parseISO } from 'date-fns'

// 不能 tree shake（匯入整個套件）
import _ from 'lodash'
_.get(obj, 'a.b')  // 把整個 lodash 都打包了

// 改成：
import get from 'lodash/get'  // 只拿一個函數
// 或用 lodash-es（ESM 版本，支援 tree shake）
import { get } from 'lodash-es'
```

#### package.json sideEffects

```json
{
  "name": "my-library",
  "sideEffects": false
}
```

```json
{
  "name": "my-app",
  "sideEffects": [
    "*.css",
    "*.scss",
    "./src/polyfills.ts",
    "./src/analytics.ts"
  ]
}
```

#### 常見「tree shake 失敗」的套件

| 套件 | 問題 | 解法 |
|------|------|------|
| `lodash` | CJS 格式 | 改用 `lodash-es` 或 `lodash/函數名` |
| `moment.js` | CJS + locale 全包 | 改用 `date-fns` 或 `dayjs` |
| `@fortawesome/fontawesome` | 全圖示載入 | 用 `@fortawesome/react-fontawesome` + 單獨 import |
| `antd` 4.x | 未完全 ESM | antd 5.x 已修正，或用 babel-plugin-import |
| `@mui/material` | 深層匯入問題 | 使用 `@mui/material/Button` 路徑匯入 |

### 3.4 字型優化

#### font-display 策略

```css
@font-face {
  font-family: 'NotoSansTC';
  src: url('/fonts/NotoSansTC-Regular.woff2') format('woff2');
  font-weight: 400;
  font-style: normal;
  font-display: swap;  /* 先顯示系統字型，載入完再替換 */
}
```

| font-display 值 | 行為 | 適用場景 |
|------------------|------|----------|
| `swap` | 先用系統字型，載入完替換（可能閃爍） | **一般首選**，內文字型 |
| `optional` | 如果沒快取就放棄，不等待 | **效能最佳**，CLS 為零 |
| `fallback` | 等 100ms，沒載完就用系統字型 | 折衷方案 |
| `block` | 等最多 3 秒（白畫面） | **避免使用** |
| `auto` | 瀏覽器決定（通常等同 block） | **避免使用** |

#### 字型子集化（中文字型必做）

```bash
# 中文字型動輒 5-20MB，子集化後可降到 100-500KB

# 安裝 pyftsubset（fonttools）
pip install fonttools brotli

# 只保留常用繁體中文字（Big5 常用字 + ASCII）
pyftsubset NotoSansTC-Regular.ttf \
  --text-file=chinese-chars.txt \
  --output-file=NotoSansTC-Regular-subset.woff2 \
  --flavor=woff2 \
  --layout-features='*'

# 用 Google Fonts 的 Unicode Range（自動分片）
# Google Fonts 會自動把中文字型拆成 ~120 個小檔案
# 瀏覽器只下載頁面實際用到的分片
```

```css
/* Google Fonts 自動分片範例（自動產生） */
@font-face {
  font-family: 'Noto Sans TC';
  src: url('/fonts/noto-sans-tc-[0].woff2') format('woff2');
  unicode-range: U+4e00-4e54, U+4e56-4e89; /* 只含這些字 */
}
@font-face {
  font-family: 'Noto Sans TC';
  src: url('/fonts/noto-sans-tc-[1].woff2') format('woff2');
  unicode-range: U+4e8a-4ecf, U+4ed1-4f00;
}
/* ... 瀏覽器會根據頁面內容只下載需要的分片 */
```

#### Variable Fonts（可變字型）

```css
/* 傳統做法：每個字重一個檔案 */
/* NotoSansTC-Light.woff2     ~200KB */
/* NotoSansTC-Regular.woff2   ~200KB */
/* NotoSansTC-Medium.woff2    ~200KB */
/* NotoSansTC-Bold.woff2      ~200KB */
/* 共 ~800KB */

/* Variable Font：一個檔案包含所有字重 */
@font-face {
  font-family: 'NotoSansTC';
  src: url('/fonts/NotoSansTC-Variable.woff2') format('woff2-variations');
  font-weight: 100 900;  /* 支援 100 到 900 的任意字重 */
  font-display: swap;
}

/* 使用時可以指定任意字重 */
h1 { font-weight: 750; }  /* 不再受限於 400/700 等固定值 */
body { font-weight: 350; }
```

#### next/font（Next.js 自動最佳化）

```tsx
// app/layout.tsx
import { Noto_Sans_TC } from 'next/font/google'

const notoSansTC = Noto_Sans_TC({
  subsets: ['latin'],       // 必填
  weight: ['400', '500', '700'],
  display: 'swap',
  preload: true,
  // Next.js 會自動：
  // 1. Build 時下載字型（不經 Google CDN）
  // 2. 內聯 @font-face CSS
  // 3. 加入 size-adjust 防止 CLS
})

export default function RootLayout({ children }) {
  return (
    <html lang="zh-TW" className={notoSansTC.className}>
      <body>{children}</body>
    </html>
  )
}

// 本地字型
import localFont from 'next/font/local'

const myFont = localFont({
  src: [
    { path: '../fonts/MyFont-Regular.woff2', weight: '400' },
    { path: '../fonts/MyFont-Bold.woff2', weight: '700' },
  ],
  display: 'swap',
  variable: '--font-my',  // CSS 變數
})
```

#### 字型最佳實踐整理

```
1. 格式用 woff2（壓縮率最高，瀏覽器支援 97%+）
2. 中文字型一定要子集化或用 Google Fonts 分片
3. font-display 用 swap 或 optional
4. preload 首屏用的字型
5. 字型檔案數量控制在 2-3 個以內
6. Variable Font 取代多個字重檔案
```

### 3.6 JavaScript 優化

#### Script 載入：defer vs async

```html
<!-- 阻塞渲染（最差，避免使用） -->
<script src="app.js"></script>

<!-- async：下載不阻塞，下載完立刻執行（執行時阻塞） -->
<!-- 適合：獨立腳本（GA、廣告），不依賴其他 script -->
<script async src="analytics.js"></script>

<!-- defer：下載不阻塞，HTML 解析完才執行，保持順序 -->
<!-- 適合：主要 app 程式碼 -->
<script defer src="app.js"></script>

<!-- type=module：自帶 defer 行為 -->
<script type="module" src="app.js"></script>
```

```
時間軸比較：

普通 script:   HTML解析 ──暫停── [下載JS] [執行JS] ──繼續解析──

async:         HTML解析 ────────────────────── 繼續解析
                        [下載JS] [執行,暫停HTML]

defer:         HTML解析 ──────────────────── 解析完成 [執行JS]
                        [下載JS]

結論：幾乎所有 <script> 都該加 defer 或放在 </body> 前
```

## 4. 渲染策略（SSR/SSG/ISR/CSR/Streaming）

### 4.1 策略總覽

| 策略 | 全名 | 渲染時機 | 適合場景 |
|------|------|----------|----------|
| **SSG** | Static Site Generation | Build 時 | 部落格、行銷頁、文件站 |
| **SSR** | Server-Side Rendering | 每次請求時 | 個人化內容、即時資料 |
| **ISR** | Incremental Static Regen | Build 時 + 背景更新 | 電商產品頁、新聞（Next.js） |
| **CSR** | Client-Side Rendering | 瀏覽器端 | SaaS 後台、Dashboard |
| **Streaming** | Streaming SSR | 請求時串流 | 複雜頁面（Next.js App Router） |

### 4.2 選擇決策流程

```
頁面內容會因使用者不同嗎？
├─ 否 → 內容多久更新一次？
│       ├─ 很少更新（每週/每月） → SSG
│       ├─ 常更新（每小時/每天） → ISR
│       └─ 即時更新（每分鐘）   → SSR
│
├─ 是 → 需要 SEO 嗎？
│       ├─ 需要 → SSR 或 Streaming SSR
│       └─ 不需要 → CSR
│
└─ 部分需要 → 混合策略（Shell SSG + 動態區塊 CSR）
```

## 5. 快取策略

### 5.1 Cache-Control Header

```
語法：Cache-Control: <指令>, <指令>, ...
```

| 指令 | 意義 | 範例 |
|------|------|------|
| `public` | 任何快取都可以存（CDN、瀏覽器） | 靜態資源 |
| `private` | 只有瀏覽器可以存（不給 CDN） | 個人化內容 |
| `no-cache` | 可以存，但每次要驗證（不是「不快取」） | HTML |
| `no-store` | 完全不存 | 敏感資料 |
| `max-age=N` | 快取 N 秒 | `max-age=31536000`（一年） |
| `s-maxage=N` | CDN 層快取 N 秒 | `s-maxage=3600`（CDN 一小時） |
| `stale-while-revalidate=N` | 過期後 N 秒內先用舊的，背景更新 | 高可用性 |
| `immutable` | 永遠不會變（搭配 hash 檔名） | `app.a1b2c3.js` |

#### 各類資源建議設定

```nginx
# Nginx 設定範例

# HTML 文件：不快取（每次檢查是否更新）
location ~* \.html$ {
    add_header Cache-Control "no-cache";
}

# 有 hash 的靜態資源：快取一年（檔名變了自然失效）
location ~* \.[a-f0-9]{8,}\.(js|css|woff2|avif|webp)$ {
    add_header Cache-Control "public, max-age=31536000, immutable";
}

# 圖片（無 hash）：快取一個月
location ~* \.(jpg|jpeg|png|gif|svg)$ {
    add_header Cache-Control "public, max-age=2592000";
}

# API 回應：不快取或短快取
location /api/ {
    add_header Cache-Control "no-store";
}

# API 回應（可快取的）：CDN 快取 60 秒
location /api/products/ {
    add_header Cache-Control "public, s-maxage=60, stale-while-revalidate=300";
}
```

#### Next.js / Vercel 設定

```ts

module.exports = {
  async headers() {
    return [
      {
        source: '/:all*(svg|jpg|png|webp|avif)',

}
```

### 5.2 ETag（驗證型快取）

```
流程：
1. Server 回應帶 ETag: "abc123"（內容的 hash）
2. 瀏覽器下次請求帶 If-None-Match: "abc123"
3. Server 比對：
   - 沒變 → 回 304 Not Modified（不傳內容，省頻寬）
   - 有變 → 回 200 + 新內容 + 新 ETag
```

```ts
// Express.js 設定
import express from 'express'
import { createHash } from 'crypto'

const app = express()

// Express 預設就有 ETag（weak），可以強化
app.set('etag', 'strong')

// 或手動設定
app.get('/api/data', async (req, res) => {
  const data = await fetchData()
  const json = JSON.stringify(data)
  const etag = createHash('md5').update(json).digest('hex')

  // 檢查 If-None-Match
  if (req.headers['if-none-match'] === etag) {
    return res.status(304).end()
  }

  res.setHeader('ETag', etag)
  res.setHeader('Cache-Control', 'no-cache') // 搭配 no-cache 使用
  res.json(data)
})
```

### 5.3 Service Worker 快取

```ts
// sw.ts — Workbox 版本（推薦）
import { precacheAndRoute } from 'workbox-precaching'
import { registerRoute } from 'workbox-routing'
import {
  CacheFirst,
  NetworkFirst,
  StaleWhileRevalidate,
} from 'workbox-strategies'
import { ExpirationPlugin } from 'workbox-expiration'
import { CacheableResponsePlugin } from 'workbox-cacheable-response'

// 預快取 build 產物
precacheAndRoute(self.__WB_MANIFEST)

// 圖片：Cache First（快取優先，沒有才去拿）
registerRoute(
  ({ request }) => request.destination === 'image',
  new CacheFirst({
    cacheName: 'images',
    plugins: [
      new CacheableResponsePlugin({ statuses: [0, 200] }),
      new ExpirationPlugin({
        maxEntries: 100,       // 最多 100 張
        maxAgeSeconds: 30 * 24 * 60 * 60, // 30 天
      }),
    ],
  })
)

// API 資料：Network First（先拿最新，失敗用快取）
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/'),
  new NetworkFirst({
    cacheName: 'api-cache',
    plugins: [
      new ExpirationPlugin({
        maxEntries: 50,
        maxAgeSeconds: 5 * 60, // 5 分鐘
      }),
    ],
  })
)

// CSS/JS：Stale While Revalidate（先用舊的，背景更新）
registerRoute(
  ({ request }) =>
    request.destination === 'style' || request.destination === 'script',
  new StaleWhileRevalidate({
    cacheName: 'static-resources',
  })
)
```

### 5.4 Stale-While-Revalidate 模式

```
概念：先給使用者舊資料（快），背景去拿新資料（準），
     下次請求就會拿到新的。犧牲一點即時性換極佳體驗。

適用：產品列表、新聞首頁、排行榜 —— 晚幾秒更新沒關係

不適用：購物車、帳戶餘額、即時通訊 —— 必須是最新的
```

```ts
// React Query / TanStack Query 的 stale-while-revalidate
import { useQuery } from '@tanstack/react-query'

function ProductList() {
  const { data, isLoading } = useQuery({
    queryKey: ['products'],
    queryFn: fetchProducts,
    staleTime: 60 * 1000,        // 60 秒內視為新鮮（不重新拿）
    gcTime: 5 * 60 * 1000,       // 5 分鐘後從快取移除
    refetchOnWindowFocus: true,   // 切回分頁時自動更新
  })

  if (isLoading) return <Skeleton />
  return <ProductGrid products={data} />
}
```

### 5.5 快取策略速查表

| 資源類型 | Cache-Control | 理由 |
|----------|---------------|------|
| HTML | `no-cache` | 每次驗證，確保拿到最新入口 |
| JS/CSS（有 hash） | `max-age=31536000, immutable` | 檔名變了就是新檔，不用驗證 |
| 圖片（無 hash） | `max-age=2592000` | 一個月，夠久又不會太舊 |
| 字型 | `max-age=31536000, immutable` | 字型幾乎不會改 |
| API（公開） | `s-maxage=60, stale-while-revalidate=300` | CDN 一分鐘，背景五分鐘 |
| API（私人） | `private, no-store` | 不給 CDN 存 |

## 6. CDN 設定

### 6.1 CDN 基本概念

```
沒有 CDN：
使用者（台灣）→ 海底電纜 → 伺服器（美西）→ 海底電纜 → 使用者
RTT: ~200ms

有 CDN：
使用者（台灣）→ CDN 節點（台北）→ 使用者
RTT: ~5ms

CDN 把靜態檔案複製到全球節點，使用者從最近的節點取得內容。
```

### 6.2 Cloudflare 設定

#### 基本設定（免費方案就很夠用）

```
1. 把網域 DNS 指向 Cloudflare
2. SSL/TLS → Full (strict)
3. Speed → Auto Minify: ✅ JavaScript, ✅ CSS, ✅ HTML
4. Caching:
   - Browser Cache TTL: Respect Existing Headers
   - Always Online: ✅（原始伺服器掛了也能看到快取版）
5. Speed → Brotli: ✅（比 gzip 壓得更小）
```

#### Page Rules（效能相關）

```
規則 1: *example.com/static/*
  - Cache Level: Cache Everything
  - Edge Cache TTL: 1 month
  - Browser Cache TTL: 1 year

規則 2: *example.com/api/*
  - Cache Level: Bypass（API 不快取）

規則 3: *example.com/
  - Cache Level: Cache Everything
  - Edge Cache TTL: 2 hours（首頁 CDN 快取 2 小時）
```

#### Cloudflare Workers（邊緣運算）

```js
      ? 'https://v2.example.com'
      : 'https://v1.example.com'

  },
```

#### 索引（Index）

```sql
-- 找出慢查詢（PostgreSQL）
SELECT query, mean_exec_time, calls
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 20;

-- 查看查詢計畫
EXPLAIN ANALYZE
SELECT * FROM products WHERE category = 'electronics' AND price < 1000;

-- 加索引
CREATE INDEX idx_products_category_price ON products(category, price);

-- 複合索引順序很重要：高選擇性的欄位放前面
-- 好：(user_id, created_at)  → 先篩出少數使用者，再排時間
-- 差：(created_at, user_id)  → 先篩日期範圍（很多），再找使用者
```

#### 常見查詢優化

```sql
-- 1. 只 SELECT 需要的欄位（不要 SELECT *）
-- 差
SELECT * FROM orders WHERE user_id = 123;
-- 好
SELECT id, status, total, created_at FROM orders WHERE user_id = 123;

-- 2. 分頁用 cursor-based，不要 OFFSET
-- 差（OFFSET 1000000 要先掃過 100 萬行）
SELECT * FROM products ORDER BY id LIMIT 20 OFFSET 1000000;
-- 好（直接跳到指定位置）
SELECT * FROM products WHERE id > 1000000 ORDER BY id LIMIT 20;

-- 3. 避免 LIKE '%keyword%'（全表掃描）
-- 改用 Full-Text Search 或 pg_trgm
CREATE INDEX idx_products_name_trgm ON products USING gin(name gin_trgm_ops);
SELECT * FROM products WHERE name ILIKE '%phone%';  -- 用了索引
```

### 7.2 N+1 問題

```
N+1 問題是後端效能殺手 No.1

場景：顯示 50 篇文章和作者
  查詢 1：SELECT * FROM posts LIMIT 50          （1 次）
  查詢 2-51：SELECT * FROM users WHERE id = ?    （50 次）
  共 51 次查詢！

解法：一次撈完
  查詢 1：SELECT * FROM posts LIMIT 50
  查詢 2：SELECT * FROM users WHERE id IN (1,2,3,...50)
  共 2 次查詢
```

#### Prisma 範例

```ts
// N+1 問題（Prisma 預設就會避免，但要注意 relation）
// 差：
const posts = await prisma.post.findMany()
for (const post of posts) {
  const author = await prisma.user.findUnique({
    where: { id: post.authorId },
  })
  // N+1！每篇文章都查一次使用者
}

// 好：用 include 一次載入
const posts = await prisma.post.findMany({
  include: {
    author: {
      select: { id: true, name: true, avatar: true },
    },
  },
  take: 50,
})
```

#### Drizzle ORM 範例

```ts
// Drizzle 用 JOIN 解決
const result = await db
  .select({
    post: posts,
    author: {
      id: users.id,
      name: users.name,
    },
  })
  .from(posts)
  .leftJoin(users, eq(posts.authorId, users.id))
  .limit(50)
```

### 7.3 Redis 快取

```ts
// redis-cache.ts
import Redis from 'ioredis'

const redis = new Redis(process.env.REDIS_URL)

// 快取包裝函數
async function cached<T>(
  key: string,
  ttl: number, // 秒
  fetcher: () => Promise<T>
): Promise<T> {
  // 1. 先查快取
  const cached = await redis.get(key)
  if (cached) {
    return JSON.parse(cached) as T
  }

  // 2. 快取沒有，去查資料庫
  const data = await fetcher()

  // 3. 存入快取
  await redis.setex(key, ttl, JSON.stringify(data))

  return data
}

// 使用
app.get('/api/products/:id', async (req, res) => {
  const product = await cached(
    `product:${req.params.id}`, // key
    300,                         // 5 分鐘
    () => db.product.findUnique({ where: { id: req.params.id } })
  )
  res.json(product)
})

// 清除快取（資料更新時）
app.put('/api/products/:id', async (req, res) => {
  const product = await db.product.update({
    where: { id: req.params.id },
    data: req.body,
  })

  // 清除相關快取
  await redis.del(`product:${req.params.id}`)
  await redis.del('products:list')  // 列表快取也要清

  res.json(product)
})
```

#### 快取模式選擇

```
Cache-Aside（旁路快取，最常用）：
  讀：查快取 → 沒有 → 查 DB → 寫入快取 → 回傳
  寫：更新 DB → 刪除快取
  適合：讀多寫少

Write-Through（直寫快取）：
  寫：同時更新 DB + 快取
  適合：寫完馬上要讀

Write-Behind（延遲寫入）：
  寫：更新快取 → 非同步寫 DB
  適合：大量寫入（需要接受資料遺失風險）
```

#### 1. 未壓縮的大圖片

```
問題：客戶給的原圖 5MB JPEG，直接丟上去
影響：LCP 從 1s 變 8s

修復：
- 轉 WebP/AVIF（省 50-80%）
- 設定 srcset 響應式尺寸

- 首屏大圖加 fetchpriority="high"
→ 詳見 cookbook/27-圖片素材優化指南.md
```

#### 2. 第三方腳本

```
問題：GA + GTM + Facebook Pixel + Hotjar + Intercom + 5 個廣告
影響：TBT 輕鬆破 2000ms，頁面操作卡頓

修復優先順序：
1. 審核：真的每個都需要嗎？（跟客戶確認）
2. 延遲載入：非必要的腳本 defer 或更晚載入
3. 用 Partytown 把第三方腳本移到 Web Worker
```

### 11.2 快速產生 Before/After 資料的腳本

```bash
#!/bin/bash
# scripts/perf-report.sh
# 用法：bash perf-report.sh https://example.com

URL=$1
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
OUTPUT_DIR="./perf-reports/${TIMESTAMP}"

mkdir -p "$OUTPUT_DIR"

echo "正在測試 ${URL}..."

# 跑 Lighthouse（桌面）
lighthouse "$URL" \
  --preset=desktop \
  --output=json \
  --output-path="${OUTPUT_DIR}/desktop.json" \
  --chrome-flags="--headless --no-sandbox" \
  --quiet

# 跑 Lighthouse（行動裝置）
lighthouse "$URL" \
  --output=json \
  --output-path="${OUTPUT_DIR}/mobile.json" \
  --chrome-flags="--headless --no-sandbox" \
  --quiet

# 提取關鍵數據
echo ""
echo "========== 效能報告 =========="
echo "網址：${URL}"
echo "時間：$(date)"
echo ""

for MODE in desktop mobile; do
  echo "--- ${MODE} ---"
  python3 -c "
import json
with open('${OUTPUT_DIR}/${MODE}.json') as f:
    d = json.load(f)
a = d['audits']
c = d['categories']['performance']
print(f\"Performance Score: {c['score'] * 100:.0f}\")
print(f\"FCP:  {a['first-contentful-paint']['displayValue']}\")
print(f\"LCP:  {a['largest-contentful-paint']['displayValue']}\")
print(f\"TBT:  {a['total-blocking-time']['displayValue']}\")
print(f\"CLS:  {a['cumulative-layout-shift']['displayValue']}\")
print(f\"SI:   {a['speed-index']['displayValue']}\")
"
  echo ""
done

echo "完整報告已存至 ${OUTPUT_DIR}/"
echo "==============================="
```
