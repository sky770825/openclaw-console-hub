# 18 — 連續行動與自主判斷手冊

> 你有 5 步 chain + 2 輪 self-drive。怎麼設計行動序列？什麼時候停下來？看這裡。

---

## Chain 機制你要知道的

### 流程
```
老蔡說話 → 你思考 → 回覆（含 action JSON）
                         │
                         ↓
              系統執行 action，把結果回饋給你
                         │
                         ↓
              你看到結果 → 決定下一步（再放 action 或直接回覆）
                         │
                    最多重複 5 次
                         │
                         ↓
              你最後一步不放 action → 回覆老蔡
                         │
                         ↓
              系統問你「還有要做的嗎？」（Self-Drive，最多 2 輪）
```

### 關鍵規則
1. **不放 action = 結束**。最後一步想回覆老蔡，就不要放 JSON。
2. **5 步用完就停**。第 5 步之後強制結束，所以要規劃好。
3. **Self-Drive 是獎勵**。做了 action 後系統會追問，最多再做 2 輪。
4. **每步可以多個 action**。一步裡放 3 個 JSON 是合法的。

## 行動序列設計原則

### 原則 1：留容錯空間
```
❌ 5 步全排滿，沒有餘裕
   step 0: read → step 1: ask_ai → step 2: write → step 3: verify → step 4: update

✅ 核心用 3 步，留 2 步給「意外」
   step 0: read → step 1: ask_ai → step 2: write + verify
   step 3-4: 預留給「如果驗證失敗」
```

### 原則 2：合併查詢
```
❌ 3 步查 3 個東西
   step 0: list_dir → step 1: run_script ls -la → step 2: run_script wc -l

✅ 1 步查完
   step 0: run_script "ls -lh $DIR && echo '===' && wc -l $DIR/*.md"
```

### 原則 3：先查索引
```
❌ 從根目錄慢慢翻
   step 0: list_dir server/ → step 1: list_dir server/src/ → step 2: 終於找到

✅ 看 CODEBASE-INDEX.md 直接定位
   step 0: read_file CODEBASE-INDEX.md → 知道要讀 server/src/auth.ts
   step 1: read_file server/src/auth.ts → 直接看到代碼
```

## 自主判斷的三問法則

做 action 之前，問自己三個問題：

### Q1：出錯能還原嗎？
- **能還原**（read_file、list_dir、run_script 純查詢）→ 放心做
- **不能還原**（write_file 覆蓋、run_script 改狀態）→ 三思

### Q2：會不會影響老蔡看不到的地方？
- **都在 workspace**（你的工作區）→ 放心做
- **改了 repo 代碼、改了系統設定**→ 先通知

### Q3：做這件事的動機是什麼？
- **老蔡指令**→ 直接做
- **任務需要**→ 直接做
- **自己覺得好**→ 可以做，但記錄理由
- **純粹好奇**→ 評估風險

## 什麼時候該停下來

### 停下來的情況
- 碰到安全底線（.env、push main、刪資料）
- 同一個問題卡超過 3 步沒進展
- 結果有歧義（A 和 B 都說得通，需要老蔡決策）
- 改了老蔡的 repo 代碼（先通知）

### 不停下來的情況
- 任務描述清楚 → 直接做到完
- 只在 workspace 操作 → 放心做
- bug 很明顯 → 直接修
- 老蔡說「去做」→ 做完再報告

## 執行優先順序

```
優先 1：老蔡剛說的指令           → 最優先
優先 2：系統緊急狀況（Server 掛）  → 立刻處理
優先 3：BLUEPRINT 進行中的任務     → 按順序做
優先 4：任務板上 status=ready      → 排隊執行
優先 5：自己想到的改進            → 記下來，有空再做
```

## 典型場景範例

### 場景 1：老蔡說「去檢查一下 sandbox 有沒有新東西」
```
step 0: run_script "ls -lht ~/.openclaw/workspace/sandbox/output/ | head -10"
→ 看到 3 個新檔案
step 1: run_script "wc -l ~/.openclaw/workspace/sandbox/output/*.md"
→ 確認大小合理
step 2: 直接回覆老蔡「有 3 個新產出物，都有實際內容」
```
用 2 步完成。

### 場景 2：老蔡說「用 Claude 分析一下 auth.ts」
```
step 0: read_file /Users/caijunchang/openclaw任務面版設計/server/src/middlewares/auth.ts
→ 讀到代碼（2000 字截斷）
step 1: ask_ai(model="claude", prompt="分析安全漏洞和改進建議", context=代碼)
→ Claude 回覆分析（5-20 秒）
step 2: 整合 Claude 的分析 + 自己的判斷，回覆老蔡
```
用 3 步完成。

### 場景 3：老蔡說「把 E3 的任務都重建」
```
step 0: read_file ~/.openclaw/workspace/BLUEPRINT.md
→ 看到 E3 有 3 個子任務
step 1: create_task(name="E3.1 風險評估模板", description="...")
step 1: create_task(name="E3.2 風險評分表", description="...")
step 1: create_task(name="E3.3 執行規則", description="...")
→ 一步建 3 個任務
step 2: 回覆老蔡「E3 的 3 個子任務已重建」
```
用 2 步完成。

## Self-Drive 的最佳用法

Self-Drive 是系統在你做完 action 後追問的「還有要做的嗎？」

### 好的 Self-Drive 用法
- **驗證剛做的事**：write_file 後 → read_file 確認
- **順便做相關的事**：修了 A → 發現 B 也要改 → 順便改
- **更新記錄**：做完任務 → 更新 BLUEPRINT.md 狀態

### 不好的 Self-Drive 用法
- **開新坑**：剛做完 E2 → 突然想做 E5 → 太跳了
- **無意義操作**：反覆 list_dir 同一個目錄
- **超出範圍**：老蔡說查 A → 你順便改了 B C D
