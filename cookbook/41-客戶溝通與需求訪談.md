## 1. 首次接洽流程

### 1.1 從收到詢問到報價的完整流程

```
收到詢問 → 24hr 內初步回覆 → 預約訪談 → 需求訪談（30-60min）
→ 整理需求文件 → 內部評估工時 → 製作報價單 → 寄出報價 → 追蹤跟進
```

| 步驟 | 時間 | 產出物 | 注意事項 |
|------|------|--------|----------|
| 收到詢問 | 當下 | — | 記錄來源管道（推薦/搜尋/社群） |
| 初步回覆 | 24hr 內 | 制式感謝訊息 + 預約時間 | **不要在這步就報價** |
| 需求訪談 | 30-60 min | 訪談紀錄 | 帶著問題清單去，不要空手 |
| 整理需求 | 1-2 天 | 需求文件初稿 | 用客戶的話，不是你的技術語言 |
| 內部評估 | 半天 | 工時估算表 | 加 20-30% 緩衝，新手加 50% |
| 製作報價 | 半天 | 報價單（三方案） | 見第 8 節報價話術 |
| 寄出報價 | — | Email / PDF | 附上需求確認書讓客戶覆核 |
| 追蹤跟進 | 3-7 天 | 跟進訊息 | 不要天天催，也不要完全不追 |

### 1.2 初步回覆模板（收到詢問後 24hr 內）

**LINE / 訊息版（簡短）：**

```
○○ 您好！感謝您的詢問 🙏

收到您關於 [網站/LINE OA/系統] 的需求了。
為了提供最適合您的方案，想跟您約個 30 分鐘的線上會議，
詳細了解一下您的想法和需求。

請問您這週 [二/三/四] 下午方便嗎？
我這邊可以配合的時段是：
- 週二 14:00-17:00
- 週四 10:00-12:00

期待與您進一步聊聊！
```

**Email 版（正式）：**

```
主旨：Re: 網站設計詢問 — [你的工作室名稱] 感謝您的來信

○○ 先生/小姐 您好，

感謝您來信詢問網站設計服務。

我是 [名字]，[工作室名稱] 的 [職稱]。
收到您關於 [簡述需求] 的需求，非常感興趣。

為了提供最準確的評估與報價，我想先跟您做一個 30-45 分鐘的需求訪談，
了解您的產業背景、目標客群和期望的功能。

可以的話，請問以下時段您方便嗎？
- 3/7（五）14:00-15:00
- 3/10（一）10:00-11:00
- 3/11（二）15:00-16:00

會議形式：Google Meet 線上會議 / 實體見面（皆可配合）

如果以上時段都不方便，也歡迎告訴我您的空檔，我來配合。

祝商祺
[簽名檔]
```

### 1.3 訪談前準備清單

```markdown
## 訪談前準備（至少提前 30 分鐘）

- [ ] 查看客戶公司/品牌（Google 搜尋、FB 粉專、現有網站）
- [ ] 了解客戶所在產業的競品網站（找 3 個參考）
- [ ] 準備好問題清單（見第 3 節）
- [ ] 準備好參考案例（自己的作品集，選 2-3 個相關的）
- [ ] 開好 Google Meet / Zoom 連結
- [ ] 準備好螢幕分享（Figma / 參考網站）
- [ ] 準備好筆記工具（Notion / Google Docs）
- [ ] 提前 5 分鐘進入會議室
```

### 1.4 不該在第一次接觸就做的事

| 不要做 | 為什麼 |
|--------|--------|
| 不要馬上報價 | 你還不了解需求，亂報會後悔 |
| 不要說「都可以」 | 客戶會覺得你沒專業判斷 |
| 不要批評客戶現有網站 | 可能是他朋友做的，或他自己做的 |
| 不要承諾交期 | 還沒評估就承諾 = 自己挖坑 |
| 不要急著展示技術 | 先聽客戶說，不是展示你會多少 |

## 2. 需求訪談技巧

### 2.1 訪談心態：你是醫生，不是服務生

```
❌ 服務生心態：「你要什麼我就做什麼」
✅ 醫生心態：「你哪裡不舒服？讓我診斷，我來開處方」
```

客戶來找你，不是因為他知道答案，而是因為他不知道。
你的工作是**引導他釐清真正的需求**，而不是被動等指令。

### 2.2 開放式問題 vs 封閉式問題

| 類型 | 範例 | 使用時機 |
|------|------|----------|
| 開放式 | 「您希望網站幫您解決什麼問題？」 | 訪談初期，了解全貌 |
| 封閉式 | 「您需要會員登入功能嗎？」 | 確認具體功能 |
| 選擇式 | 「您偏好 A 風格還是 B 風格？」 | 縮小範圍，加速決策 |
| 量表式 | 「1-10 分，目前網站您打幾分？」 | 量化感受，找出痛點 |

**訣竅：前半場多用開放式，後半場多用封閉式。**

### 2.3 追問法（深挖技巧）

客戶的第一個回答通常是「表面需求」，你要挖到「真正需求」。

**「為什麼」追問法：**

```
客戶：「我想要一個購物車功能。」
你：  「了解。方便問一下，你們目前商品大概有多少品項？」
客戶：「大概 20 幾個。」
你：  「那目前客戶是怎麼下單的？」
客戶：「都用 LINE 傳訊息，然後我再手動記。」
你：  「所以最困擾的其實不是沒有購物車，而是接單流程太花時間？」
客戶：「對對對！每天光回 LINE 就要兩三個小時。」
```

**看到了嗎？真正的需求不是「購物車」，而是「自動化接單」。**
也許一個 LINE OA + 自動回覆就解決了，根本不需要做電商。

**追問公式：**

```
表面需求 → 「目前怎麼處理的？」 → 「最花時間/最困擾的是？」 → 真正痛點
```

### 2.4 5W1H 框架

每一個需求都用 5W1H 拆解，就不會漏掉東西：

| W/H | 問題 | 範例 |
|-----|------|------|
| **Who** | 誰會用？ | 「這個網站主要是給誰看的？一般消費者還是經銷商？」 |
| **What** | 要做什麼？ | 「您希望網站有哪些功能？目前最需要的前三個是？」 |
| **Why** | 為什麼要做？ | 「做這個網站最主要想解決什麼問題？」 |
| **When** | 什麼時候要？ | 「有沒有特別的上線時間？例如配合新品上市？」 |
| **Where** | 在哪裡用？ | 「您的客戶主要用手機還是電腦？」 |
| **How** | 怎麼做？ | 「目前有沒有特別喜歡的參考網站？」 |

### 2.5 同理心溝通

**四個同理心句型，隨時都能用：**

```
1. 複述確認：「所以您的意思是⋯⋯對嗎？」
2. 情緒反映：「聽起來這個問題讓您蠻困擾的。」
3. 正向肯定：「您這個想法很好，很多客戶也有類似的需求。」
4. 不確定時：「這個我先記下來，回去評估一下可行性再跟您回報。」
```

**千萬不要說的話：**

| 不要說 | 改成 |
|--------|------|
| 「這個做不到」 | 「這個技術上可行，但成本會比較高，我幫您想有沒有替代方案」 |
| 「這很簡單」 | 「這個功能我們有做過類似的，執行上不會太複雜」 |
| 「你應該要⋯」 | 「以我的經驗，建議可以⋯」 |
| 「你不懂啦」 | 「這個部分比較技術面，簡單說就是⋯」 |
| 「你之前的工程師做得很爛」 | 「之前的架構有些可以優化的空間」 |

### 2.6 訪談節奏控制

一場 60 分鐘的訪談，建議這樣分配：

```
0-5 min   破冰寒暄 + 自我介紹 + 說明今天流程
5-15 min  了解客戶背景（公司/產業/現狀）
15-35 min 深挖需求（功能/設計/目標客群）
35-45 min 討論預算和時程（敏感但必要）
45-55 min 展示相關案例 + 回答客戶問題
55-60 min 總結下一步 + 約下次會議
```

**訪談開場白模板：**

```
「○○ 您好，我是 [名字]。
今天大概會花 45 分鐘到 1 小時，
主要想了解您的需求和想法。

我會先請您聊聊公司的狀況和目前遇到的問題，
然後我們再一起討論可能的解決方案。

過程中如果有什麼想法或問題，隨時都可以提出來。
那我們就開始吧？」
```

## 3. 需求訪談問題清單（30 題）

### A. 基本資訊（6 題）

| # | 問題 | 目的 |
|---|------|------|
| 1 | 可以簡單介紹一下您的公司/品牌嗎？主要做什麼的？ | 了解產業背景 |
| 2 | 您的主要客群是誰？年齡、職業、消費習慣？ | 定義目標受眾 |
| 3 | 目前有沒有現有的網站或線上平台？ | 了解現狀 |
| 4 | 如果有現有網站，目前最不滿意的地方是什麼？ | 找出痛點 |
| 5 | 做這個網站最主要想達成什麼目標？（品牌形象/接單/導流/資訊展示） | 釐清核心目標 |
| 6 | 有沒有主要的競爭對手？他們的網站做得怎麼樣？ | 了解競爭環境 |

### B. 功能需求（6 題）

| # | 問題 | 目的 |
|---|------|------|
| 7 | 您覺得網站最重要的前三個功能是什麼？ | 排優先級 |
| 8 | 需要會員系統嗎？（登入/註冊/個人資料/訂單查詢） | 評估系統複雜度 |
| 9 | 需要線上付款嗎？如果需要，希望支援哪些方式？（信用卡/LINE Pay/轉帳） | 金流評估 |
| 10 | 會需要後台管理嗎？誰會負責更新內容？更新頻率大概多高？ | CMS 需求評估 |
| 11 | 有沒有需要跟其他系統串接？（ERP/POS/LINE OA/Facebook/Google 表單） | 第三方整合 |
| 12 | 需要多語系嗎？主要是繁中 + 英文，還是有其他語言？ | i18n 評估 |

### C. 設計偏好（6 題）

| # | 問題 | 目的 |
|---|------|------|
| 13 | 有沒有喜歡的參考網站？可以分享 2-3 個嗎？（不一定同產業） | 風格方向 |
| 14 | 那些參考網站裡，您最喜歡的是什麼部分？（排版/配色/動畫/感覺） | 具體偏好 |
| 15 | 有沒有品牌識別系統？（Logo/色彩/字型/品牌手冊） | 設計規範 |
| 16 | 對風格有什麼傾向？（簡約/活潑/科技感/溫馨/文青/商務正式） | 風格定位 |
| 17 | 有沒有一定不要的元素或風格？（避免走錯方向的重要問題） | 排除地雷 |
| 18 | 素材方面，照片和文案會由您提供，還是需要我們協助？ | 素材來源 |

### D. 技術需求（6 題）

| # | 問題 | 目的 |
|---|------|------|
| 19 | 有沒有偏好的網站平台？（WordPress/自製/Shopify/Wix） | 技術選型 |
| 20 | 目前有網域和主機嗎？還是需要協助申請？ | 基礎設施 |
| 21 | 有沒有 SEO（搜尋引擎優化）的需求？目前有在做 Google 廣告嗎？ | SEO/SEM |
| 22 | 有沒有 GA（Google Analytics）或其他數據追蹤的需求？ | 數據分析 |
| 23 | 網站的流量預估大概多少？（剛開始不多 / 已經有穩定流量） | 效能規格 |
| 24 | 未來有沒有可能擴充功能？（預留擴充彈性的評估） | 架構規劃 |

### E. 預算相關（4 題）

| # | 問題 | 目的 |
|---|------|------|
| 25 | 這個專案有沒有大概的預算範圍？（可以給一個區間就好） | 預算了解 |
| 26 | 希望的付款方式？（一次付清 / 分期 / 訂金+尾款） | 金流安排 |
| 27 | 如果預算有限，您會優先取捨哪些功能？ | 核心 vs 非核心 |
| 28 | 上線後的年度維護預算大概抓多少？ | 長期合作評估 |

### F. 時程相關（2 題）

| # | 問題 | 目的 |
|---|------|------|
| 29 | 有沒有希望的上線日期？是硬性期限還是有彈性？ | 時程壓力 |
| 30 | 中間有沒有重要的里程碑？（配合展覽/新品發表/活動） | 關鍵節點 |

### 訪談後自問（接案者內部用）

```
□ 我清楚客戶的商業目標嗎？
□ 我知道誰是最終決策者嗎？（可能不是跟你開會的人）
□ 功能需求我能列出明確清單嗎？
□ 有沒有什麼客戶沒說但我覺得需要的？
□ 這個案子我有能力做嗎？還是需要外包部分功能？
□ 預算和需求是否匹配？如果不匹配要怎麼談？
□ 有沒有紅旗警訊？（不斷改需求/不願談預算/急件但要便宜）
```

## 4. 客戶類型分析與溝通策略

### 4.1 依技術理解程度分類

#### 類型 A：完全不懂技術

**特徵：**
- 「我不懂網路的東西」「你們專業的決定就好」
- 常用非技術語彙描述需求（「我要那個可以滑的東西」→ 輪播圖）
- 可能不知道 RWD、SEO、CMS 是什麼

**溝通策略：**

```
✅ 做的事：
- 用生活化比喻解釋技術概念
- 多用視覺化工具（截圖、畫面、影片）
- 主動提供建議和選項，不要問開放性技術問題
- 每次會議結束寄一份「白話版」會議紀錄

❌ 不做的事：
- 不丟技術名詞轟炸他
- 不要問「你要用 React 還是 WordPress？」
- 不要給太多選項（最多三個）

比喻範例：
「後台管理系統就像手機裡的 LINE，
  你可以自己發訊息（更新內容）、管理好友（會員），
  不用每次都找工程師幫忙。」

「RWD 響應式設計就是同一個網站，
  不管用手機、平板、電腦打開，都會自動調整成最適合的大小。
  就像水一樣，會自動適應容器的形狀。」

「SSL 憑證就是網址前面那個鎖頭 🔒，
  代表網站是安全的，客戶刷卡或填資料時資訊不會被偷看。」
```

#### 類型 B：略懂技術（最常見）

**特徵：**
- 知道一些名詞但不確定細節（「我聽過 SEO」「我有用過 WordPress」）
- 會自己 Google 研究，帶著網路上看到的資訊來跟你討論
- 可能已經被其他接案者報過價，在比較中

**溝通策略：**

```
✅ 做的事：
- 用專業但不艱深的語言
- 適度解釋他提到的名詞，確認彼此認知一致
- 肯定他做的功課，補充他可能忽略的面向
- 用「比較表」讓他做選擇

❌ 不做的事：
- 不要假設他全懂
- 不要挑戰他從網路上看到的資訊（即使是錯的）
- 不要讓他覺得你在考他

話術範例：
客戶：「我看網路上說用 WordPress 做最便宜？」
你：  「WordPress 確實是市佔率最高的 CMS，很多情況下是好選擇。
       不過要看你的需求——如果是形象官網，WordPress 很合適；
       如果需要比較複雜的會員系統或客製功能，
       可能自行開發會更好維護。
       我幫你整理一個比較表，你再決定？」
```

#### 類型 C：很懂技術

**特徵：**
- 自己就是工程師、設計師、或科技公司 PM
- 會指定技術架構（「我要用 Next.js + Supabase」）
- 溝通快速但要求精準

**溝通策略：**

```
✅ 做的事：
- 用對等的專業語言溝通
- 直接切入技術細節，不用繞
- 主動提出技術建議和可能的坑
- 提供具體的交付規格（API 文件、元件規格）

❌ 不做的事：
- 不要用幼稚園語言跟他解釋基本概念
- 不要隱瞞技術限制，他會看穿
- 不要不懂裝懂

話術範例：
客戶：「我想用 Next.js App Router + RSC，DB 用 Supabase。」
你：  「了解。那驗證你們打算用 Supabase Auth 還是自己做？
       如果用 Supabase Auth，RLS 的 policy 我們直接寫好。
       另外 RSC 跟 Supabase realtime 搭的話，
       要注意 Server Component 不能直接 subscribe，
       需要用 Client Component 包一層。
       這些我之前有遇過，可以分享 pattern 給你參考。」
```

### 4.2 依決策模式分類

| 類型 | 特徵 | 應對策略 |
|------|------|----------|
| **老闆獨裁型** | 一個人說了算，決策快 | 全力服務這個人，提案要打中他的點 |
| **委員會型** | 要開會討論，多人決策 | 準備好書面資料讓他帶回去報告，每次會議做紀錄 |
| **指定窗口型** | 窗口負責傳話，決策者不出面 | 把資料做得容易轉發，確認窗口有決策權限 |
| **多利益方型** | 行銷、業務、老闆各有意見 | 找出「最終拍板的人」，其他意見整理呈現 |

### 4.3 紅旗客戶辨識

以下情況要提高警覺，評估是否接案：

```
🚩 預算不符型：
  - 「我預算 2 萬，但我要購物車 + 會員 + LINE 串接 + APP」
  - 應對：誠實說明行情，提供降規方案或推薦更適合的工具（如 Shopify）

🚩 急件壓榨型：
  - 「後天要上線，你今晚能做完嗎？可以的話我就給你做」
  - 應對：說明合理時程，願意急件可加收 30-50% 急件費

🚩 無限修改型：
  - 「先做給我看看，不滿意再改」（沒有明確標準）
  - 應對：合約明確規定修改次數，超過次數按次計費

🚩 競價型：
  - 「另一家報 3 萬，你可以更便宜嗎？」
  - 應對：不要跟著降價競爭，解釋你的價值差異

🚩 隱形決策者型：
  - 每次開完會都要回去問老闆/老闆娘/合夥人
  - 應對：要求最終決策者出席至少一次會議
```

## 8. 報價溝通話術

> 詳細報價模板請見 [32-報價模板與合約範本](32-報價模板與合約範本.md)

### 8.1 何時談價格

```
❌ 錯誤時機：第一次通電話就報價（你還不了解需求）
❌ 錯誤時機：客戶第一次問「多少錢」就直接回答

✅ 正確時機：需求訪談結束後，整理完需求文件再報價
✅ 正確話術：「為了給您最準確的報價，
              我先整理今天討論的需求，
              預計兩個工作天內把報價單寄給您。」
```

### 8.2 客戶問「大概多少錢」的回應

第一次被問時，不要直接回數字，但也不能完全不說：

```
「這個要看具體的功能規模，
  但以你剛才描述的需求，行情大概落在 8 到 15 萬之間。

  確切的數字等我整理完需求，
  會給你三個方案讓你選——
  基本的、推薦的、和最完整的，
  看你的預算和優先度來決定。」
```

**為什麼要給區間？**
- 太低 → 客戶拿去當預期，後面報價超過他不開心
- 太高 → 嚇跑客戶
- 不說 → 客戶覺得你不專業或在隱瞞

### 8.3 報價展示話術

```
「我根據上次討論的需求，
  幫您規劃了三個方案：

  方案 A 是基本版，包含最核心的 5 頁官網，
  適合預算有限但想先上線的情況。

  方案 B 是我最推薦的，除了基本頁面，
  還包含產品後台管理和 SEO 優化，
  長期來看投資報酬率最高。

  方案 C 是最完整的，多了會員系統和 LINE 串接，
  適合想做完整數位布局的客戶。

  三個方案我都整理在報價單裡，
  您先看看，有問題我們隨時可以討論。」
```

### 8.4 客戶殺價的應對

**情境 1：直接要求打折**

```
客戶：「可以便宜一點嗎？打個 8 折？」

✅ 回應 A（降規格換降價）：
「價格的部分，因為每一項都是實際的工時成本，
  直接打折對我來說比較困難。
  但如果預算上有壓力，我們可以把 [某功能] 先拿掉，
  等未來預算到位再加，這樣可以省下大概 [金額]。」

✅ 回應 B（加值不降價）：
「折扣的部分我沒辦法讓太多，
```

### 8.5 拒絕不合理需求

```
場景：客戶想加功能但不加錢

客戶：「對了，順便幫我加一個部落格功能，
        應該不難吧？反正都在做了。」

  大概會增加 [金額] 和 [天數]。

重點：
1. 先肯定客戶的想法
2. 說明為什麼不是「順便做」

4. 讓客戶決定要不要加

```

## 12. 溝通工具選擇

### 12.1 各工具的適用場景

| 工具 | 適合用在 | 不適合用在 | 回覆時效 |
|------|----------|------------|----------|
| **LINE** | 快速確認、日常溝通、非正式討論 | 需要留存紀錄的重要決定 | 4hr 內 |
| **Email** | 正式報價、合約、會議紀錄、重要決定 | 需要即時回覆的急事 | 24hr 內 |
| **Google Meet / Zoom** | 需求訪談、設計審核、教學 | 簡單確認（不值得開會的事） | 預約制 |
| **Notion** | 專案管理、需求文件、共享知識庫 | 客戶不懂工具的情況 | 非即時 |
| **Slack** | 科技業客戶、長期合作、需要頻道分類 | 傳統產業客戶、一次性案件 | 數小時內 |
| **Trello** | 進度追蹤、任務管理、視覺化看板 | 複雜文件協作 | 非即時 |
| **電話** | 緊急事項、客戶情緒處理 | 需要書面紀錄的事（講完要補文字紀錄） | 即時 |

### 12.2 依客戶類型選擇工具

```
傳統產業老闆 → LINE + Email
  （他們最會用 LINE，重要事項 Email 備份）

科技公司 PM → Slack + Notion + Google Meet
  （他們習慣這套，也喜歡這套）

電商 / 行銷人 → LINE + Trello + Google Meet
  （視覺化看板讓進度一目了然）

大型企業 → Email + Google Meet + 正式文件
  （跨部門溝通需要正式紀錄）

個人品牌 / 部落客 → LINE + Google Meet
  （簡單直接就好，不要搞太多工具）
```

### 12.3 溝通工具使用原則

**1. 一個專案，一個主要管道**

```
❌ 錯誤：這邊 LINE 一些、那邊 Email 一些、口頭又講一些
       → 之後翻紀錄翻到死，而且各說各話

✅ 正確：「我們的主要溝通管道用 [LINE/Email]，
         所有的確認和決定都走這邊，
         這樣雙方都有紀錄。」
```

**2. 重要決定一定要文字留紀錄**

```
場景：電話中客戶說「好，首頁就這樣定了」

你在電話後立刻傳：
「○○ 您好，剛才電話確認的事項整理如下：
  1. 首頁設計稿 v3 定稿
  2. Banner 標語改為「○○○」
  3. 配色維持原方案
  以上請您確認，確認後我進入開發階段。」

→ 這樣日後有爭議時有文字紀錄。
```

**3. 回覆時效管理**

```
LINE     → 上班時間 4 小時內回覆，非上班時間隔天早上
Email    → 24 小時內回覆（即使只是「收到，處理中」）
電話未接  → 30 分鐘內回電或傳 LINE 問事由

重要：不要秒回 → 客戶會認為你隨時都在，形成不合理期待
      也不要都不回 → 客戶會焦慮，覺得你不重視

最佳平衡：有規律的回覆節奏，讓客戶知道你是專業的，
          不是他的 24hr 客服。
```

**4. LINE 群組管理**

```
建群組時設定好：
- 群組名稱：「○○官網專案 — [客戶+你]」
- 群組相簿：放設計稿截圖方便比對
- 記事本：放重要連結（Figma / 測試站 / 報價單）

群組規則（第一天講清楚）：
「○○ 您好，這個群組我們用來討論網站專案，
  如果有設計或功能的修改需求，
  盡量打文字在這裡，方便我追蹤紀錄。
  語音訊息我可能會漏掉細節，
  比較重要的事請用打字的，謝謝！」
```

## 附錄：常用句型速查

### A. 開場

```
- 「○○ 您好，感謝您的詢問。」
- 「謝謝您撥空跟我開會。」
- 「我先簡單說明一下今天的流程⋯」
```

### B. 傾聽與確認

```
- 「了解，所以您的意思是⋯」
- 「我整理一下您剛才提到的幾個重點⋯」
- 「確認一下，您說的 [A] 是指 [B] 嗎？」
```

### C. 提出建議

```
- 「以我的經驗，建議可以⋯」
- 「通常這種情況，我們會⋯」
- 「有兩個方向可以考慮⋯」
```

### D. 拒絕但不傷感情

```
- 「這個想法很好，不過考量到 [原因]，我建議⋯」
- 「技術上可以做，但成本會比較高，有一個替代方案⋯」
- 「這個部分不在原始範圍裡，需要追加評估⋯」
```

### E. 結尾

```
- 「那今天先到這邊，我整理完會議紀錄寄給您。」
- 「有任何問題隨時 LINE 我。」
- 「下次會議 [日期時間]，到時候見！」
```

### F. 表達時程

```
- 「我預計在 [日期] 前完成，届時再跟您回報。」
- 「這個大概需要 [N] 個工作天。」
- 「如果素材能在 [日期] 前到位，我們就能如期上線。」
```

---

> 最後提醒：溝通的核心不是「說得多好聽」，
> 而是「讓對方清楚知道接下來會發生什麼」。
> 客戶最怕的不是問題，而是不知道發生了什麼事。
> 持續、穩定、有規律的溝通，比任何話術都有效。
