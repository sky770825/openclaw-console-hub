## 1. 報價原則

### 1.1 三種計費模式

| 計費方式 | 說明 | 適合場景 | 優點 | 缺點 |
|----------|------|----------|------|------|
| **時間計費** | 按工時收費（時薪或日薪） | 需求不明確、長期合作、顧問型 | 不怕需求變更、彈性高 | 客戶怕開口支票、要記工時 |
| **專案計費** | 固定總價，做完為止 | 範圍明確的專案、一次性案件 | 客戶安心、容易比價 | 範圍蔓延風險、加班自己吞 |
| **價值計費** | 按產出的商業價值收費 | 高價值客戶、能量化效益的案件 | 利潤最高、脫離工時陷阱 | 需要懂商業、客戶不一定買單 |

**實務建議：**

- 新手先用「專案計費」，累積案例後逐步轉向「價值計費」
- 長期維護用「時間計費」或「月費制」
- 混合模式最常見：專案固定價 + 額外需求按工時計

**時薪參考（2025-2026 台灣）：**

| 經驗等級 | 時薪範圍 | 日薪（8hr） | 說明 |
|----------|----------|-------------|------|
| 新手（< 1 年） | NT$ 500-800 | NT$ 4,000-6,400 | 有基本作品集 |
| 中階（1-3 年） | NT$ 800-1,500 | NT$ 6,400-12,000 | 能獨立完成專案 |
| 資深（3-5 年） | NT$ 1,500-2,500 | NT$ 12,000-20,000 | 有技術深度 + 溝通能力 |
| 專家（5 年以上） | NT$ 2,500-5,000 | NT$ 20,000-40,000 | 業界知名度、特殊技術 |

### 1.2 報價心理學

#### 錨定效應（Anchoring）

先報一個高的數字，後面的數字看起來就便宜了。

```
❌ 錯誤：「這個案子大概 5 萬」

✅ 正確：「依照需求規模，市場行情在 12-18 萬之間。
        考量到我們長期合作的可能，我為您規劃了三個方案⋯」
```

**關鍵**：先說市場行情（高錨點），再給出你的報價，客戶會覺得划算。

#### 三方案選擇法（Good / Better / Best）

永遠提供三個方案，大多數客戶會選中間那個。

```
方案 A（基本型）：NT$ 35,000  ← 太陽春，多數人不選
方案 B（推薦型）：NT$ 65,000  ← 你真正想賣的 ★
方案 C（旗艦型）：NT$ 120,000 ← 讓方案 B 看起來划算
```

**比例建議**：A 約為 B 的 50-60%，C 約為 B 的 170-200%

#### 報價數字技巧

```
❌ NT$ 50,000（太整，像亂喊的）
✅ NT$ 48,500（有零頭，像真的算過）

❌ NT$ 100,000（心理門檻高）
✅ NT$ 98,000（破十萬的感覺完全不同）
```

### 1.3 什麼時候報高、什麼時候報低

#### 報高的情況

| 信號 | 原因 |
|------|------|
| 客戶是中大型企業 / 有品牌 | 預算充足，太低反而被看不起 |
| 客戶強調品質、不在乎價格 | 直接給最好的方案 |
| 趕時間（「下週要上線」） | 急件加價 30-50% |
| 客戶被前一個廠商搞砸 | 他正在付「痛苦代價」，願意付更多 |
| 技術難度高 / 你是少數能做的 | 稀缺性 = 溢價 |
| 客戶找了很多間、回來找你 | 你是他最後的選擇，不必打折 |
| 這個案子你不是很想接 | 報高一點，接到就賺、沒接到也不虧 |

#### 報低的情況

| 信號 | 原因 |
|------|------|
| 對方是值得長期合作的好客戶 | 用首案優惠換長期關係 |
| 你需要這個作品集 | 品牌價值比金錢重要 |
| 案子簡單、可以模板化 | 邊際成本低，薄利多銷 |
| 非營利組織 / 你認同的事業 | 社會價值考量 |
| 你剛入行，需要累積口碑 | 但也不要太低，低價客戶往往最難搞 |

### 1.4 殺價怎麼應對

#### 原則：永遠不要「直接降價」，要「減規格」

```
❌ 「好啦，算你便宜一點，5 萬變 4 萬。」
   → 客戶學到：你的價格有水分，下次繼續殺

✅ 「我理解您的預算考量。如果我們把首頁動畫簡化、
    內頁從 8 頁調整為 5 頁，可以控制在 4 萬以內。」
   → 客戶學到：便宜 = 少東西，你的報價合理
```

#### 常見殺價話術 vs 應對

| 客戶說 | 你回 |
|--------|------|
| 「別人報 3 萬就可以」 | 「3 萬的方案通常是套模板，如果您的需求適合模板，我也可以用模板方案報 3 萬給您。不過您剛才提到的 X、Y、Z 功能需要客製，這部分是成本差異所在。」 |
| 「你算便宜一點啦」 | 「這樣好了，我幫您看看哪些功能可以第二期再做。第一期先上核心功能，預算可以先控制在 N 萬。」 |
| 「我們預算只有 X 萬」 | 「了解。我來調整方案，在 X 萬的預算內把最重要的功能先做好。（附上精簡版方案）」 |
| 「做完這個，後面還有很多案子」 | 「太好了。這樣我們可以簽半年合作約，享有 85 折優惠。（白紙黑字才算）」 |
| 「我朋友做這個只要 1 萬」 | 「1 萬的品質我也能做，但不包含 RWD、SEO、SSL、後續修改。如果您需要完整專業服務，我的報價已含這些。」 |

#### 可以讓步的地方（不砍價格）

- 多送一個月維護（你的邊際成本低）
- 免費做簡易 SEO 設定（反正你本來就會做）
- 送 Google Analytics 安裝（5 分鐘的事）
- 分期付款（不降總價，拉長收款）
- 提供簡易操作教學（半小時線上教）

### 5.1 標準付款流程

```
簽約 → 訂金（30%）→ 設計確認 → 中期款（40%）→ 開發完成 → 驗收 → 尾款（30%）
  │                      │                              │            │
  └──── 3 工作天內 ──────┘──── 設計確認後 5 天 ──────────┘── 7 天內 ──┘
```

#### 三種常見付款方式

| 方式 | 訂金 | 中期款 | 尾款 | 適用場景 |
|------|------|--------|------|----------|
| 30/40/30 | 30% | 40% | 30% | **最推薦**，平衡雙方風險 |
| 50/50 | 50% | — | 50% | 小案子、信任度高 |
| 40/30/30 | 40% | 30% | 30% | 前期投入大（設計為主） |
| 30/30/20/20 | 30% | 30% | 20% + 20% | 大案子，多個里程碑 |

#### 每期觸發條件

| 期數 | 觸發條件 | 甲方需確認 | 付款期限 |
|------|----------|-----------|----------|
| 訂金 | 合約簽署完成 | 合約條款 | 簽約後 3 個工作天 |
| 中期款 | 設計稿或前端 Demo 確認 | 設計稿 / 頁面截圖 | 確認後 5 個工作天 |
| 尾款 | 驗收通過、正式上線 | 驗收清單勾選完畢 | 驗收後 7 個工作天 |

## 7. 變更管理

### 7.1 為什麼需要變更管理

```
沒有變更管理的結果：

客戶：「對了，可以加一個 XX 功能嗎？」
你  ：「好啊，小事。」
客戶：「那 YY 也順便改一下。」
你  ：「嗯⋯好吧。」
客戶：「ZZ 我覺得要整個重做。」
你  ：「⋯⋯」

→ 做了 3 倍的工作量，收一樣的錢
→ 時程延後 2 個月
→ 你怨、客戶也不滿意
→ 以後再也不想接案了
```

**變更管理 = 保護你也保護客戶。**

### 7.3 追加費用計算方式

#### 計算公式

```
追加費用 = 預估工時 × 時薪 × 複雜度係數
```

| 變更類型 | 複雜度係數 | 說明 |
|----------|-----------|------|
| 文字/圖片更換 | 0（免費） | 不影響架構的小改 |
| 版面微調 | 1.0 | 位置、大小、顏色 |
| 新增頁面 | 1.0 | 按原報價的頁面單價 |
| 新增功能 | 1.2 | 需設計 + 開發 + 測試 |
| 修改已確認的設計 | 1.3 | 超出免費修改次數 |
| 修改已完成的功能 | 1.5 | 要拆掉重做，浪費前期工作 |
| 整體重新設計 | 2.0 | 等於重來，高係數 |
| 急件變更（3 天內要） | +50% | 急件加成 |

#### 範例

```
情境：客戶在前端完成後要求新增一個「線上預約」功能

預估工時：16 小時
時薪：NT$ 1,200
複雜度係數：1.2（新增功能）
時程影響：延後 5 個工作天

追加費用 = 16 × 1,200 × 1.2 = NT$ 23,040
→ 報價 NT$ 23,000（取整數）
```

### 7.4 避免「做到死」的七個方法

#### 1. 需求規格書要明確

```
❌ 模糊的需求：「做一個好看的網站」
✅ 明確的需求：「首頁含 Hero Banner（1920×800，輪播 3 張）、
    服務特色區塊（3 欄 icon + 文字）、最新消息區（取 3 筆）、
    CTA 按鈕連至聯絡頁」
```

每一個功能、每一個頁面、每一個區塊都要寫清楚。

#### 2. 合約寫清楚修改次數

- 設計稿：2 次免費修改
- 前端 / 功能：1 次免費修改
- 超出的每次收費
- 「一次修改」= 彙整後一次提出，不是零星丟

#### 3. 每個階段要簽確認

```
設計稿確認 → 客戶簽名 → 進入開發
↓
開發中客戶說「設計要改」→ 拿出確認書 → 這是 Change Request → 另外收費
```

沒有簽確認就進下一階段 = 自找麻煩。

#### 4. 區分「Bug」和「新需求」

| 類型 | 定義 | 費用 |
|------|------|------|
| Bug | 原需求規格書上有寫，但做出來不對 | 免費修 |
| 新需求 | 原需求規格書上沒寫 | Change Request |
| 改主意 | 原本確認的設計，現在想改 | Change Request |

**關鍵**：需求規格書就是判斷標準。

#### 5. 學會說「好的，這是一個新需求」

```
客戶：「可以加一個購物車功能嗎？」

❌ 「好啊，我做。」
✅ 「好的，購物車是一個新功能，不在原始報價範圍內。
    我評估一下工時和費用，明天給您一份變更報價。」
```

不要害怕說「這要加錢」，專業的客戶會理解。不專業的客戶，你更需要保護自己。

#### 6. 設定「凍結日」

在合約中加入這一條：

```
設計稿經甲方確認後即進入「設計凍結」狀態。
凍結後如需修改視覺設計，依變更管理流程辦理。

功能開發完成經甲方確認後即進入「功能凍結」狀態。
凍結後如需修改功能，依變更管理流程辦理。
```

#### 7. 保留緩衝時間

```
內部評估：20 個工作天
對外報價：30 個工作天（多 50% 緩衝）

→ 提早完成 = 客戶覺得你效率好
→ 剛好完成 = 正常
→ 被變更影響 = 還有緩衝可用
```
