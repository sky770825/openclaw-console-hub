## 1. 資料庫選型決策

### 七種方案比較

| 資料庫 | 類型 | 免費額度 | 適合場景 | 學習成本 | 台灣延遲 | 推薦度 |
|--------|------|---------|---------|---------|---------|--------|
| **PostgreSQL + Supabase** | 關聯式 (託管) | 500MB / 2 專案 | 中小型專案全能 | 中 | ~50ms (新加坡) | ★★★★★ |
| **PostgreSQL (自架)** | 關聯式 | 完全免費 | 需要完整控制 | 高 | 看主機 | ★★★★ |
| **MySQL** | 關聯式 | 完全免費 | WordPress / 傳統 PHP | 中 | 看主機 | ★★★ |
| **MongoDB** | 文件式 (NoSQL) | 512MB (Atlas) | 彈性 Schema / IoT | 中 | ~80ms (Atlas) | ★★★ |
| **SQLite** | 嵌入式 | 完全免費 | 小工具 / 本地應用 | 低 | 本地 0ms | ★★★ |
| **PlanetScale** | MySQL 託管 | 已停免費方案 | 需水平擴展 MySQL | 中 | ~120ms | ★★ |
| **Firebase Firestore** | 文件式 (NoSQL) | 1GB / 50K 讀/天 | 純前端 / 即時同步 | 低 | ~60ms | ★★★ |

### 選擇決策樹

```
你的專案需要什麼？
  │
  ├─ 快速上線 + 不想管 infra？
  │    ├─ 需要 SQL + Auth + Storage？ → Supabase ★
  │    ├─ 純前端 + 即時同步？ → Firebase
  │    └─ 只需要簡單 key-value？ → Firebase 或 Upstash Redis
  │
  ├─ 傳統 PHP / WordPress？ → MySQL
  │
  ├─ Schema 經常變動 / 巢狀文件多？ → MongoDB
  │
  ├─ 嵌入式 / CLI 工具 / Electron？ → SQLite
  │
  ├─ 需要完整控制 + 進階功能？ → 自架 PostgreSQL
  │
  └─ 接案客戶大部分情境？ → Supabase（一站搞定）★
```

### 各資料庫詳細比較

#### PostgreSQL vs MySQL

| 面向 | PostgreSQL | MySQL |
|------|-----------|-------|
| JSON 支援 | 原生 `jsonb`，可索引查詢 | JSON 型別，功能較弱 |
| 全文搜尋 | 內建 `tsvector` + 中文需外掛 | 內建 FULLTEXT |
| 擴展性 | 外掛生態豐富 (PostGIS, pgvector) | 有限 |
| 交易隔離 | MVCC，預設 Read Committed | 依引擎，InnoDB 支援 |
| 授權 | PostgreSQL License (超寬鬆) | GPL (注意商用) |
| 效能 | 複雜查詢強 | 簡單讀取快 |
| 託管服務 | Supabase / Neon / Railway | PlanetScale / Railway |

**結論**：新專案用 PostgreSQL，除非客戶指定 MySQL 或已有 MySQL 環境。

#### MongoDB 何時用

適合：
- Schema 經常變動的敏捷開發初期
- 深度巢狀的文件結構（商品規格、表單欄位動態生成）
- IoT 大量寫入的時序資料
- 內容管理系統的彈性欄位

不適合：
- 需要複雜 JOIN 的業務邏輯（訂單 + 商品 + 用戶）
- 需要嚴格交易一致性（金流）
- 團隊熟悉 SQL

#### SQLite 何時用

適合：
- Electron 桌面應用的本地資料
- CLI 工具的設定/快取
- 開發/測試環境的快速原型
- 邊緣運算 / Cloudflare D1

不適合：
- 多用戶同時寫入的 Web 應用
- 需要網路存取的雲端服務

## 2. PostgreSQL + Supabase（推薦方案）

### 為什麼推薦 Supabase

1. **免費額度夠用**：500MB 資料庫 + 1GB 檔案 + 50K Auth MAU
2. **一站式**：DB + Auth + Storage + Realtime + Edge Functions
3. **開源**：不怕被綁定，可自架
4. **PostgreSQL 完整功能**：RLS、Functions、Triggers 都有
5. **Dashboard 好用**：Table Editor 可以直接操作資料

### 建立專案

1. 到 [supabase.com](https://supabase.com) 註冊
2. New Project → 選 Region: **Singapore (Southeast Asia)**（台灣最近）
3. 設定資料庫密碼（記好，後面連線要用）
4. 等 2-3 分鐘初始化完成

```typescript
// 查詢（SELECT）
const { data, error } = await supabase
  .from('products')
  .select('*')
  .eq('category', '3C')
  .order('created_at', { ascending: false })

  .from('orders')
  .select(`

    total,
    created_at,

    items:order_items(
      quantity,
      product:products(name, price)

// 新增（INSERT）
const { data, error } = await supabase
  .from('products')
  .insert({
    name: 'MacBook Pro 14"',

    category: '3C',
    stock: 50
  })
  .select()       // 回傳新增的資料
  .single();      // 只取一筆

// 批次新增
const { data, error } = await supabase
  .from('products')
  .insert([
    { name: '商品A', price: 100 },
    { name: '商品B', price: 200 },
    { name: '商品C', price: 300 },
  ])
  .select();

// 更新（UPDATE）
const { data, error } = await supabase
  .from('products')
  .update({ price: 54900, stock: 45 })
  .eq('id', productId)
  .select()
  .single();

// 刪除（DELETE）
const { error } = await supabase
  .from('products')
  .delete()
  .eq('id', productId);

// ===== 進階查詢 =====

// 模糊搜尋
const { data } = await supabase
  .from('products')
  .select('*')
  .ilike('name', '%macbook%');

// 範圍查詢
const { data } = await supabase
  .from('products')
  .select('*')
  .gte('price', 1000)
  .lte('price', 5000);

// 多條件 OR
const { data } = await supabase
  .from('products')
  .select('*')
  .or('category.eq.3C,category.eq.家電');

// 計數
const { count } = await supabase
  .from('products')
  .select('*', { count: 'exact', head: true })

// ===== Realtime 即時監聽 =====

  .channel('orders-changes')

// 取消監聽

// ===== Storage（檔案上傳）=====
```

## 3. 關聯式資料庫設計原則

### 正規化（Normalization）

#### 第一正規化（1NF）— 欄位原子化

每個欄位只放一個值，不要放逗號分隔的清單。

```
❌ 錯誤：
| id | name | tags              |
|----|------|-------------------|
| 1  | 商品A | 3C,熱賣,新品       |

✅ 正確（拆成關聯表）：
products:
| id | name |
|----|------|
| 1  | 商品A |

product_tags:
| product_id | tag  |
|------------|------|
| 1          | 3C   |
| 1          | 熱賣 |
| 1          | 新品 |
```

**例外**：PostgreSQL 的 `text[]` 陣列或 `jsonb` 可以存多值，適合不需要 JOIN 的簡單標籤。接案時間緊迫可以用：

```sql
-- 用 PostgreSQL 陣列（簡單標籤場景可接受）
CREATE TABLE products (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  tags text[] DEFAULT '{}'
);

-- 查詢含特定標籤的商品
SELECT * FROM products WHERE '熱賣' = ANY(tags);
```

#### 第二正規化（2NF）— 非鍵欄位完全依賴主鍵

適用於複合主鍵的表。所有非鍵欄位必須依賴「完整主鍵」，不能只依賴部分。

```
❌ 錯誤（order_items 裡放了 product_name）：
order_items:
| order_id | product_id | quantity | product_name |
                                     ↑ 只依賴 product_id，不依賴 order_id

✅ 正確：product_name 放在 products 表，需要時 JOIN
order_items:
| order_id | product_id | quantity | unit_price |
                                     ↑ 下單當時的價格，依賴整筆訂單項目
```

**注意**：`unit_price`（下單當時的單價）應該存在 order_items 裡，因為商品價格會變動。這不算違反 2NF。

#### 第三正規化（3NF）— 非鍵欄位不互相依賴

```
❌ 錯誤：
users:
| id | city    | city_zip_code |
                 ↑ 依賴 city，不是依賴 id

✅ 正確：
users:
| id | city_id |

cities:
| id | name   | zip_code |
```

**實務建議**：接案專案通常做到 3NF 就夠了。過度正規化會增加 JOIN 複雜度，反而影響開發效率和查詢效能。

### ER Diagram 畫法

推薦工具：
- **dbdiagram.io** — 線上免費，用 DSL 語法畫，可匯出 SQL
- **DrawSQL** — 視覺化拖拉，免費版 15 張表
- **pgAdmin 4** — 直接從現有資料庫逆向產生 ERD

dbdiagram.io 語法範例：

```
// dbdiagram.io 語法
Table users {
  id uuid [pk, default: `gen_random_uuid()`]
  email varchar(255) [unique, not null]
  name varchar(100) [not null]
  role varchar(20) [default: 'user']
  created_at timestamptz [default: `now()`]
}

Table products {
  id uuid [pk, default: `gen_random_uuid()`]
  name varchar(200) [not null]
  price integer [not null]
  category_id uuid [ref: > categories.id]
  created_at timestamptz [default: `now()`]
}

Table categories {
  id uuid [pk]
  name varchar(100) [not null]
  parent_id uuid [ref: > categories.id]
}

Table orders {
  id uuid [pk]
  user_id uuid [ref: > users.id]
  status varchar(20) [not null]
  total integer [not null]
  created_at timestamptz [default: `now()`]
}

Table order_items {
  id uuid [pk]
  order_id uuid [ref: > orders.id]
  product_id uuid [ref: > products.id]
  quantity integer [not null]
  unit_price integer [not null]
}

// 關係圖會自動畫出線條
```

### 命名規範

| 項目 | 規範 | 範例 |
|------|------|------|
| 表名 | 複數、snake_case | `users`, `order_items` |
| 欄位名 | 單數、snake_case | `user_id`, `created_at` |
| 主鍵 | `id`（UUID 優先） | `id uuid PRIMARY KEY` |
| 外鍵 | `{單數表名}_id` | `user_id`, `product_id` |
| 時間戳 | `_at` 結尾 + `timestamptz` | `created_at`, `updated_at` |
| 布林值 | `is_` 或 `has_` 開頭 | `is_active`, `has_paid` |
| 狀態欄位 | 用 `varchar` 不要用 `enum` | `status varchar(20)` |
| 金額 | 用 `integer` 存「分」或「元」 | `price integer` (單位：元) |
| 關聯表 | `{表A}_{表B}` 按字母序 | `product_tags`, `user_roles` |

**為什麼金額用 integer 不用 decimal？**
- 台幣大部分情境沒有小數點（元為單位）
- 整數運算比浮點數精確、效能好
- 如需處理外幣小數，用 `integer` 存「分」（1 美元 = 100）

**為什麼 status 用 varchar 不用 enum？**
- PostgreSQL 的 `enum` 改值很麻煩（要 ALTER TYPE）
- `varchar` + CHECK 約束更靈活
- 或者乾脆在應用層驗證

```sql
-- 用 CHECK 約束取代 enum
CREATE TABLE orders (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  status varchar(20) NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'paid', 'shipped', 'completed', 'cancelled')),
  -- ...
);
```

### UUID vs 自增 ID

| 面向 | UUID (`gen_random_uuid()`) | 自增 (`SERIAL` / `BIGSERIAL`) |
|------|---------------------------|------------------------------|
| 全域唯一 | 是 | 否（單表唯一） |
| 可猜測性 | 不可猜 | 可猜（/api/users/1, 2, 3...） |
| 分散式 | 天生支援 | 需要額外處理 |
| 儲存大小 | 16 bytes | 4/8 bytes |
| 索引效能 | 稍慢（隨機寫入） | 快（順序寫入） |
| URL 友好 | 不友好（太長） | 友好 |

**建議**：
- API 暴露給外部的表 → UUID（安全）
- 內部表 / 高寫入量 → 自增 ID（效能）
- 接案預設用 UUID，除非有效能考量

## 4. 常見 Schema 範例

### 4.1 用戶系統（User & Auth）

```sql
-- 用戶表（配合 Supabase Auth 使用）
CREATE TABLE public.profiles (
  id uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email text UNIQUE NOT NULL,
  display_name text,
  avatar_url text,
  phone text,
  role text NOT NULL DEFAULT 'user'
    CHECK (role IN ('user', 'admin', 'editor')),
  is_active boolean DEFAULT true,
  metadata jsonb DEFAULT '{}',
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);

-- 自動建立 profile（Supabase Auth Trigger）
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public.profiles (id, email, display_name, avatar_url)
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NEW.raw_user_meta_data->>'full_name', split_part(NEW.email, '@', 1)),
    NEW.raw_user_meta_data->>'avatar_url'
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

-- 自動更新 updated_at
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS trigger AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER profiles_updated_at
  BEFORE UPDATE ON profiles
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();

-- RLS（Row Level Security）
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

-- 用戶只能看自己的資料
CREATE POLICY "Users can view own profile"
  ON profiles FOR SELECT
  USING (auth.uid() = id);

-- 用戶只能改自己的資料
CREATE POLICY "Users can update own profile"
  ON profiles FOR UPDATE
  USING (auth.uid() = id)
  WITH CHECK (auth.uid() = id);

-- Admin 可以看所有人
CREATE POLICY "Admins can view all profiles"
  ON profiles FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin'
    )
  );
```

### 4.2 商品目錄（Product Catalog）

```sql
-- 商品分類（支援多層巢狀）
CREATE TABLE categories (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  slug text UNIQUE NOT NULL,
  parent_id uuid REFERENCES categories(id) ON DELETE SET NULL,
  sort_order integer DEFAULT 0,
  is_active boolean DEFAULT true,
  created_at timestamptz DEFAULT now()
);

-- 商品
CREATE TABLE products (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  slug text UNIQUE NOT NULL,
  description text,
  price integer NOT NULL CHECK (price >= 0),           -- 原價（元）
  sale_price integer CHECK (sale_price >= 0),           -- 特價
  cost integer CHECK (cost >= 0),                       -- 成本
  sku text UNIQUE,                                      -- 庫存編號
  category_id uuid REFERENCES categories(id) ON DELETE SET NULL,
  stock integer NOT NULL DEFAULT 0 CHECK (stock >= 0),
  is_active boolean DEFAULT true,
  is_featured boolean DEFAULT false,
  images jsonb DEFAULT '[]',  -- ["url1", "url2", ...]
  specs jsonb DEFAULT '{}',   -- {"顏色": "黑", "尺寸": "M", ...}
  seo_title text,
  seo_description text,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);

-- 商品變體（如不同顏色/尺寸有不同價格庫存）
CREATE TABLE product_variants (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id uuid NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  name text NOT NULL,              -- "黑色 / L"
  sku text UNIQUE,
  price integer NOT NULL,
  stock integer NOT NULL DEFAULT 0,
  attributes jsonb DEFAULT '{}',   -- {"color": "黑", "size": "L"}
  is_active boolean DEFAULT true,
  created_at timestamptz DEFAULT now()
);

-- 索引
CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_active ON products(is_active) WHERE is_active = true;
CREATE INDEX idx_products_slug ON products(slug);
CREATE INDEX idx_products_price ON products(price);
CREATE INDEX idx_product_variants_product ON product_variants(product_id);

-- 全文搜尋索引（商品名稱 + 描述）
ALTER TABLE products ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    setweight(to_tsvector('simple', coalesce(name, '')), 'A') ||
    setweight(to_tsvector('simple', coalesce(description, '')), 'B')
  ) STORED;

CREATE INDEX idx_products_search ON products USING gin(search_vector);

-- 全文搜尋查詢
SELECT id, name, price,
  ts_rank(search_vector, to_tsquery('simple', 'macbook & pro')) AS rank
FROM products
WHERE search_vector @@ to_tsquery('simple', 'macbook & pro')
ORDER BY rank DESC;
```

```sql

CREATE TABLE schedules (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  provider_id uuid NOT NULL REFERENCES providers(id) ON DELETE CASCADE,
  day_of_week integer NOT NULL CHECK (day_of_week BETWEEN 0 AND 6),
    -- 0=Sunday, 1=Monday, ...
  start_time time NOT NULL,
  end_time time NOT NULL,
  CHECK (start_time < end_time)

-- 特殊休假 / 加班
CREATE TABLE schedule_overrides (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  provider_id uuid NOT NULL REFERENCES providers(id) ON DELETE CASCADE,
  date date NOT NULL,
  is_available boolean NOT NULL DEFAULT false,  -- false=休假, true=加班
  start_time time,
  end_time time,
  reason text
);

-- 預約
CREATE TABLE bookings (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  booking_number text UNIQUE NOT NULL,
  user_id uuid REFERENCES profiles(id),
  provider_id uuid NOT NULL REFERENCES providers(id),
  service_id uuid NOT NULL REFERENCES services(id),
  status text NOT NULL DEFAULT 'pending'
    CHECK (status IN (
      'pending',     -- 待確認
      'confirmed',   -- 已確認
      'in_progress', -- 進行中
      'completed',   -- 完成
      'cancelled',   -- 取消
      'no_show'      -- 未到
    )),
  start_time timestamptz NOT NULL,
  end_time timestamptz NOT NULL,
  -- 客戶資訊（未登入用戶也可預約）
  customer_name text NOT NULL,
  customer_phone text NOT NULL,
  customer_email text,
  note text,
  total_price integer NOT NULL,
  cancelled_at timestamptz,
  cancel_reason text,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now(),
  -- 防止同一 provider 同時段重複預約
  EXCLUDE USING gist (
    provider_id WITH =,
    tstzrange(start_time, end_time) WITH &&
  ) WHERE (status NOT IN ('cancelled', 'no_show'))
);

-- 安裝 btree_gist 擴展（Supabase 已內建）
CREATE EXTENSION IF NOT EXISTS btree_gist;

-- 索引
CREATE INDEX idx_bookings_provider ON bookings(provider_id);
CREATE INDEX idx_bookings_user ON bookings(user_id);
CREATE INDEX idx_bookings_time ON bookings(start_time);
CREATE INDEX idx_bookings_status ON bookings(status);
CREATE INDEX idx_schedules_provider ON schedules(provider_id, day_of_week);

-- 查詢某 provider 某天的可用時段
-- （實務上通常在後端程式碼處理，但可以用 SQL 做）
SELECT
  s.start_time,
  s.end_time,
  COALESCE(
    array_agg(
      jsonb_build_object(
        'start', b.start_time,
        'end', b.end_time,
        'status', b.status
      )
    ) FILTER (WHERE b.id IS NOT NULL), '{}'
  ) AS existing_bookings
FROM schedules s
LEFT JOIN bookings b ON b.provider_id = s.provider_id
  AND b.start_time::date = '2026-03-15'
  AND b.status NOT IN ('cancelled', 'no_show')
WHERE s.provider_id = '指定provider的uuid'
  AND s.day_of_week = EXTRACT(DOW FROM DATE '2026-03-15')
GROUP BY s.start_time, s.end_time;
```

### 子查詢 & CTE

```sql
-- 子查詢（WHERE 裡）
SELECT * FROM products
WHERE category_id IN (
  SELECT id FROM categories WHERE parent_id IS NULL
);

-- 子查詢（FROM 裡）
SELECT category_name, product_count
FROM (
  SELECT
    c.name AS category_name,
    COUNT(p.id) AS product_count
  FROM categories c
  LEFT JOIN products p ON c.id = p.category_id
  GROUP BY c.name
) sub
WHERE product_count > 5;

-- EXISTS（有下過單的用戶）
SELECT * FROM profiles p
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.user_id = p.id
);

-- CTE（Common Table Expression）— 更易讀的子查詢
WITH monthly_revenue AS (
  SELECT
    date_trunc('month', created_at) AS month,
    SUM(total) AS revenue
  FROM orders
  WHERE status IN ('paid', 'completed')
  GROUP BY date_trunc('month', created_at)
),
monthly_growth AS (
  SELECT
    month,
    revenue,
    LAG(revenue) OVER (ORDER BY month) AS prev_revenue,
    ROUND(
      (revenue - LAG(revenue) OVER (ORDER BY month))::numeric /
      NULLIF(LAG(revenue) OVER (ORDER BY month), 0) * 100,
      1
    ) AS growth_pct
  FROM monthly_revenue
)
SELECT * FROM monthly_growth ORDER BY month DESC;

-- 遞迴 CTE（分類樹展開）
WITH RECURSIVE category_tree AS (
  -- 根節點
  SELECT id, name, parent_id, 0 AS depth, name::text AS path
  FROM categories
  WHERE parent_id IS NULL

  UNION ALL

  -- 子節點
  SELECT c.id, c.name, c.parent_id, ct.depth + 1,
    (ct.path || ' > ' || c.name)::text
  FROM categories c
  JOIN category_tree ct ON c.parent_id = ct.id
)
SELECT * FROM category_tree ORDER BY path;
```

## 6. 索引設計

### 什麼時候建索引

**建索引的時機**：
1. WHERE 常用的篩選欄位
2. JOIN 的關聯欄位（外鍵）
3. ORDER BY 的排序欄位
4. 常被搜尋的欄位（用戶名、email）
5. UNIQUE 約束的欄位（自動建索引）

**不要建索引**：
1. 很少查詢的欄位
2. 值的基數很低（如 boolean：只有 true/false，但 partial index 除外）
3. 經常大量寫入的表（索引拖慢寫入）
4. 小表（< 1000 筆，全表掃描更快）

### PostgreSQL 索引類型

| 類型 | 適用場景 | 範例 |
|------|---------|------|
| **B-tree**（預設） | 等值、範圍、排序查詢 | `WHERE price > 1000` |
| **Hash** | 純等值查詢 | `WHERE status = 'active'` |
| **GIN** | 陣列、jsonb、全文搜尋 | `WHERE tags @> '{"3C"}'` |
| **GiST** | 幾何、範圍、全文搜尋 | `WHERE location <-> point` |
| **BRIN** | 自然排序的大表（時間戳） | 按時間查詢的 log 表 |

### 索引建立範例

```sql
-- 基本索引（B-tree，最常用）
CREATE INDEX idx_products_category ON products(category_id);

-- 複合索引（多欄位，順序很重要）
-- 原則：篩選條件在前，排序在後
CREATE INDEX idx_orders_user_status ON orders(user_id, status);
-- 這個索引同時優化：
--   WHERE user_id = 'x'
--   WHERE user_id = 'x' AND status = 'paid'
-- 但不優化：
--   WHERE status = 'paid'（沒用到第一個欄位）

-- 部分索引（Partial Index）— 只索引符合條件的行
CREATE INDEX idx_products_active ON products(name, price)
  WHERE is_active = true;
-- 適合：大部分查詢只查 is_active = true 的商品

-- UNIQUE 索引
CREATE UNIQUE INDEX idx_products_sku ON products(sku)
  WHERE sku IS NOT NULL;

-- 表達式索引（函數索引）
CREATE INDEX idx_users_email_lower ON profiles(lower(email));
-- 優化：WHERE lower(email) = 'user@example.com'

-- GIN 索引（jsonb 查詢）
CREATE INDEX idx_products_specs ON products USING gin(specs);
-- 優化：WHERE specs @> '{"顏色": "黑"}'

-- GIN 索引（陣列查詢）
CREATE INDEX idx_products_tags ON products USING gin(tags);
-- 優化：WHERE '3C' = ANY(tags)

-- GIN 索引（全文搜尋）
CREATE INDEX idx_products_search ON products USING gin(search_vector);

-- BRIN 索引（時間序列大表）
CREATE INDEX idx_logs_created ON access_logs USING brin(created_at);
-- 適合：千萬筆以上的 log 表，按時間範圍查詢
```

### EXPLAIN ANALYZE 查詢分析

```sql
-- 看執行計畫（不實際執行）
EXPLAIN SELECT * FROM products WHERE category_id = 'xxx';

-- 看執行計畫 + 實際執行時間
EXPLAIN ANALYZE SELECT * FROM products WHERE category_id = 'xxx';

-- 詳細版本（含 buffer 使用量）
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM products WHERE category_id = 'xxx';
```

**EXPLAIN 輸出重點**：

```
Seq Scan on products  (cost=0.00..35.50 rows=1000 width=200)
                       ↑ 預估成本    ↑ 預估行數  ↑ 每行大小
  Filter: (category_id = 'xxx')
  Rows Removed by Filter: 9000
  ↑ 如果 Rows Removed 很大，代表需要索引
```

```
Index Scan using idx_products_category on products  (cost=0.29..8.31 rows=10 width=200)
  Index Cond: (category_id = 'xxx')
  ↑ 用到索引了，效能好
```

**讀 EXPLAIN 的關鍵字**：

| 操作 | 說明 | 好壞 |
|------|------|------|
| Seq Scan | 全表掃描 | 小表OK，大表糟糕 |
| Index Scan | 用索引找到行再回表取資料 | 好 |
| Index Only Scan | 只讀索引不回表 | 最好 |
| Bitmap Index Scan | 先用索引篩選再批次回表 | 中大結果集好 |
| Nested Loop | 巢狀迴圈 JOIN | 小結果集好 |
| Hash Join | 雜湊 JOIN | 中大結果集好 |
| Sort | 排序 | 有索引可避免 |

### 索引維護

```sql
-- 查看未使用的索引（浪費空間 + 拖慢寫入）
SELECT
  schemaname, tablename, indexname,
  idx_scan AS times_used,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;

-- 查看索引大小
SELECT
  indexname,
  pg_size_pretty(pg_relation_size(indexname::regclass)) AS size
FROM pg_indexes
WHERE tablename = 'products'
ORDER BY pg_relation_size(indexname::regclass) DESC;

-- 重建索引（不鎖表）
REINDEX INDEX CONCURRENTLY idx_products_category;

-- 刪除不需要的索引
DROP INDEX IF EXISTS idx_unused;
```

### Prisma 實用技巧

```typescript
// 1. Soft Delete（用 middleware 攔截）
prisma.$use(async (params, next) => {
  if (params.model === 'Product') {
    if (params.action === 'delete') {
      params.action = 'update';
      params.args.data = { isActive: false, updatedAt: new Date() };
    }
    if (params.action === 'findMany' || params.action === 'findFirst') {
      if (!params.args.where) params.args.where = {};
      if (params.args.where.isActive === undefined) {
        params.args.where.isActive = true;
      }
    }
  }
  return next(params);
});

// 2. 查詢日誌（開發除錯用）
const prisma = new PrismaClient({
```

## 8. Drizzle ORM（輕量替代方案）

### Prisma vs Drizzle

| 面向 | Prisma | Drizzle |
|------|--------|---------|
| 型別安全 | 自動產生 | 手寫 schema 即型別 |
| Bundle 大小 | ~2MB | ~50KB |
| 查詢方式 | 自訂 API | 接近 SQL 語法 |
| Migration | 內建完整方案 | drizzle-kit |
| Edge Runtime | 部分支援 | 完整支援 |
| 學習成本 | 高 | 中（要會 SQL） |
| 適合 | 大專案、快速開發 | 輕量專案、Serverless |

```typescript
// db/schema.ts
import {
  pgTable, uuid, text, integer, boolean, timestamp,
  varchar, jsonb, index, uniqueIndex, primaryKey
} from 'drizzle-orm/pg-core';
import { relations } from 'drizzle-orm';

// ===== 用戶 =====

export const users = pgTable('users', {
  id: uuid('id').defaultRandom().primaryKey(),
  email: varchar('email', { length: 255 }).unique().notNull(),
  name: varchar('name', { length: 100 }),
  avatarUrl: text('avatar_url'),
  role: varchar('role', { length: 20 }).default('user').notNull(),
  isActive: boolean('is_active').default(true),
  createdAt: timestamp('created_at', { withTimezone: true }).defaultNow(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).defaultNow(),
}, (table) => ({
  emailIdx: uniqueIndex('idx_users_email').on(table.email),
}));

// ===== 商品 =====

export const products = pgTable('products', {
  id: uuid('id').defaultRandom().primaryKey(),
  name: varchar('name', { length: 200 }).notNull(),
  slug: varchar('slug', { length: 200 }).unique().notNull(),
  description: text('description'),
  price: integer('price').notNull(),
  salePrice: integer('sale_price'),
  sku: varchar('sku', { length: 50 }).unique(),
  categoryId: uuid('category_id').references(() => categories.id),
  stock: integer('stock').default(0).notNull(),
  isActive: boolean('is_active').default(true),
  images: jsonb('images').default([]),
  specs: jsonb('specs').default({}),
  createdAt: timestamp('created_at', { withTimezone: true }).defaultNow(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).defaultNow(),
}, (table) => ({
  categoryIdx: index('idx_products_category').on(table.categoryId),
  activeIdx: index('idx_products_active').on(table.isActive),
}));

export const categories = pgTable('categories', {
  id: uuid('id').defaultRandom().primaryKey(),
  name: varchar('name', { length: 100 }).notNull(),
  slug: varchar('slug', { length: 100 }).unique().notNull(),
  parentId: uuid('parent_id'),
  sortOrder: integer('sort_order').default(0),
});

// ===== 關聯定義 =====

export const usersRelations = relations(users, ({ many }) => ({
  orders: many(orders),
  posts: many(posts),
}));

export const productsRelations = relations(products, ({ one, many }) => ({
  category: one(categories, {
    fields: [products.categoryId],
    references: [categories.id],
  }),
  orderItems: many(orderItems),
}));

// ===== 訂單 =====

export const orders = pgTable('orders', {
  id: uuid('id').defaultRandom().primaryKey(),
  orderNumber: varchar('order_number', { length: 30 }).unique().notNull(),
  userId: uuid('user_id').references(() => users.id).notNull(),
  status: varchar('status', { length: 20 }).default('pending').notNull(),
  total: integer('total').default(0).notNull(),
  createdAt: timestamp('created_at', { withTimezone: true }).defaultNow(),
}, (table) => ({
  userIdx: index('idx_orders_user').on(table.userId),
  statusIdx: index('idx_orders_status').on(table.status),
}));

export const orderItems = pgTable('order_items', {
  id: uuid('id').defaultRandom().primaryKey(),
  orderId: uuid('order_id').references(() => orders.id, { onDelete: 'cascade' }).notNull(),
  productId: uuid('product_id').references(() => products.id, { onDelete: 'set null' }),
  productName: text('product_name').notNull(),
  unitPrice: integer('unit_price').notNull(),
  quantity: integer('quantity').notNull(),
  subtotal: integer('subtotal').notNull(),
});

// ===== 文章 =====

export const posts = pgTable('posts', {
  id: uuid('id').defaultRandom().primaryKey(),
  title: varchar('title', { length: 200 }).notNull(),
  slug: varchar('slug', { length: 200 }).unique().notNull(),
  content: text('content'),
  authorId: uuid('author_id').references(() => users.id).notNull(),
  status: varchar('status', { length: 20 }).default('draft').notNull(),
  publishedAt: timestamp('published_at', { withTimezone: true }),
  viewCount: integer('view_count').default(0),
  createdAt: timestamp('created_at', { withTimezone: true }).defaultNow(),
});
```

```bash
# 只備份特定表

  "postgresql://postgres.xxxxx:密碼@aws-0-ap-southeast-1.pooler.supabase.com:5432/postgres" \

# 只備份 public schema（排除 Supabase 系統表）

  "postgresql://postgres.xxxxx:密碼@aws-0-ap-southeast-1.pooler.supabase.com:5432/postgres" \
  > supabase_public_$(date +%Y%m%d).dump

# ===== 恢復 =====

# 恢復自訂格式備份

# 恢復純 SQL 備份

# 恢復到新資料庫

pg_restore -h host -U postgres -d new_database backup.dump

# 只恢復特定表
pg_restore -h host -U postgres -d postgres -t products backup.dump
```

```bash
# scripts/db-backup.sh — 自動備份到本地 + 上傳 S3

  aws s3 cp "${BACKUP_FILE}" "${S3_BUCKET}/"
  echo "[$(date)] 已上傳至 ${S3_BUCKET}"
fi

# 清理舊備份
find "${BACKUP_DIR}" -name "backup_*.dump.gz" -mtime +${RETENTION_DAYS} -delete
echo "[$(date)] 已清理 ${RETENTION_DAYS} 天前的備份"
```

## 10. 效能優化

### 連線池（Connection Pooling）

**為什麼需要**：每次建立 PostgreSQL 連線要 ~50ms + 記憶體。Serverless 環境每個 request 都建新連線會爆掉。

```
                 無連線池                      有連線池
Client 1 ──→ [新連線] ──→ DB          Client 1 ──→ [Pool] ──→ DB
Client 2 ──→ [新連線] ──→ DB          Client 2 ──→ [Pool] ──→ (複用)
Client 3 ──→ [新連線] ──→ DB          Client 3 ──→ [Pool] ──→ (排隊)
  (每個 request 新連線)                   (固定連線數，複用)
```

**Supabase 內建 PgBouncer**：用 port 6543 就是走連線池。

```env
# 直連（Migration/Admin 用）— port 5432
DIRECT_URL="postgresql://postgres.xxx:密碼@host:5432/postgres"

# 連線池（應用程式用）— port 6543
DATABASE_URL="postgresql://postgres.xxx:密碼@host:6543/postgres?pgbouncer=true"
```

**Node.js 連線池設定**：

```typescript
// postgres.js
import postgres from 'postgres';

const sql = postgres(process.env.DATABASE_URL!, {
  max: 10,            // 最大連線數（根據方案調整）
  idle_timeout: 20,   // 閒置多久釋放（秒）
  connect_timeout: 10, // 連線超時（秒）
});

// Prisma（prisma 自帶連線池）
const prisma = new PrismaClient({
  datasources: {
    db: { url: process.env.DATABASE_URL },
  },
});
// Prisma 預設 connection_limit = (CPU核心數 * 2) + 1
// 可在 URL 指定：?connection_limit=10
```

### 查詢優化

```sql
-- ❌ 不好：SELECT *（取回所有欄位，含大 text/json）
SELECT * FROM products;

-- ✅ 好：只取需要的欄位
SELECT id, name, price, stock FROM products;

-- ❌ 不好：在 WHERE 裡用函數（索引失效）
SELECT * FROM users WHERE UPPER(email) = 'USER@EXAMPLE.COM';

-- ✅ 好：建函數索引，或存正規化的值
CREATE INDEX idx_users_email_lower ON users(lower(email));
SELECT * FROM users WHERE lower(email) = 'user@example.com';

-- ❌ 不好：LIKE '%keyword%'（前面有 % 索引失效）
SELECT * FROM products WHERE name LIKE '%手機%';

-- ✅ 好：用全文搜尋
SELECT * FROM products WHERE search_vector @@ to_tsquery('simple', '手機');

-- ❌ 不好：大量 OFFSET 分頁（越後面越慢）
SELECT * FROM products ORDER BY created_at DESC LIMIT 20 OFFSET 10000;
-- PostgreSQL 要先掃 10000 行再丟掉

-- ✅ 好：Cursor-based 分頁（游標分頁）
SELECT * FROM products
WHERE created_at < '2026-03-01T00:00:00Z'  -- 上一頁最後一筆的時間
ORDER BY created_at DESC
LIMIT 20;

-- ❌ 不好：NOT IN 子查詢（效能差）
SELECT * FROM products WHERE id NOT IN (SELECT product_id FROM order_items);

-- ✅ 好：用 LEFT JOIN IS NULL 或 NOT EXISTS
SELECT p.* FROM products p
LEFT JOIN order_items oi ON p.id = oi.product_id
WHERE oi.id IS NULL;
-- 或
SELECT * FROM products p
WHERE NOT EXISTS (SELECT 1 FROM order_items oi WHERE oi.product_id = p.id);
```

### N+1 問題

**什麼是 N+1**：查 N 筆主資料，每筆再跑一次子查詢 = 1 + N 次查詢。

```typescript
// ❌ N+1（假設有 100 筆訂單，會跑 101 次查詢）
const orders = await db.select().from(ordersTable); // 1 次
for (const order of orders) {
  const items = await db.select()
    .from(orderItemsTable)
    .where(eq(orderItemsTable.orderId, order.id)); // N 次
}

// ✅ 改用 JOIN 或批次查詢（1-2 次查詢）
const ordersWithItems = await db.select()
  .from(ordersTable)
  .leftJoin(orderItemsTable, eq(ordersTable.id, orderItemsTable.orderId));

// ✅ Prisma 用 include（自動處理 N+1）
const orders = await prisma.order.findMany({
  include: { items: true },  // Prisma 只跑 2 次查詢
});

// ✅ 或用 DataLoader（GraphQL 常用模式）
import DataLoader from 'dataloader';

const orderItemsLoader = new DataLoader(async (orderIds: string[]) => {
  const items = await db.select()
    .from(orderItemsTable)
    .where(inArray(orderItemsTable.orderId, orderIds));

  // 分組回傳
  return orderIds.map(id => items.filter(item => item.orderId === id));
});

// 使用
const items = await orderItemsLoader.load(orderId); // 自動批次
```

### 快取策略

```typescript

// 2. Redis 快取（多 instance 共享）
// 適合：Session、排行榜、購物車、API 限流

async function getProduct(slug: string) {
  // 先查快取
  const cached = await redis.get(`product:${slug}`);
  if (cached) return JSON.parse(cached);

  // 查資料庫
  const product = await prisma.product.findUnique({
    where: { slug },
    include: { category: true, variants: true },
  });

  if (product) {
    // 寫入快取，10 分鐘過期
    await redis.set(`product:${slug}`, JSON.stringify(product), 'EX', 600);
  }

  return product;
}

// 3. Stale-While-Revalidate（SWR 模式）
// 適合：可以容忍短暫舊資料的場景

async function getProductsWithSWR() {
  const cacheKey = 'products:active';
  const cached = await redis.get(cacheKey);

  if (cached) {
    const { data, expireAt } = JSON.parse(cached);

    // 快取還沒過期，直接回傳
    if (Date.now() < expireAt) return data;

    // 已過期但還有資料：先回傳舊資料，背景更新
    refreshProductsCache(); // 非同步更新
    return data;
  }

  // 完全沒快取，同步查詢
  return refreshProductsCache();
}
```

### PostgreSQL 效能監控

```sql
-- 最慢的查詢（需要 pg_stat_statements 擴展）
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

SELECT
  calls,
  round(total_exec_time::numeric, 2) AS total_ms,
  round(mean_exec_time::numeric, 2) AS avg_ms,
  round(max_exec_time::numeric, 2) AS max_ms,
  query
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;

-- 表的讀寫統計
SELECT
  relname AS table_name,
  seq_scan,          -- 全表掃描次數
  idx_scan,          -- 索引掃描次數
  n_tup_ins AS inserts,
  n_tup_upd AS updates,
  n_tup_del AS deletes,
  n_live_tup AS live_rows,
  n_dead_tup AS dead_rows
FROM pg_stat_user_tables
ORDER BY seq_scan DESC;
-- 如果 seq_scan 很高但 idx_scan 很低，代表缺少索引

-- 連線數監控
SELECT
  state,
  COUNT(*) AS connections
FROM pg_stat_activity
GROUP BY state;
-- 正常應該大部分是 idle，如果 active 太多代表查詢太慢

-- 鎖等待
SELECT
  blocked.pid AS blocked_pid,
  blocked.query AS blocked_query,
  blocking.pid AS blocking_pid,
  blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_locks bl ON blocked.pid = bl.pid
JOIN pg_locks rl ON bl.relation = rl.relation AND bl.pid != rl.pid
JOIN pg_stat_activity blocking ON rl.pid = blocking.pid
WHERE NOT bl.granted;

-- 資料庫大小
SELECT
  pg_size_pretty(pg_database_size('postgres')) AS db_size;

-- 各表大小排名
SELECT
  tablename,
  pg_size_pretty(pg_total_relation_size(tablename::text)) AS total_size,
  pg_size_pretty(pg_relation_size(tablename::text)) AS data_size,
  pg_size_pretty(
    pg_total_relation_size(tablename::text) - pg_relation_size(tablename::text)
  ) AS index_size
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(tablename::text) DESC;
```

## 11. 資料庫安全

### Row Level Security（RLS）

RLS 是 PostgreSQL 原生功能，Supabase 重度依賴它。可以在資料庫層面控制「誰能看到哪些行」。

```sql
-- 啟用 RLS
ALTER TABLE products ENABLE ROW LEVEL SECURITY;

-- ===== 常用 Policy 模式 =====

-- 1. 公開讀取，登入才能寫入
CREATE POLICY "Public read products"
  ON products FOR SELECT
  USING (true);  -- 所有人都能讀

CREATE POLICY "Auth users insert products"
  ON products FOR INSERT
  WITH CHECK (auth.role() = 'authenticated');

-- 2. 用戶只能存取自己的資料
CREATE POLICY "Users manage own data"
  ON user_settings FOR ALL
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

-- 3. 角色權限控制
CREATE POLICY "Admins manage all products"
  ON products FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM profiles
      WHERE id = auth.uid() AND role = 'admin'
    )
  );

CREATE POLICY "Editors can update products"
  ON products FOR UPDATE
  USING (
    EXISTS (
      SELECT 1 FROM profiles
      WHERE id = auth.uid() AND role IN ('admin', 'editor')
    )
  );

-- 4. 組織 / 租戶隔離（Multi-tenant）
CREATE POLICY "Tenant isolation"
  ON orders FOR ALL
  USING (
    tenant_id = (
      SELECT tenant_id FROM profiles WHERE id = auth.uid()
    )
  );

-- 5. 時間限制（只能修改 24 小時內的資料）
CREATE POLICY "Edit recent only"
  ON comments FOR UPDATE
  USING (
    auth.uid() = user_id
    AND created_at > now() - INTERVAL '24 hours'
  );

-- ===== 測試 RLS =====

-- 切換角色測試
SET LOCAL ROLE anon;  -- 匿名用戶
SELECT * FROM products;  -- 應該能看到

SET LOCAL ROLE authenticated;
SET LOCAL request.jwt.claims = '{"sub": "user-uuid-here"}';
SELECT * FROM user_settings;  -- 只看到自己的
```

### 最小權限原則

```sql
-- 建立應用程式專用角色（不要用 postgres 超級使用者）
CREATE ROLE app_user WITH LOGIN PASSWORD '強密碼';

-- 只給必要的權限
GRANT USAGE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO app_user;
-- 不給 DELETE 權限（用 soft delete）

-- 特定表的權限
GRANT SELECT ON products TO app_user;
GRANT SELECT, INSERT ON orders, order_items TO app_user;
GRANT UPDATE (status, tracking_number) ON orders TO app_user;  -- 只能更新特定欄位

-- 不要給 DROP / TRUNCATE / ALTER 權限
-- 不要給 SUPERUSER 權限

-- ===== Supabase 角色 =====
-- anon：匿名用戶（未登入），通常只給 SELECT
-- authenticated：登入用戶，受 RLS 控制
-- service_role：後端服務用，繞過 RLS（絕對不要暴露到前端）
```

### 加密

```sql
-- 1. 傳輸加密：SSL/TLS
```
