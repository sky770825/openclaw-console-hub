# Cookbook 31: 根因分析框架 (RCA Framework)

> 當系統出現異常、錯誤或非預期行為時，啟動此框架進行結構化問題排查。

## 核心原則

不要猜，要驗證。每一步都要基於證據，而不是假設。

## 分析流程

### Phase 1: 釐清問題 (Problem Definition)
1.  這是什麼問題 (What)？
    - 用一句話精確描述現象。例如：「run_script action 執行 curl 命令時返回空結果，而不是預期的 API 輸出。」
2.  在哪裡發生 (Where)？
    - 定位到具體的模組、功能、API 端點或程式碼檔案。例如：「問題發生在 action-handlers.ts 的 handleRunScript 函數中。」
3.  什麼時候發生 (When)？
    - 問題是持續性的，還是間歇性的？是否有特定的觸發條件？例如：「每次呼叫 run_script 都會復現。」
4.  影響範圍多大 (Impact)？
    - 這個問題影響了哪些功能？對系統的整體目標有何衝擊？例如：「影響了我的所有診斷和輕量腳本執行能力，大幅降低了排查效率。」

### Phase 2: 收集證據 (Evidence Collection)
- 工具組合：query_supabase, read_file, run_script, semantic_search
1.  查日誌：read_file 讀取 /tmp/openclaw-server.log，尋找錯誤訊息或異常堆疊。
2.  查數據：query_supabase 檢查 openclaw_runs 或 openclaw_audit_logs 表，看是否有相關的失敗記錄。
3.  查狀態：run_script 執行 ps aux, lsof -i, curl 等命令，檢查進程、端口和網路連線。
4.  查知識：semantic_search 搜尋內部知識庫，看是否有類似問題的解決方案。
5.  查程式碼：read_file 讀取 CODEBASE-INDEX.md 中標示的相關源碼，理解其邏輯。

### Phase 3: 歸納假設 (Hypothesis Generation)
- 工具：ask_ai model=claude (深度分析)
- 根據收集到的證據，提出 2-3 個最可能的原因假設。例如：
    - 假設A：沙盒環境的網路設定問題，導致 curl 無法對外連線。
    - 假設B：child_process.exec 的輸出捕獲邏輯有 bug，命令執行了但結果沒回傳。
    - 假設C：傳遞給 run_script 的命令字串在某個環節被錯誤地轉義或截斷。

### Phase 4: 驗證/排除 (Verification / Elimination)
- 針對每個假設，設計一個最小化的驗證實驗。
- 驗證A：run_script 執行 ping 8.8.8.8 看網路是否通。
- 驗證B：create_task 派工，寫一個簡單的 node 腳本直接在 server 環境執行 child_process.exec，看結果是否正常。
- 驗證C：run_script 執行一個帶有特殊字符的 echo 命令，看輸出是否完整。

### Phase 5: 確定根因與方案 (Conclusion & Solution)
- 當一個假設被證實後，就找到了根本原因。
- 基於根因，提出解決方案，並透過 create_task 派工修復。
