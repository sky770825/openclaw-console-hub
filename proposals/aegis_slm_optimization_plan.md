# 神盾掃描器 (Aegis Scanner) SLM 效能優化方案

## 1. 背景與分析總結
根據前次 SLM 效能測試結果，目前系統在處理長文本掃描時存在以下瓶頸：
- **首字延遲 (TTFT):** 在 Prompt 超過 2k tokens 時顯著增加。
- **推論吞吐量:** 單執行緒處理速度受限於模型參數加載。
- **記憶體占用:** 在併發請求下，KV Cache 占用過高導致 Swap 發生。

## 2. 優化策略與目標

### 策略 A: 量化技術優化 (Quantization)
- **行動:** 將 FP16 模型轉換為 4-bit 或 8-bit 量化版本 (如 GGUF Q4_K_M)。
- **目標:** 降低 50% 記憶體占用，提升 30% 推論速度。

### 策略 B: Prompt Caching (提示詞緩存)
- **行動:** 針對固定格式的系統提示詞 (System Prompt) 與掃描規則實作緩存機制。
- **目標:** 重複請求的首字延遲降低 80%。

### 策略 C: 併發處理與批次推論 (Batching)
- **行動:** 實作動態批次處理 (Continuous Batching)，優化多文件同時掃描的效率。
- **目標:** 系統總吞吐量 (Requests Per Minute) 提升 2 倍。

### 策略 D: 串流輸出優化 (Streaming)
- **行動:** 全面導入 SSE (Server-Sent Events) 串流輸出。
- **目標:** 用戶感知延遲 (Perceived Latency) 降低至 200ms 以內。

## 3. 實施計畫 (W1-W2)

| 階段 | 項目 | 負責人 | 預計時程 |
| :--- | :--- | :--- | :--- |
| 第一階段 | 模型量化與評估 | 開發小組 | Day 1-2 |
| 第二階段 | Prompt Cache 整合 | 架構師 | Day 3-4 |
| 第三階段 | 併發負載均衡調優 | 運維小組 | Day 5-7 |
| 第四階段 | 效能驗收測試 | 測試小組 | Day 8-10 |

## 4. 驗收標準
1. 在相同硬體環境下，掃描單個 10KB 原始碼文件的時間從 5s 降低至 2s。
2. 支持至少 5 個併發掃描任務而不發生 OOM (Out of Memory)。
3. 輸出結果的準確率誤差與 FP16 版本相比小於 1%。
