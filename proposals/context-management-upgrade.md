# OpenClaw 進階上下文管理升級提案 (v1.0)

## 1. 背景與目標
隨著 NEUXA (OpenClaw) 處理的任務複雜度提升，現有的基於 `NOW.md` 與 `SESSION-STATE.md` 的線性上下文管理方式面臨以下挑戰：
- **Token 膨脹**：長期對話導致 Context Window 迅速填滿。
- **記憶碎片化**：關鍵決策資訊被淹沒在瑣碎的執行日誌中。
- **協作效率**：多代理環境下，冗餘的上下文傳遞增加了延遲與成本。

本提案旨在設計一套「分層式動態記憶架構」，優化長期記憶與多代理協同。

---

## 2. 技術方案建議

### A. 動態上下文壓縮 (Dynamic Context Compression)
*   **技術說明**：利用 LLM 對歷史對話進行遞歸式摘要。
*   **實作細節**：
    *   **滑動窗口 (Sliding Window)**：保留最近 10 輪原始對話。
    *   **語意壓縮**：將超過窗口的內容轉化為「關鍵事件摘要」，更新至 `SESSION-STATE.md`。
    *   **Token 觸發器**：當上下文達到 70% 閾值時自動觸發壓縮任務。

### B. 智能 RAG 策略 (Smart RAG Evolution)
*   **技術說明**：從簡單的相似度檢索進化為預判與多跳檢索。
*   **實作細節**：
    *   **預判式檢索 (Predictive Retrieval)**：在生成答案前，先分析問題意圖，預先檢索關聯知識庫。
    *   **多跳檢索 (Multi-hop Retrieval)**：針對複雜問題，第一步檢索出的資訊若不足，則生成二級查詢語句進一步深入。
    *   **動態 Re-ranking**：對檢索出的 Chunk 進行重新評分，只將高相關性內容注入 Prompt。

### C. 多代理上下文共享與協調 (Multi-Agent Blackboard)
*   **技術說明**：建立「共享黑板」模式，減少 Agent 間的資訊冗餘。
*   **實作細節**：
    *   **層次化上下文**：區分「全局上下文」(Shared) 與「私有上下文」(Private)。
    *   **狀態傳播機制**：當一個 Agent 更新 `NOW.md` 中的重大目標時，向其他活躍 Agent 發送輕量級狀態更新通知，而非完整日誌。

---

## 3. 與現有架構整合分析

| 組件 | 整合方式 | 挑戰 |
| :--- | :--- | :--- |
| **NOW.md** | 轉型為「當前短期焦點」，僅存放進行中任務。 | 需確保寫入頻率不會造成 I/O 瓶頸。 |
| **SESSION-STATE.md** | 作為「結構化長期記憶」，存放壓縮後的決策鏈。 | 需要強大的 Parsing 規則防止結構損壞。 |
| **向量數據庫** | 導入分層索引 (Hierarchical Indexing)，支援摘要級檢索。 | 索引同步的時效性。 |

---

## 4. 預期效益
1.  **降低 Token 成本**：通過動態壓縮，預計可節省 30% - 50% 的長文本消耗。
2.  **提升響應速度**：減少 Prompt 長度，顯著降低首字生成時間 (TTFT)。
3.  **增強記憶持久度**：關鍵資訊不再因 Context 溢出而遺失，NEUXA 能更準確回溯數天前的決策。
4.  **優化協作品質**：多代理間的干擾減少，意圖對齊更精確。

---

## 5. 風險與解決方案
*   **風險 1：摘要失真**。LLM 在壓縮過程中可能遺漏細節。
    *   *解決方案*：保留原始日誌的向量索引，摘要中附帶索引路徑，必要時可還原細節。
*   **風險 2：計算開銷**。壓縮與多跳檢索會增加額外的 LLM 調用。
    *   *解決方案*：使用較小、較快的模型 (如 GPT-4o-mini 或本地 Llama3-8B) 處理壓縮任務。
