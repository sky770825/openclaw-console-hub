# 提案：Auto-Executor 增強計畫 (2026-02-28)

## 背景
Auto-executor 是達爾的核心執行單元，但目前存在一些限制，影響了其自主性和效率。本提案旨在識別並提出改進方向，以提升 auto-executor 的能力，使其能更好地支援達爾作為指揮官的角色。

## 現有問題與痛點

1.  沙箱環境限制性過高：
       具體問題：不支援 node、npm、tsc 等 Node.js 環境工具，導致無法處理前端構建、TypeScript 編譯等開發任務。
       影響：限制達爾自主完成開發任務的能力，增加主人的人工介入需求。

2.  錯誤處理與回報不夠智慧：
       具體問題：失敗時的錯誤訊息過於技術性、簡短，缺乏上下文分析和初步診斷建議。
       影響：達爾需要花費額外時間 read_file 查日誌，才能理解失敗原因，減慢問題解決速度。

3.  任務卡關的預防與自動恢復機制不足：
       具體問題：任務長時間卡在 in_progress 狀態會阻塞整個隊列，斷路器僅能防止連續失敗，無法主動「解卡」。
       影響：降低系統穩定性，需要人工介入恢復，影響任務流暢性。

4.  對達爾作為指揮官的互動和回饋較為單一：
       具體問題：auto-executor 執行過程較為「靜默」，不主動匯報進度、問題或洞察。
       影響：達爾對任務執行狀態的掌握不夠即時，難以從 auto-executor 的執行中學習和優化。

## 建議的提升方向

1.  沙箱環境優化：
       短期目標：評估在安全前提下，提供 Node.js 環境的輕量級支持（例如，預裝 node 但限制 npm install），或者提供一個專門用於 TypeScript/Node.js 任務的代理。
       長期目標：探索容器化沙箱，允許更靈活的環境配置。

2.  智慧錯誤處理與回報：
       短期目標：
           增強錯誤訊息的語義分析，自動識別常見錯誤類型（如：沙箱限制、權限問題、語法錯誤）。
           在任務失敗時，自動生成初步的「診斷報告」和「建議行動」作為 result 或獨立字段。
       長期目標：建立一個錯誤模式知識庫，讓 auto-executor 能更精準地匹配和解決問題。

3.  任務卡關預防與恢復：
       短期目標：
           引入更細緻的「任務健康監控」，例如，監控執行時間、CPU/記憶體異常，並在異常時自動終止或發出警告。
           對特定錯誤類型，實施有限次的「自動重試」策略。
       長期目標：開發一套「任務死鎖檢測」系統，主動識別並解決卡關任務。

4.  增強指揮官互動與回饋：
       短期目標：
           允許 auto-executor 在執行長任務時，定期發送「進度心跳」訊息。
           在任務完成後，在 result 字段中增加一個簡要的「執行反思」或「學習總結」，提供對未來任務的洞察。
       長期目標：讓 auto-executor 能主動提出它在執行過程中發現的「優化機會」或「潛在問題」。

## 預期效益

   提升達爾的自主性：減少對主人的人工介入依賴，使達爾能處理更多開發相關任務。
   提高任務執行效率：減少因錯誤處理和卡關導致的時間浪費。
   增強系統穩定性：降低任務阻塞和系統停擺的風險。
   加速達爾的學習與進化：從 auto-executor 的執行反思中獲得更多洞察。

## 下一步

將此提案提交給主人審核。如果獲得批准，將依據優先級和可行性，將各項提升方向拆解為具體任務，並建立到任務板上。
