# 提案：優化 ask_ai 指揮策略與內部流程整合

## 提出者：達爾
## 日期：2026-02-28

## 背景
根據主人的指示，我的核心能力是指揮與調度。我已經讀取了 cookbook/17-ask_ai協作指南.md，並認識到需要更精準地利用 ask_ai 這個強大的顧問團，以提升整體指揮效率和任務品質。

## 目標
1.  強化 ask_ai 模型選擇的準確性與效益。
2.  提升 prompt 撰寫的精準度，確保子代理能更有效地理解和執行任務。
3.  優化 context 管理，確保提供給子代理的資訊完整且相關。
4.  將 ask_ai 更流暢地整合到複雜的任務指揮鏈中，特別是與 create_task 的協作。

## 實施方案

### 1. 建立「ask_ai 模型選擇決策流程圖」
   我將基於 cookbook/17-ask_ai協作指南.md 中的模型決策樹，進一步細化並內化一套決策流程。當需要使用 ask_ai 時，我會依循此流程圖來選擇最適合的模型 (Flash/Pro/Claude/Opus)，以平衡成本、速度和品質。
   
      具體做法：將此流程圖以文字形式儲存為內部參考，並在每次調用 ask_ai 前進行快速自檢。

### 2. 導入「Prompt 寫作核對清單」
   為了確保 prompt 的品質，我將建立一個內部核對清單，每次撰寫 prompt 前都進行檢查：
      是否明確指出「動作」？(e.g., 分析、摘要、設計)
      是否明確指出「對象」？(e.g., 這段代碼、這份報告)
      是否明確指出「面向」？(e.g., 安全漏洞、架構問題、優缺點)
      是否明確指出「輸出格式」？(e.g., 列點、表格、3 個重點)
      是否避免了模糊詞彙？(e.g., 看看、好不好)
   
      具體做法：將此清單儲存為內部參考，並在每次撰寫 ask_ai 的 prompt 時參考。

### 3. 優化 Context 傳遞機制
   我將更嚴格地執行 context 傳遞的原則：
      永遠不要給空的 context。
      對於大型檔案，優先使用 head、tail、grep 等指令提取關鍵片段，而非盲目傳遞整個檔案。
      當需要對整個大型檔案進行深度分析時，考慮將檔案內容分段傳遞給 ask_ai，或將任務拆解為多個子任務，並利用 create_task 派發給更適合處理長內容的模型 (如 Claude Opus)。
   
      具體做法：在每次 read_file 後，增加一個內部判斷環節，評估檔案大小和內容，再決定如何構造 context。

### 4. ask_ai 與 create_task 的協同指揮模式
   對於需要 ask_ai 進行初步分析，然後再由 auto-executor 執行後續步驟的複雜任務，我將設計一個標準的協同指揮模式：
   1.  問題分解：將大問題分解為「分析決策」和「執行操作」兩部分。
   2.  ask_ai 決策：先用 ask_ai 進行分析、方案評估、代碼審查，獲取關鍵的決策依據或執行建議。
   3.  create_task 執行：根據 ask_ai 的分析結果，建立一個或多個清晰的 create_task 任務，詳細描述執行步驟和預期產出，交由 auto-executor 執行。
   
      具體做法：在面對複雜任務時，優先考慮這種兩階段的指揮模式，並在 create_task 的 description 中明確引用 ask_ai 的分析結果。

## 預期效益
   提升 ask_ai 的調用效率和準確性，減少錯誤和重複工作。
   降低模型使用成本，確保資源被精準利用。
   加速複雜任務的處理流程，提高整體任務完成率。
   強化達爾作為指揮官的戰略規劃和執行能力。

## 結論
此提案旨在系統性地優化我利用 ask_ai 的方式，使其成為我更強大的指揮助手。期待主人審核。
