# 自動任務生成規則 (Auto-Task Generation Rules)\n\n## 1. 發現缺口時的主動資料收集\n\n### 資訊收集優先級\n```\n優先級 1 - 內部資源查詢（免費、快速）\n├─ 搜尋 workspace 本地檔案\n├─ 查詢 MEMORY.md 歷史記錄  \n├─ 檢查現有文檔庫\n└─ 時間預算：< 5 分鐘\n\n優先級 2 - 網路搜尋和爬蟲（需要 API）\n├─ 使用搜尋引擎\n├─ 查詢官方文件和 API 文檔\n├─ 網路爬蟲取得資訊\n└─ 時間預算：5-15 分鐘\n\n優先級 3 - 專業服務/外包（成本較高）\n├─ 查詢知識庫\n├─ 外部諮詢\n└─ 時間預算：視情況而定\n```\n\n### 資訊收集任務模板\n```yaml\nid: INFO-GATHER-{YYYYMMDD}-{NNN}\ntitle: \"收集 [主題] 資訊\"\nsource_gap: [GAP-ID]\ndescription: |\n  為了解決 [缺口描述]，需要收集 [資訊類型]\n  \nsteps:\n  1. \"搜尋本地檔案和記憶\"\n  2. \"查詢網路資源\"\n  3. \"整理成結構化摘要\"\n  4. \"產出至 taskboard/info/[topic]-findings.md\"\n  \nexpected_output: |\n  - 關鍵發現清單\n  - 來源連結\n  - 資訊驗證結果\n  - 應用建議\n\ntime_estimate: \"15-30m\"\npriority_boost: 5  # 如果缺口阻塞其他任務\n```\n\n---\n\n## 2. 資訊缺口轉化為具體任務\n\n### 轉化決策樹\n```\n發現缺口 (from self-check.md)\n    ↓\n是否可以立即解決？\n    ├─ NO  → 是否需要外部資源？\n    │        ├─ YES → \"等待/依賴\" 任務\n    │        └─ NO  → \"調查/研究\" 任務\n    │\n    └─ YES → 直接生成 \"立即行動\" 任務\n```\n\n### 任務生成模板\n```markdown\n---\nid: TASK-{YYYYMMDD}-{NNN}\ntype: [research|implementation|documentation|coordination]\ngap_source: [GAP-ID]\npriority: [P0|P1|P2|P3]\ncreated_at: {ISO8601}\n---\n\n## 標題\n[簡潔但完整的任務目標]\n\n## 背景\n[為什麼這個任務存在？與哪個缺口相關？]\n\n## 目標\n[SMART 原則定義的具體目標]\n\n## Acceptance Criteria\n- [ ] 條件 1\n- [ ] 條件 2  \n- [ ] 條件 3\n\n## 執行步驟\n1. [具體步驟]\n2. [具體步驟]\n3. [具體步驟]\n\n## 預估工時\n- 時間：[X 分鐘/小時]\n- 複雜度：[LOW|MEDIUM|HIGH]\n- 必需資源：[資源列表]\n\n## 相關資源\n- 缺口報告：[gap-analysis-XX.md]\n- 參考文件：[list]\n```\n\n### 缺口到任務映射規則\n| 缺口類型 | 生成任務類型 | 命名規則 | 初始優先級 |\n|---------|-----------|---------|----------|\n| INFO_GAP | 資訊收集 | INFO-{topic}-{id} | P1 |\n| SKILL_GAP | 學習/評估 | LEARN-{skill}-{id} | P2 |\n| PROCESS_GAP | 流程設計 | DESIGN-{process}-{id} | P1 |\n| DEPENDENCY_GAP | 協調/等待 | COORD-{dep}-{id} | P0 |\n\n---\n\n## 3. 任務優先級自動判定規則\n\n### 優先級計算維度\n\n#### 維度 A：阻塞性 (BLOCKER_SCORE) [0-5]\n```\n5 = 阻塞多個進行中任務\n4 = 阻塞 1-2 個進行中任務  \n3 = 有依賴任務等待\n2 = 影響計畫進度\n1 = 無阻塞\n```\n\n#### 維度 B：時間緊迫性 (URGENCY_SCORE) [0-5]\n```\n5 = 24 小時內截止\n4 = 1-3 天內截止\n3 = 1 週內截止\n2 = 1-2 週內截止\n1 = 無明確期限\n```\n\n#### 維度 C：業務價值 (VALUE_SCORE) [0-5]\n```\n5 = 核心功能/高度優先目標\n4 = 重要功能/項目里程碑\n3 = 中等功能/輔助目標\n2 = 小改善\n1 = 可選優化\n```\n\n#### 維度 D：工作量 (EFFORT_SCORE) [1-5]（反向）\n```\n5 = 快速 (< 10 分鐘)\n4 = 較快 (10-30 分鐘)\n3 = 中等 (30-120 分鐘)\n2 = 較長 (2-4 小時)\n1 = 長期 (> 4 小時)\n```\n\n### 優先級計算公式\n```\nPRIORITY_SCORE = (\n    BLOCKER_SCORE × 2.0 +      # 阻塞性最重要\n    URGENCY_SCORE × 1.5 +      # 時間緊迫次之\n    VALUE_SCORE × 1.0 +        # 業務價值\n    EFFORT_SCORE × 0.5         # 快速完成加分\n) / 6.0\n\n結果範圍：0-5\n```\n\n### 優先級分級\n```\nP0 (CRITICAL):      >= 4.0\n  ├─ 立即執行（在任何當前工作前）\n  ├─ 典型場景：阻塞重要任務、關鍵決策點待決\n  └─ 最多 2-3 個同時進行\n\nP1 (HIGH):          3.0-3.9\n  ├─ 本輪優先執行\n  ├─ 典型場景：重要功能、明確期限任務\n  └─ 可並行 2-3 個\n\nP2 (MEDIUM):        2.0-2.9  \n  ├─ 下輪執行\n  ├─ 典型場景：改進項目、中期計畫\n  └─ 可積壓\n\nP3 (LOW):           < 2.0\n  ├─ 擱置池（資源有餘才考慮）\n  ├─ 典型場景：優化建議、將來功能\n  └─ 定期評估是否需要提升\n```\n\n### 優先級動態調整\n```\n每小時檢查一次：\n\n❌ 任務超期 → Urgency +1\n   - pending 中超過預計時間 50% \n   - 自動升一級\n\n❌ 新阻塞發現 → Blocker +1\n   - 發現新的依賴任務\n   - 自動升一級\n\n✅ 快速可完成 → Effort +1\n   - 預計時間 < 10 分鐘\n   - 優先執行（quick-win）\n\n⏰ 週期評估 (每天)\n   - P0 任務是否 > 3 個？如是則警告\n   - 是否有任務被長期延遲？\n   - 優先級與實際執行是否匹配？\n```\n\n---\n\n## 4. 任務生成工作流程\n\n```\n┌─────────────────────────┐\n│  self-check.md 識別缺口  │\n└──────────┬──────────────┘\n           ↓\n┌──────────────────────────┐\n│  分類缺口類型            │\n│  INFO/SKILL/PROCESS/DEP  │\n└──────────┬───────────────┘\n           ↓\n┌──────────────────────────┐\n│  收集相關資訊            │\n│  (如需要)                │\n└──────────┬───────────────┘\n           ↓\n┌──────────────────────────┐\n│  生成具體任務檔案        │\n│  taskboard/pending/      │\n└──────────┬───────────────┘\n           ↓\n┌──────────────────────────┐\n│  計算優先級分數          │\n│  自動分類至 P0-P3        │\n└──────────┬───────────────┘\n           ↓\n┌──────────────────────────┐\n│  添加標籤 (tags)         │\n│  critical/quick-win/etc  │\n└──────────┬───────────────┘\n           ↓\n┌──────────────────────────┐\n│  報告新任務清單          │\n│  等待分配或執行          │\n└──────────────────────────┘\n```\n\n---\n\n## 5. 任務生成觸發規則\n\n### 自動觸發條件\n✅ self-check 流程完成  \n✅ 發現新缺口時  \n✅ 現有任務標記為「已完成」  \n✅ 用戶指令中含有新需求  \n✅ 技術債或風險升級時  \n\n### 手動觸發\n💬 用戶直接要求：「建立任務...\"  \n💬 任務板檢視後的後續動作  \n"
