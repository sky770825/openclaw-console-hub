## 目的

多個 Agent 同時工作會撞車。規則：一個任務一個人負責，透過任務板協調，不要直接互相叫。

---

## 核心原則

1. **一個任務一個 Agent** — 不可兩個 Agent 同時做同一個任務
2. **透過任務板溝通** — 不直接對話，用任務板 + 檔案系統
3. **不搶別人的任務** — 看到別人 running 的任務，不要碰
4. **發現問題回報老蔡** — 不要自己修別人的問題
5. **專業分工優於通才** — 每個 Agent 深耕自己的領域，不要什麼都搶著做
6. **同時最多 3 個 Agent 並行** — 超過 3 個協調成本太高，品質會下降

---

## 協調模式

根據任務性質選擇合適的協調方式：

### 模式 1：Orchestrator（指揮官模式）— 最常用

小蔡當指揮官，拆解任務、分派工作、整合結果。

```
老蔡下指令
  → 小蔡拆解任務
    → Claude Code 做後端
    → Cursor 做前端
    → Ollama 做翻譯/摘要
  → 小蔡整合結果 → 回報老蔡
```

**適用場景：** 大部分任務（有明確目標、可拆分）
**注意：** 小蔡要確認每個子任務的「輸入」和「期望輸出」都寫清楚

### 模式 2：Pipeline（接力模式）

前一個 Agent 的輸出是下一個 Agent 的輸入，依序執行。

```
Ollama 蒐集資料 → 整理成 markdown
  → Claude Code 根據資料設計方案
    → Cursor 根據方案實作 UI
```

**適用場景：** 有固定步驟的流程（研究→設計→實作）
**注意：** 每個階段的產出要放在明確的檔案路徑，下一個 Agent 才找得到

### 模式 3：Review（審查模式）

一個做，另一個檢查。降低錯誤率。

```
Cursor 寫前端程式碼
  → Claude Code 審查品質、找 bug
    → Cursor 修正 → 完成
```

**適用場景：** 重要的程式碼變更、上線前檢查
**注意：** 審查者不要直接改，而是列出問題讓原作者修

### 選擇指南

| 任務特性 | 用哪個模式 |
|---------|-----------|
| 可以拆成獨立子任務 | Orchestrator |
| 有先後順序依賴 | Pipeline |
| 品質要求高、怕出錯 | Review |
| 探索性、不確定怎麼做 | Orchestrator + 小範圍試做 |

## 開始工作前的檢查

### Step 1: 檢查任務板

```bash
# 查相關任務有沒有別人在做
curl -s http://localhost:3011/api/tasks?status=running
```

- 有人在做相同的事 → **停止，回報老蔡**
- 沒人在做 → 繼續

### Step 2: 檢查 git 狀態

```bash
git status
git log --oneline -5
```

- 有未 commit 的變更 → 可能是別的 Agent 在改 → **不要覆蓋**
- 乾淨 → 繼續

### Step 3: 檢查目標檔案

```bash
# 確認沒人在同時改這個檔案
find . -name "{target_file}" -mmin -5 2>/dev/null
```

- 最近 5 分鐘被改過 → 可能有人在改 → **等一下或問老蔡**

## 檔案鎖定規則

| 檔案 | 同時修改 |
|------|---------|
| AGENTS.md | ❌ 每次只能一個 Agent 改 |
| MEMORY.md | ❌ 每次只能一個 Agent 改 |
| 知識庫文件 | ❌ 同一檔案不可兩人改 |
| 任務板 task | ❌ 一個 task 一個 Agent |
| scripts/ | ⚠️ 不同腳本可以，同一腳本不行 |
| logs/ | ✅ 各寫各的 log |

## 結構化通訊協議

Agent 之間傳遞資訊要用統一格式，不要用模糊的自然語言。

### 派工訊息格式

建任務時，description 必須包含以下結構：

```json
{
  "from": "小蔡",
  "to": "Claude Code",
  "task_type": "bug_fix | feature | review | research",
  "priority": "P0 | P1 | P2 | P3",
  "input": {
    "files": ["server/src/index.ts"],
    "context": "什麼背景、為什麼要做",
    "constraints": "不能改什麼、要注意什麼"
  },
  "expected_output": {
    "deliverables": ["修好的檔案", "RESULT.md"],
    "acceptance_criteria": ["通過測試", "不影響現有功能"]
  },
  "deadline": "2 小時內 | 今天 | 不急"
}
```

### 回報結果格式

完成任務時，RESULT.md 必須包含：

```markdown
## Summary
（一句話說做了什麼）

## 做了什麼
- 改了哪些檔案
- 為什麼這樣改

## 驗證結果
- 跑了什麼測試
- 結果是什麼

## 發現的問題（如果有）
- 發現什麼額外問題
- 建議怎麼處理

## Next Steps
- 下一步該做什麼
- 誰來做
```

## 共享上下文管理

### 原則：需要知道的才給，不要塞全部

每個 Agent 只需要跟自己任務相關的上下文：

| Agent | 需要知道 | 不需要知道 |
|-------|---------|-----------|
| Cursor（做前端 UI） | 設計稿、元件 API、目前的頁面結構 | 後端資料庫 schema、部署設定 |
| Claude Code（做後端） | API 規格、資料庫結構、現有程式碼 | 前端 CSS 細節、UI 排版 |
| Ollama（做翻譯） | 原文、目標語言、風格要求 | 整個專案架構 |

### 上下文傳遞方式

```
方式 1：檔案路徑（最推薦）
  → 告訴 Agent「讀 docs/API-SPEC.md 第 50-80 行」

方式 2：摘要（適合大量資訊）
  → 自己先用 Ollama 摘要，再把摘要傳給下游

方式 3：完整傳遞（只在必要時）
  → 除非 Agent 確實需要全部歷史，否則不要這樣做
```

## 回饋迴路

### 下游 Agent 發現問題時的回報機制

不是只有上→下的單向流，下游也可以回報問題：

```
小蔡派任務給 Cursor
  → Cursor 發現「API 還沒實作，前端做不了」
    → Cursor 回報：
        ⚠️ 任務阻塞
        原因：GET /api/reviews 回 404，後端還沒做
        建議：先讓 Claude Code 做後端 API，再繼續前端
    → 小蔡看到 → 調整優先順序 → 先派 Claude Code 做 API
    → API 做好 → 通知 Cursor 繼續
```

### 回饋觸發條件

| 狀況 | 處理 |
|------|------|
| 缺少前置依賴（API 沒做、資料沒準備好） | 回報阻塞，建議先做什麼 |
| 發現上游的設計有缺陷 | 回報問題 + 建議修改方案 |
| 任務範圍比預期大很多 | 回報重新評估，建議拆分 |
| 做到一半發現更好的做法 | 回報替代方案，等決定 |

### 回報格式

```
⚠️ 回饋迴路觸發

任務：{task ID}
階段：{目前在做什麼}
問題：{發現什麼}
影響：{不處理會怎樣}
建議：{建議怎麼處理}
需要誰：{哪個 Agent 或老蔡}
```

## 失敗降級策略

單一 Agent 失敗時，不讓整個流程崩潰：

### 降級鏈

```
第一選擇失敗 → 換人做（降級）

Claude Code 掛了/太貴
  → 試 Cursor（如果是程式碼任務）
  → 試 Ollama qwen2.5:14b（如果是分析任務）
  → 回報老蔡手動處理

Cursor 掛了/品質不好
  → 試 Claude Code（小量改動）
  → 回報老蔡手動處理

Ollama 掛了
  → 試 Gemini Flash 免費額度
  → 回報老蔡

Gemini Flash 額度用完
  → 用 Ollama 本地模型替代
  → 回報老蔡
```

### 失敗判定標準

| 失敗類型 | 判定方式 | 處理 |
|---------|---------|------|
| 超時 | 10 分鐘沒回應 | 重試 1 次，再失敗就降級 |
| 品質不合格 | 產出不符合 acceptance criteria | 退回重做，給更多上下文 |
| 無法完成 | Agent 回報「做不了」 | 降級到下一個 Agent |
| 連線失敗 | API/服務不可用 | 等 2 分鐘重試，再不行就降級 |

### 重試規則

- 同一任務最多重試 2 次
- 重試時要加上「之前失敗的原因」讓 Agent 避開
- 重試 2 次仍失敗 → 降級或回報老蔡

## 需要別人做事時

**不要直接叫別的 Agent。建任務讓老蔡分派。**

```
POST /api/tasks
Body: {
  "title": "需要 {Agent} 做 {什麼}",
  "description": "（用上面的結構化格式）",
  "projectPath": "projects/openclaw"
}
```

然後回報老蔡：
```
我需要 {Agent} 幫忙做 {什麼}，已建任務 {taskId}，請分派。
```

---

## 發現別人的問題時

**不要自己修。回報老蔡。**（這就是回饋迴路的一種）

```
⚠️ 發現問題

位置：{檔案路徑}
問題：{描述}
可能是 {Agent} 的操作造成的
建議：{修復方式}

等你決定由誰來修。
```

---

## 回報格式

```
👥 跨 Agent 協作狀態

我的任務：{task ID} — {名稱}
工作區域：{目標檔案/目錄}
其他 Agent 活動：{有/無}
衝突風險：{低/中/高}
```

## 錯誤處理

| 狀況 | 處理方式 |
|------|----------|
| 兩個 Agent 同時改同一檔案 | 立即停止，回報老蔡，用 git diff 看衝突 |
| 別的 Agent 覆蓋了我的產出 | 回報老蔡，用 git log 找回 |
| 任務板顯示兩人在做同一任務 | 回報老蔡，其中一個要退出 |
| 子代理沒回來 | 等 10 分鐘，沒回就回報老蔡 |

## 子代理管理

### 派工時（SOP-4 整合）

```
sessions_spawn(
  task="...",
  label="...",
  model="..."  # 見 11-本地模型子派 選模型
)
```

### 子代理超時

- 10 分鐘沒回應 → 回報老蔡
- 30 分鐘沒回應 → 建議殺掉重派

### 子代理回報後

1. 驗證產出（ls -la、wc -c、cat 抽查）
2. 不可只看「完成」就信
3. 品質不過 → 退回重做或換模型
