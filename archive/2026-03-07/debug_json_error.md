## JSON 解析錯誤分析與解決計畫

### 1. 問題現象

伺服器日誌中出現大量 `SyntaxError: Bad control character in string literal in JSON` 錯誤，指出在處理傳入的 API 請求時，`body-parser` 無法成功解析請求主體的 JSON 內容。

### 2. 問題根源

- **錯誤類型**: JSON 格式錯誤。
- **具體原因**: JSON 字串中包含了未經轉義 (unescaped) 的控制字元，例如換行符 (`\n`) 或定位符 (`\t`)。一個常見的例子是，前端將一個包含多行文字的 `textarea` 內容直接放入 JSON，而沒有將換行符轉換為 `\n`。
- **錯誤位置**: 錯誤發生在 Express 的 `body-parser` 中介軟體層，這是在我們的 API 路由處理器執行之前。

### 3. 解決方案

為了定位並解決問題，需要採取以下步驟：

1.  **攔截錯誤**: 在 Express 應用中，加入一個專門處理 `body-parser` 語法錯誤的錯誤處理中介軟體。
2.  **日誌記錄**: 在該中介軟體中，記錄下導致錯誤的原始請求主體 (raw request body)。這將幫助我們看到是哪個欄位的資料格式不正確。
3.  **修正來源**: 根據日誌，找到發送錯誤格式資料的前端或其他客戶端，並修正其資料處理邏輯，確保在 `JSON.stringify` 之前，所有字串都符合 JSON 規範。
4.  **增強健壯性**: 錯誤處理中介軟體應回傳一個標準的 `400 Bad Request` 回應，告知客戶端其請求格式有誤，而不是讓伺服器崩潰或掛起。
