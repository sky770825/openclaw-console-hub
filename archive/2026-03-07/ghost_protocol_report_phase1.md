
# 幽靈-01 (Ghost-01) 任務報告 - 代理伺服器整合方案 (第一階段)

## 目標
研究將代理伺服器 (Proxy) 整合進 OpenClaw 子代理網路請求的最佳方案，重點關注基於 Shell 或腳本的 HTTP 請求。

## 1. 理解環境
OpenClaw 子代理主要透過 `web_search`, `web_fetch` 等工具發出網路請求。這些工具底層通常使用 `curl` 或類似的 HTTP 客戶端，因此本研究聚焦於如何為這類**基於 Shell 或腳本的 HTTP 請求**設定代理。

## 2. 技術方案調研摘要

### cURL 代理配置
*   使用 `-x` 或 `--proxy` 選項指定代理伺服器地址，例如 `curl -x http://proxy.example.com:8080 http://example.com`。
*   使用 `-U` 或 `--proxy-user` 指定代理認證，例如 `curl -x http://user:pass@proxy.example.com:8080 http://example.com`。
*   對於 HTTP 代理隧道 (HTTP proxy tunnels)，特別是涉及 TLS/HTTPS 的請求，可以使用 `-p` 或 `--proxytunnel`。
*   環境變數 `HTTP_PROXY`, `HTTPS_PROXY`, `NO_PROXY` 廣泛支援，許多 shell 工具和程式庫會自動讀取這些變數。

### 開源代理伺服器與隱私
*   `Squid`: 功能強大的快取代理伺服器，常用於大型網路環境。
*   `mitmproxy`: 互動式 HTTPS 代理，適合開發、調試和安全測試，可編寫 Python 腳本實現自定義邏輯，如代理輪換。
*   `Tinyproxy`: 輕量級 HTTP/HTTPS 代理，資源消耗低。

### 輪換代理服務 API
*   許多第三方服務 (如 ScraperAPI, Zyte Smart Proxy Manager, Proxyway, ZenRows) 提供輪換代理，自動管理大量 IP、地理位置選擇、IP 輪換和避免封鎖。這些服務通常提供 API 接口供程式化整合。

## 3. 情報分析與報告

### **本地代理 vs. 代理服務**

| 特性       | 本地代理 (Squid, mitmproxy)                               | 代理服務 (Bright Data, Smartproxy, ScraperAPI, Zyte)              |
| :--------- | :-------------------------------------------------------- | :---------------------------------------------------------------- |
| **優點**   | - 完全控制，高度自訂義<br>- 成本較低 (無服務費)<br>- 可用於內部網路監控/快取<br>- `mitmproxy` 適合開發與調試 | - 大量輪換 IP (住宅、數據中心、移動)<br>- 全球地理位置選擇<br>- 自動處理 IP 輪換和封鎖<br>- 通常有較高的匿名性與成功率<br>- 提供 API 方便整合 |
| **缺點**   | - 需自行維護<br>- IP 輪換和地理位置選擇困難<br>- 單一 IP 易被偵測/封鎖<br>- 需處理 HTTPS 憑證問題           | - 成本較高 (按流量/請求計費)<br>- 控制權較少<br>- 依賴第三方服務穩定性與隱私政策 |
| **適用場景** | - 內部測試、開發<br>- 對成本敏感、需求簡單的代理<br>- 需要深度控制代理行為的特殊場景 | - 大規模網頁抓取 (Web Scraping)<br>- 需要高匿名性、頻繁 IP 變更的需求<br>- 需要繞過嚴格反爬機制 |

### **配置方式**

1.  **環境變數 (`HTTP_PROXY`, `HTTPS_PROXY`):**
    *   **優點:**
        *   廣泛支援，許多 Shell 工具 (如 `curl`, `wget`) 和程式庫會自動讀取這些變數。
        *   易於設置和管理，無需修改現有工具程式碼。
        *   在 OpenClaw 的 `exec` 工具中，可以透過 `env` 參數為特定命令設定這些變數，實現彈性配置。
    *   **缺點:**
        *   對於不支援這些環境變數的應用程式或工具無效。
        *   如果需要複雜的代理邏輯（例如針對不同 URL 使用不同代理），單純的環境變數可能不足。

2.  **修改工具的程式碼 (例如 OpenClaw `web_search`, `web_fetch` 底層):**
    *   **優點:** 提供最精確的控制，可以實現非常複雜和定制化的代理邏輯。
    *   **缺點:**
        *   需要修改 OpenClaw 工具的原始碼，增加維護複雜性。
        *   可能在 OpenClaw 更新時造成衝突，不推薦子代理直接修改。
        *   子代理的職責是執行任務，而非修改 OpenClaw 核心工具。

### **推薦方案**

初步推薦採用 **結合環境變數與第三方輪換代理服務** 的方案：

1.  **預設與簡單需求:**
    *   透過設定 OpenClaw 子代理執行環境的 **`HTTP_PROXY` 和 `HTTPS_PROXY` 環境變數**。這些變數可以指向一個本地自建的輕量級代理 (如 `Tinyproxy`) 或是一個固定的第三方代理節點。
    *   這種方式無需修改工具程式碼，兼容性好，適用於一般性的網路請求。

2.  **高匿名性與複雜需求:**
    *   對於 `web_search`, `web_fetch` 等可能面臨反爬或地理限制的請求，建議使用 **第三方輪換代理服務**。
    *   這些服務通常提供易於整合的 API 或單一閘道入口。子代理可以透過在請求中包含這些服務提供的特殊標頭 (Headers) 或將環境變數指向這些服務的閘道來使用。
    *   這樣可以將 IP 輪換、地理位置選擇等複雜性交由專業服務處理，提高請求成功率。

### **潛在挑戰**

1.  **性能下降:** 代理引入額外網路跳轉，可能增加請求延遲。輪換代理切換 IP 也會耗時。
2.  **配置複雜性:** 管理多個代理設定 (不同環境變數、不同認證、不同代理類型) 可能會增加配置複雜性。
3.  **HTTPS 憑證問題:** 如果使用自建代理攔截 HTTPS 流量進行解密和重新加密 (例如 `mitmproxy` 的某些功能)，需要將代理的根憑證安裝到 OpenClaw 運行環境中，這涉及到安全和信任配置。
4.  **成本考量:** 第三方代理服務通常按流量或請求量收費，需要仔細評估和控制使用成本。
5.  **偵測與封鎖:** 即使使用代理，目標網站仍可能偵測到代理流量 (例如透過瀏覽器指紋、JS 執行等) 並進行封鎖。需要持續優化請求行為。
6.  **穩定性與可靠性:** 自建代理需要確保伺服器穩定運行；第三方代理服務也需要考慮其服務品質和潛在的停機時間。
