# PR 審查檢查清單 v0.1

> 當提交 Pull Request (PR) 或進行程式碼審查時，請使用此清單確保品質與一致性。

---

## 1. 提交者檢查清單 (Submitter's Checklist)

*   [ ] **PR 標題清晰**: 標題是否遵循 `[類型](範圍): 簡短描述` 的格式？ (e.g., `feat(DebugMaster): 新增日誌分析功能`)
*   [ ] **描述完整**: 是否清楚說明了此 PR 的目的、解決的問題以及實現方式？
*   [ ] **自我審查**: 是否已自己審查過程式碼，移除不必要的 `console.log` 或註解？
*   [ ] **測試相關**:
    *   [ ] 是否已新增或更新相關的單元測試？
    *   [ ] 所有測試是否都已通過？
*   [ ] **文件更新**: 是否已更新相關的文件 (e.g., `README.md`, `SKILL.md`)？
*   [ ] **與主分支同步**: 是否已將最新的 `main` 分支合併到此分支，解決所有衝突？

---

## 2. 審查者檢查清單 (Reviewer's Checklist)

### 架構與設計
*   [ ] **符合設計**: 程式碼是否符合最初的架構設計和 `EXPERT.md` 的職責定義？
*   [ ] **單一職責**: 每個函式和模組的職責是否單一且明確？
*   [ ] **可擴展性**: 設計是否考慮了未來的擴展性？

### 程式碼品質
*   [ ] **風格一致**: 程式碼是否遵循 `CODING_STYLE.md` 的規範？
*   [ ] **命名清晰**: 變數、函式、類別的命名是否清晰易懂？
*   [ ] **錯誤處理**: 是否有適當的錯誤處理機制 (e.g., `try-catch`, `set -e`)？
*   [ ] **效能考量**: 是否存在明顯的效能瓶頸 (e.g., 巢狀迴圈、不必要的 I/O)？
*   [ ] **安全性**: 是否存在潛在的安全風險 (e.g., 命令注入、未驗證的輸入)？

### 文件與測試
*   [ ] **註解必要**: 複雜或關鍵的邏輯是否有必要的註解？
*   [ ] **測試覆蓋**: 測試是否充分覆蓋了核心邏輯與邊界情況？
